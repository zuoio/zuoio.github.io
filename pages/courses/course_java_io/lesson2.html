第二章：Unix IO 模型——理解一切的理论基础

学习目标：从操作系统层面，搞清楚一次 IO 到底发生了什么，为理解 BIO/NIO/AIO 打下地基。

2.1 先搞清楚：操作系统是怎么读数据的？

在学 IO 模型之前，我们必须先弄懂几个基础概念，否则后面全是空中楼阁。

什么是用户空间和内核空间？

你的电脑内存并不是所有程序都能随便访问的。操作系统把内存划分为两个区域：

┌──────────────────────────────────┐
│          内核空间                  │  ← 操作系统专用，你的程序不能直接碰
│  (操作系统内核、驱动程序住在这里)    │
├──────────────────────────────────┤
│          用户空间                  │  ← 你的 Java 程序住在这里
│  (你的应用程序、JVM 住在这里)       │
└──────────────────────────────────┘

为什么要分开？ 安全。如果你的 Java 程序能直接操作硬件（磁盘、网卡），一个 bug 就可能搞崩整个系统。所以操作系统把硬件操作权垄断在内核空间，你的程序想读文件、收网络数据，必须"求"操作系统帮你做。

什么是系统调用（System Call）？

你的程序想读一个文件，不能自己去操作磁盘，必须通过系统调用请求操作系统来做。

你的 Java 代码：inputStream.read()
        │
        ▼
    JVM 内部调用
        │
        ▼
    系统调用 read()   ← 这一步从"用户态"切换到"内核态"
        │
        ▼
    操作系统内核去操作磁盘/网卡

关键点：发起系统调用后，仍然是你这个线程在执行，只不过它的身份从"用户态"切换到了"内核态"。操作系统不会另起一个线程帮你读。你可以理解为：你的线程走进了"操作系统的办公室"，用操作系统的权限去干活。

一次读文件到底发生了什么？

当你执行 FileInputStream.read() 时：

步骤1：你的线程发起系统调用 read()
       → 线程从"用户态"切换到"内核态"

步骤2：操作系统命令 DMA 控制器去磁盘读数据
       → DMA 控制器独立工作，把数据从磁盘搬到"内核缓冲区"
       → 这个过程中 CPU 不需要参与搬运，可以干别的事
       → 但你的线程在等 DMA 搬完，所以线程是挂起的（阻塞）

步骤3：DMA 搬完了，数据在"内核缓冲区"中
       → CPU 把数据从"内核缓冲区"复制到"用户缓冲区"（你程序的 byte[] 数组）
       → 这一步需要 CPU 参与，但很快（内存到内存的复制）

步骤4：复制完毕，线程从"内核态"切回"用户态"
       → read() 方法返回，你可以用数据了

什么是 DMA？为什么说"不消耗 CPU"？

DMA（Direct Memory Access，直接内存访问） 是硬件上的一个控制器。它的工作是：在不需要 CPU 参与的情况下，把数据在磁盘/网卡和内存之间搬运。

可以把 CPU 想象成经理，DMA 想象成搬运工。经理（CPU）下了一个命令"把仓库（磁盘）里的货搬到办公室（内存）"，然后搬运工（DMA）就去干活了，经理可以去做别的事。搬完了，搬运工通知经理"搞定了"。

所以当我们说"阻塞等待期间不消耗 CPU"时，意思是：

  • CPU 不需要亲自搬数据（DMA 在搬）
  • 但你的线程确实被挂起了，在那等着，什么都干不了
  • CPU 可以被操作系统调度给其他线程使用

2.2 为什么要讨论 IO 模型？——从"读文件"到"网络通信"

读文件时，阻塞其实问题不大

你说得对——如果你的主线程要读一个文件，读完才能继续执行，那确实只能等着。这种场景下讨论"阻塞还是非阻塞"意义不大。

// 读文件：我就是需要这个数据，不读完没法往下走
byte[] data = Files.readAllBytes(Paths.get("config.json"));
// 必须等 data 有了，下面才能用
processConfig(data);

但问题出在网络通信上。 网络通信和读文件有一个巨大的区别：

网络通信的特殊性：等待是不确定的

读文件时，数据就在你自己的磁盘上，读取速度虽然比内存慢，但至少是可预测的、很快就能完成的。

网络通信就不一样了：

  • 你是一个服务器，有 1000 个客户端连着你
  • 客户端什么时候发数据？不知道！ 可能 1 秒后发，可能 1 小时后才发，可能永远不发
  • 你的线程在那里 read()，等的是对方从网络发来的数据
读文件：数据在本地磁盘，几毫秒就能搬到内存 → 等一下没关系
网络IO：数据在对方电脑上，什么时候发过来不确定 → 一直等就出大问题

这就是 IO 模型真正要解决的问题

想象你是一个服务器程序，同时连着 1000 个客户端：

用阻塞模型：你给每个客户端分配一个线程等着，1000 个客户端就要 1000 个线程，而且大部分线程都在干等（因为客户端不是每时每刻都在发数据）。

线程1: 等客户端A发数据... 等啊等... 等了3秒... 来了！处理！又开始等...
线程2: 等客户端B发数据... 等啊等... 等了10秒... 来了！处理！又开始等...
线程3: 等客户端C发数据... 等啊等... 还在等... 一直等...
...
线程1000: 等客户端ZZZ发数据...

1000 个线程，990 个在发呆。 这就是浪费。

IO 模型就是在解决这个问题：如何高效地等待和处理大量网络连接的数据。

总结：五种 IO 模型主要针对的是网络 IO 场景。读文件时一般不需要纠结模型，但网络通信中，"如何高效地等待不确定什么时候到来的数据"是一个核心问题。

2.3 一次网络 IO 操作的两个阶段

当服务器要从一个网络连接中读取数据时，分两个阶段：

                          你的线程
                             │
                     ┌───────┴───────┐
                     │  发起read()   │
                     │  系统调用      │
                     └───────┬───────┘
                             │
              ═══════════════╪═══════════════  （用户态 / 内核态 分界线）
                             │
                  ┌──────────▼──────────┐
                  │    阶段一：等待数据    │  ← 等对方把数据通过网络发过来
                  │  （数据从网卡 → 内核） │     数据到达网卡后 DMA 搬到内核缓冲区
                  │                      │     这个过程可能很快，也可能很慢
                  │   耗时：不确定！       │     取决于对方什么时候发数据
                  └──────────┬──────────┘
                             │
                  ┌──────────▼──────────┐
                  │    阶段二：复制数据    │  ← 数据到了内核缓冲区，
                  │  （内核缓冲 → 用户缓冲）│     CPU 复制到你程序的 byte[] 中
                  │                      │
                  │   耗时：很快（微秒级） │     内存到内存的复制
                  └──────────┬──────────┘
                             │
              ═══════════════╪═══════════════
                             │
                     ┌───────▼───────┐
                     │  read() 返回   │
                     │  你拿到数据了   │
                     └───────────────┘

核心认知：

  • 阶段一是最耗时的，因为要等对方发数据，这个等待时间可能很长
  • 阶段二很快，就是内存之间的拷贝
  • 五种 IO 模型的区别，就在于这两个阶段中，你的线程是在等还是去做别的事

2.4 五种 Unix IO 模型

现在你已经理解了背景知识，我们来看五种模型。为了便于理解，我统一用一个生活场景来类比：

场景：你开了一家快递驿站，有 100 个快递柜。你需要知道哪个柜子里有新快递到了，然后取出来处理。

模型一：阻塞式 IO（Blocking IO）

你的做法：站在第 1 个快递柜前等。快递到了就取出来处理，然后走到第 2 个柜子前等……

你的线程                          操作系统内核
   │                               │
   │──── read(连接A) ────────────→│
   │                               │
   │    线程挂起，啥都干不了          │  等客户端A的数据从网络到达...
   │    （CPU可以去服务其他线程）      │  ...可能等几毫秒，可能等几秒...
   │                               │  数据到了！DMA搬到内核缓冲区
   │                               │  CPU把数据复制到你的byte[]
   │                               │
   │←── 返回数据 ──────────────────│
   │                               │
   ▼ 处理数据，然后去read(连接B)...   │

问题：你同时只能盯一个柜子。100 个柜子就要 100 个人（线程），95 个在发呆。

模型二：非阻塞式 IO（Non-blocking IO）

你的做法：挨个柜子看一眼"有快递没？"，没有就立刻看下一个，一直循环。

你的线程                          操作系统内核
   │                               │
   │──── read(连接A) ────────────→│
   │←── "没数据" ──────────────────│  数据还没到
   │                               │
   │──── read(连接B) ────────────→│
   │←── "没数据" ──────────────────│  数据还没到
   │                               │
   │──── read(连接A) ────────────→│
   │←── "没数据" ──────────────────│  数据还没到
   │                               │
   │──── read(连接A) ────────────→│
   │←── "有数据！" ────────────────│  数据到了，返回给你
   │                               │
   ▼ 处理数据                       │

问题：虽然一个线程可以"照顾"多个连接，但你的 CPU 一直在做"跑过去看一眼、没有、跑回来"的无用功。就像你在 100 个柜子之间来回跑，每次跑一圈发现都没快递——累死你但没产出。

模型三：IO 多路复用（IO Multiplexing）

你的做法：在驿站装了一个监控大屏（Selector/epoll），大屏能同时监控 100 个柜子。你只需要盯着大屏——大屏亮了"柜子 3 有快递"，你就去柜子 3 取。

你的线程                          操作系统内核
   │                               │
   │──── select(100个连接) ──────→│ 帮你同时盯着100个连接
   │                               │
   │    线程挂起，等着               │  盯着... 盯着...
   │    （但这次是同时等100个！）     │  连接A没数据、连接B没数据...
   │                               │  连接C有数据到了！
   │                               │
   │←── "连接C就绪" ────────────── │
   │                               │
   │──── read(连接C) ────────────→│  读连接C的数据
   │←── 返回数据 ──────────────────│
   │                               │
   ▼ 处理连接C的数据                 │

优势：一个线程就能管 100 个（甚至上万个）连接！不用为每个连接开一个线程，也不用做无用的轮询。

和前面两个的关键区别：

  • 和阻塞 IO 比：虽然也在等，但一次等的是多个连接，任何一个有数据就能处理
  • 和非阻塞 IO 比：不用做无用的轮询，而是内核帮你监控，有数据才通知

这就是 Java NIO 的底层原理。

模型四：信号驱动 IO（Signal-driven IO）

你的做法：给每个柜子装一个门铃。你不用盯着任何柜子，该干嘛干嘛。快递到了柜子会响铃（发信号），你听到铃声再去取。

你的线程                          操作系统内核
   │                               │
   │──── "连接A有数据就通知我" ────→│  注册信号
   │←── "好的" ────────────────────│
   │                               │
   │  你去做其他事...                │  盯着...
   │  处理别的业务...                │  数据到了！
   │                               │
   │←── 信号："连接A有数据了！" ────│  内核发信号通知你
   │                               │
   │──── read(连接A) ────────────→│  你来取数据
   │←── 返回数据 ──────────────────│
   │                               │
   ▼ 处理数据                       │

和 IO 多路复用的区别：

  • IO 多路复用：你盯着监控大屏等（select 阻塞），大屏亮了告诉你"数据好了，你来取"
  • 信号驱动：你不盯大屏，柜子自己会按门铃通知你"数据好了，你来取"

共同点：都需要你自己去取数据（调用 read）。通知你的只是"数据到了内核缓冲区"，你还要把数据从内核搬到你的程序里。

模型五：异步 IO（Asynchronous IO）

你的做法：告诉快递系统"有快递就直接送到我办公桌上"。你完全不用管，快递系统把快递取出柜子、拆好包装、放到你桌上，然后通知你"全搞定了，直接用吧"。

你的线程                          操作系统内核
   │                               │
   │──── "读连接A，完了直接          │
   │      放到我的byte[]里" ───────→│  注册异步读取
   │←── "好的" ────────────────────│
   │                               │
   │  你去做其他事...                │  等数据...
   │  处理别的业务...                │  数据到了！（阶段一完成）
   │  完全不用管连接A...             │  复制到你的byte[]里！（阶段二完成）
   │                               │
   │←── "全搞定了，数据已经          │
   │     在你的byte[]里了" ─────────│  通知你
   │                               │
   ▼ 直接使用数据                    │

和前面所有模型的本质区别：

  • 前四种：内核只负责"通知你数据到了"或者"等你来取"，你必须自己调 read() 把数据从内核搬到你的程序
  • 异步 IO：内核把所有事情都做了——等数据、搬数据——全搞定后才通知你，你拿到通知时数据已经在你的 byte[] 里了

2.5 用一张图区分五种模型

到这里你可能已经有点晕了。让我们把五种模型放在一起对比，关键看两点：阶段一你在干嘛？通知你的时候数据是在内核还是已经到你手里了？

                      阶段一（等数据到达）        阶段二（搬到你程序里）     你拿到通知时
───────────────────────────────────────────────────────────────────────────────────
阻塞IO            你在等（线程挂起）              你在等              数据到手
非阻塞IO          你在轮询（不停问）              你在等              数据到手
IO多路复用         你在等（但同时等多个）           你在等              数据到手
信号驱动IO        你在做别的事                    你在等              数据到手
异步IO            你在做别的事                   内核帮你搬好了       数据到手
───────────────────────────────────────────────────────────────────────────────────
                                                  ↑
                                          前四种都要你自己搬
                                          第五种内核帮你搬好了

2.6 同步 IO vs 异步 IO（面试必问）

这两个概念容易混淆。关键就一句话：

看"阶段二"（把数据从内核搬到你的程序里）是谁干的。

  • 你自己搬 → 同步 IO
  • 内核帮你搬好了再通知 → 异步 IO
IO 模型 阶段一你在干嘛 阶段二谁搬数据 属于
阻塞式 IO 等着 你搬（线程阻塞等搬完） 同步
非阻塞式 IO 不停问 你搬（线程阻塞等搬完） 同步
IO 多路复用 等着（但等多个） 你搬（线程阻塞等搬完） 同步
信号驱动 IO 做别的事 你搬（线程阻塞等搬完） 同步
异步 IO 做别的事 内核帮你搬好了 异步

关键结论：前四种都是同步 IO！ 因为最终都要你的线程自己去把数据从内核搬到用户空间。

阻塞/非阻塞 vs 同步/异步

这两组概念是不同维度的：

  • 阻塞/非阻塞：问的是"阶段一你等不等？"——等数据到达时你是干等还是做别的
  • 同步/异步：问的是"阶段二谁干活？"——搬数据这活是你干还是内核帮你干
            │   同步                异步
            │  （你自己搬数据）    （内核帮你搬好）
────────────┼──────────────────────────────
 阻塞       │  阻塞式 IO            ──
            │  IO 多路复用
────────────┼──────────────────────────────
 非阻塞     │  非阻塞式 IO          异步 IO
            │  信号驱动 IO

面试常见误区

误区：非阻塞 IO 就是异步 IO

纠正：非阻塞 IO 只是在阶段一不用等（通过轮询），但在阶段二（搬数据）你的线程仍然要等着搬完，所以它还是同步 IO。就像快递到了门口（阶段一完成），但你还是要自己下楼去拿（阶段二）。异步 IO 是快递直接送到你桌上。

2.7 IO 多路复用的三种实现：select / poll / epoll

IO 多路复用是最重要的模型（Java NIO 就是基于它），它有三种操作系统实现。

三者的核心区别

还是用驿站的比喻：

  • select：你给前台一张手写清单（最多写 1024 个柜子号），前台挨个检查清单上的柜子，查完告诉你"有 3 个柜子有快递"，但不告诉你具体哪 3 个——你得自己再挨个看。每次查都要重新写清单、重新交给前台。

  • poll：和 select 一样，只是清单不限制长度了。但前台还是挨个查，还是不告诉你具体是哪个。

  • epoll：前台有一块智能电子屏。你只需要注册一次"我关心柜子 1~100"，之后电子屏自动监控。有快递时，屏幕直接显示"柜子 3、柜子 7 有快递"——你直接去取，不用自己遍历。而且不需要每次重新注册。

对比表

特性 select poll epoll
最大连接数 1024 无限制 无限制
每次调用是否要重传清单 是 是 否（只注册一次）
结果 只告诉有几个就绪，你自己遍历 同 select 直接告诉你哪几个就绪
时间复杂度 O(n)（要遍历所有） O(n) O(1)（只返回就绪的）
平台支持 跨平台 跨平台 Linux 独有

适用场景

  • 连接数少（<1000），要跨平台 → select/poll
  • 连接数多（上万），运行在 Linux → epoll

Java NIO 在 Linux 上的底层实现就是 epoll。 这也是为什么高性能 Java 服务器通常部署在 Linux 上。

epoll 的两种触发模式（了解即可）

  • LT（水平触发）：柜子里有快递，每次你去看电子屏都会显示。安全，不怕你忘。默认模式。
  • ET（边缘触发）：柜子里来了新快递时只通知一次。你必须一次取完，否则不会再通知。效率高但编程难度大。

2.8 到底该选哪个模型？

场景 推荐模型 原因
读写本地文件 直接阻塞读就行 文件在本地，速度可预测，阻塞时间短
少量网络连接（<100） 阻塞 IO + 多线程 简单好写，线程数可控
大量网络连接（上万） IO 多路复用 一个线程管所有连接，资源占用小
Windows 平台 + 大量连接 异步 IO IOCP 是真正的异步
实际项目 Netty（基于 IO 多路复用） 封装好了，不用自己踩坑

附录：你可能也会有的疑问

Q1：应用程序读文件时，是应用程序的线程在读，还是操作系统另起一个线程在读？

是你自己的线程在读。 操作系统不会为你创建新线程。

当你调用 inputStream.read() 时，你的线程会通过系统调用（System Call）从"用户态"切换到"内核态"。切换后，仍然是你这个线程在执行，只不过它现在拥有了操作系统的权限，可以操作硬件了。

具体来说：你的线程进入内核态后，命令 DMA 控制器去磁盘搬数据。DMA 是硬件，它独立工作不需要 CPU。在 DMA 搬数据的过程中，你的线程就挂在那里等（阻塞），CPU 被操作系统调度给别的线程用。DMA 搬完后，你的线程被唤醒，CPU 帮你把数据从内核缓冲区复制到你的 byte[] 中，然后 read() 返回。

全程只有你这一个线程，没有"系统线程"帮你读。

Q2：如果主线程读文件，必须等数据读完才能往下执行，那阻塞不是很正常吗？讨论"阻塞"好像没意义？

你说得完全对！ 对于读文件来说，阻塞确实是很自然的事。

五种 IO 模型主要是为网络通信场景设计的，不是为读文件。网络通信的特殊之处是：

  • 你不知道对方什么时候发数据（可能等 1 毫秒，可能等 1 小时）
  • 你可能同时连着 1000 个客户端，每个都可能随时发数据

如果每个连接都阻塞一个线程等着，1000 个连接就要 1000 个线程。而且这 1000 个线程大部分时间都在发呆（等对方发数据），这是巨大的浪费。

所以 IO 多路复用（NIO）的价值在于：一个线程就能同时等 1000 个连接，谁先来数据就先处理谁。 这在文件读取场景下确实没啥用，但在网络服务器场景下是革命性的。

Q3："等待期间线程挂起，不消耗 CPU"——把数据加载到内存不消耗 CPU 吗？

从磁盘/网卡搬数据到内核缓冲区这一步，确实不怎么消耗 CPU。 因为这是 DMA 控制器（硬件）在干活，CPU 不需要参与。

但"把数据从内核缓冲区复制到用户缓冲区"这一步确实需要 CPU。好消息是这一步非常快（内存到内存的复制，微秒级别），所以通常不是性能瓶颈。

所以更准确的描述是：

  • 等待数据阶段（阶段一）：DMA 干活，CPU 不参与 → 不消耗 CPU
  • 复制数据阶段（阶段二）：CPU 干活 → 消耗 CPU，但很快
  • 合在一起说"不消耗 CPU"指的是最耗时的"等待"阶段

Q4：IO 多路复用、信号驱动 IO、异步 IO 看起来都是"数据到了通知应用程序"，有什么区别？

区别在于通知你的时机不同和通知后你要不要自己搬数据。用取快递来比喻：

模型 通知你的时机 通知后你要做什么
IO 多路复用 "柜子里有快递了"（数据到了内核缓冲区） 你自己去柜子取出来（调 read，阻塞等搬完）
信号驱动 IO "柜子里有快递了"（数据到了内核缓冲区） 你自己去柜子取出来（调 read，阻塞等搬完）
异步 IO "快递已经放你桌上了"（数据已经在你的 byte[] 里） 什么都不用做，直接用

IO 多路复用和信号驱动 IO 通知的内容是一样的——"数据到了内核"，区别只是通知方式：

  • IO 多路复用：你坐在监控大屏前等着看（select 阻塞），大屏亮了你去取
  • 信号驱动 IO：你不用盯大屏，大屏主动给你发消息（信号），你收到消息去取

而异步 IO 是质的飞跃：通知你的时候，连"取"这步都帮你做好了，数据直接在你的程序内存里了。

所以本质区别是：

  • 前两个通知的是"半成品"（数据到了内核，你还要自己搬）
  • 异步 IO 通知的是"成品"（数据已经到你手里了）
