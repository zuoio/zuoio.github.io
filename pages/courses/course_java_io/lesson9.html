9.1 基础概念类

Q1：Java IO 流的分类有哪些？

参考答案：

Java IO 流可以从两个维度分类：

按传输方式分，分为字节流和字符流。字节流以 InputStream/OutputStream 为基类，处理二进制数据；字符流以 Reader/Writer 为基类，处理文本数据。核心区别是字节流操作的最小单位是字节，字符流操作的最小单位是字符，内部涉及编解码。

按操作对象分，分为文件流（FileInputStream）、数组流（ByteArrayInputStream）、缓冲流（BufferedInputStream）、数据类型流（DataInputStream）、对象流（ObjectInputStream）、管道流（PipedInputStream）、转换流（InputStreamReader）等。

另外，这些 IO 类大量使用了装饰者模式，可以灵活组合功能。

可能追问：装饰者模式是怎么体现的？→ 见 Q3

Q2：字节流和字符流的区别？什么时候用哪个？

参考答案：

字节流处理原始的二进制数据，一次读写一个字节；字符流处理文本数据，一次读写一个字符，内部自动处理编解码。

使用原则：

  • 处理二进制文件（图片、视频、压缩包）→ 用字节流
  • 处理文本文件（.txt、.java、.xml）→ 用字符流
  • 不确定内容类型 → 用字节流（更通用）

如果需要在字节流和字符流之间转换，可以使用桥梁类 InputStreamReader 和 OutputStreamWriter，它们在转换时需要指定字符编码，编解码不一致就会出现乱码。

Q3：Java IO 用了什么设计模式？请举例说明。

参考答案：

Java IO 大量使用了装饰者模式。

以 InputStream 为例：InputStream 是抽象组件，FileInputStream 是具体组件负责真正从文件读取字节，FilterInputStream 是抽象装饰者，BufferedInputStream 和 DataInputStream 是具体装饰者，分别增加了缓冲和读取基本数据类型的功能。

使用时可以像"套娃"一样组合：new DataInputStream(new BufferedInputStream(new FileInputStream("file.txt")))，每套一层就增加一个能力。

使用装饰者模式而不是继承的好处是避免了类爆炸——不需要为每种功能组合创建子类，通过运行时的动态组合就能灵活扩展。

Q4：序列化和反序列化是什么？Serializable 和 Externalizable 的区别？

参考答案：

序列化是把 Java 对象转换为字节序列的过程，反序列化是把字节序列恢复为对象的过程。主要用于网络传输和持久化存储。

Serializable 是标记接口，不需要实现任何方法，JVM 自动处理序列化逻辑。Externalizable 继承自 Serializable，但需要自己实现 writeExternal() 和 readExternal() 方法，可以精确控制哪些字段被序列化，性能更好。

实际开发中还需要注意：

  • 建议显式声明 serialVersionUID，否则类结构变化后反序列化会失败
  • transient 修饰的字段不参与序列化
  • static 字段属于类而非对象，也不会被序列化

9.2 IO 模型类（重点！）

Q5：阻塞、非阻塞、同步、异步四个概念分别是什么？

参考答案：

这是两组不同维度的概念：

阻塞/非阻塞描述的是应用程序的等待状态——发起 IO 请求后，如果数据没准备好，是一直等着（阻塞），还是立即返回去做别的事（非阻塞）。

同步/异步描述的是 IO 操作的通知机制——同步是应用程序自己去检查（轮询或阻塞等待）IO 是否完成；异步是操作系统做完后主动通知应用程序。

一个常见的误区是认为非阻塞就是异步。其实非阻塞 IO 只是在等待数据阶段不阻塞（通过轮询），但在数据从内核复制到用户空间时仍然阻塞，所以它还是同步 IO。

在 Unix 五种 IO 模型中，前四种（阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO）都是同步 IO，只有异步 IO 是真正的异步。

追问预判：面试官可能接着问"五种模型分别是什么？"

Q6：五种 Unix IO 模型分别是什么？

参考答案：

一次 IO 操作分两个阶段：等待数据准备好、把数据从内核复制到用户空间。五种模型的区别就在于这两个阶段如何处理。

  1. 阻塞 IO：两个阶段都阻塞，最简单
  2. 非阻塞 IO：阶段一不阻塞（轮询检查），阶段二阻塞
  3. IO 多路复用：用 select/poll/epoll 同时监听多个连接，阶段一阻塞在 select 上，阶段二阻塞
  4. 信号驱动 IO：阶段一注册信号后不阻塞，数据就绪时内核发信号通知，阶段二阻塞
  5. 异步 IO：两个阶段都不阻塞，内核完成所有操作后通知应用程序

Java 中 BIO 对应阻塞 IO，NIO 对应 IO 多路复用，AIO 对应异步 IO。

Q7：BIO、NIO、AIO 分别是什么？有什么区别？

参考答案：

**BIO（Blocking IO）**是 JDK 1.0 的传统 IO，同步阻塞模型。每个连接需要一个线程处理，accept() 和 read() 都会阻塞。适合连接数少的场景，高并发时线程资源会耗尽。

**NIO（New IO）**是 JDK 1.4 引入的，基于 IO 多路复用。核心是三大组件：Channel（双向通道）、Buffer（缓冲区）、Selector（选择器）。一个线程通过 Selector 可以同时监听多个 Channel 的事件，大大减少了线程数。底层在 Linux 上用 epoll 实现。

**AIO（Asynchronous IO）**是 JDK 1.7 引入的异步 IO。采用订阅-通知模式，应用程序注册回调后不用管，操作系统完成 IO 后主动通知。但在 Linux 上底层还是 epoll 模拟的，没有真正的异步，所以实际使用不如 NIO 广泛。

简单用比喻来说：BIO 是在饭店干等上菜，NIO 是取号后看叫号屏，AIO 是点外卖等骑手送上门。

Q8：为什么说非阻塞 IO 仍然是同步 IO？

参考答案：

关键在于理解 IO 的两个阶段。

非阻塞 IO 在第一阶段（等待数据）确实不阻塞——它通过轮询不断询问操作系统"数据好了没"。但在第二阶段（把数据从内核复制到用户空间），应用程序仍然要阻塞等待这个拷贝完成。

同步和异步的判断标准是：第二阶段是否阻塞。只要应用程序在数据复制阶段需要等待，就是同步 IO。异步 IO 是操作系统把两个阶段都做完后才通知应用程序。

Q9：NIO 中的 Channel、Buffer、Selector 分别是什么？

参考答案：

**Channel（通道）**是数据传输的载体，类似于传统 IO 的 Stream，但它是双向的（可读可写），而且可以设置为非阻塞模式。常用的有 FileChannel、SocketChannel、ServerSocketChannel。

**Buffer（缓冲区）**是数据的容器，本质是一个数组加上一套读写状态管理。所有数据必须先放入 Buffer 才能通过 Channel 传输。核心有三个状态变量：capacity（容量）、position（当前位置）、limit（界限）。写完数据后需要调用 flip() 切换到读模式。

**Selector（选择器）**是 NIO 的核心，实现了 IO 多路复用。一个 Selector 可以同时监听多个 Channel 上的事件（ACCEPT、READ、WRITE、CONNECT），让一个线程就能处理多个连接。底层在 Linux 上是 epoll 实现。

Q10：select、poll、epoll 的区别和适用场景？

参考答案：

三者都是 IO 多路复用的系统调用实现：

select：最古老，跨平台。最多监听 1024 个文件描述符，每次调用需要把 fd 集合从用户空间拷贝到内核，返回后需要 O(n) 遍历所有 fd 找出就绪的。适合连接数少且对跨平台有要求的场景。

poll：和 select 类似但去掉了 1024 的限制，用链表替代位图。但仍然需要 O(n) 遍历和每次拷贝 fd 集合。

epoll：Linux 独有。fd 只需注册一次（epoll_ctl），就绪时只返回有事件的 fd，O(1) 效率。在大量连接且大多数空闲的场景下性能远超 select/poll。Java NIO 在 Linux 上就是基于 epoll。

Q11：什么是 Reactor 模型？有几种变体？

参考答案：

Reactor 是一种事件驱动的设计模式，核心思想是用一个或少量线程监听 IO 事件，事件到达后分发给对应的处理器处理。

有三种变体：

  1. 单线程 Reactor：一个线程负责所有事——accept、IO 读写、业务处理。简单但无法利用多核，一个 Handler 慢了所有连接都受影响。

  2. 多线程 Reactor：一个 Reactor 线程负责 accept 和 IO 事件分发，业务处理交给线程池。解决了业务处理阻塞的问题，但 Reactor 线程仍是单点瓶颈。

  3. 主从多线程 Reactor：Boss Reactor 负责 accept，Worker Reactor 组负责 IO 读写，业务处理可以再交给单独的线程池。Netty 默认使用这种模型。

9.3 性能优化类

Q12：什么是零拷贝？Java 中如何实现？

参考答案：

零拷贝不是"完全不拷贝"，而是减少不必要的 CPU 参与的数据拷贝。传统 IO 中，数据从磁盘到网络要经历 4 次拷贝（磁盘→内核→用户→Socket→网卡），其中用户空间的两次拷贝是多余的，因为应用程序并不修改数据。

Java 中有两种零拷贝实现：

  1. MappedByteBuffer（基于 mmap）：通过 FileChannel.map() 将文件映射到内存，内核和用户空间共享一块物理内存，省去了一次 CPU 拷贝。适合需要操作文件内容的场景。

  2. FileChannel.transferTo()（基于 sendfile）：数据直接从内核的文件缓冲区传到 Socket 缓冲区，完全不经过用户空间。在 Linux 2.4+ 上可以实现真正的零 CPU 拷贝。适合纯数据转发场景。

Q13：Kafka 和 RocketMQ 的零拷贝有什么区别？

参考答案：

Kafka 使用 sendfile 方式（FileChannel.transferTo()）。消费者读取消息时，数据从磁盘直接传到网卡，不经过 JVM 堆内存。优点是性能最高，缺点是数据不经过用户空间，无法在传输过程中修改。

RocketMQ 使用 mmap 方式（MappedByteBuffer）。将 CommitLog 文件映射到内存，读写都通过内存映射。优点是可以在用户空间操作数据（比如构建索引），灵活性更高。缺点是比 sendfile 多一次 CPU 拷贝。

选择取决于场景：纯消息转发选 sendfile，需要处理数据选 mmap。

9.4 框架类

Q14：为什么要用 Netty？原生 NIO 有什么问题？

参考答案：

原生 NIO 有几个主要问题：

  1. API 复杂：Buffer 的 flip/clear 容易出错
  2. TCP 粘包/拆包需要自己处理
  3. Selector 空轮询 Bug：Linux 上 epoll 的 bug 会导致 CPU 100%
  4. 缺少高级功能：编解码、心跳、SSL 等都需要自己实现

Netty 解决了所有这些问题：ByteBuf 用读写双指针代替了 flip；内置多种粘包/拆包解码器；通过计数 + 重建 Selector 解决空轮询 bug；提供完善的编解码和连接管理支持。

而且 Netty 内置了主从 Reactor 模型，性能经过了工业级验证。Dubbo、gRPC、Elasticsearch 等大量项目都基于 Netty。

Q15：Netty 的线程模型是怎样的？

参考答案：

Netty 默认使用主从多线程 Reactor 模型。

Boss Group（通常 1 个线程）负责接收客户端连接（accept），收到新连接后把它分配给 Worker Group。

Worker Group（通常 CPU 核数 × 2 个线程）负责处理 IO 读写和业务逻辑。每个 Worker 线程（EventLoop）通过 Selector 管理多个 Channel。

关键设计：一个 Channel 在整个生命周期内只由一个 EventLoop 处理，避免了多线程并发问题，也不需要加锁。

Q16：Netty 如何解决 TCP 粘包/拆包问题？

参考答案：

TCP 是流协议，没有消息边界，所以会出现粘包（多条消息粘在一起）和拆包（一条消息被拆成多次传输）。

Netty 提供了多种开箱即用的解码器：

  • FixedLengthFrameDecoder：按固定长度拆分
  • LineBasedFrameDecoder：按换行符拆分
  • DelimiterBasedFrameDecoder：按自定义分隔符拆分
  • LengthFieldBasedFrameDecoder：按长度字段拆分（最常用，适合自定义二进制协议）

实际项目中最常用的是 LengthFieldBasedFrameDecoder，在消息头中包含一个长度字段，接收方根据长度字段来确定消息边界。

Q17：Netty 的 ByteBuf 和 Java NIO 的 ByteBuffer 有什么区别？

参考答案：

主要有四点区别：

  1. 读写模式：ByteBuffer 只有一个 position 指针，读写需要 flip 切换；ByteBuf 有读指针和写指针，读写独立，不需要 flip。

  2. 动态扩容：ByteBuffer 大小固定，创建后不能改变；ByteBuf 支持自动扩容。

  3. 内存池化：ByteBuffer 不支持池化；ByteBuf 提供 PooledByteBufAllocator，可以复用内存，减少 GC 压力。

  4. 零拷贝支持：ByteBuf 支持 CompositeByteBuf（逻辑合并多个 Buffer）和 slice（逻辑切分 Buffer），在应用层减少了数据拷贝。

9.5 综合场景题

Q18：如果让你设计一个高并发聊天服务器，你会怎么选择 IO 模型？

参考思路：

聊天服务器的特点：大量长连接，每个连接大部分时间空闲，偶尔收发消息。

我会选择 NIO（IO 多路复用）+ Netty 框架。

原因：

  1. BIO 不行——一个连接一个线程，1 万个在线用户就要 1 万个线程
  2. NIO 的 Selector 可以让少量线程管理大量连接，适合"连接多、活跃少"的场景
  3. 不直接用原生 NIO 而用 Netty，因为 Netty 提供了完善的编解码、心跳检测、线程模型
  4. 服务器部署在 Linux 上，NIO 底层是 epoll，性能足够
  5. AIO 在 Linux 上没有性能优势，不选

架构设计：Netty 主从 Reactor 模型，Boss 线程接受连接，Worker 线程处理 IO，业务逻辑交给独立的业务线程池。使用 LengthFieldBasedFrameDecoder 解决粘包问题，Protobuf 做序列化。

Q19：说一下你对 Java IO 知识体系的整体理解。

参考思路：（这种开放题可以展现你的知识广度）

Java IO 可以分三个层次来理解：

基础层是传统的 IO 流（InputStream/OutputStream/Reader/Writer），核心设计是装饰者模式，解决的是"如何读写数据"的问题。

模型层是 BIO/NIO/AIO 三种 IO 模型，解决的是"如何高效地处理网络 IO"的问题。BIO 一连接一线程，NIO 通过 Selector 实现多路复用，AIO 是异步回调。底层对应 Unix 的五种 IO 模型，关键是理解同步/异步、阻塞/非阻塞的区别。

框架层是 Netty 这样的高性能网络框架，在 NIO 之上提供了线程模型（Reactor）、编解码、连接管理等能力，是实际项目中的首选。

另外，零拷贝是贯穿各层的性能优化技术，Kafka 用 sendfile，RocketMQ 用 mmap，Netty 也有应用层的零拷贝支持。

面试答题锦囊

1. 不要背书，要讲故事

不要一上来就"BIO 是 Blocking IO，NIO 是 New IO..."，而是用比喻和场景引入。比如先说"想象你去饭店吃饭..."，这样面试官更容易理解，也显得你真正掌握了。

2. 主动延伸，展示知识面

回答完一个问题后，可以自然地引出相关知识。比如说完 NIO 后提一句"不过原生 NIO 有空轮询 bug，所以实际项目中一般用 Netty"，面试官可能就顺着问 Netty，而你已经准备好了。

3. 画图说明

IO 模型的对比、Reactor 模型的架构，如果面试官提供白板，画图比纯口述效果好 10 倍。面试前可以练习画：

  • 五种 IO 模型的时序图
  • BIO/NIO/AIO 的对比表
  • Netty 的主从 Reactor 架构图

4. 承认不足

如果遇到不会的问题，诚实说"这块我了解不深"比乱答好得多。你可以说"虽然没深入研究过，但我理解它大概是..."
