第三章：Java BIO 详解——从阻塞说起

学习目标：深入理解 BIO 的工作方式和瓶颈所在。通过代码实战感受"阻塞"到底意味着什么。

前置知识回顾

上一章我们学了 Unix 的五种 IO 模型。Java BIO 对应的就是第一种——阻塞式 IO。

  • 阶段一（等待数据）：阻塞
  • 阶段二（复制数据）：阻塞
  • 整个过程线程完全挂起，直到操作完成

3.1 什么是 BIO

BIO = Blocking IO，Java 中最传统的网络编程方式。

核心流程如下：

服务器启动
    │
    ▼
ServerSocket.accept()  ←── 阻塞！等待客户端连接
    │
    ▼ 收到连接
InputStream.read()     ←── 阻塞！等待客户端发送数据
    │
    ▼ 收到数据
处理业务逻辑
    │
    ▼
OutputStream.write()   ←── 发送响应
    │
    ▼
回到 accept() 等下一个连接

关键点：accept() 和 read() 都是阻塞的系统调用，线程会一直等待。

看一下 JDK 源码中 ServerSocket.accept() 的注释：

Listens for a connection to be made to this socket and accepts it. The method blocks until a connection is made.

翻译：监听连接，该方法会阻塞直到有连接到来。

3.2 BIO 的通信模型

模型一：单线程 BIO

最简单的模型——服务器只有一个线程，同一时间只能服务一个客户端。

客户端A ──请求──→ ┌────────────┐
                  │  服务器线程  │ ←── 正在处理 A 的请求
客户端B ──请求──→ │  (单线程)   │ ←── B 只能等待
客户端C ──请求──→ │            │ ←── C 只能等待
                  └────────────┘

问题：当服务器在处理客户端 A 的请求时，客户端 B 和 C 只能干等。处理完 A 才能处理 B。这在高并发场景下完全不可接受。

模型二：多线程 BIO（伪异步）

自然的改进思路——每个连接分配一个线程：

客户端A ──请求──→ ┌────────────┐   ┌──────────┐
                  │            │──→│ 线程1处理A │
客户端B ──请求──→ │  主线程     │   └──────────┘
                  │  (accept)  │   ┌──────────┐
客户端C ──请求──→ │            │──→│ 线程2处理B │
                  └────────────┘   └──────────┘
                                   ┌──────────┐
                               ──→ │ 线程3处理C │
                                   └──────────┘

改进了什么：多个客户端可以同时被服务。

但仍有问题：

  1. 线程是昂贵资源：每个线程默认占用 128K~1M 的栈空间，1 万个连接就要 1 万个线程
  2. 线程切换有开销：线程越多，CPU 花在上下文切换上的时间越多
  3. 线程数有上限：Linux 系统的线程数有限（cat /proc/sys/kernel/threads-max 可查看）
  4. 即使用线程池：线程池大小固定，超出的请求只能排队等待（BlockingQueue 积压）
  5. accept 仍然是串行的：主线程接受连接还是一个一个来的

BIO 的根本问题

问题的根源不在于是否使用多线程，而在于 accept() 和 read() 这两个操作本身就是阻塞的。

  • accept()：底层使用的是操作系统的阻塞式同步 IO，如果没有新连接到来，线程就挂起
  • read()：如果客户端还没有发送数据，线程也挂起

多线程只是把"处理过程"并行化了，但 接受连接的环节仍然是串行的。

3.3 BIO 代码实战

单线程 BIO 服务器

public class BioServer {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);
        System.out.println("服务器启动，监听 8080 端口...");
        
        while (true) {
            // 阻塞点1：等待客户端连接
            Socket socket = serverSocket.accept();
            System.out.println("收到连接：" + socket.getPort());
            
            // 阻塞点2：等待客户端发送数据
            InputStream in = socket.getInputStream();
            byte[] buffer = new byte[1024];
            int len = in.read(buffer);  // 阻塞！
            
            String message = new String(buffer, 0, len);
            System.out.println("收到消息：" + message);
            
            // 发送响应
            OutputStream out = socket.getOutputStream();
            out.write("已收到你的消息".getBytes());
            
            socket.close();
        }
    }
}

测试结果：如果同时有 20 个客户端发请求，它们只能逐个被处理。假设每个请求处理 1 秒，第 20 个客户端要等 20 秒才能得到响应。

多线程 BIO 服务器

public class BioServerMultiThread {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);
        System.out.println("服务器启动，监听 8080 端口...");
        
        while (true) {
            // 主线程仍然在这里阻塞等待连接
            Socket socket = serverSocket.accept();
            
            // 把业务处理交给新线程
            new Thread(() -> handleClient(socket)).start();
        }
    }
    
    private static void handleClient(Socket socket) {
        try (InputStream in = socket.getInputStream();
             OutputStream out = socket.getOutputStream()) {
             
            byte[] buffer = new byte[1024];
            int len = in.read(buffer);  // 在子线程中阻塞
            
            String message = new String(buffer, 0, len);
            System.out.println("[" + Thread.currentThread().getName() + "] 收到：" + message);
            
            // 模拟业务处理耗时
            Thread.sleep(1000);
            
            out.write("已处理你的消息".getBytes());
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            try { socket.close(); } catch (IOException e) { }
        }
    }
}

改进：多个客户端的请求可以并行处理了。

但是：

  • 每个连接创建一个线程，如果有 10000 个并发连接，就要创建 10000 个线程
  • 如果是长连接，线程一直不释放

线程池版 BIO 服务器

public class BioServerThreadPool {
    public static void main(String[] args) throws IOException {
        // 最多 50 个线程，超出的排队
        ExecutorService threadPool = Executors.newFixedThreadPool(50);
        ServerSocket serverSocket = new ServerSocket(8080);
        
        while (true) {
            Socket socket = serverSocket.accept();
            threadPool.execute(() -> handleClient(socket));
        }
    }
    
    private static void handleClient(Socket socket) {
        // 和上面相同...
    }
}

改进：控制了线程数量，不会无限创建线程。

但是：线程池大小是固定的（比如 50），如果 50 个线程都在处理长连接，第 51 个客户端只能排队等。而且这 50 个线程大部分时间可能都在阻塞等待 read()，造成了严重的资源浪费。

3.4 BIO 的性能瓶颈分析

为什么 BIO 在高并发下不行？

┌──────────────────────────────────────────────────────┐
│                   BIO 服务器                          │
│                                                      │
│  线程1: [████ 等待read ████ 处理 ██ 响应]              │
│  线程2: [██████████ 等待read ██████████ 处理 ██]      │
│  线程3: [████ 等待read ████████████ 处理 ████]        │
│  ...                                                 │
│  线程N: [████████████████ 等待read ████ 处理 ██]      │
│                                                      │
│  观察：大部分时间线程都在等待，真正干活的时间很短          │
└──────────────────────────────────────────────────────┘

核心矛盾：

  • 每个连接需要一个线程
  • 每个线程大部分时间都在阻塞等待（IO 操作远慢于 CPU）
  • 线程是操作系统的稀缺资源
  • 线程切换有成本

结论：BIO 模型下，线程数 ≈ 连接数。当连接数达到成千上万时，线程资源会被耗尽。

C10K 问题

这就是著名的 C10K 问题（如何在单机上支持 1 万个并发连接）。BIO 模型根本无法解决这个问题，因为你不可能创建 1 万个线程。

解决 C10K 问题的关键是：用少量线程管理大量连接——这正是 NIO 的核心思想。

3.5 BIO 的适用场景

BIO 不是完全没有用武之地：

  • 连接数少（几十到几百个）
  • 每个连接的数据交互频繁、处理时间短
  • 开发简单性优先于性能
  • 内部系统通信，如管理后台
