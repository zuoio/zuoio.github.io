第六章：BIO vs NIO vs AIO 全面对比（面试核心章节）

学习目标：用一个比喻、一张表、多个维度，彻底说清三者区别。这一章是面试回答的直接弹药。

6.1 一个比喻彻底搞懂

想象你要吃午饭：

BIO（阻塞式 IO）—— 堂食干等型

你走进饭店，坐下来点了一份牛肉面。然后你就坐在那里干等，什么都不干——不刷手机、不聊天、不看书——就盯着厨房方向等。直到面端上来，你才开始吃。

特点：全程等待，人力（线程）浪费严重。

NIO（非阻塞 IO / IO 多路复用）—— 取号等叫型

你走进饭店，取了个号码牌坐下。你可以刷手机、看书，但是你需要时不时抬头看看叫号屏（Selector 轮询）。屏幕上显示"32 号请取餐"的时候，你才起身去取。

而且，一个叫号屏可以同时管理所有人的订单——这就是一个 Selector 管理多个 Channel。

特点：不用干等，但需要自己定期检查。一个叫号屏（线程）管所有人（连接）。

AIO（异步 IO）—— 外卖到家型

你在手机上点了外卖，然后该干嘛干嘛——打游戏、做家务、睡觉都行。骑手到你门口时按门铃通知你（操作系统回调），你直接开门拿到吃的。

全程不需要等待，不需要检查，骑手（操作系统）搞定一切。

特点：完全不管，有人送上门。但骑手（操作系统支持）很重要——如果你在的城市没有外卖服务（Linux 没有原生 AIO），那就只能退回到叫号模式。

6.3 用代码对比三者

同一个功能（服务器接收客户端消息并回显），用三种方式实现的核心逻辑对比：

BIO 核心代码

// 简单直接，但一个连接一个线程
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();           // 阻塞！等待连接
    new Thread(() -> {
        InputStream in = socket.getInputStream();
        int data = in.read();                   // 阻塞！等待数据
        // 处理...
    }).start();
}

NIO 核心代码

// 一个线程管理所有连接
Selector selector = Selector.open();
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.configureBlocking(false);
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();                          // 阻塞，但同时等多个事件
    for (SelectionKey key : selector.selectedKeys()) {
        if (key.isAcceptable()) { /* 处理新连接 */ }
        if (key.isReadable())   { /* 读取数据 */ }
    }
}

AIO 核心代码

// 完全回调驱动，不需要事件循环
AsynchronousServerSocketChannel server = AsynchronousServerSocketChannel.open();
server.bind(new InetSocketAddress(8080));
server.accept(null, new CompletionHandler<AsynchronousSocketChannel, Void>() {
    @Override
    public void completed(AsynchronousSocketChannel client, Void att) {
        server.accept(null, this);              // 重新注册
        ByteBuffer buf = ByteBuffer.allocate(1024);
        client.read(buf, buf, new CompletionHandler<>() {  // 回调里套回调
            @Override
            public void completed(Integer len, ByteBuffer b) { /* 处理数据 */ }
            @Override
            public void failed(Throwable e, ByteBuffer b) { /* 处理异常 */ }
        });
    }
    @Override
    public void failed(Throwable exc, Void att) { /* 处理异常 */ }
});

6.4 适用场景总结

BIO 适用场景

  • 连接数少且固定（几十到几百）
  • 每个连接的数据交互频繁、处理时间短
  • 简单的内部管理系统
  • 开发效率优先于运行效率

NIO 适用场景

  • 高并发、大量连接（聊天服务器、推送服务）
  • 连接多但每个连接的活跃度不高（大部分时间空闲）
  • 需要长连接保持
  • 服务器运行在 Linux 上

AIO 适用场景

  • Windows 平台（能发挥 IOCP 的真正异步能力）
  • 连接数多且每个连接 IO 操作重（文件操作密集）
  • 实际中很少单独使用，通常被 Netty 等框架替代

6.5 面试经典问题逐个击破

Q1：BIO 和 NIO 的区别是什么？

BIO 是阻塞式 IO，每个连接需要一个独立线程处理，accept() 和 read() 都会阻塞线程。当并发连接数很多时，需要创建大量线程，资源消耗严重。

NIO 是基于 IO 多路复用的非阻塞 IO，核心是 Selector 机制。一个线程通过 Selector 可以同时监听多个 Channel 的事件，大大减少了线程数量。NIO 面向 Buffer 操作数据，Channel 是双向的。底层在 Linux 上使用 epoll 实现。

Q2：NIO 为什么比 BIO 性能好？

主要在于线程模型的区别。BIO 是一连接一线程，1 万个连接就要 1 万个线程，线程创建、切换、内存消耗都是开销。NIO 通过 Selector 实现了 IO 多路复用，一个线程就可以管理上万个连接。而且 NIO 的 Channel 是非阻塞的，不会有线程在空等。

底层上，NIO 在 Linux 上使用 epoll，它用事件驱动的方式通知就绪的文件描述符，而不需要遍历所有连接，所以在连接数很多的情况下性能远好于 BIO。

Q3：同步和异步的区别？阻塞和非阻塞的区别？

这是两组不同维度的概念：

阻塞/非阻塞描述的是应用程序的等待方式——发起 IO 后是干等（阻塞）还是可以先做别的事（非阻塞）。

同步/异步描述的是 IO 操作的完成通知方式——同步是应用程序自己去检查 IO 是否完成，异步是操作系统完成后主动通知应用程序。

BIO 是同步阻塞，NIO 是同步非阻塞（虽然 select 本身会阻塞，但 Channel 是非阻塞的），AIO 是异步非阻塞。

Q4：select、poll、epoll 的区别？

三者都是 IO 多路复用的实现：

  • select：最古老，最多监听 1024 个 fd，每次调用要把 fd 集合拷贝到内核，返回后需要遍历所有 fd 找出就绪的，O(n) 复杂度。
  • poll：去掉了 1024 的限制，但本质和 select 一样需要遍历。
  • epoll：Linux 独有，fd 只需注册一次，只返回就绪的 fd，O(1) 复杂度。在大量连接场景下性能远超 select/poll。

Java NIO 在 Linux 上底层就是 epoll。

Q5：NIO 的 Selector 是怎么工作的？

Selector 实现了 Reactor 模式。工作流程是：

  1. 创建 Selector 并将多个 Channel 注册上去，每个 Channel 声明关心的事件（ACCEPT/READ/WRITE）
  2. 调用 select() 方法阻塞等待，任何注册的 Channel 有事件就绪就返回
  3. 遍历返回的 SelectionKey 集合，根据事件类型做对应处理
  4. 处理完后回到 step 2，形成事件循环

底层在 Linux 上通过 epoll 实现，一个线程就可以高效管理上万个连接。

Q6：什么是 Reactor 模型？

Reactor 是一种事件驱动的设计模式，核心思想是：用一个或少量线程监听事件，事件发生后分发给对应的处理器。

三种变体：

  • 单线程 Reactor：一个线程做所有事（accept + read + 业务处理），简单但无法利用多核
  • 多线程 Reactor：一个线程监听事件，业务处理交给线程池，能利用多核但 accept 仍是瓶颈
  • 主从多线程 Reactor：Boss 线程负责 accept，Worker 线程组负责 IO 和业务处理。Netty 默认使用这种模型

Q7：为什么 Netty 使用 NIO 而不是 AIO？

两个原因：

  1. Linux 上 AIO 没有性能优势：Java AIO 在 Linux 底层还是 epoll 模拟的，跟 NIO 相比没有本质提升。而大多数服务器运行在 Linux 上。
  2. AIO 编程模型更复杂：回调嵌套回调容易产生"回调地狱"，代码可读性和可维护性差。NIO 的事件循环模型更直观。

Netty 曾在 4.0 版本支持 AIO，但后来移除了，因为实测证明在 Linux 上没有性能提升。

6.6 一张思维导图总结全部

Java IO 体系
│
├── BIO（JDK 1.0）
│   ├── 阻塞同步
│   ├── 面向流
│   ├── 一连接一线程
│   ├── 适合：低并发、简单场景
│   └── 问题：线程资源受限，C10K 无解
│
├── NIO（JDK 1.4）
│   ├── 非阻塞同步（IO 多路复用）
│   ├── 面向 Buffer
│   ├── 三大组件：Channel + Buffer + Selector
│   ├── 一线程管多连接
│   ├── 底层：Linux epoll
│   ├── 适合：高并发、长连接
│   └── 问题：API 复杂、有 bug
│
├── AIO（JDK 1.7）
│   ├── 异步非阻塞
│   ├── 订阅-通知模式
│   ├── CompletionHandler 回调
│   ├── 不需要 Selector
│   ├── 底层：Windows IOCP / Linux epoll 模拟
│   ├── 适合：Windows 平台、IO 密集
│   └── 问题：Linux 无原生支持、回调地狱
│
└── Netty（框架）
    ├── 基于 NIO 封装
    ├── 修复了 NIO 的 bug
    ├── 提供编解码、线程模型、连接管理
    ├── 主从 Reactor 模式
    └── 事实标准：Dubbo, gRPC, ES 等都在用
