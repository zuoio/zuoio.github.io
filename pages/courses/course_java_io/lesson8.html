第八章：Netty 入门——为什么不直接用 NIO？

学习目标：理解 Netty 存在的意义，掌握其核心概念和线程模型，知道它解决了原生 NIO 的哪些问题。

前言

前面我们学了原生 NIO，你可能已经感受到了——代码量大、容易出错、要自己处理的东西太多。实际项目中，几乎不会直接使用 Java 原生 NIO，而是使用 Netty 框架。

面试中 Netty 是高频考点，即使你没深入用过，也需要了解它的核心概念和设计理念。

8.1 原生 NIO 有什么问题？

问题一：API 复杂，容易出错

Buffer 的 flip()、clear()、compact() 容易搞混。忘了 flip 就写不出数据，忘了 clear 就读不到新数据。

buffer.put(data);
// 忘了 flip()...
channel.write(buffer);  // 写出去的是空的！因为 position 还在末尾

问题二：需要自己处理 TCP 粘包/拆包

TCP 是流协议，没有消息边界。发送方发了两条消息 "Hello" 和 "World"，接收方可能一次性收到 "HelloWorld"（粘包），也可能分两次收到 "Hel" 和 "loWorld"（拆包）。

原生 NIO 需要你自己实现消息的拆分和组装逻辑，这非常容易出 bug。

问题三：Selector 空轮询 Bug

这是 Java NIO 在 Linux 上的一个著名 bug：

正常情况下，selector.select() 应该阻塞到有事件就绪才返回。但由于底层 epoll 的 bug，select() 可能在没有任何事件的情况下返回 0，导致 while 循环空转，CPU 飙到 100%。

这个 bug 从 JDK 1.6 就存在，在 JDK 的 bug 追踪系统中编号为 6670302 和 2147719，至今没有完全修复。

问题四：缺少高级功能

原生 NIO 没有提供：

  • 编解码框架（JSON、Protobuf、自定义协议）
  • 心跳检测
  • 连接超时管理
  • 异常处理和重连机制
  • SSL/TLS 支持
  • HTTP/WebSocket 协议支持

所有这些都需要自己从零实现。

8.2 Netty 是什么？

Netty 是一个高性能、异步事件驱动的网络应用框架，基于 Java NIO 封装，极大地简化了 TCP/UDP 网络编程。

Netty 的定位

┌─────────────────────────────────────────────┐
│            你的业务代码                        │
├─────────────────────────────────────────────┤
│               Netty                          │
│  编解码 | 线程模型 | 连接管理 | 异常处理 | SSL  │
├─────────────────────────────────────────────┤
│            Java NIO                          │
│  Channel | Buffer | Selector                 │
├─────────────────────────────────────────────┤
│           操作系统                             │
│  epoll (Linux) | kqueue (macOS)              │
└─────────────────────────────────────────────┘

Netty 在 NIO 之上提供了一层优雅的抽象，让你只需关注业务逻辑。

谁在用 Netty？

  • Dubbo：阿里的 RPC 框架，底层通信用 Netty
  • gRPC：Google 的 RPC 框架，Java 版底层用 Netty
  • Elasticsearch：节点间通信用 Netty
  • RocketMQ：消息通信用 Netty
  • Spark：节点间数据传输用 Netty
  • Zookeeper：客户端与服务端通信可选 Netty

可以说，Netty 是 Java 网络编程的事实标准。

8.3 Netty 核心概念

核心组件一览

EventLoopGroup（线程组）
├── EventLoop（事件循环 = 一个线程）
│   ├── 管理多个 Channel
│   └── 处理 IO 事件 + 定时任务
│
Channel（通道）
├── 对 Java NIO Channel 的封装
├── NioServerSocketChannel
├── NioSocketChannel
└── 关联一个 ChannelPipeline
│
ChannelPipeline（处理器链）
├── ChannelHandler（处理器）
│   ├── ChannelInboundHandler（入站处理：解码、业务逻辑）
│   └── ChannelOutboundHandler（出站处理：编码、发送）
│
ByteBuf（缓冲区）
└── 增强版 Buffer，读写双指针，不需要 flip

EventLoopGroup 和 EventLoop

EventLoop = 一个线程 + 一个 Selector + 一个任务队列

一个 EventLoop 负责处理多个 Channel 的 IO 事件。这就是 NIO 中"一个线程管多个连接"的封装。

EventLoopGroup = 一组 EventLoop（线程池）

// 创建两个 EventLoopGroup
EventLoopGroup bossGroup = new NioEventLoopGroup(1);    // 1 个线程，负责 accept
EventLoopGroup workerGroup = new NioEventLoopGroup(4);  // 4 个线程，负责 IO

Channel

Netty 的 Channel 是对 Java NIO Channel 的封装，常用的有：

Netty Channel 对应 NIO Channel 用途
NioServerSocketChannel ServerSocketChannel TCP 服务端监听
NioSocketChannel SocketChannel TCP 客户端/服务端连接
NioDatagramChannel DatagramChannel UDP

ChannelHandler 和 ChannelPipeline

这是 Netty 最精妙的设计——责任链模式。

每个 Channel 都有一个 ChannelPipeline，Pipeline 中包含多个 ChannelHandler，数据流经 Pipeline 时依次被每个 Handler 处理：

入站（接收数据）方向 →
┌─────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│网络  │──→│解码器 │──→│日志   │──→│业务   │──→│...   │
│数据  │   │Handler│   │Handler│   │Handler│   │      │
└─────┘   └──────┘   └──────┘   └──────┘   └──────┘

← 出站（发送数据）方向
┌──────┐   ┌──────┐   ┌─────┐
│编码器 │←──│日志   │←──│业务  │
│Handler│   │Handler│   │数据  │
└──────┘   └──────┘   └─────┘

好处：每个 Handler 只关注一件事（解码、编码、业务逻辑、日志等），通过组合实现复杂功能。想加功能？往 Pipeline 里加一个 Handler 就行。

ByteBuf：比 ByteBuffer 好用 100 倍

原生 NIO 的 ByteBuffer 需要手动 flip/clear，容易出错。Netty 的 ByteBuf 用读写双指针解决了这个问题：

ByteBuffer（NIO）：一个 position 指针，读写需要 flip 切换

ByteBuf（Netty）：两个指针，读写独立
┌──────────────────────────────────────────────────┐
│  已读区域    │     可读区域      │    可写区域      │
│ (discardable)│   (readable)     │   (writable)    │
└──────────────────────────────────────────────────┘
0          readerIndex      writerIndex         capacity
特性 ByteBuffer (NIO) ByteBuf (Netty)
读写切换 需要 flip() 不需要，读写指针独立
动态扩容 不支持，固定大小 支持自动扩容
引用计数 不支持 支持（手动管理内存释放）
池化 不支持 支持（PooledByteBufAllocator）
零拷贝 有限支持 CompositeByteBuf、slice 等

8.4 Netty 的线程模型

Reactor 模式回顾

第二章讲过 Reactor 模式有三种变体。Netty 默认使用的是最强的一种——主从多线程 Reactor。

主从多线程 Reactor

                    ┌──────────────────────────────────┐
                    │         Boss Group                │
                    │   (1个线程，负责 accept)            │
  客户端连接 ──────→│   EventLoop: selector.select()    │
                    │         ↓ 新连接到达               │
                    └─────────┼────────────────────────┘
                              │ 将新连接分配给 Worker
                    ┌─────────▼────────────────────────┐
                    │         Worker Group               │
                    │   (N个线程，负责 IO 读写)           │
                    │                                    │
                    │   EventLoop-1: channel A, B, C     │
                    │   EventLoop-2: channel D, E, F     │
                    │   EventLoop-3: channel G, H, I     │
                    │   EventLoop-N: ...                  │
                    │                                    │
                    │   每个 EventLoop 处理分配给它的      │
                    │   Channel 的所有 IO 事件             │
                    └────────────────────────────────────┘

工作流程：

  1. Boss Group 中有一个 EventLoop（线程），只负责监听 OP_ACCEPT 事件（接收新连接）
  2. 新连接到达后，Boss 把这个连接分配给 Worker Group 中的某个 EventLoop
  3. Worker EventLoop 负责这个连接后续所有的 IO 操作（读、写、编解码、业务处理）
  4. 一个 Worker EventLoop 可以管理多个 Channel，但一个 Channel 的整个生命周期只由一个 EventLoop 负责（避免并发问题）

为什么高效：

  • Boss 和 Worker 分离，accept 不会阻塞 IO 处理
  • Worker 线程数通常设为 CPU 核数 × 2
  • 一个 Channel 绑定一个 EventLoop，不需要加锁

8.5 Netty 服务器示例

import io.netty.bootstrap.ServerBootstrap;
import io.netty.channel.*;
import io.netty.channel.nio.NioEventLoopGroup;
import io.netty.channel.socket.SocketChannel;
import io.netty.channel.socket.nio.NioServerSocketChannel;
import io.netty.handler.codec.string.StringDecoder;
import io.netty.handler.codec.string.StringEncoder;

public class NettyServer {
    public static void main(String[] args) throws InterruptedException {
        // Boss 线程组：接收连接
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        // Worker 线程组：处理 IO
        EventLoopGroup workerGroup = new NioEventLoopGroup();
        
        try {
            ServerBootstrap bootstrap = new ServerBootstrap();
            bootstrap.group(bossGroup, workerGroup)
                .channel(NioServerSocketChannel.class)         // 使用 NIO
                .childHandler(new ChannelInitializer<SocketChannel>() {
                    @Override
                    protected void initChannel(SocketChannel ch) {
                        ChannelPipeline pipeline = ch.pipeline();
                        pipeline.addLast(new StringDecoder());  // 解码器
                        pipeline.addLast(new StringEncoder());  // 编码器
                        pipeline.addLast(new BusinessHandler()); // 业务处理器
                    }
                });
            
            // 绑定端口并启动
            ChannelFuture future = bootstrap.bind(8080).sync();
            System.out.println("Netty 服务器启动，监听 8080 端口...");
            
            // 等待服务器关闭
            future.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}

class BusinessHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        String message = (String) msg;
        System.out.println("收到消息：" + message);
        ctx.writeAndFlush("Echo: " + message);
    }
    
    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}

对比第四章的原生 NIO 服务器代码——Netty 版本更简洁、更清晰、更健壮。

8.6 Netty 如何解决 NIO 的问题

NIO 的问题 Netty 的解决方案
Buffer flip/clear 容易出错 ByteBuf 读写双指针，不需要 flip
TCP 粘包/拆包 内置多种解码器：LineBasedFrameDecoder、LengthFieldBasedFrameDecoder 等
Selector 空轮询 bug 检测空轮询次数，超过阈值自动重建 Selector
缺少编解码支持 内置 String、JSON、Protobuf 等编解码器
需要自己管理线程模型 内置主从 Reactor 模型，开箱即用
缺少连接管理 内置心跳检测、超时管理
缺少 SSL 支持 SslHandler 一行代码搞定

Netty 如何解决空轮询 Bug？

Netty 的策略：
1. 记录 select() 返回 0 的次数（空轮询计数器）
2. 如果连续空轮询次数超过阈值（默认 512 次），认为触发了 bug
3. 重建一个新的 Selector，把旧 Selector 上的所有 Channel 迁移过去
4. 关闭旧 Selector

Netty 如何解决粘包/拆包？

Netty 提供了多种开箱即用的解码器：

解码器 策略 适用场景
FixedLengthFrameDecoder 固定长度 每条消息长度固定
LineBasedFrameDecoder 换行符分割 文本协议（按行）
DelimiterBasedFrameDecoder 自定义分隔符 自定义文本协议
LengthFieldBasedFrameDecoder 长度字段 二进制协议（最常用）

8.7 为什么 Netty 选择 NIO 而不是 AIO？

这个问题前面章节提过，这里做一个完整的总结：

  1. Linux 上 AIO 没有性能优势：Java AIO 在 Linux 底层还是 epoll，和 NIO 一样
  2. Netty 的线程模型已经足够高效：主从 Reactor + EventLoop，一个线程管理多个 Channel
  3. AIO 增加了编程复杂度：回调嵌套、状态管理更复杂
  4. Netty 的历史验证：4.0 版本曾实现 AIO 支持，实测无性能提升后移除
