第五章：Java AIO 详解——真正的异步

学习目标：理解 AIO 的"订阅-通知"模型，掌握 CompletionHandler 回调机制，以及它与 NIO 的本质区别。

前言

前面我们学了：

  • BIO：两个阶段都阻塞，一连接一线程
  • NIO：通过 Selector 实现 IO 多路复用，一个线程管理多个连接，但 select() 仍然要阻塞等待

它们都属于同步 IO——应用程序需要主动去检查/等待 IO 是否完成。

AIO 则是另一种思路：你别来问我，我做完了主动通知你。

5.1 什么是 AIO

AIO = Asynchronous IO，JDK 1.7 引入，也叫 NIO.2。

核心思想：订阅-通知模式

NIO 的思路（轮询模式）：
应用程序 ──→ "数据好了吗？" ──→ 操作系统
应用程序 ──→ "数据好了吗？" ──→ 操作系统
应用程序 ──→ "数据好了吗？" ──→ 操作系统
应用程序 ←── "好了！"       ←── 操作系统

AIO 的思路（订阅-通知模式）：
应用程序 ──→ "数据好了叫我" ──→ 操作系统
应用程序去做其他事...
应用程序 ←── "搞定了！数据在这" ←── 操作系统主动通知

NIO vs AIO 的核心区别

对比项 NIO AIO
谁来检查 IO 完成 应用程序（通过 Selector 轮询） 操作系统（完成后回调通知）
是否需要 Selector 需要 不需要
IO 模型 同步非阻塞（IO 多路复用） 异步
编程模式 事件循环 回调/Future

一句话总结：NIO 是"我去问"，AIO 是"你来告诉我"。

5.2 AIO 的操作系统支持

AIO 是否真正异步，取决于操作系统的支持：

操作系统 实现方式 是否真正异步
Windows IOCP（IO Completion Port） 是
Linux epoll 模拟 不是（伪异步）
macOS kqueue 模拟 不是（伪异步）

这一点非常重要：在 Linux 上，Java AIO 底层仍然是 epoll（和 NIO 一样），只是 JDK 帮你封装了回调的"语法糖"。真正的异步性能优势只有在 Windows 的 IOCP 上才能体现。

这也是为什么 AIO 在 Java 服务器开发中没有大规模流行——大多数服务器运行在 Linux 上，AIO 并没有比 NIO 快。

5.3 AIO 核心 API

核心类

AsynchronousServerSocketChannel  # 异步服务器监听通道（类比 NIO 的 ServerSocketChannel）
AsynchronousSocketChannel        # 异步套接字通道（类比 NIO 的 SocketChannel）
AsynchronousFileChannel          # 异步文件通道（类比 NIO 的 FileChannel）
AsynchronousChannelGroup         # 通道组，管理一组异步通道共享的线程池
CompletionHandler<V, A>          # 回调接口

CompletionHandler 接口

这是 AIO 编程的核心——所有异步操作完成后，都通过这个接口回调：

public interface CompletionHandler<V, A> {
    // IO 操作成功完成时调用
    // V: 操作结果（如 read 返回读取的字节数）
    // A: 附件对象（可以传递上下文信息）
    void completed(V result, A attachment);
    
    // IO 操作失败时调用
    void failed(Throwable exc, A attachment);
}

两种异步操作方式

方式一：CompletionHandler 回调

channel.read(buffer, attachment, new CompletionHandler<Integer, Object>() {
    @Override
    public void completed(Integer bytesRead, Object attachment) {
        // 读取完成，处理数据
    }
    
    @Override
    public void failed(Throwable exc, Object attachment) {
        // 读取失败，处理异常
    }
});

方式二：Future

Future<Integer> future = channel.read(buffer);
// 做其他事...
Integer bytesRead = future.get();  // 需要结果时再阻塞获取

实际开发中 CompletionHandler 方式更常用，因为 Future.get() 会阻塞，失去了异步的意义。

5.4 AIO 服务器完整实战

服务器端

import java.io.IOException;
import java.net.InetSocketAddress;
import java.nio.ByteBuffer;
import java.nio.channels.AsynchronousChannelGroup;
import java.nio.channels.AsynchronousServerSocketChannel;
import java.nio.channels.AsynchronousSocketChannel;
import java.nio.channels.CompletionHandler;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class AioServer {
    
    public static void main(String[] args) throws Exception {
        // 创建线程池给异步通道使用
        ExecutorService threadPool = Executors.newFixedThreadPool(4);
        AsynchronousChannelGroup group = AsynchronousChannelGroup.withThreadPool(threadPool);
        
        // 创建异步服务器通道
        AsynchronousServerSocketChannel serverChannel = 
            AsynchronousServerSocketChannel.open(group);
        serverChannel.bind(new InetSocketAddress(8080));
        System.out.println("AIO 服务器启动，监听 8080 端口...");
        
        // 注册 accept 监听（异步！不阻塞！）
        serverChannel.accept(null, new AcceptHandler(serverChannel));
        
        // 主线程不退出
        Thread.currentThread().join();
    }
}

/**
 * 处理新连接到达的回调
 */
class AcceptHandler implements CompletionHandler<AsynchronousSocketChannel, Void> {
    
    private AsynchronousServerSocketChannel serverChannel;
    
    public AcceptHandler(AsynchronousServerSocketChannel serverChannel) {
        this.serverChannel = serverChannel;
    }
    
    @Override
    public void completed(AsynchronousSocketChannel clientChannel, Void attachment) {
        // 重要！每次回调后必须重新注册，因为 AIO 是"一次注册，一次响应"
        serverChannel.accept(null, this);
        
        try {
            System.out.println("新客户端连接：" + clientChannel.getRemoteAddress());
        } catch (IOException e) {
            e.printStackTrace();
        }
        
        // 为这个客户端连接注册读事件
        ByteBuffer buffer = ByteBuffer.allocate(1024);
        clientChannel.read(buffer, buffer, new ReadHandler(clientChannel));
    }
    
    @Override
    public void failed(Throwable exc, Void attachment) {
        System.err.println("接受连接失败：" + exc.getMessage());
    }
}

/**
 * 处理数据读取的回调
 */
class ReadHandler implements CompletionHandler<Integer, ByteBuffer> {
    
    private AsynchronousSocketChannel clientChannel;
    
    public ReadHandler(AsynchronousSocketChannel clientChannel) {
        this.clientChannel = clientChannel;
    }
    
    @Override
    public void completed(Integer bytesRead, ByteBuffer buffer) {
        if (bytesRead == -1) {
            try {
                clientChannel.close();
            } catch (IOException e) {
                e.printStackTrace();
            }
            return;
        }
        
        // 读取数据
        buffer.flip();
        byte[] data = new byte[buffer.remaining()];
        buffer.get(data);
        String message = new String(data);
        System.out.println("收到消息：" + message);
        
        // 回写响应
        ByteBuffer writeBuffer = ByteBuffer.wrap(("Echo: " + message).getBytes());
        clientChannel.write(writeBuffer, null, new CompletionHandler<Integer, Void>() {
            @Override
            public void completed(Integer result, Void attachment) {
                // 响应写入完成，继续监听读事件
                ByteBuffer newBuffer = ByteBuffer.allocate(1024);
                clientChannel.read(newBuffer, newBuffer, ReadHandler.this);
            }
            
            @Override
            public void failed(Throwable exc, Void attachment) {
                try { clientChannel.close(); } catch (IOException e) { }
            }
        });
    }
    
    @Override
    public void failed(Throwable exc, ByteBuffer attachment) {
        System.err.println("读取数据失败：" + exc.getMessage());
        try { clientChannel.close(); } catch (IOException e) { }
    }
}

关键要点分析

1. "一次注册，一次响应"

AIO 的每次异步操作（accept、read、write）都是一次性的——回调触发后，如果还想继续监听，必须重新注册：

@Override
public void completed(AsynchronousSocketChannel clientChannel, Void attachment) {
    // 必须重新注册 accept，否则不会再接收新连接
    serverChannel.accept(null, this);
    // ...
}

2. 没有 Selector

注意 AIO 代码中完全没有 Selector 的身影。因为不需要轮询了——操作系统会在 IO 完成时直接回调你的 CompletionHandler。

3. 回调地狱

观察上面的代码，你会发现回调嵌套回调（accept → read → write → read → ...），代码结构变得复杂。这就是所谓的"回调地狱"（Callback Hell），也是 AIO 编程复杂的原因之一。

4. 线程池的用途

AIO 中的线程池不是用来处理业务的，而是用来执行回调的。操作系统通知 IO 完成后，JDK 从线程池中取一个线程来执行你的 CompletionHandler。

5.5 AIO 文件操作

AIO 也支持异步文件操作：

import java.nio.ByteBuffer;
import java.nio.channels.AsynchronousFileChannel;
import java.nio.channels.CompletionHandler;
import java.nio.file.Paths;
import java.nio.file.StandardOpenOption;

public class AioFileDemo {
    public static void main(String[] args) throws Exception {
        AsynchronousFileChannel fileChannel = AsynchronousFileChannel.open(
            Paths.get("test.txt"), StandardOpenOption.READ
        );
        
        ByteBuffer buffer = ByteBuffer.allocate(1024);
        
        // 异步读取文件，从位置0开始
        fileChannel.read(buffer, 0, buffer, new CompletionHandler<Integer, ByteBuffer>() {
            @Override
            public void completed(Integer bytesRead, ByteBuffer buf) {
                buf.flip();
                byte[] data = new byte[buf.remaining()];
                buf.get(data);
                System.out.println("文件内容：" + new String(data));
            }
            
            @Override
            public void failed(Throwable exc, ByteBuffer buf) {
                System.err.println("读取失败：" + exc.getMessage());
            }
        });
        
        // 主线程可以做其他事
        System.out.println("主线程继续执行...");
        Thread.sleep(1000);
    }
}

5.6 为什么 AIO 没有大规模流行？

面试中经常会问到这个问题。原因有四：

1. Linux 上没有真正的异步 IO

Java AIO 在 Linux 上底层仍然是 epoll，只是 JDK 封装了回调接口。跟直接用 NIO + Selector 相比，性能没有本质提升，反而多了一层封装的开销。

2. 编程模型复杂

回调嵌套回调，代码可读性差，调试困难。NIO 的事件循环模型虽然也不简单，但至少代码是线性的。

3. Netty 放弃了 AIO

Netty（最流行的 Java 网络框架）曾在 4.0 版本尝试支持 AIO，但最终移除了。Netty 的作者 Trustin Lee 认为，在 Linux 上 AIO 没有带来性能提升，反而增加了复杂性。

4. epoll 已经足够好

IO 多路复用（NIO + epoll）在 Linux 上已经能够处理百万级并发连接（C10M 问题），对于绝大多数场景来说性能足够。

面试回答模板

Q：为什么 AIO 没有大规模使用？

A：主要有两个原因。第一，Linux 上没有原生的异步 IO 支持，Java AIO 底层还是 epoll 模拟的，性能和 NIO 差不多。第二，AIO 的回调编程模型比较复杂，代码可读性和可维护性不如 NIO 的事件循环模式。所以 Netty 最终也选择了 NIO 而不是 AIO。
