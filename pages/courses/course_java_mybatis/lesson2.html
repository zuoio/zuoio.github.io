大纲

第二章 全局配置文件 mybatis-config.xml 📖

定位：理解 MyBatis 的完整配置体系。日常开发中大部分配置已由 Spring Boot 自动接管，但在排查问题、阅读源码或需要精细调优时，必须清楚每个配置项的含义与生效原理。

2.1 configuration 根标签与子标签加载顺序

mybatis-config.xml 的根标签是 <configuration>，其内部子标签必须严格按照以下顺序排列，否则 XML 校验不通过，MyBatis 启动时会报错：

<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE configuration
  PUBLIC "-//mybatis.org//DTD Config 3.0//EN"
  "https://mybatis.org/dtd/mybatis-3-config.dtd">

<configuration>
  <!-- ① properties：外部化配置 -->
  <!-- ② settings：全局行为开关 -->
  <!-- ③ typeAliases：类型别名 -->
  <!-- ④ typeHandlers：类型处理器 -->
  <!-- ⑤ objectFactory：对象工厂 -->
  <!-- ⑥ objectWrapperFactory：对象包装工厂 -->
  <!-- ⑦ reflectorFactory：反射工厂 -->
  <!-- ⑧ plugins：插件（拦截器） -->
  <!-- ⑨ environments：环境配置（数据源 + 事务管理器） -->
  <!-- ⑩ databaseIdProvider：多数据库厂商标识 -->
  <!-- ⑪ mappers：映射器注册 -->
</configuration>

完整顺序口诀（按首字母缩写）：P-S-T-T-O-O-R-P-E-D-M

序号 子标签 用途
① properties 加载外部 .properties 文件或直接定义键值对
② settings 控制 MyBatis 的全局行为（最常用）
③ typeAliases 为 Java 全限定类名定义短别名
④ typeHandlers 注册自定义类型处理器
⑤ objectFactory 自定义结果对象的实例化方式（极少用）
⑥ objectWrapperFactory 自定义对象包装器工厂（极少用）
⑦ reflectorFactory 自定义反射工厂（极少用）
⑧ plugins 注册拦截器插件
⑨ environments 配置数据库环境（数据源、事务管理器）
⑩ databaseIdProvider 多数据库厂商支持
⑪ mappers 注册 Mapper XML 或 Mapper 接口

⚠️ 常见错误：把 <settings> 写在 <typeAliases> 之后，或把 <mappers> 写在 <environments> 之前，都会导致 DTD 校验失败。错误信息类似： The content of element type "configuration" must match "(properties?,settings?,typeAliases?,...)".

Spring Boot 中的对应：使用 Spring Boot 时不需要 mybatis-config.xml，所有配置通过 application.yml 完成。但如果仍想使用独立配置文件，可以通过以下方式指定：

# application.yml
mybatis:
  config-location: classpath:mybatis-config.xml

2.2 properties —— 外部化配置与占位符

<properties> 标签用于引入外部配置文件或直接定义键值对，在后续的配置中通过 ${key} 占位符引用。

方式一：引用外部 .properties 文件

<properties resource="db.properties" />

db.properties 内容：

db.driver=com.mysql.cj.jdbc.Driver
db.url=jdbc:mysql://localhost:3306/mydb?useSSL=false&serverTimezone=Asia/Shanghai
db.username=root
db.password=123456

方式二：直接在标签内定义

<properties>
  <property name="db.driver" value="com.mysql.cj.jdbc.Driver"/>
  <property name="db.username" value="root"/>
</properties>

方式三：两者混用

<properties resource="db.properties">
  <property name="db.username" value="admin"/>
</properties>

属性加载优先级（由低到高）

  1. <property> 子标签中定义的值
  2. resource / url 指向的外部文件中的值
  3. 通过 SqlSessionFactoryBuilder.build(reader, props) 以方法参数传入的值

高优先级覆盖低优先级，即方法参数 > 外部文件 > 标签内定义。

在后续标签中使用占位符

<environments default="dev">
  <environment id="dev">
    <transactionManager type="JDBC"/>
    <dataSource type="POOLED">
      <property name="driver" value="${db.driver}"/>
      <property name="url" value="${db.url}"/>
      <property name="username" value="${db.username}"/>
      <property name="password" value="${db.password}"/>
    </dataSource>
  </environment>
</environments>

默认值特性（MyBatis 3.4.2+）

从 3.4.2 版本开始，占位符支持默认值，但需要先在 <properties> 中启用：

<properties resource="db.properties">
  <property name="org.apache.ibatis.parsing.PropertyParser.enable-default-value" value="true"/>
</properties>

启用后可以这样使用：

<property name="username" value="${db.username:root}"/>

如果 db.username 未定义，则使用默认值 root。

Spring Boot 对应写法

Spring Boot 本身就使用 application.yml / application.properties 管理所有配置，数据源配置如下：

spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/mydb?useSSL=false&serverTimezone=Asia/Shanghai
    username: root
    password: 123456

外部化配置可以通过 Spring 的 Profile、环境变量、命令行参数等多种方式实现，比 MyBatis 原生的 <properties> 灵活得多。

2.3 settings（⭐ 重点掌握常用项）

<settings> 是 MyBatis 中最重要的配置标签，控制框架的全局运行行为。基本语法：

<settings>
  <setting name="配置项名" value="值"/>
  <!-- 可以有多个 setting -->
</settings>

2.3.1 mapUnderscoreToCamelCase —— 驼峰映射

作用：将数据库字段名中的下划线风格（user_name）自动映射为 Java 属性的驼峰风格（userName），省去手写 <resultMap> 或列别名。

<settings>
  <setting name="mapUnderscoreToCamelCase" value="true"/>
</settings>
属性 说明
默认值 false
推荐值 true（几乎所有项目都建议开启）

示例对比：

-- 数据库字段
SELECT user_name, create_time FROM t_user

public class User {
    private String userName;    // 自动匹配 user_name
    private Date createTime;    // 自动匹配 create_time
}

开启后无需额外配置即可完成映射，未开启时需要手动写别名或 <resultMap>。

Spring Boot 对应写法：

mybatis:
  configuration:
    map-underscore-to-camel-case: true

2.3.2 lazyLoadingEnabled / aggressiveLazyLoading —— 懒加载

作用：控制关联对象的加载时机。懒加载可以减少不必要的数据库查询，提升性能。

<settings>
  <setting name="lazyLoadingEnabled" value="true"/>
  <setting name="aggressiveLazyLoading" value="false"/>
</settings>
属性 说明 默认值 推荐值
lazyLoadingEnabled 全局懒加载开关 false 按需开启
aggressiveLazyLoading 激进懒加载：任意方法调用都会触发全部懒加载属性加载 false（3.4.1+） false

工作原理：

  • lazyLoadingEnabled=true：关联查询（<association> / <collection> 的嵌套查询）不会立即执行，而是在访问关联属性时才触发 SQL
  • aggressiveLazyLoading=true：只要调用了对象的任意方法（包括 toString()、hashCode()），就会立即加载所有懒加载属性。在 3.4.1 之前默认为 true，会导致懒加载形同虚设

注意：懒加载依赖动态代理（cglib 或 javassist），需要确保相关依赖可用。

Spring Boot 对应写法：

mybatis:
  configuration:
    lazy-loading-enabled: true
    aggressive-lazy-loading: false

2.3.3 cacheEnabled —— 全局缓存开关

作用：控制 MyBatis 二级缓存的全局开关。

<settings>
  <setting name="cacheEnabled" value="true"/>
</settings>
属性 说明 默认值 推荐值
cacheEnabled 二级缓存全局开关 true 视项目情况，分布式环境慎用

注意事项：

  • 此设置仅控制二级缓存（namespace 级别），一级缓存（SqlSession 级别）始终开启，无法通过此配置关闭
  • 即使全局开启，还需要在具体 Mapper XML 中添加 <cache/> 标签才能激活该 namespace 的二级缓存
  • 在分布式环境下，本地二级缓存可能导致脏读，建议使用 Redis 等分布式缓存方案或直接关闭

Spring Boot 对应写法：

mybatis:
  configuration:
    cache-enabled: true

2.3.4 defaultExecutorType —— SIMPLE / REUSE / BATCH

作用：设定默认的 SQL 执行器类型，影响 Statement 的复用策略和批量执行行为。

<settings>
  <setting name="defaultExecutorType" value="SIMPLE"/>
</settings>
类型 说明
SIMPLE 每次执行都创建新的 PreparedStatement（默认）
REUSE 复用相同 SQL 的 PreparedStatement，减少编译开销
BATCH 批量执行模式，将多条 SQL 攒在一起一次性发送给数据库
属性 说明 默认值 推荐值
defaultExecutorType 默认执行器类型 SIMPLE 一般用 SIMPLE，批量场景临时切换 BATCH

BATCH 模式的注意事项：

  • BATCH 模式下 insert / update / delete 的返回值不可靠（返回的是固定值 BATCH_UPDATE_RETURN_VALUE）
  • 通常不建议全局设为 BATCH，而是在需要批量操作时通过 SqlSessionFactory.openSession(ExecutorType.BATCH) 临时获取

Spring Boot 对应写法：

mybatis:
  configuration:
    default-executor-type: simple

2.3.5 logImpl —— 日志实现选择

作用：指定 MyBatis 使用的日志框架实现，方便在开发阶段输出 SQL 语句、参数和结果。

<settings>
  <setting name="logImpl" value="SLF4J"/>
</settings>

可选值（按优先级自动探测顺序排列）：

值 对应日志框架
SLF4J SLF4J（推荐，配合 Logback）
LOG4J Log4j 1.x（已过时）
LOG4J2 Log4j 2.x
JDK_LOGGING JDK 内置日志（java.util.logging）
COMMONS_LOGGING Apache Commons Logging
STDOUT_LOGGING 直接输出到标准输出（开发调试用）
NO_LOGGING 关闭日志
属性 说明 默认值 推荐值
logImpl 日志实现类 未设置（自动探测） SLF4J（Spring Boot 项目）

开发阶段查看 SQL：如果不想引入额外依赖，可以临时使用 STDOUT_LOGGING：

<setting name="logImpl" value="STDOUT_LOGGING"/>

Spring Boot 对应写法：

Spring Boot 项目一般使用 SLF4J + Logback，无需显式指定 logImpl，只需在 application.yml 中配置日志级别即可看到 SQL：

# 方式一：通过 mybatis 配置
mybatis:
  configuration:
    log-impl: org.apache.ibatis.logging.slf4j.Slf4jImpl

# 方式二（推荐）：通过 Spring Boot 日志级别控制
logging:
  level:
    com.example.mapper: debug   # 将 Mapper 包的日志级别设为 debug

2.4 typeAliases —— 类型别名

类型别名可以为 Java 全限定类名定义一个短名称，在 Mapper XML 中使用时更简洁。

方式一：逐个定义别名

<typeAliases>
  <typeAlias alias="User" type="com.example.entity.User"/>
  <typeAlias alias="Order" type="com.example.entity.Order"/>
</typeAliases>

方式二：包扫描（推荐）

<typeAliases>
  <package name="com.example.entity"/>
</typeAliases>

扫描指定包下所有类，默认别名为类名首字母小写（实际上 MyBatis 对别名不区分大小写）。例如 com.example.entity.User 的默认别名为 user。

如果需要自定义别名，可在实体类上添加 @Alias 注解：

@Alias("sysUser")
public class User {
    // ...
}

MyBatis 内置别名

MyBatis 预置了一批 Java 常用类型的别名，可以直接在 XML 中使用：

别名 Java 类型 别名 Java 类型
int Integer _int int（基本类型）
long Long _long long（基本类型）
double Double _double double（基本类型）
boolean Boolean _boolean boolean（基本类型）
string String date Date
map Map hashmap HashMap
list List arraylist ArrayList
object Object byte[] byte[]

注意：带 _ 前缀的是基本类型，不带前缀的是包装类型。

在 Mapper XML 中使用别名

<!-- 使用全限定名 -->
<select id="findById" resultType="com.example.entity.User">
  SELECT * FROM t_user WHERE id = #{id}
</select>

<!-- 使用别名（配置 typeAliases 后） -->
<select id="findById" resultType="User">
  SELECT * FROM t_user WHERE id = #{id}
</select>

Spring Boot 对应写法

mybatis:
  type-aliases-package: com.example.entity

多个包用逗号分隔：

mybatis:
  type-aliases-package: com.example.entity,com.example.dto

2.5 typeHandlers —— 类型处理器（⭐ 自定义 TypeHandler 实战）

TypeHandler 是 MyBatis 中负责 Java 类型 与 JDBC 类型 之间相互转换的桥梁。每当 MyBatis 设置 PreparedStatement 参数或从 ResultSet 读取数据时，都会通过 TypeHandler 完成类型转换。

内置 TypeHandler

MyBatis 内置了大量 TypeHandler 覆盖常见类型，例如：

TypeHandler Java 类型 JDBC 类型
StringTypeHandler String VARCHAR / CHAR
IntegerTypeHandler Integer / int INTEGER
LongTypeHandler Long / long BIGINT
DateTypeHandler Date TIMESTAMP
BooleanTypeHandler Boolean / boolean BOOLEAN / BIT
EnumTypeHandler Enum VARCHAR（存枚举名）
EnumOrdinalTypeHandler Enum INTEGER（存枚举序号）

自定义 TypeHandler 实战：逗号分隔字符串 ↔ List<String>

一个常见需求：数据库中用 VARCHAR 存储逗号分隔的标签（如 "java,mybatis,spring"），Java 中用 List<String> 操作。

第一步：编写 TypeHandler

package com.example.handler;

import org.apache.ibatis.type.BaseTypeHandler;
import org.apache.ibatis.type.JdbcType;
import org.apache.ibatis.type.MappedJdbcTypes;
import org.apache.ibatis.type.MappedTypes;

import java.sql.CallableStatement;
import java.sql.PreparedStatement;
import java.sql.ResultSet;
import java.sql.SQLException;
import java.util.Arrays;
import java.util.Collections;
import java.util.List;

@MappedTypes(List.class)
@MappedJdbcTypes(JdbcType.VARCHAR)
public class StringListTypeHandler extends BaseTypeHandler<List<String>> {

    private static final String DELIMITER = ",";

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
                                     List<String> parameter, JdbcType jdbcType)
            throws SQLException {
        ps.setString(i, String.join(DELIMITER, parameter));
    }

    @Override
    public List<String> getNullableResult(ResultSet rs, String columnName)
            throws SQLException {
        return toList(rs.getString(columnName));
    }

    @Override
    public List<String> getNullableResult(ResultSet rs, int columnIndex)
            throws SQLException {
        return toList(rs.getString(columnIndex));
    }

    @Override
    public List<String> getNullableResult(CallableStatement cs, int columnIndex)
            throws SQLException {
        return toList(cs.getString(columnIndex));
    }

    private List<String> toList(String value) {
        if (value == null || value.isEmpty()) {
            return Collections.emptyList();
        }
        return Arrays.asList(value.split(DELIMITER));
    }
}

第二步：注册 TypeHandler

方式一 —— 全局注册（mybatis-config.xml）：

<typeHandlers>
  <!-- 逐个注册 -->
  <typeHandler handler="com.example.handler.StringListTypeHandler"/>

  <!-- 或包扫描注册 -->
  <package name="com.example.handler"/>
</typeHandlers>

方式二 —— 在 Mapper XML 中局部指定（不需要全局注册）：

<!-- 写入时指定 -->
<insert id="insertArticle">
  INSERT INTO t_article (title, tags)
  VALUES (#{title}, #{tags, typeHandler=com.example.handler.StringListTypeHandler})
</insert>

<!-- 读取时在 resultMap 中指定 -->
<resultMap id="articleMap" type="Article">
  <id property="id" column="id"/>
  <result property="title" column="title"/>
  <result property="tags" column="tags"
          typeHandler="com.example.handler.StringListTypeHandler"/>
</resultMap>

第三步：实体类定义

public class Article {
    private Long id;
    private String title;
    private List<String> tags;  // 数据库中存储为 "java,mybatis,spring"
    // getter / setter 省略
}

第四步：使用效果

-- 数据库中的数据
-- id=1, title='MyBatis入门', tags='java,mybatis,spring'

Article article = mapper.findById(1L);
System.out.println(article.getTags());
// 输出: [java, mybatis, spring]

article.setTags(Arrays.asList("java", "spring-boot"));
mapper.updateArticle(article);
// 数据库中 tags 列更新为: "java,spring-boot"

自定义 TypeHandler 实战（补充）：通用枚举处理器

MyBatis 内置的 EnumTypeHandler（存名称）和 EnumOrdinalTypeHandler（存序号）都有局限性。实际项目中更常见的是用自定义的 code 值存储枚举。

定义枚举接口：

public interface CodeEnum {
    int getCode();
}

定义具体枚举：

public enum OrderStatus implements CodeEnum {
    CREATED(0, "已创建"),
    PAID(1, "已支付"),
    SHIPPED(2, "已发货"),
    COMPLETED(3, "已完成");

    private final int code;
    private final String desc;

    OrderStatus(int code, String desc) {
        this.code = code;
        this.desc = desc;
    }

    @Override
    public int getCode() { return code; }

    public String getDesc() { return desc; }

    public static OrderStatus fromCode(int code) {
        for (OrderStatus status : values()) {
            if (status.code == code) return status;
        }
        throw new IllegalArgumentException("Unknown OrderStatus code: " + code);
    }
}

通用 TypeHandler：

package com.example.handler;

import com.example.enums.CodeEnum;
import org.apache.ibatis.type.BaseTypeHandler;
import org.apache.ibatis.type.JdbcType;

import java.sql.CallableStatement;
import java.sql.PreparedStatement;
import java.sql.ResultSet;
import java.sql.SQLException;

public class CodeEnumTypeHandler<E extends Enum<E> & CodeEnum> extends BaseTypeHandler<E> {

    private final Class<E> type;
    private final E[] enums;

    public CodeEnumTypeHandler(Class<E> type) {
        this.type = type;
        this.enums = type.getEnumConstants();
        if (this.enums == null) {
            throw new IllegalArgumentException(type.getSimpleName() + " does not represent an enum type.");
        }
    }

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, E parameter, JdbcType jdbcType)
            throws SQLException {
        ps.setInt(i, parameter.getCode());
    }

    @Override
    public E getNullableResult(ResultSet rs, String columnName) throws SQLException {
        int code = rs.getInt(columnName);
        return rs.wasNull() ? null : codeOf(code);
    }

    @Override
    public E getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        int code = rs.getInt(columnIndex);
        return rs.wasNull() ? null : codeOf(code);
    }

    @Override
    public E getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        int code = cs.getInt(columnIndex);
        return cs.wasNull() ? null : codeOf(code);
    }

    private E codeOf(int code) {
        for (E e : enums) {
            if (e.getCode() == code) return e;
        }
        throw new IllegalArgumentException("Unknown enum code " + code + " for " + type.getSimpleName());
    }
}

注册方式：

<typeHandlers>
  <typeHandler handler="com.example.handler.CodeEnumTypeHandler"
               javaType="com.example.enums.OrderStatus"/>
</typeHandlers>

Spring Boot 对应写法

mybatis:
  type-handlers-package: com.example.handler

💡 提示：如果使用 MyBatis-Plus，可以直接使用内置的 MybatisEnumTypeHandler，配合 @EnumValue 注解更加简洁，详见第十一章。

2.6 plugins —— 插件注册入口（详见第八章）

<plugins> 标签用于注册 MyBatis 拦截器插件。MyBatis 允许在 SQL 执行过程中的四个关键节点进行拦截：

  • Executor：拦截执行器方法（update、query、commit、rollback 等）
  • StatementHandler：拦截 SQL 语句的构建与执行
  • ParameterHandler：拦截参数设置
  • ResultSetHandler：拦截结果集处理

配置示例

<plugins>
  <!-- 自定义 SQL 耗时统计插件 -->
  <plugin interceptor="com.example.plugin.SqlCostInterceptor">
    <property name="threshold" value="100"/>
  </plugin>

  <!-- PageHelper 分页插件 -->
  <plugin interceptor="com.github.pagehelper.PageInterceptor">
    <property name="helperDialect" value="mysql"/>
    <property name="reasonable" value="true"/>
  </plugin>
</plugins>

⚠️ 注意：插件的注册顺序会影响执行顺序。多个插件形成责任链，先注册的后执行（类似栈的 LIFO 结构）。

Spring Boot 对应写法

在 Spring Boot 中，插件通过 Bean 注册更为常见：

@Configuration
public class MyBatisConfig {

    @Bean
    public SqlCostInterceptor sqlCostInterceptor() {
        SqlCostInterceptor interceptor = new SqlCostInterceptor();
        Properties props = new Properties();
        props.setProperty("threshold", "100");
        interceptor.setProperties(props);
        return interceptor;
    }
}

PageHelper 在 Spring Boot 中只需引入 starter 即可自动注册：

<!-- pom.xml -->
<dependency>
  <groupId>com.github.pagehelper</groupId>
  <artifactId>pagehelper-spring-boot-starter</artifactId>
  <version>最新版本</version>
</dependency>

# application.yml
pagehelper:
  helper-dialect: mysql
  reasonable: true

插件的详细原理与自定义实战见第八章。

2.7 environments / dataSource / transactionManager

<environments> 用于配置数据库环境，支持定义多个环境（开发、测试、生产），但运行时只能选择其中一个。

完整配置结构

<environments default="dev">

  <!-- 开发环境 -->
  <environment id="dev">
    <transactionManager type="JDBC"/>
    <dataSource type="POOLED">
      <property name="driver" value="${db.driver}"/>
      <property name="url" value="${db.url}"/>
      <property name="username" value="${db.username}"/>
      <property name="password" value="${db.password}"/>
    </dataSource>
  </environment>

  <!-- 测试环境 -->
  <environment id="test">
    <transactionManager type="JDBC"/>
    <dataSource type="POOLED">
      <property name="driver" value="com.mysql.cj.jdbc.Driver"/>
      <property name="url" value="jdbc:mysql://test-server:3306/mydb"/>
      <property name="username" value="test_user"/>
      <property name="password" value="test_pass"/>
    </dataSource>
  </environment>

</environments>

transactionManager 事务管理器

类型 说明
JDBC 直接使用 JDBC 的事务管理（Connection.commit() / rollback()）
MANAGED 将事务交给容器管理（如 JEE 应用服务器），MyBatis 自身不做任何事务操作

实际项目：与 Spring 整合后，事务管理器由 Spring 的 DataSourceTransactionManager 接管，这里的配置不再生效。

dataSource 数据源

类型 说明
UNPOOLED 无连接池，每次请求都打开/关闭连接。仅适合测试或非常简单的应用
POOLED 内置连接池实现，支持基本的池化配置
JNDI 通过 JNDI 查找数据源，适用于 JEE 应用服务器环境

POOLED 数据源的常用属性：

属性 说明 默认值
poolMaximumActiveConnections 最大活跃连接数 10
poolMaximumIdleConnections 最大空闲连接数 5
poolMaximumCheckoutTime 连接被强制归还前的最大使用时间（毫秒） 20000
poolTimeToWait 获取连接失败时的等待时间（毫秒） 20000
poolPingEnabled 是否启用连接探活 false
poolPingQuery 探活 SQL NO PING QUERY SET

Spring Boot 对应写法

Spring Boot 不使用 MyBatis 的 <environments> 配置，而是使用 Spring 自己的数据源管理。推荐使用 HikariCP（Spring Boot 2.x+ 默认）：

spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/mydb?useSSL=false&serverTimezone=Asia/Shanghai
    username: root
    password: 123456
    type: com.zaxxer.hikari.HikariDataSource
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000

多环境切换通过 Spring Profile 实现：

# application-dev.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb

# application-prod.yml
spring:
  datasource:
    url: jdbc:mysql://prod-server:3306/mydb

# 启动时指定环境
java -jar app.jar --spring.profiles.active=prod

2.8 mappers —— 四种注册 Mapper 的方式

<mappers> 标签用于告诉 MyBatis 去哪里找 Mapper XML 文件或 Mapper 接口。有四种注册方式：

方式一：resource —— 指定 classpath 下的 XML 文件路径

<mappers>
  <mapper resource="mapper/UserMapper.xml"/>
  <mapper resource="mapper/OrderMapper.xml"/>
</mappers>
特点 说明
路径格式 classpath 相对路径，使用 / 分隔
适用场景 XML 文件与 Mapper 接口不在同一目录下
缺点 每个 Mapper 都要单独配置，文件多时繁琐

方式二：url —— 指定文件系统绝对路径或网络 URL

<mappers>
  <mapper url="file:///D:/config/mapper/UserMapper.xml"/>
</mappers>
特点 说明
路径格式 file:/// 开头的绝对路径，或 http:// 网络地址
适用场景 极少使用，仅在 Mapper XML 不在 classpath 中时才需要
缺点 路径硬编码，可移植性差

方式三：class —— 指定 Mapper 接口的全限定名

<mappers>
  <mapper class="com.example.mapper.UserMapper"/>
  <mapper class="com.example.mapper.OrderMapper"/>
</mappers>
特点 说明
路径格式 Mapper 接口的全限定类名
适用场景 使用注解开发（@Select 等），或 XML 文件与接口在同包同名
要求 如果同时使用 XML，XML 文件必须与接口同包同名（如接口 com.example.mapper.UserMapper 对应 com/example/mapper/UserMapper.xml）

方式四：package —— 包扫描（推荐）

<mappers>
  <package name="com.example.mapper"/>
</mappers>
特点 说明
路径格式 包名
适用场景 Mapper 数量多时推荐使用，一行搞定
要求 与方式三相同，XML 文件必须与接口同包同名
优势 新增 Mapper 无需修改配置

四种方式对比总结

方式 配置目标 格式要求 推荐度
resource XML 文件 classpath 相对路径 ⭐⭐⭐ 中等
url XML 文件 绝对路径 / URL ⭐ 不推荐
class 接口类 全限定类名；XML 需同包同名 ⭐⭐⭐ 中等
package 接口包 包名；XML 需同包同名 ⭐⭐⭐⭐⭐ 推荐

XML 与接口同包同名的含义

假设 Mapper 接口为 com.example.mapper.UserMapper，那么 XML 文件应放在 classpath 的 com/example/mapper/ 目录下，且文件名为 UserMapper.xml。

在 Maven 项目中，常见的两种文件组织方式：

方式 A：XML 放在 src/main/resources 的对应包路径下

src/main/java/com/example/mapper/UserMapper.java
src/main/resources/com/example/mapper/UserMapper.xml

方式 B：XML 放在 src/main/java 的同包目录下（需要配置 Maven 资源过滤）

src/main/java/com/example/mapper/UserMapper.java
src/main/java/com/example/mapper/UserMapper.xml

方式 B 需要在 pom.xml 中添加资源配置，否则 XML 不会被打包：

<build>
  <resources>
    <resource>
      <directory>src/main/java</directory>
      <includes>
        <include>**/*.xml</include>
      </includes>
    </resource>
    <resource>
      <directory>src/main/resources</directory>
    </resource>
  </resources>
</build>

Spring Boot 对应写法

Spring Boot 中有两种方式注册 Mapper：

方式一：使用 @MapperScan 注解（推荐）

@SpringBootApplication
@MapperScan("com.example.mapper")
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

方式二：在每个 Mapper 接口上添加 @Mapper 注解

@Mapper
public interface UserMapper {
    User findById(Long id);
}

XML 文件路径通过 mapper-locations 指定（不要求与接口同包同名）：

mybatis:
  mapper-locations: classpath:mapper/**/*.xml

这是 Spring Boot 中最常见的写法，XML 统一放在 src/main/resources/mapper/ 目录下：

src/main/resources/mapper/UserMapper.xml
src/main/resources/mapper/OrderMapper.xml

本章小结

配置项 核心要点 日常开发影响
子标签顺序 必须严格遵守 DTD 定义的顺序 Spring Boot 下无需关心
properties 外部化配置，支持优先级覆盖 被 Spring 配置体系完全替代
settings 控制全局行为，mapUnderscoreToCamelCase 几乎必开 mybatis.configuration.*
typeAliases 简化 XML 中的类型引用 mybatis.type-aliases-package
typeHandlers Java ↔ JDBC 类型转换桥梁 mybatis.type-handlers-package
plugins 拦截器入口（分页、审计等） 通过 Spring Bean 注册
environments 数据源与事务管理 被 Spring 数据源管理完全替代
mappers 四种方式注册，推荐包扫描 @MapperScan + mapper-locations

📌 记住：在 Spring Boot 项目中，以上大部分配置通过 application.yml 的 mybatis.* 前缀完成，无需编写 mybatis-config.xml。但理解原生配置的含义和加载机制，是排查问题和深入理解框架的基础。
