第十一章 MyBatis-Plus 增强框架 ⭐

MyBatis-Plus（简称 MP）是 MyBatis 的增强工具，在 MyBatis 的基础上只做增强不做改变，为简化开发、提高效率而生。它已成为国内 Java 项目中使用 MyBatis 的事实标准，几乎所有新项目都会直接引入 MP 而非裸用 MyBatis。本章将系统讲解 MyBatis-Plus 的核心功能。

11.1 快速上手

11.1.1 依赖引入与自动配置

MyBatis-Plus 提供了 Spring Boot Starter，引入后会自动配置 SqlSessionFactory、MapperScannerConfigurer 等组件，无需再单独引入 MyBatis 的 Starter。

Maven 依赖

<dependencies>
    <!-- MyBatis-Plus Spring Boot Starter（已包含 MyBatis + MyBatis-Spring） -->
    <dependency>
        <groupId>com.baomidou</groupId>
        <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
        <version>3.5.7</version>
    </dependency>

    <!-- 如果是 Spring Boot 2.x 项目，使用以下依赖 -->
    <!--
    <dependency>
        <groupId>com.baomidou</groupId>
        <artifactId>mybatis-plus-boot-starter</artifactId>
        <version>3.5.7</version>
    </dependency>
    -->

    <!-- 数据库驱动（以 MySQL 为例） -->
    <dependency>
        <groupId>com.mysql</groupId>
        <artifactId>mysql-connector-j</artifactId>
        <scope>runtime</scope>
    </dependency>

    <!-- 连接池（Spring Boot 默认 HikariCP，通常无需额外引入） -->

    <!-- Lombok（可选，但强烈推荐） -->
    <dependency>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
        <optional>true</optional>
    </dependency>
</dependencies>

注意：引入 mybatis-plus-spring-boot3-starter 后不要再引入 mybatis-spring-boot-starter，否则会产生版本冲突。MP 的 Starter 已经包含了 MyBatis 的所有依赖。

application.yml 配置

spring:
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/mybatis_demo?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: 123456

mybatis-plus:
  # Mapper XML 文件位置（默认值就是 classpath*:/mapper/**/*.xml）
  mapper-locations: classpath*:/mapper/**/*.xml
  # 实体类别名包扫描
  type-aliases-package: com.example.entity
  configuration:
    # 开启驼峰命名自动映射（默认已开启）
    map-underscore-to-camel-case: true
    # 日志实现（开发环境建议开启，可以看到完整 SQL）
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
  global-config:
    db-config:
      # 全局主键策略（默认 ASSIGN_ID，即雪花算法）
      id-type: assign_id
      # 全局逻辑删除字段名
      logic-delete-field: deleted
      # 逻辑删除值
      logic-delete-value: 1
      # 逻辑未删除值
      logic-not-delete-value: 0
      # 全局表名前缀
      # table-prefix: t_

启动类配置

@SpringBootApplication
@MapperScan("com.example.mapper")  // 扫描 Mapper 接口包
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

实体类与 Mapper 接口

假设数据库中有一张 user 表：

CREATE TABLE user (
    id       BIGINT       NOT NULL AUTO_INCREMENT,
    name     VARCHAR(50)  NOT NULL,
    age      INT          NULL,
    email    VARCHAR(100) NULL,
    deleted  TINYINT      NOT NULL DEFAULT 0,
    create_time DATETIME  NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME  NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer age;
    private String email;
    @TableLogic
    private Integer deleted;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
}

public interface UserMapper extends BaseMapper<User> {
    // 无需编写任何方法，继承 BaseMapper 即可获得完整的 CRUD 能力
}

到这里，一个完整的 MyBatis-Plus 项目就搭建完成了。你已经可以对 user 表执行增删改查操作，无需编写任何 XML 或 SQL 语句。

11.1.2 BaseMapper —— 内置 CRUD 方法一览

BaseMapper<T> 是 MyBatis-Plus 提供的通用 Mapper 接口，所有自定义 Mapper 只需继承它，即可获得以下内置方法：

插入

方法签名 说明
int insert(T entity) 插入一条记录，null 字段不会出现在 SQL 中（字段策略）

删除

方法签名 说明
int deleteById(Serializable id) 根据主键删除
int deleteById(T entity) 根据实体主键删除（3.5.0+）
int deleteBatchIds(Collection<?> idList) 根据主键批量删除
int deleteByMap(Map<String, Object> columnMap) 根据 columnMap 条件删除（key 为列名）
int delete(Wrapper<T> queryWrapper) 根据条件构造器删除

更新

方法签名 说明
int updateById(T entity) 根据主键更新，null 字段不更新
int update(T entity, Wrapper<T> updateWrapper) 根据条件构造器更新
int update(Wrapper<T> updateWrapper) 根据 UpdateWrapper 更新（可用 set 方法指定更新字段）

查询

方法签名 说明
T selectById(Serializable id) 根据主键查询
List<T> selectBatchIds(Collection<?> idList) 根据主键批量查询
List<T> selectByMap(Map<String, Object> columnMap) 根据 columnMap 条件查询
T selectOne(Wrapper<T> queryWrapper) 根据条件查询一条记录（结果多条会抛异常）
T selectOne(Wrapper<T> queryWrapper, boolean throwEx) 同上，throwEx=false 时多条结果取第一条
Long selectCount(Wrapper<T> queryWrapper) 根据条件查询总记录数
List<T> selectList(Wrapper<T> queryWrapper) 根据条件查询列表
List<Map<String, Object>> selectMaps(Wrapper<T> queryWrapper) 查询结果封装为 Map 列表（适合部分字段查询）
List<Object> selectObjs(Wrapper<T> queryWrapper) 查询结果只返回第一列的值
<P extends IPage<T>> P selectPage(P page, Wrapper<T> queryWrapper) 分页查询
<P extends IPage<Map<String, Object>>> P selectMapsPage(P page, Wrapper<T> queryWrapper) 分页查询，结果封装为 Map
boolean exists(Wrapper<T> queryWrapper) 判断是否存在满足条件的记录（3.5.3.2+）

使用示例

@SpringBootTest
class UserMapperTest {

    @Autowired
    private UserMapper userMapper;

    @Test
    void testInsert() {
        User user = new User();
        user.setName("张三");
        user.setAge(25);
        user.setEmail("zhangsan@example.com");
        int rows = userMapper.insert(user);
        // 插入后主键会自动回填到 user.id
        System.out.println("影响行数: " + rows);
        System.out.println("回填主键: " + user.getId());
    }

    @Test
    void testSelectById() {
        User user = userMapper.selectById(1L);
        System.out.println(user);
    }

    @Test
    void testSelectBatchIds() {
        List<User> users = userMapper.selectBatchIds(Arrays.asList(1L, 2L, 3L));
        users.forEach(System.out::println);
    }

    @Test
    void testSelectByMap() {
        Map<String, Object> columnMap = new HashMap<>();
        columnMap.put("name", "张三");
        columnMap.put("age", 25);
        List<User> users = userMapper.selectByMap(columnMap);
        users.forEach(System.out::println);
    }

    @Test
    void testUpdateById() {
        User user = new User();
        user.setId(1L);
        user.setName("张三（已更新）");
        int rows = userMapper.updateById(user);
        System.out.println("影响行数: " + rows);
    }

    @Test
    void testDeleteById() {
        int rows = userMapper.deleteById(1L);
        System.out.println("影响行数: " + rows);
    }
}

11.1.3 IService / ServiceImpl —— 服务层封装

MyBatis-Plus 还提供了通用 Service 层接口 IService<T> 及其实现类 ServiceImpl<M extends BaseMapper<T>, T>，在 BaseMapper 基础上进一步封装了批量操作和链式调用等便利方法。

定义 Service

public interface UserService extends IService<User> {
    // 可在此定义业务方法
}

@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements UserService {
    // 自动注入 UserMapper，无需手动 @Autowired
    // 可通过 this.baseMapper 或 getBaseMapper() 访问
}

保存

方法签名 说明
boolean save(T entity) 插入一条记录
boolean saveBatch(Collection<T> entityList) 批量插入（默认 1000 条一批）
boolean saveBatch(Collection<T> entityList, int batchSize) 批量插入，自定义批次大小
boolean saveOrUpdate(T entity) 主键有值则更新，否则插入
boolean saveOrUpdateBatch(Collection<T> entityList) 批量 saveOrUpdate

删除

方法签名 说明
boolean removeById(Serializable id) 根据主键删除
boolean removeBatchByIds(Collection<?> idList) 批量删除
boolean removeByMap(Map<String, Object> columnMap) 根据列值条件删除

更新

方法签名 说明
boolean updateById(T entity) 根据主键更新
boolean updateBatchById(Collection<T> entityList) 批量更新
boolean update(Wrapper<T> updateWrapper) 根据条件构造器更新
boolean update(T entity, Wrapper<T> updateWrapper) entity 提供 SET 值，wrapper 提供 WHERE

查询

方法签名 说明
T getById(Serializable id) 根据主键查询

T getOne(Wrapper<T> queryWrapper, boolean throwEx) 同上，throwEx=false 取第一条
List<T> list() 查询全部

链式调用

方法签名 说明

LambdaQueryChainWrapper<T> lambdaQuery() Lambda 链式查询
UpdateChainWrapper<T> update() 链式更新
LambdaUpdateChainWrapper<T> lambdaUpdate() Lambda 链式更新

链式调用示例

@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements UserService {

    public List<User> findActiveAdults() {
        // 链式 Lambda 查询 —— 极其简洁
        return lambdaQuery()
                .ge(User::getAge, 18)
                .eq(User::getDeleted, 0)
                .orderByDesc(User::getCreateTime)
                .list();
    }

    public boolean deactivateUser(Long userId) {
        // 链式 Lambda 更新
        return lambdaUpdate()
                .eq(User::getId, userId)
                .set(User::getDeleted, 1)
                .update();
    }
}

IService vs BaseMapper 如何选择？

  • 简单 CRUD → 直接在 Controller 或其他地方注入 Mapper 调用 BaseMapper 方法即可
  • 需要批量操作、事务控制、复杂业务逻辑 → 使用 IService / ServiceImpl
  • 实际项目中推荐统一使用 Service 层，保持分层清晰

11.2 注解体系

MyBatis-Plus 通过一套简洁的注解，实现实体类与数据库表之间的映射。这比原生 MyBatis 的 resultMap 配置要简单得多。

11.2.1 @TableName —— 表名映射

当实体类名与数据库表名不一致时，使用 @TableName 指定表名。

@Data
@TableName("sys_user")  // 映射到 sys_user 表
public class User {
    private Long id;
    private String name;
    private Integer age;
}

常用属性：

属性 类型 默认值 说明
value String "" 表名
schema String "" schema
resultMap String "" XML 中 resultMap 的 id（复杂映射时使用）
autoResultMap boolean false 是否自动构建 resultMap（含 typeHandler 时需开启）

如果表名有统一前缀（如 t_user、t_order），可以在 application.yml 中设置 mybatis-plus.global-config.db-config.table-prefix: t_，这样就不需要每个实体都写 @TableName 了。

11.2.2 @TableId —— 主键策略

@TableId 用于标识主键字段，并指定主键生成策略。

@Data
@TableName("user")
public class User {
    @TableId(value = "id", type = IdType.AUTO)
    private Long id;
    // ...
}

主键策略一览：

策略 枚举值 说明
AUTO IdType.AUTO 数据库自增，MP 不参与主键生成，依赖数据库 AUTO_INCREMENT
ASSIGN_ID IdType.ASSIGN_ID 默认策略，MP 自动分配 ID（雪花算法），适用于 Long / String
ASSIGN_UUID IdType.ASSIGN_UUID 分配 UUID（去掉横线的 32 位字符串），适用于 String
INPUT IdType.INPUT 手动输入，MP 不参与，需要在 insert 前自行设置主键值
NONE IdType.NONE 无策略，等同于 INPUT 
// 数据库自增
@TableId(type = IdType.AUTO)
private Long id;

// 雪花算法（默认，推荐用于分布式系统）
@TableId(type = IdType.ASSIGN_ID)
private Long id;

// UUID
@TableId(type = IdType.ASSIGN_UUID)
private String id;

// 手动输入
@TableId(type = IdType.INPUT)
private Long id;

生产建议：分布式系统推荐使用 ASSIGN_ID（雪花算法），生成的 ID 是趋势递增的 Long 型，既保证全局唯一，又对数据库索引友好。单机小项目用 AUTO 即可。

11.2.3 @TableField —— 字段映射

@TableField 用于标注非主键字段，处理字段名不一致、字段排除、自动填充等场景。

基础映射 —— 字段名不一致

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;

    @TableField("user_name")  // 数据库字段名为 user_name，Java 属性名为 name
    private String name;

    private Integer age;
}

exist —— 排除非数据库字段

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;

    @TableField(exist = false)  // 该字段在数据库中不存在，查询和插入时忽略
    private String remark;
}

fill —— 自动填充策略（配合 MetaObjectHandler，见 11.6）

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;

    @TableField(fill = FieldFill.INSERT)          // 插入时自动填充
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)   // 插入和更新时自动填充
    private LocalDateTime updateTime;
}

FieldFill 枚举值：

值 说明
DEFAULT 不自动填充（默认）
INSERT 插入时填充
UPDATE 更新时填充
INSERT_UPDATE 插入和更新时都填充

typeHandler —— 指定类型处理器

@Data
@TableName(value = "user", autoResultMap = true)  // 必须开启 autoResultMap
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;

    // 数据库 JSON 字段 ↔ Java 对象
    @TableField(typeHandler = JacksonTypeHandler.class)
    private Map<String, Object> extraInfo;
}

注意：使用 typeHandler 时，必须在 @TableName 中设置 autoResultMap = true，否则查询时 typeHandler 不生效。

condition —— 控制默认条件生成方式

@TableField(condition = SqlCondition.LIKE)  // 当此字段作为查询条件时，默认使用 LIKE
private String name;

11.2.4 @TableLogic —— 逻辑删除

逻辑删除不会真正删除数据库记录，而是通过标记字段（如 deleted）来表示已删除。MP 会自动在所有查询中追加 WHERE deleted = 0 条件，在删除操作中将 DELETE 改为 UPDATE。

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;

    @TableLogic  // 标记逻辑删除字段
    private Integer deleted;  // 0-未删除 1-已删除
}

配置全局逻辑删除（推荐在 application.yml 中统一配置，这样不需要每个实体都加 @TableLogic）：

mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: deleted
      logic-delete-value: 1
      logic-not-delete-value: 0

逻辑删除的效果：

// 调用 deleteById
userMapper.deleteById(1L);
// 实际执行的 SQL: UPDATE user SET deleted = 1 WHERE id = 1 AND deleted = 0

// 调用 selectList
userMapper.selectList(null);
// 实际执行的 SQL: SELECT * FROM user WHERE deleted = 0

// 调用 updateById
userMapper.updateById(user);
// 实际执行的 SQL: UPDATE user SET name = '...' WHERE id = 1 AND deleted = 0

注意：逻辑删除只对 MP 自动生成的 SQL 生效。如果你在 XML 中手写 SQL，需要自行添加 WHERE deleted = 0 条件。

11.2.5 @Version —— 乐观锁

乐观锁通过版本号机制防止并发更新冲突：更新时检查版本号是否一致，不一致则更新失败。

第 1 步：配置乐观锁插件

@Configuration
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        interceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());
        return interceptor;
    }
}

第 2 步：实体类添加 @Version

@Data
@TableName("product")
public class Product {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer price;

    @Version  // 乐观锁版本号字段
    private Integer version;
}

第 3 步：使用

// 先查询获取当前版本号
Product product = productMapper.selectById(1L);
// product.version = 1

product.setPrice(200);
int rows = productMapper.updateById(product);
// 实际执行的 SQL:
// UPDATE product SET name='...', price=200, version=2
// WHERE id=1 AND version=1
//                 ^^^^^^^^^^^ 乐观锁条件

if (rows == 0) {
    // 更新失败，说明数据已被其他线程修改，需要重试或提示用户
    System.out.println("更新失败，数据已被修改");
}

支持的类型：int、Integer、long、Long、Date、Timestamp、LocalDateTime。整数类型每次更新自动 +1，时间类型自动更新为当前时间。

11.2.6 @EnumValue —— 枚举映射

在 MP 中，枚举类型可以直接映射到数据库字段，通过 @EnumValue 标注枚举中与数据库存储值对应的属性。

定义枚举

@Getter
public enum GenderEnum {
    MALE(1, "男"),
    FEMALE(2, "女");

    @EnumValue  // 标记数据库存储的值
    private final int code;

    private final String desc;

    GenderEnum(int code, String desc) {
        this.code = code;
        this.desc = desc;
    }
}

实体类中使用

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private GenderEnum gender;  // 直接使用枚举类型
}

配置枚举包扫描

mybatis-plus:
  type-enums-package: com.example.enums

使用时 MP 会自动将 GenderEnum.MALE 存为 1，查询时将 1 映射回 GenderEnum.MALE。

User user = new User();
user.setName("张三");
user.setGender(GenderEnum.MALE);  // 存储值为 1
userMapper.insert(user);
// SQL: INSERT INTO user (name, gender) VALUES ('张三', 1)

User result = userMapper.selectById(1L);
System.out.println(result.getGender());  // 输出: MALE
System.out.println(result.getGender().getDesc());  // 输出: 男

替代方案：也可以让枚举实现 IEnum<T> 接口，效果相同。选择 @EnumValue 注解方式更简洁，推荐优先使用。

11.3 条件构造器（⭐ 核心）

条件构造器是 MyBatis-Plus 最强大的功能之一，它让你用 Java 代码构建 SQL 条件，告别手写动态 SQL 的烦恼。这是 MP 日常使用中出现频率最高的 API，必须熟练掌握。

11.3.1 QueryWrapper —— 基础条件构造

QueryWrapper 是最基础的条件构造器，以字符串形式指定列名。

// 查询年龄大于 18 且邮箱不为空的用户
QueryWrapper<User> wrapper = new QueryWrapper<>();
wrapper.gt("age", 18)
       .isNotNull("email");
List<User> users = userMapper.selectList(wrapper);
// SQL: SELECT * FROM user WHERE age > 18 AND email IS NOT NULL

// 查询姓张的用户，按年龄降序
QueryWrapper<User> wrapper2 = new QueryWrapper<>();
wrapper2.likeRight("name", "张")
        .orderByDesc("age");
List<User> users2 = userMapper.selectList(wrapper2);
// SQL: SELECT * FROM user WHERE name LIKE '张%' ORDER BY age DESC

// 只查询部分字段
QueryWrapper<User> wrapper3 = new QueryWrapper<>();
wrapper3.select("id", "name", "age")  // 指定查询字段
        .lt("age", 30);
List<User> users3 = userMapper.selectList(wrapper3);
// SQL: SELECT id, name, age FROM user WHERE age < 30

// 条件判断：根据参数动态拼接条件
String name = "张三";
Integer minAge = null;
QueryWrapper<User> wrapper4 = new QueryWrapper<>();
wrapper4.eq(StringUtils.isNotBlank(name), "name", name)   // name 非空才拼接
        .ge(minAge != null, "age", minAge);                // minAge 非 null 才拼接
List<User> users4 = userMapper.selectList(wrapper4);
// 如果 minAge 为 null，SQL: SELECT * FROM user WHERE name = '张三'

condition 参数：几乎所有条件方法的第一个参数都支持 boolean condition，为 true 时才拼接该条件。这是 MP 替代 <if> 动态 SQL 的方式，非常实用。

11.3.2 LambdaQueryWrapper —— 类型安全的条件构造

强烈推荐优先使用 LambdaQueryWrapper，原因如下：

  1. 编译期检查：用方法引用（如 User::getName）代替字符串列名，拼写错误在编译时就能发现
  2. 重构友好：修改字段名时 IDE 会自动同步更新所有引用
  3. IDE 提示：写条件时有代码补全，不需要记列名
// 基础查询
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.gt(User::getAge, 18)
       .isNotNull(User::getEmail);
List<User> users = userMapper.selectList(wrapper);
// SQL: SELECT * FROM user WHERE age > 18 AND email IS NOT NULL

// 便捷写法 —— 静态构造
LambdaQueryWrapper<User> wrapper2 = Wrappers.<User>lambdaQuery()
        .eq(User::getName, "张三")
        .between(User::getAge, 18, 30)
        .orderByAsc(User::getAge);
List<User> users2 = userMapper.selectList(wrapper2);
// SQL: SELECT * FROM user WHERE name = '张三' AND age BETWEEN 18 AND 30 ORDER BY age ASC

// 动态条件
String name = getUserInput();
Integer age = getAgeInput();
LambdaQueryWrapper<User> wrapper3 = new LambdaQueryWrapper<>();
wrapper3.like(StringUtils.isNotBlank(name), User::getName, name)
        .eq(age != null, User::getAge, age)
        .orderByDesc(User::getCreateTime);
List<User> users3 = userMapper.selectList(wrapper3);

// 查询指定字段
LambdaQueryWrapper<User> wrapper4 = new LambdaQueryWrapper<>();
wrapper4.select(User::getId, User::getName, User::getAge)
        .lt(User::getAge, 30);
List<User> users4 = userMapper.selectList(wrapper4);
// SQL: SELECT id, name, age FROM user WHERE age < 30

// 查询数量
Long count = userMapper.selectCount(
    new LambdaQueryWrapper<User>().eq(User::getAge, 25)
);

// 查询单条记录
User user = userMapper.selectOne(
    new LambdaQueryWrapper<User>()
        .eq(User::getName, "张三")
        .eq(User::getAge, 25)
);

QueryWrapper vs LambdaQueryWrapper 对比

// QueryWrapper —— 字符串列名，容易拼错，重构不友好
wrapper.eq("user_name", "张三");

// LambdaQueryWrapper —— 方法引用，编译期安全，IDE 可提示
wrapper.eq(User::getName, "张三");

结论：除非需要使用 last()、apply() 等少数仅字符串方式支持的方法，否则一律使用 LambdaQueryWrapper。

11.3.3 UpdateWrapper / LambdaUpdateWrapper

UpdateWrapper 和 LambdaUpdateWrapper 用于构建更新操作的条件和 SET 子句。

// 将年龄大于 30 的用户的邮箱置空
LambdaUpdateWrapper<User> wrapper = new LambdaUpdateWrapper<>();
wrapper.gt(User::getAge, 30)
       .set(User::getEmail, null);  // SET email = null
userMapper.update(wrapper);
// SQL: UPDATE user SET email = null WHERE age > 30

// 使用 setSql 拼接原生 SQL 片段
LambdaUpdateWrapper<User> wrapper2 = new LambdaUpdateWrapper<>();
wrapper2.eq(User::getId, 1L)
        .setSql("age = age + 1");  // 原生 SQL 表达式
userMapper.update(wrapper2);
// SQL: UPDATE user SET age = age + 1 WHERE id = 1

// 同时提供实体和条件：实体中非 null 字段作为 SET 值，wrapper 提供 WHERE 条件
User updateEntity = new User();
updateEntity.setName("新名字");
LambdaUpdateWrapper<User> wrapper3 = new LambdaUpdateWrapper<>();
wrapper3.eq(User::getId, 1L);
userMapper.update(updateEntity, wrapper3);
// SQL: UPDATE user SET name = '新名字' WHERE id = 1

// LambdaUpdateWrapper 综合示例 —— 批量更新状态
LambdaUpdateWrapper<User> wrapper4 = new LambdaUpdateWrapper<>();
wrapper4.in(User::getId, Arrays.asList(1L, 2L, 3L))
        .set(User::getEmail, "updated@example.com")
        .set(User::getAge, 30);
userMapper.update(wrapper4);
// SQL: UPDATE user SET email = 'updated@example.com', age = 30 WHERE id IN (1, 2, 3)

何时用 updateById，何时用 UpdateWrapper？

  • updateById：根据主键更新，简单直接，适合已知主键的单条更新
  • UpdateWrapper：支持复杂条件更新、批量更新、字段自增等场景

11.3.5 条件构造器嵌套与 OR 条件

默认情况下，所有条件之间是 AND 关系。要使用 OR、嵌套括号等复杂条件，需要使用以下方法：

or() —— 切换为 OR 连接

// 查询名字为张三 OR 年龄大于 30
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getName, "张三")
       .or()
       .gt(User::getAge, 30);
// SQL: WHERE name = '张三' OR age > 30

or(Consumer) —— OR 嵌套（带括号）

// 查询邮箱不为空 AND (名字为张三 OR 年龄大于30)
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.isNotNull(User::getEmail)
       .or(w -> w.eq(User::getName, "张三")
                 .gt(User::getAge, 30));
// SQL: WHERE email IS NOT NULL OR (name = '张三' AND age > 30)
// ⚠️ 注意：or(Consumer) 是将整个 Consumer 内部的条件用 OR 连接到外部

and(Consumer) —— AND 嵌套（带括号）

// 查询名字为张三 AND (年龄小于 20 OR 年龄大于 40)
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getName, "张三")
       .and(w -> w.lt(User::getAge, 20)
                  .or()
                  .gt(User::getAge, 40));
// SQL: WHERE name = '张三' AND (age < 20 OR age > 40)

nested(Consumer) —— 纯嵌套（不带 AND/OR 前缀）

// nested() 只生成括号，不自动加 AND/OR
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.nested(w -> w.eq(User::getName, "张三")
                     .or()
                     .eq(User::getName, "李四"))
       .ge(User::getAge, 18);
// SQL: WHERE (name = '张三' OR name = '李四') AND age >= 18

综合实战示例

// 场景：后台用户搜索接口
// 条件：状态为启用 AND (用户名模糊匹配 OR 邮箱模糊匹配) AND 注册时间在指定范围内
public List<User> searchUsers(String keyword, LocalDateTime startTime, LocalDateTime endTime) {
    return userMapper.selectList(
        new LambdaQueryWrapper<User>()
            .eq(User::getDeleted, 0)
            .and(StringUtils.isNotBlank(keyword), w -> w
                .like(User::getName, keyword)
                .or()
                .like(User::getEmail, keyword)
            )
            .ge(startTime != null, User::getCreateTime, startTime)
            .le(endTime != null, User::getCreateTime, endTime)
            .orderByDesc(User::getCreateTime)
    );
    // 假设 keyword="张", startTime=2024-01-01
    // SQL: WHERE deleted = 0
    //      AND (name LIKE '%张%' OR email LIKE '%张%')
    //      AND create_time >= '2024-01-01'
    //      ORDER BY create_time DESC
}

// 场景：复杂多条件嵌套
// 查询 VIP 用户中（年龄在 25-35 之间 且 有邮箱）或（年龄大于 50 的）
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getVip, true)
       .and(w -> w
           .nested(n -> n
               .between(User::getAge, 25, 35)
               .isNotNull(User::getEmail)
           )
           .or()
           .gt(User::getAge, 50)
       );
// SQL: WHERE vip = true AND ((age BETWEEN 25 AND 35 AND email IS NOT NULL) OR age > 50)

11.4 分页插件

MyBatis-Plus 内置了分页插件，配置简单且功能强大，是替代 PageHelper 的首选方案。

11.4.1 MybatisPlusInterceptor 配置

@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 分页插件（必须指定数据库类型）
        PaginationInnerInterceptor paginationInterceptor = new PaginationInnerInterceptor(DbType.MYSQL);
        paginationInterceptor.setMaxLimit(500L); // 单页最大条数限制，防止恶意大分页
        interceptor.addInnerInterceptor(paginationInterceptor);

        // 可以继续添加其他插件，如乐观锁
        // interceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());

        return interceptor;
    }
}

注意：MybatisPlusInterceptor 中各插件的添加顺序有要求。建议按照官方文档推荐的顺序：多租户 → 动态表名 → 分页 → 乐观锁 → SQL 性能规范 → 防全表更新删除。

11.4.2 Page 对象与分页查询

// 基础分页查询
@Test
void testPage() {
    // 参数：当前页码（从 1 开始）、每页大小
    Page<User> page = new Page<>(1, 10);
    userMapper.selectPage(page, null);  // null 表示无条件

    System.out.println("总记录数: " + page.getTotal());
    System.out.println("总页数: " + page.getPages());
    System.out.println("当前页码: " + page.getCurrent());
    System.out.println("每页大小: " + page.getSize());
    System.out.println("是否有上一页: " + page.hasPrevious());
    System.out.println("是否有下一页: " + page.hasNext());
    System.out.println("当前页数据: " + page.getRecords());
}

// 带条件的分页查询
@Test
void testPageWithCondition() {
    Page<User> page = new Page<>(2, 5);  // 第 2 页，每页 5 条
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    wrapper.ge(User::getAge, 18)
           .orderByDesc(User::getCreateTime);

    userMapper.selectPage(page, wrapper);
    // SQL: SELECT * FROM user WHERE age >= 18 ORDER BY create_time DESC LIMIT 5 OFFSET 5
    // 同时会自动执行 COUNT 查询: SELECT COUNT(*) FROM user WHERE age >= 18

    page.getRecords().forEach(System.out::println);
}

// 不查询 COUNT（适用于不需要总数的场景，如滚动加载）
@Test
void testPageWithoutCount() {
    Page<User> page = new Page<>(1, 10, false);  // 第三个参数 false 表示不查 COUNT
    userMapper.selectPage(page, null);
    // 只会执行 LIMIT 查询，不会执行 COUNT 查询
}

11.4.3 自定义 SQL 分页

有些复杂查询需要手写 SQL，同样可以使用 MP 的分页功能。

Mapper 接口

public interface UserMapper extends BaseMapper<User> {

    // 方式一：Page 作为第一个参数，MP 会自动拦截并分页
    @Select("SELECT u.*, d.name AS dept_name FROM user u LEFT JOIN department d ON u.dept_id = d.id WHERE u.age > #{age}")
    Page<UserVO> selectUserWithDept(Page<UserVO> page, @Param("age") Integer age);

    // 方式二：使用 XML
    Page<UserVO> selectPageByCondition(Page<UserVO> page, @Param("query") UserQuery query);
}

Mapper XML

<!-- UserMapper.xml -->
<select id="selectPageByCondition" resultType="com.example.vo.UserVO">
    SELECT u.id, u.name, u.age, u.email, d.name AS deptName
    FROM user u
    LEFT JOIN department d ON u.dept_id = d.id
    <where>
        <if test="query.name != null and query.name != ''">
            AND u.name LIKE CONCAT('%', #{query.name}, '%')
        </if>
        <if test="query.minAge != null">
            AND u.age >= #{query.minAge}
        </if>
        <if test="query.deptId != null">
            AND u.dept_id = #{query.deptId}
        </if>
    </where>
    ORDER BY u.create_time DESC
</select>

使用方式

@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements UserService {

    public Page<UserVO> getUserPage(int pageNum, int pageSize, UserQuery query) {
        Page<UserVO> page = new Page<>(pageNum, pageSize);
        return baseMapper.selectPageByCondition(page, query);
        // MP 会自动在 SQL 后追加 LIMIT 和执行 COUNT 查询
    }
}

关键点：自定义分页的核心是将 Page 对象作为 Mapper 方法的第一个参数，MP 分页插件会自动识别并进行分页处理。

11.6 自动填充 —— MetaObjectHandler

在实际项目中，create_time、update_time、create_by 等字段在每次插入或更新时都需要自动填充，手动设置既繁琐又容易遗漏。MP 提供了 MetaObjectHandler 来统一处理。

完整实现

第 1 步：实体类标注填充策略

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer age;

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;

    @TableField(fill = FieldFill.INSERT)
    private String createBy;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private String updateBy;
}

第 2 步：实现 MetaObjectHandler

@Component
public class MyMetaObjectHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        // 插入时填充 createTime 和 updateTime
        this.strictInsertFill(metaObject, "createTime", LocalDateTime::now, LocalDateTime.class);
        this.strictInsertFill(metaObject, "updateTime", LocalDateTime::now, LocalDateTime.class);
        // 填充创建人（从安全上下文获取当前用户）
        this.strictInsertFill(metaObject, "createBy", this::getCurrentUsername, String.class);
        this.strictInsertFill(metaObject, "updateBy", this::getCurrentUsername, String.class);
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        // 更新时填充 updateTime
        this.strictUpdateFill(metaObject, "updateTime", LocalDateTime::now, LocalDateTime.class);
        this.strictUpdateFill(metaObject, "updateBy", this::getCurrentUsername, String.class);
    }

    private String getCurrentUsername() {
        // 实际项目中从 SecurityContext / ThreadLocal 获取当前登录用户
        // 示例：SecurityContextHolder.getContext().getAuthentication().getName()
        return "system";
    }
}

strictInsertFill vs fillStrategy：

  • strictInsertFill：严格模式，只有当实体的该字段值为 null 时才会填充（不覆盖手动设置的值）
  • setFieldValByName：直接强制设置值（会覆盖已有值，已废弃）

推荐使用 strictInsertFill / strictUpdateFill。

使用效果

User user = new User();
user.setName("张三");
user.setAge(25);
// 无需手动设置 createTime、updateTime、createBy、updateBy
userMapper.insert(user);
// INSERT INTO user (name, age, create_time, update_time, create_by, update_by)
// VALUES ('张三', 25, '2024-12-01 10:30:00', '2024-12-01 10:30:00', 'system', 'system')

11.7 多租户插件 💡

多租户（Multi-Tenancy）是 SaaS 系统的核心需求，MP 提供了开箱即用的多租户插件，基于共享数据库、共享 Schema、通过租户 ID 隔离数据的模式。

基本原理

插件会自动在所有 SQL 的 WHERE 条件中追加 tenant_id = ?，在 INSERT 语句中自动插入 tenant_id 字段。

配置示例

@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 多租户插件（必须在分页插件之前添加）
        interceptor.addInnerInterceptor(new TenantLineInnerInterceptor(new TenantLineHandler() {
            @Override
            public Expression getTenantId() {
                // 从当前上下文获取租户 ID（如 ThreadLocal、请求头等）
                Long tenantId = TenantContext.getCurrentTenantId();
                return new LongValue(tenantId);
            }

            @Override
            public String getTenantIdColumn() {
                return "tenant_id";  // 租户 ID 字段名
            }

            @Override
            public boolean ignoreTable(String tableName) {
                // 某些公共表不需要租户隔离
                return "sys_config".equals(tableName) || "sys_dict".equals(tableName);
            }
        }));

        // 分页插件
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));

        return interceptor;
    }
}

配置后的效果：

userMapper.selectList(null);
// 原始 SQL: SELECT * FROM user
// 实际执行: SELECT * FROM user WHERE tenant_id = 1

userMapper.insert(user);
// 原始 SQL: INSERT INTO user (name, age) VALUES ('张三', 25)
// 实际执行: INSERT INTO user (name, age, tenant_id) VALUES ('张三', 25, 1)

多租户插件对业务代码完全透明，无需在每个 SQL 中手写租户条件。如果需要某个方法绕过租户过滤（如超级管理员查看所有数据），可以使用 @InterceptorIgnore(tenantLine = "true") 注解。

11.8 数据权限插件 💡

数据权限是指不同角色/用户只能看到属于自己权限范围内的数据，例如：部门经理只能看到本部门的数据、区域经理只能看到本区域的数据。

MP 提供了 DataPermissionInterceptor，允许你通过自定义处理器在 SQL 中动态追加数据权限条件。

配置示例

@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        // 数据权限插件
        DataPermissionInterceptor dataPermissionInterceptor = new DataPermissionInterceptor();
        dataPermissionInterceptor.setDataPermissionHandler(new MyDataPermissionHandler());
        interceptor.addInnerInterceptor(dataPermissionInterceptor);

        // 分页插件
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}

public class MyDataPermissionHandler implements DataPermissionHandler {
    @Override
    public Expression getSqlSegment(Expression where, String mappedStatementId) {
        // 根据当前用户角色构造数据权限条件
        // 例如：普通用户只能查自己的数据，部门经理能查本部门数据
        // 具体实现取决于业务需求
        // 返回 null 表示不追加条件
        return where;
    }
}

数据权限的实现相对复杂，通常需要结合具体业务场景和权限模型来设计。在实际项目中，往往需要配合 @DataPermission 自定义注解、AOP 等机制，实现细粒度的数据权限控制。

11.9 MyBatis-Plus 与原生 MyBatis XML 混合使用

MyBatis-Plus 的设计理念是只做增强不做改变，因此你完全可以在 MP 项目中继续使用原生 MyBatis 的 XML 映射文件，两者完美共存。

使用场景

  • 复杂多表联查（多个 JOIN + 子查询 + 聚合函数）
  • 需要精确控制 SQL 的性能优化场景
  • 已有大量 MyBatis XML 的遗留项目迁移到 MP
  • 需要使用 <resultMap> 处理复杂对象映射
  • 统计报表类的特殊查询

配置

确保 application.yml 中配置了 XML 文件扫描路径：

mybatis-plus:
  mapper-locations: classpath*:/mapper/**/*.xml

示例

Mapper 接口

public interface UserMapper extends BaseMapper<User> {

    // ===== MP 内置方法（继承自 BaseMapper） =====
    // selectById、insert、updateById 等 —— 无需编写，直接使用

    // ===== 自定义方法（使用 XML 实现） =====

    /**
     * 多表联查 —— 查询用户及其部门信息
     */
    List<UserDetailVO> selectUserDetail(@Param("query") UserQuery query);

    /**
     * 复杂统计 —— 按部门统计用户数量
     */
    List<DeptUserCountVO> countByDept();

    /**
     * 自定义分页 —— MP 分页 + 自定义 SQL
     */
    Page<UserDetailVO> selectUserDetailPage(Page<UserDetailVO> page, @Param("query") UserQuery query);
}

Mapper XML

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.UserMapper">

    <!-- 复杂多表联查 -->
    <select id="selectUserDetail" resultType="com.example.vo.UserDetailVO">
        SELECT
            u.id,
            u.name,
            u.age,
            u.email,
            d.name AS deptName,
            r.name AS roleName
        FROM user u
        LEFT JOIN department d ON u.dept_id = d.id
        LEFT JOIN user_role ur ON u.id = ur.user_id
        LEFT JOIN role r ON ur.role_id = r.id
        <where>
            u.deleted = 0
            <if test="query.name != null and query.name != ''">
                AND u.name LIKE CONCAT('%', #{query.name}, '%')
            </if>
            <if test="query.deptId != null">
                AND u.dept_id = #{query.deptId}
            </if>
            <if test="query.roleName != null and query.roleName != ''">
                AND r.name = #{query.roleName}
            </if>
        </where>
        ORDER BY u.create_time DESC
    </select>

    <!-- 统计查询 -->
    <select id="countByDept" resultType="com.example.vo.DeptUserCountVO">
        SELECT
            d.id AS deptId,
            d.name AS deptName,
            COUNT(u.id) AS userCount
        FROM department d
        LEFT JOIN user u ON d.id = u.dept_id AND u.deleted = 0
        GROUP BY d.id, d.name
        ORDER BY userCount DESC
    </select>

    <!-- 自定义分页（Page 作为第一个参数，MP 自动拦截） -->
    <select id="selectUserDetailPage" resultType="com.example.vo.UserDetailVO">
        SELECT u.id, u.name, u.age, d.name AS deptName
        FROM user u
        LEFT JOIN department d ON u.dept_id = d.id
        <where>
            u.deleted = 0
            <if test="query.name != null and query.name != ''">
                AND u.name LIKE CONCAT('%', #{query.name}, '%')
            </if>
        </where>
        ORDER BY u.create_time DESC
    </select>

</mapper>

Service 层混合使用

@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements UserService {

    // 使用 MP 内置方法
    public User getUser(Long id) {
        return getById(id);  // IService 方法
    }

    // 使用 MP 条件构造器
    public List<User> findByAge(Integer minAge) {
        return lambdaQuery()
                .ge(User::getAge, minAge)
                .list();
    }

    // 使用自定义 XML 方法
    public List<UserDetailVO> getUserDetail(UserQuery query) {
        return baseMapper.selectUserDetail(query);
    }

    // MP 分页 + 自定义 XML
    public Page<UserDetailVO> getUserDetailPage(int pageNum, int pageSize, UserQuery query) {
        return baseMapper.selectUserDetailPage(new Page<>(pageNum, pageSize), query);
    }
}

混合使用的最佳实践

场景 推荐方式
单表 CRUD MP 内置方法（BaseMapper / IService）
单表条件查询 LambdaQueryWrapper / LambdaUpdateWrapper
简单两表 JOIN 可用 MP 的 selectMaps 或自定义 XML
复杂多表联查 自定义 Mapper XML
统计报表类查询 自定义 Mapper XML
批量操作 MP 的 saveBatch / updateBatchById
需精确控制 SQL 的性能场景 自定义 Mapper XML

总结：MP 和原生 MyBatis XML 是互补关系，不是替代关系。简单操作用 MP 省时省力，复杂查询用 XML 灵活可控。在一个 Mapper 接口中，可以同时拥有 MP 内置方法和自定义 XML 方法，两者互不干扰。
