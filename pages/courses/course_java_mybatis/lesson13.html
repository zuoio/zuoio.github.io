13.1 Mapper XML 编写规范

良好的编写规范能让团队协作更顺畅，也能减少低级错误。以下规范在多个大型项目中验证过，推荐直接采纳。

13.1.2 SQL 编写建议

1. 禁止 SELECT *

<!-- ❌ 不推荐 -->
<select id="selectById" resultMap="BaseResultMap">
    SELECT * FROM user WHERE id = #{id}
</select>

<!-- ✅ 推荐：明确列出所需字段 -->
<sql id="Base_Column_List">
    id, username, email, phone, status, create_time, update_time
</sql>

<select id="selectById" resultMap="BaseResultMap">
    SELECT <include refid="Base_Column_List"/> FROM user WHERE id = #{id}
</select>

SELECT * 的问题：

  • 性能浪费：查询了不需要的列，尤其是 TEXT / BLOB 类型字段
  • 覆盖索引失效：索引覆盖扫描（Index Only Scan）在 SELECT * 下无法生效
  • 字段变更风险：新增字段后可能导致映射异常或数据泄露

2. 合理使用索引

<!-- ❌ 索引失效：对列使用函数 -->
<select id="selectByDate">
    SELECT id, username FROM user
    WHERE DATE_FORMAT(create_time, '%Y-%m-%d') = #{date}
</select>

<!-- ✅ 改为范围查询，索引可命中 -->
<select id="selectByDate">
    SELECT id, username FROM user
    WHERE create_time >= #{startTime} AND create_time &lt; #{endTime}
</select>

<!-- ❌ 索引失效：隐式类型转换（phone 是 VARCHAR，传入数字） -->
<select id="selectByPhone">
    SELECT id, username FROM user WHERE phone = #{phone, jdbcType=INTEGER}
</select>

<!-- ✅ 确保类型一致 -->
<select id="selectByPhone">
    SELECT id, username FROM user WHERE phone = #{phone, jdbcType=VARCHAR}
</select>

3. 其他 SQL 编写建议

建议 说明
避免在 WHERE 子句中使用 OR 可能导致索引失效，改用 UNION ALL 或 IN
LIKE 模糊查询避免前缀 % LIKE '%keyword' 无法利用索引
大表查询必须带 LIMIT 防止全表扫描拖垮数据库
INSERT 语句显式指定列名 避免表结构变更导致插入错位
UPDATE 语句使用动态 <set> 只更新变更字段，避免覆盖其他字段
避免在循环中执行单条 SQL 改为批量操作（见 13.2）

13.2 批量操作优化

批量操作是 MyBatis 性能优化中最常见的场景。方案选择不当，轻则性能差 10 倍，重则直接 OOM 或超过数据库限制。

13.2.1 foreach 批量插入的上限与分批策略

MySQL 的限制

限制项 默认值 配置参数
单条 SQL 最大长度 4MB（4,194,304 字节） max_allowed_packet
PreparedStatement 参数占位符上限 65535 个 ? MySQL 协议限制，不可配置
单条 INSERT 的 VALUES 行数 无硬限制，受上面两项约束 —

如何估算单批数量上限：

  • 假设每行有 10 个字段，每批插入 N 行，则参数占位符数为 10 × N
  • 不超过 65535 → N ≤ 6553
  • 同时需要 SQL 文本长度不超过 max_allowed_packet
  • 经验值：每批 500~2000 条比较安全，具体视字段数和字段内容长度而定

foreach 批量插入示例

<insert id="insertBatch">
    INSERT INTO user (username, email, phone, status)
    VALUES
    <foreach collection="list" item="item" separator=",">
        (#{item.username}, #{item.email}, #{item.phone}, #{item.status})
    </foreach>
</insert>

分批策略代码

public class BatchUtils {

    private static final int BATCH_SIZE = 1000;

    /**
     * 通用分批执行，适用于 foreach 拼 SQL 的批量插入
     */
    public static <T> void batchExecute(List<T> dataList, Consumer<List<T>> consumer) {
        if (dataList == null || dataList.isEmpty()) {
            return;
        }
        List<List<T>> partitions = partition(dataList, BATCH_SIZE);
        for (List<T> batch : partitions) {
            consumer.accept(batch);
        }
    }

    private static <T> List<List<T>> partition(List<T> list, int size) {
        List<List<T>> partitions = new ArrayList<>();
        for (int i = 0; i < list.size(); i += size) {
            partitions.add(list.subList(i, Math.min(i + size, list.size())));
        }
        return partitions;
    }
}

调用示例：

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    @Transactional
    public void importUsers(List<User> users) {
        BatchUtils.batchExecute(users, batch -> userMapper.insertBatch(batch));
    }
}

Google Guava 用户可以直接使用 Lists.partition(list, batchSize) 代替手写分片。

13.2.2 BATCH 模式的 Executor

foreach 方式是将多行 VALUES 拼成一条大 SQL。而 BATCH 模式是另一种思路：每条 INSERT 仍然是单独的 SQL，但通过 JDBC 的 addBatch() + executeBatch() 批量发送，减少网络往返。

使用方式

@Service
public class UserBatchService {

    @Autowired
    private SqlSessionFactory sqlSessionFactory;

    @Transactional
    public void batchInsert(List<User> users) {
        // 手动开启 BATCH 模式的 SqlSession
        try (SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
            UserMapper mapper = sqlSession.getMapper(UserMapper.class);

            for (int i = 0; i < users.size(); i++) {
                mapper.insert(users.get(i));

                // 每 1000 条刷新一次，防止内存积压
                if ((i + 1) % 1000 == 0) {
                    sqlSession.flushStatements();
                }
            }
            // 处理剩余的
            sqlSession.flushStatements();
        }
    }
}

BATCH 模式注意事项

要点 说明
无法获取自增主键 flushStatements() 后无法通过实体对象获取生成的 ID
必须手动管理 SqlSession 不能使用 Spring 注入的 Mapper（它用的是 SIMPLE 模式）
需要配合 flushStatements() 否则所有 SQL 堆积在内存中直到 commit()
JDBC URL 需加 rewriteBatchedStatements=true MySQL 驱动默认不会真正批量发送，加此参数后性能显著提升

foreach 拼 SQL vs BATCH 模式对比

维度 foreach 拼 SQL BATCH 模式
SQL 条数 1 条（多行 VALUES） N 条（每行一条）
网络往返 1 次 1 次（executeBatch 合并发送）
适合场景 字段少、数据量适中 字段多、数据量大
灵活性 只能用于 INSERT INSERT / UPDATE / DELETE 均可
获取自增 ID 支持 不支持
SQL 长度风险 有（可能超 max_allowed_packet） 无

13.2.3 MyBatis-Plus 的 saveBatch 实现分析

MyBatis-Plus 的 IService.saveBatch() 是开发中最常用的批量插入方法，了解其内部实现对于性能调优很有帮助。

源码核心逻辑（简化版）

// ServiceImpl.java 中的 saveBatch 实现
@Transactional(rollbackFor = Exception.class)
public boolean saveBatch(Collection<T> entityList, int batchSize) {
    // 默认 batchSize = 1000
    String sqlStatement = getSqlStatement(SqlMethod.INSERT_ONE);

    return executeBatch(entityList, batchSize, (sqlSession, entity) -> {
        sqlSession.insert(sqlStatement, entity);
    });
}

// 底层 executeBatch 方法
public static <E> boolean executeBatch(Collection<E> list, int batchSize,
        BiConsumer<SqlSession, E> consumer) {
    // 1. 打开 BATCH 模式的 SqlSession
    SqlSession sqlSession = sqlSessionFactory.openSession(ExecutorType.BATCH);
    try {
        int i = 0;
        for (E element : list) {
            // 2. 逐条调用 insert
            consumer.accept(sqlSession, element);
            if ((i + 1) % batchSize == 0) {
                // 3. 每积累 batchSize 条，刷新一次
                sqlSession.flushStatements();
            }
            i++;
        }
        // 4. 处理不足一批的尾部数据
        sqlSession.flushStatements();
        return true;
    } finally {
        sqlSession.close();
    }
}

关键细节

要点 说明
默认批次大小 DEFAULT_BATCH_SIZE = 1000
执行模式 BATCH 模式，不是 foreach 拼 SQL
事务控制 方法上有 @Transactional，整体提交或回滚
性能关键 必须在 JDBC URL 中加 rewriteBatchedStatements=true，否则性能与逐条插入无异
自增 ID 回填 默认不支持，需要 MP 3.4.3+ 并配置 executeBatch 参数

最佳实践

# application.yml —— 确保开启批量重写
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb?rewriteBatchedStatements=true&useServerPrepStmts=false

// 使用 saveBatch，指定批次大小
userService.saveBatch(userList, 2000);

性能参考：同样插入 10 万条数据，开启 rewriteBatchedStatements 前后，耗时可从 60 秒降至 3~5 秒。

13.3 多表查询策略选择

13.3.1 JOIN 查询 vs 分步查询 vs 业务层组装

维度 JOIN 查询 分步查询（嵌套 Select） 业务层组装
SQL 数量 1 条 N+1 条（可懒加载优化） N 条（可控）
实现方式 XML 中写 JOIN + 嵌套 resultMap association/collection + select 属性 代码中分别查询再组装
性能 数据量小时最优 懒加载时按需查询 可精确控制，可并行查询
可维护性 SQL 复杂度高 中等 最清晰，易单测
缓存友好 差（整体缓存） 好（子查询可命中缓存） 最好（各层独立缓存）
跨库支持 ❌ 不支持 ✅ 支持 ✅ 支持
分库分表 ❌ 困难 ⚠️ 需注意路由 ✅ 天然支持
适用场景 简单关联、报表查询 关联表数据访问频率低 微服务、分库分表、复杂聚合

业务层组装示例

public OrderDetailVO getOrderDetail(Long orderId) {
    // 1. 查主表
    Order order = orderMapper.selectById(orderId);

    // 2. 查关联数据（可并行）
    CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(
        () -> userMapper.selectById(order.getUserId())
    );
    CompletableFuture<List<OrderItem>> itemsFuture = CompletableFuture.supplyAsync(
        () -> orderItemMapper.selectByOrderId(orderId)
    );

    // 3. 组装结果
    OrderDetailVO vo = new OrderDetailVO();
    vo.setOrder(order);
    vo.setUser(userFuture.join());
    vo.setItems(itemsFuture.join());
    return vo;
}

13.3.2 N+1 问题的识别与解决

什么是 N+1 问题：查询一组父数据（1 次查询），然后为每条父数据分别查询子数据（N 次查询），总计 N+1 次 SQL。

识别方法

开启 MyBatis SQL 日志，观察控制台输出：

logging:
  level:
    com.example.mapper: DEBUG

如果看到类似以下日志模式，就是典型的 N+1：

==> Preparing: SELECT * FROM order WHERE user_id = ?
==> Preparing: SELECT * FROM order_item WHERE order_id = ?   ← 重复 N 次
==> Preparing: SELECT * FROM order_item WHERE order_id = ?
==> Preparing: SELECT * FROM order_item WHERE order_id = ?
...

解决方案

方案 做法 适用场景
改用 JOIN 一条 SQL 完成关联查询 数据量不大，关联简单
开启懒加载 fetchType="lazy"，用到时才查 不一定每次都需要子数据
批量 IN 查询 先收集所有父 ID，一次 IN 查询所有子数据 业务层组装模式的首选
@BatchSize（JPA） MyBatis 无此注解，需手动实现 —

批量 IN 查询示例（推荐）：

public List<OrderVO> getOrderList(List<Long> orderIds) {
    // 1. 批量查订单
    List<Order> orders = orderMapper.selectBatchIds(orderIds);

    // 2. 收集所有 orderId，一次查询所有明细
    List<Long> allOrderIds = orders.stream().map(Order::getId).collect(Collectors.toList());
    List<OrderItem> allItems = orderItemMapper.selectByOrderIds(allOrderIds);

    // 3. 按 orderId 分组
    Map<Long, List<OrderItem>> itemMap = allItems.stream()
        .collect(Collectors.groupingBy(OrderItem::getOrderId));

    // 4. 组装
    return orders.stream().map(order -> {
        OrderVO vo = new OrderVO();
        BeanUtils.copyProperties(order, vo);
        vo.setItems(itemMap.getOrDefault(order.getId(), Collections.emptyList()));
        return vo;
    }).collect(Collectors.toList());
}

13.4 分页方案对比

13.4.1 物理分页 vs 逻辑分页

维度 物理分页 逻辑分页
实现方式 数据库层 LIMIT offset, size 查全部数据，内存中截取
数据传输量 只传当前页数据 传输全部数据
内存占用 低 高（数据量大时可能 OOM）
实时性 高（每次查库） 低（首次查询后的缓存数据）
适用数据量 任意（大表必须用） 仅适合数据量极小的场景（< 几百条）
MyBatis 实现 插件拦截 SQL 拼接 LIMIT RowBounds（不推荐）

结论：生产环境统一使用物理分页，逻辑分页仅在特殊场景（如配置表、字典表全量缓存）中考虑。

13.4.2 PageHelper vs MyBatis-Plus 分页插件

维度 PageHelper MyBatis-Plus PaginationInnerInterceptor
引入方式 独立依赖 pagehelper-spring-boot-starter MyBatis-Plus 内置
使用方式 PageHelper.startPage(page, size) 方法参数传入 Page 对象
原理 ThreadLocal 存储分页参数，拦截器拼 SQL 拦截器识别 IPage 参数，拼 SQL
COUNT 查询 自动优化（去 ORDER BY） 自动优化
多数据库支持 自动识别方言 需配置 DbType
安全性 ⚠️ ThreadLocal 可能泄漏 ✅ 参数传递，无泄漏风险
与 MP 集成 可以但有冗余 天然集成

PageHelper 使用

public PageInfo<User> getUserPage(int pageNum, int pageSize) {
    // 必须紧跟在查询方法前调用，中间不能有其他查询
    PageHelper.startPage(pageNum, pageSize);
    List<User> list = userMapper.selectAll();
    return new PageInfo<>(list);
}

PageHelper 的 ThreadLocal 陷阱：如果 startPage() 后发生异常导致查询未执行，分页参数会残留在 ThreadLocal 中，影响下一次查询。建议用 try-finally 或者直接切换到 MyBatis-Plus 分页。

MyBatis-Plus 分页插件使用

// 配置
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
    return interceptor;
}

// 使用
public IPage<User> getUserPage(int current, int size) {
    Page<User> page = new Page<>(current, size);
    return userMapper.selectPage(page, new LambdaQueryWrapper<User>()
        .eq(User::getStatus, 1)
        .orderByDesc(User::getCreateTime));
}

13.4.3 深分页（大偏移量）优化

当 OFFSET 值很大时（如第 10000 页），MySQL 需要先扫描并丢弃 OFFSET 条记录，性能急剧下降。

-- 深分页：OFFSET 100000 会扫描 100010 行，丢弃前 100000 行
SELECT * FROM user ORDER BY id LIMIT 100000, 10;

方案一：基于游标（记住上次位置）

最推荐的方案，前端传入上一页最后一条记录的 ID，避免 OFFSET。

<select id="selectPageAfterId" resultMap="BaseResultMap">
    SELECT <include refid="Base_Column_List"/>
    FROM user
    WHERE id > #{lastId}
    ORDER BY id ASC
    LIMIT #{pageSize}
</select>

public List<User> getNextPage(Long lastId, int pageSize) {
    return userMapper.selectPageAfterId(lastId, pageSize);
}

优点：利用主键索引，无论第几页都是常数时间。

限制：只能上一页/下一页翻页，不能跳页。适合瀑布流、无限滚动场景。

方案二：子查询优化 OFFSET（延迟关联）

先在覆盖索引上定位 ID，再回表取数据。

<select id="selectPageOptimized" resultMap="BaseResultMap">
    SELECT u.*
    FROM user u
    INNER JOIN (
        SELECT id FROM user ORDER BY id LIMIT #{offset}, #{pageSize}
    ) t ON u.id = t.id
</select>

原理：子查询 SELECT id FROM user ORDER BY id LIMIT offset, size 走覆盖索引（只扫描 ID），速度很快。外层 JOIN 只回表 pageSize 条记录。

方案三：使用 BETWEEN 范围限定

适用于 ID 连续递增（无删除）的场景：

SELECT * FROM user WHERE id BETWEEN 100001 AND 100010 ORDER BY id;

三种方案对比

方案 性能 跳页支持 适用场景
游标（记住上次位置） ⭐⭐⭐ 最优 ❌ 不支持 无限滚动、瀑布流
子查询延迟关联 ⭐⭐ 较优 ✅ 支持 后台管理系统分页
BETWEEN 范围 ⭐⭐⭐ 最优 ✅ 支持 ID 连续递增的特殊表

13.5 枚举类型的优雅处理方案

数据库中常用 TINYINT 存储状态值（如 0-禁用、1-启用），Java 中用枚举可以提高类型安全性和可读性。

完整方案

第一步：定义枚举

@Getter
public enum UserStatus {
    DISABLED(0, "禁用"),
    ENABLED(1, "启用"),
    LOCKED(2, "锁定");

    @EnumValue  // MyBatis-Plus 标记数据库存储值
    private final int code;

    @JsonValue  // Jackson 序列化时使用
    private final String desc;

    UserStatus(int code, String desc) {
        this.code = code;
        this.desc = desc;
    }

    private static final Map<Integer, UserStatus> CODE_MAP = Arrays.stream(values())
            .collect(Collectors.toMap(UserStatus::getCode, Function.identity()));

    @JsonCreator
    public static UserStatus fromCode(Integer code) {
        return Optional.ofNullable(CODE_MAP.get(code))
                .orElseThrow(() -> new IllegalArgumentException("未知的用户状态码: " + code));
    }
}

第二步：实体类使用枚举

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String username;
    private UserStatus status;  // 直接使用枚举类型
}

第三步（方案 A）：MyBatis-Plus @EnumValue（推荐）

只需全局配置，零额外代码：

mybatis-plus:
  configuration:
    default-enum-type-handler: com.baomidou.mybatisplus.core.handlers.MybatisEnumTypeHandler

@EnumValue 标注的字段值会自动与数据库值互转。

第三步（方案 B）：自定义 TypeHandler（原生 MyBatis）

@MappedTypes(UserStatus.class)
public class UserStatusTypeHandler extends BaseTypeHandler<UserStatus> {

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
            UserStatus parameter, JdbcType jdbcType) throws SQLException {
        ps.setInt(i, parameter.getCode());
    }

    @Override
    public UserStatus getNullableResult(ResultSet rs, String columnName) throws SQLException {
        int code = rs.getInt(columnName);
        return rs.wasNull() ? null : UserStatus.fromCode(code);
    }

    @Override
    public UserStatus getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        int code = rs.getInt(columnIndex);
        return rs.wasNull() ? null : UserStatus.fromCode(code);
    }

    @Override
    public UserStatus getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        int code = cs.getInt(columnIndex);
        return cs.wasNull() ? null : UserStatus.fromCode(code);
    }
}

注册方式：

# 全局注册（扫描包下所有 TypeHandler）
mybatis:
  type-handlers-package: com.example.handler

或在 XML 中局部使用：

<result column="status" property="status" typeHandler="com.example.handler.UserStatusTypeHandler"/>

通用枚举接口（进阶）

如果项目中有很多枚举类型，可以抽象一个通用接口：

public interface BaseEnum<T> {
    T getCode();
    String getDesc();
}

public class GenericEnumTypeHandler<E extends BaseEnum<?>> extends BaseTypeHandler<E> {
    private final Map<Object, E> enumMap;

    public GenericEnumTypeHandler(Class<E> type) {
        E[] constants = type.getEnumConstants();
        this.enumMap = Arrays.stream(constants)
                .collect(Collectors.toMap(BaseEnum::getCode, Function.identity()));
    }

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
            E parameter, JdbcType jdbcType) throws SQLException {
        ps.setObject(i, parameter.getCode());
    }

    @Override
    public E getNullableResult(ResultSet rs, String columnName) throws SQLException {
        Object code = rs.getObject(columnName);
        return code == null ? null : enumMap.get(code);
    }

    @Override
    public E getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        Object code = rs.getObject(columnIndex);
        return code == null ? null : enumMap.get(code);
    }

    @Override
    public E getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        Object code = cs.getObject(columnIndex);
        return code == null ? null : enumMap.get(code);
    }
}

13.6 JSON 字段的映射处理

MySQL 5.7+ 支持 JSON 字段类型，但 MyBatis 默认不能自动将 JSON 字符串与 Java 对象互转。

场景

CREATE TABLE user_profile (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    extra_info JSON COMMENT '扩展信息，如 {"hobby":["reading","coding"],"level":3}'
);

@Data
public class ExtraInfo {
    private List<String> hobby;
    private Integer level;
}

方案一：自定义 TypeHandler（原生 MyBatis）

@MappedTypes(ExtraInfo.class)
@MappedJdbcTypes(JdbcType.VARCHAR)
public class ExtraInfoTypeHandler extends BaseTypeHandler<ExtraInfo> {

    private static final ObjectMapper MAPPER = new ObjectMapper();

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
            ExtraInfo parameter, JdbcType jdbcType) throws SQLException {
        try {
            ps.setString(i, MAPPER.writeValueAsString(parameter));
        } catch (JsonProcessingException e) {
            throw new SQLException("序列化 ExtraInfo 失败", e);
        }
    }

    @Override
    public ExtraInfo getNullableResult(ResultSet rs, String columnName) throws SQLException {
        return parse(rs.getString(columnName));
    }

    @Override
    public ExtraInfo getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        return parse(rs.getString(columnIndex));
    }

    @Override
    public ExtraInfo getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        return parse(cs.getString(columnIndex));
    }

    private ExtraInfo parse(String json) throws SQLException {
        if (json == null || json.isEmpty()) {
            return null;
        }
        try {
            return MAPPER.readValue(json, ExtraInfo.class);
        } catch (JsonProcessingException e) {
            throw new SQLException("反序列化 ExtraInfo 失败: " + json, e);
        }
    }
}

方案二：通用 JSON TypeHandler（推荐）

为避免每个 JSON 字段都写一个 TypeHandler，可以做一个泛型版：

public class JsonTypeHandler<T> extends BaseTypeHandler<T> {

    private static final ObjectMapper MAPPER = new ObjectMapper();
    private final Class<T> type;

    public JsonTypeHandler(Class<T> type) {
        this.type = type;
    }

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
            T parameter, JdbcType jdbcType) throws SQLException {
        try {
            ps.setString(i, MAPPER.writeValueAsString(parameter));
        } catch (JsonProcessingException e) {
            throw new SQLException("JSON 序列化失败", e);
        }
    }

    @Override
    public T getNullableResult(ResultSet rs, String columnName) throws SQLException {
        return parse(rs.getString(columnName));
    }

    @Override
    public T getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        return parse(rs.getString(columnIndex));
    }

    @Override
    public T getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        return parse(cs.getString(columnIndex));
    }

    private T parse(String json) throws SQLException {
        if (json == null || json.isEmpty()) {
            return null;
        }
        try {
            return MAPPER.readValue(json, type);
        } catch (JsonProcessingException e) {
            throw new SQLException("JSON 反序列化失败", e);
        }
    }
}

方案三：MyBatis-Plus @TableField + JacksonTypeHandler

MyBatis-Plus 内置了 JacksonTypeHandler，使用最简单：

13.7 乐观锁与悲观锁实践

并发场景下的数据一致性是永恒的课题。MyBatis / MyBatis-Plus 对两种锁方案都有良好支持。

乐观锁（Optimistic Locking）

核心思想：不加锁，更新时检查版本号是否一致，不一致则认为被其他线程修改过，更新失败。

MyBatis-Plus @Version 实现

@Data
@TableName("product")
public class Product {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer stock;
    private BigDecimal price;

    @Version
    private Integer version;  // 版本号字段
}

-- 对应表结构
CREATE TABLE product (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    stock INT DEFAULT 0,
    price DECIMAL(10,2),
    version INT DEFAULT 0
);

配置乐观锁拦截器：

@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());
    interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
    return interceptor;
}

使用示例：

public boolean deductStock(Long productId, int quantity) {
    // 1. 先查询，获取当前 version
    Product product = productMapper.selectById(productId);
    if (product.getStock() < quantity) {
        throw new BusinessException("库存不足");
    }

    // 2. 扣减库存
    product.setStock(product.getStock() - quantity);

    // 3. 更新（MyBatis-Plus 自动在 WHERE 中加 version 条件，并自增 version）
    // 生成 SQL: UPDATE product SET stock=?, version=version+1 WHERE id=? AND version=?
    int rows = productMapper.updateById(product);

    // 4. 判断是否更新成功
    if (rows == 0) {
        // 版本号不匹配，更新失败，可重试
        throw new OptimisticLockException("数据已被其他操作修改，请重试");
    }
    return true;
}

乐观锁重试机制

public boolean deductStockWithRetry(Long productId, int quantity) {
    int maxRetry = 3;
    for (int i = 0; i < maxRetry; i++) {
        try {
            return deductStock(productId, quantity);
        } catch (OptimisticLockException e) {
            if (i == maxRetry - 1) {
                throw e;
            }
            // 短暂等待后重试
        }
    }
    return false;
}

@Version 支持的字段类型

类型 更新策略
int / Integer newVersion = oldVersion + 1
long / Long newVersion = oldVersion + 1
Date newVersion = new Date()
Timestamp newVersion = new Timestamp(...)
LocalDateTime newVersion = LocalDateTime.now()

悲观锁（Pessimistic Locking）

核心思想：先加锁，再操作。使用 SELECT ... FOR UPDATE 在数据库层面加排他锁。

实现示例

<select id="selectForUpdate" resultMap="BaseResultMap">
    SELECT <include refid="Base_Column_List"/>
    FROM product
    WHERE id = #{id}
    FOR UPDATE
</select>

@Transactional  // 必须在事务中，否则 FOR UPDATE 无效
public boolean deductStockPessimistic(Long productId, int quantity) {
    // 1. 加锁查询（其他事务会被阻塞等待）
    Product product = productMapper.selectForUpdate(productId);

    if (product.getStock() < quantity) {
        throw new BusinessException("库存不足");
    }

    // 2. 扣减库存
    product.setStock(product.getStock() - quantity);
    productMapper.updateById(product);
    return true;
}

乐观锁 vs 悲观锁

维度 乐观锁 悲观锁
加锁时机 不加锁，更新时校验 查询时加锁
并发性能 高（无阻塞） 低（阻塞等待）
冲突处理 失败重试 排队等待
死锁风险 无 有（多表加锁时需注意顺序）
适用场景 读多写少，冲突概率低 写多读少，冲突概率高
实现复杂度 需版本号字段和重试逻辑 SQL 加 FOR UPDATE 即可
典型场景 商品信息修改、配置更新 库存扣减、账户余额变更

13.8 逻辑删除的利弊与实施方案

逻辑删除不是物理删除数据行，而是通过一个标记字段（如 is_deleted）标识数据是否已删除。

利弊分析

优势 劣势
数据可恢复，误删不慌 数据库存储量持续增长
保留数据审计轨迹 唯一索引需要特殊处理
关联数据完整性不受影响 所有查询都需要加 WHERE is_deleted = 0
满足合规要求（如金融、医疗） 数据量大后影响查询性能

唯一索引问题

逻辑删除后，username 的唯一索引会阻止同名用户重新注册：

-- 问题：删除用户 "zhangsan" 后，新注册 "zhangsan" 会唯一键冲突
-- 因为旧记录还在，只是 is_deleted = 1

-- 解决方案：联合唯一索引包含删除标记
-- 方案 1：is_deleted 存 0 或 主键ID（而非 0 和 1）
ALTER TABLE user ADD UNIQUE INDEX uk_username_deleted (username, is_deleted);
-- 删除时设置 is_deleted = id，这样同一 username 可以有多条已删除记录

-- 方案 2：MySQL 8.0+ 函数索引
ALTER TABLE user ADD UNIQUE INDEX uk_username_active ((CASE WHEN is_deleted = 0 THEN username END));

MyBatis-Plus @TableLogic 配置

第一步：数据库字段

ALTER TABLE user ADD COLUMN is_deleted TINYINT(1) DEFAULT 0 COMMENT '逻辑删除 0-未删除 1-已删除';

第二步：实体配置

@Data
@TableName("user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String username;

    @TableLogic
    private Integer isDeleted;
}

第三步：全局配置

mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: isDeleted  # 全局逻辑删除字段名
      logic-delete-value: 1          # 删除后的值
      logic-not-delete-value: 0      # 未删除的值

配置后的效果

操作 原始 SQL 配置后 SQL
selectById(1) SELECT * FROM user WHERE id = 1 SELECT * FROM user WHERE id = 1 AND is_deleted = 0
selectList(wrapper) SELECT * FROM user WHERE ... SELECT * FROM user WHERE ... AND is_deleted = 0
deleteById(1) DELETE FROM user WHERE id = 1 UPDATE user SET is_deleted = 1 WHERE id = 1
update(entity, wrapper) UPDATE user SET ... UPDATE user SET ... WHERE ... AND is_deleted = 0

查询已删除数据

@TableLogic 会自动过滤已删除记录。如果确实需要查询已删除数据（如管理后台回收站功能），需要手写 SQL：

<select id="selectDeletedUsers" resultMap="BaseResultMap">
    SELECT <include refid="Base_Column_List"/>
    FROM user
    WHERE is_deleted = 1
</select>

13.9 多数据源与读写分离

生产环境中，数据库读写分离和多数据源是常见需求。

方案一：Spring AbstractRoutingDataSource

原理：通过 ThreadLocal 动态切换数据源 Key。

public class DynamicDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContextHolder.getDataSourceKey();
    }
}

public class DataSourceContextHolder {
    private static final ThreadLocal<String> CONTEXT = new ThreadLocal<>();

    public static void setDataSourceKey(String key) { CONTEXT.set(key); }
    public static String getDataSourceKey() { return CONTEXT.get(); }
    public static void clear() { CONTEXT.remove(); }
}

@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface DataSource {
    String value() default "master";
}

@Aspect
@Component
public class DataSourceAspect {

    @Around("@annotation(ds)")
    public Object around(ProceedingJoinPoint point, DataSource ds) throws Throwable {
        DataSourceContextHolder.setDataSourceKey(ds.value());
        try {
            return point.proceed();
        } finally {
            DataSourceContextHolder.clear();
        }
    }
}

使用：

@Service
public class UserService {

    @DataSource("master")
    public void createUser(User user) { /* 写主库 */ }

    @DataSource("slave")
    public User getUser(Long id) { /* 读从库 */ }
}

方案二：dynamic-datasource-spring-boot-starter（推荐）

苞米豆团队（MyBatis-Plus 同团队）出品，开箱即用：

<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>dynamic-datasource-spring-boot-starter</artifactId>
    <version>4.3.1</version>
</dependency>

spring:
  datasource:
    dynamic:
      primary: master
      strict: false  # 未匹配到数据源时是否抛异常
      datasource:
        master:
          url: jdbc:mysql://master-host:3306/mydb
          username: root
          password: root
          driver-class-name: com.mysql.cj.jdbc.Driver
        slave_1:
          url: jdbc:mysql://slave1-host:3306/mydb
          username: root
          password: root
        slave_2:
          url: jdbc:mysql://slave2-host:3306/mydb
          username: root
          password: root

使用 @DS 注解切换数据源：

@Service
public class UserService {

    @DS("master")
    public void createUser(User user) {
        userMapper.insert(user);
    }

    @DS("slave_1")
    public User getUser(Long id) {
        return userMapper.selectById(id);
    }

    @DS("slave")  // 不指定具体从库，负载均衡轮询所有 slave 开头的数据源
    public List<User> listUsers() {
        return userMapper.selectList(null);
    }
}

两种方案对比

维度 AbstractRoutingDataSource dynamic-datasource-starter
侵入性 需自己写 AOP、Holder、配置类 引入依赖 + 注解即可
负载均衡 需自己实现 内置（轮询）
多数据源事务 需自行处理 内置 @DSTransactional
数据源分组 不支持 支持（如 slave 组）
适用场景 对三方依赖有严格限制的项目 大多数项目推荐

13.10 SQL 注入防范 checklist

SQL 注入是 Web 应用最常见、最危险的安全漏洞之一。MyBatis 提供了良好的防护机制，但如果使用不当，仍然会暴露风险。

Checklist

# 检查项 说明
1 优先使用 #{} 而非 ${} #{} 使用 PreparedStatement 预编译参数，自动防注入；${} 是字符串拼接，存在注入风险
2 ${} 仅用于不可参数化的位置 如动态表名 ${tableName}、动态排序 ORDER BY ${column}——此时必须在 Java 层校验白名单
3 动态表名/列名必须白名单校验 在 Java 代码中限定允许的值，禁止直接传入用户输入
4 LIKE 查询使用 CONCAT + #{} ✅ LIKE CONCAT('%', #{keyword}, '%') / ❌ LIKE '%${keyword}%'
5 IN 查询使用 <foreach> ✅ <foreach> 自动用 #{} 绑定参数 / ❌ 手动拼接 IN (${ids})
6 ORDER BY 做枚举限定 前端传入排序字段后，后端用 Map 或 Enum 映射到允许的列名
7 禁止 Wrapper.last() 拼接用户输入 last("LIMIT " + userInput) 极易注入，LIMIT 值必须用数字校验
8 代码审查关注 ${} 使用 在 Code Review 中将 ${} 作为必查项，每处使用都必须有正当理由和安全校验
9 开启 SQL 日志监控 通过日志或慢查询日志，发现异常 SQL 模式（如超长 SQL、异常关键字）
10 使用 Web 层参数校验 对请求参数做长度限制、类型校验、正则校验，从入口拦截恶意输入
11 数据库账号最小权限 应用使用的数据库账号不应有 DROP、ALTER、GRANT 等权限
12 定期安全扫描 使用 SQLMap 等工具对接口做注入测试

安全 vs 不安全写法对照

<!-- ❌ 危险：${}直接拼接用户输入 -->
<select id="search">
    SELECT * FROM user WHERE username = '${username}'
</select>
<!-- 攻击者传入: ' OR '1'='1 -->

<!-- ✅ 安全：#{}预编译参数 -->
<select id="search">
    SELECT * FROM user WHERE username = #{username}
</select>

<!-- ❌ 危险：动态表名未校验 -->
<select id="selectByTable">
    SELECT * FROM ${tableName} WHERE id = #{id}
</select>

<!-- ✅ 安全：Java 层白名单校验 -->

private static final Set<String> ALLOWED_TABLES = Set.of("user", "order", "product");

public List<Map<String, Object>> queryByTable(String tableName, Long id) {
    if (!ALLOWED_TABLES.contains(tableName)) {
        throw new IllegalArgumentException("非法表名: " + tableName);
    }
    return commonMapper.selectByTable(tableName, id);
}

<!-- ❌ 危险：ORDER BY 直接拼接 -->
<select id="selectList">
    SELECT * FROM user ORDER BY ${orderBy}
</select>

<!-- ✅ 安全：枚举映射 -->

private static final Map<String, String> ORDER_COLUMN_MAP = Map.of(
    "name", "username",
    "time", "create_time",
    "id", "id"
);

public List<User> selectList(String orderBy) {
    String column = ORDER_COLUMN_MAP.getOrDefault(orderBy, "id");
    return userMapper.selectListOrderBy(column);
}
