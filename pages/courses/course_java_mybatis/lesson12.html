三层职责说明

层级 职责 关键包/类
接口层 提供面向开发者的 API，接收调用请求并委托给核心处理层 SqlSession、SqlSessionFactory、MapperProxy
核心处理层 完成从配置解析、SQL 解析、参数绑定、SQL 执行到结果映射的全部流程 Executor、StatementHandler、ParameterHandler、ResultSetHandler、MappedStatement、SqlSource

设计思想：核心处理层的四大组件（Executor / StatementHandler / ParameterHandler / ResultSetHandler）均通过接口抽象 + 插件拦截的方式实现，这正是 MyBatis 插件机制的扩展点所在（详见第八章）。

Configuration 对象关键属性

Configuration 是 MyBatis 最核心的数据结构，几乎承载了所有运行时信息：

public class Configuration {
    // 全局配置
    protected Environment environment;           // 数据源 + 事务管理器
    protected boolean cacheEnabled = true;       // 二级缓存开关
    protected boolean lazyLoadingEnabled;         // 懒加载开关
    protected boolean mapUnderscoreToCamelCase;   // 驼峰映射
    protected ExecutorType defaultExecutorType = ExecutorType.SIMPLE;

    // 类型体系
    protected final TypeHandlerRegistry typeHandlerRegistry;
    protected final TypeAliasRegistry typeAliasRegistry;

    // Mapper 注册中心
    protected final MapperRegistry mapperRegistry;

    // SQL 语句注册（key = namespace + id）
    protected final Map<String, MappedStatement> mappedStatements;

    // 结果映射注册
    protected final Map<String, ResultMap> resultMaps;

    // 插件链
    protected final InterceptorChain interceptorChain;

    // 缓存注册（二级缓存，key = namespace）
    protected final Map<String, Cache> caches;

    // 对象工厂、反射工厂等
    protected ObjectFactory objectFactory;
    protected ReflectorFactory reflectorFactory;
}

排查技巧：很多"找不到 SQL"、"TypeHandler 未注册"的问题，本质上都是 Configuration 对象中对应的注册表里缺少了相应条目。理解这个对象的结构，是定位配置类问题的关键。

12.3 Mapper 代理机制 —— MapperProxy 与 MapperMethod

我们在代码中只定义了 Mapper 接口（没有实现类），MyBatis 如何让它"跑起来"？答案是 JDK 动态代理。

MapperProxy —— InvocationHandler 实现

// 简化后的核心逻辑
public class MapperProxy<T> implements InvocationHandler, Serializable {

    private final SqlSession sqlSession;
    private final Class<T> mapperInterface;
    private final Map<Method, MapperMethodInvoker> methodCache;

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // Object 类的方法（toString/hashCode 等）直接放行
        if (Object.class.equals(method.getDeclaringClass())) {
            return method.invoke(this, args);
        }
        // 其他方法通过 MapperMethodInvoker 执行
        return cachedInvoker(method).invoke(proxy, method, args, sqlSession);
    }
}

关键要点：

  • 注册时机：Configuration.addMapper(Class) → MapperRegistry.addMapper() → 为每个 Mapper 接口创建 MapperProxyFactory
  • 创建代理：SqlSession.getMapper() → MapperRegistry.getMapper() → MapperProxyFactory.newInstance() → Proxy.newProxyInstance()
  • 方法缓存：MapperProxy 内部维护了 methodCache，每个接口方法只需解析一次，后续直接复用

MapperMethod —— 方法签名到 SQL 执行的路由

MapperMethod 是连接 Mapper 接口方法与底层 SQL 执行的桥梁，由两个内部类组成：

public class MapperMethod {

    private final SqlCommand command;       // SQL 命令信息
    private final MethodSignature method;   // 方法签名信息

    public Object execute(SqlSession sqlSession, Object[] args) {
        switch (command.getType()) {
            case INSERT:
                // 参数转换 → sqlSession.insert(statement, param)
                break;
            case UPDATE:
                // 参数转换 → sqlSession.update(statement, param)
                break;
            case DELETE:
                // 参数转换 → sqlSession.delete(statement, param)
                break;
            case SELECT:
                if (method.returnsVoid() && method.hasResultHandler()) {
                    executeWithResultHandler(sqlSession, args);
                } else if (method.returnsMany()) {
                    result = executeForMany(sqlSession, args);
                } else if (method.returnsMap()) {
                    result = executeForMap(sqlSession, args);
                } else if (method.returnsCursor()) {
                    result = executeForCursor(sqlSession, args);
                } else {
                    result = sqlSession.selectOne(command.getName(), param);
                }
                break;
            case FLUSH:
                result = sqlSession.flushStatements();
                break;
        }
        return result;
    }
}
  • SqlCommand：解析接口方法对应的 MappedStatement，确定 SQL 类型（INSERT / UPDATE / DELETE / SELECT）和语句 ID
  • MethodSignature：解析方法的返回类型、参数名、是否有 RowBounds（分页）和 ResultHandler

代理调用链总结

userMapper.findById(1L)
    │
    ▼ JDK 动态代理
MapperProxy.invoke()
    │
    ▼ 从 methodCache 获取或创建
MapperMethod.execute(sqlSession, args)
    │
    ├─ SqlCommand: 确定是 SELECT 类型，语句 ID 为 "com.example.mapper.UserMapper.findById"
    ├─ MethodSignature: 返回类型是 User（单个对象）
    │
    ▼ 路由到
sqlSession.selectOne("com.example.mapper.UserMapper.findById", 1L)
    │
    ▼
进入 Executor 执行流程...

12.4 SQL 执行全流程

从 SqlSession 收到请求开始，SQL 执行会经过四大核心组件的流水线处理：

SqlSession.selectList(statementId, parameter)
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│ Executor                                                     │
│  ① 查询二级缓存（CachingExecutor，如果开启）                    │
│  ② 查询一级缓存（BaseExecutor.localCache）                     │
│  ③ 缓存未命中 → 调用 doQuery()                                │
│     └─ 创建 StatementHandler                                  │
├──────────────────────────────────────────────────────────────┤
│ StatementHandler                                             │
│  ④ 调用 prepare() → 创建 JDBC Statement（PreparedStatement）  │
│  ⑤ 调用 parameterize() → 委托 ParameterHandler 设置参数       │
├──────────────────────────────────────────────────────────────┤
│ ParameterHandler                                             │
│  ⑥ 遍历参数映射 → 通过 TypeHandler 将 Java 值写入占位符         │
│     如: ps.setLong(1, 1L)                                    │
├──────────────────────────────────────────────────────────────┤
│ JDBC 执行                                                     │
│  ⑦ PreparedStatement.execute() → 数据库返回 ResultSet          │
├──────────────────────────────────────────────────────────────┤
│ ResultSetHandler                                             │
│  ⑧ 遍历 ResultSet → 根据 resultMap/resultType 创建对象         │
│  ⑨ 通过 TypeHandler 逐列转换 → 反射填充属性                     │
│  ⑩ 处理嵌套映射、懒加载代理等                                   │
├──────────────────────────────────────────────────────────────┤
│ 返回                                                          │
│  ⑪ 结果写入一级缓存 → 返回给调用方                              │
└──────────────────────────────────────────────────────────────┘

记住这条链路：Executor → StatementHandler → ParameterHandler → JDBC → ResultSetHandler，这也是插件可以拦截的四个组件（Executor 和后三者）。

12.4.1 Executor —— SimpleExecutor / ReuseExecutor / BatchExecutor

Executor 是 SQL 执行的入口，MyBatis 提供了三种实现（均继承自 BaseExecutor），外加一个装饰器 CachingExecutor。

继承关系

Executor (接口)
    │
    ├── BaseExecutor (抽象类，实现一级缓存逻辑)
    │       ├── SimpleExecutor
    │       ├── ReuseExecutor
    │       └── BatchExecutor
    │
    └── CachingExecutor (装饰器，实现二级缓存逻辑)
              └── 内部持有一个 BaseExecutor 实例

三种 Executor 对比

特性 SimpleExecutor ReuseExecutor BatchExecutor
Statement 复用 每次创建新的 Statement，执行后关闭 以 SQL 字符串为 key 缓存 Statement，相同 SQL 复用 将多条 SQL 添加到批处理队列
执行方式 逐条执行 逐条执行（但复用 Statement） 调用 flushStatements() 时批量发送
适用场景 默认选择，通用场景 同一 SqlSession 中频繁执行相同 SQL 大批量写操作（插入/更新/删除）
性能特点 每条 SQL 都有完整的 prepare → execute 开销 减少了 Statement 创建和编译开销 网络往返次数最少，批量写性能最优
注意事项 无特殊限制 Statement 在 SqlSession 关闭时统一释放 无法立即获取自增主键和影响行数，需 flush 后获取

源码关键方法：

  • BaseExecutor.query() — 一级缓存查询逻辑
  • SimpleExecutor.doQuery() — 创建 Statement → 执行 → 关闭
  • ReuseExecutor.prepareStatement() — 从 statementMap 中复用
  • BatchExecutor.doUpdate() — 调用 Statement.addBatch()

配置方式：全局通过 <setting name="defaultExecutorType" value="REUSE"/> 设置，也可在打开 SqlSession 时指定：sqlSessionFactory.openSession(ExecutorType.BATCH)。

12.4.2 StatementHandler —— 三种实现

StatementHandler 负责创建 JDBC Statement 并驱动执行，三种实现对应 JDBC 的三种 Statement 类型。

StatementHandler (接口)
    │
    ├── BaseStatementHandler (抽象类)
    │       ├── SimpleStatementHandler    → java.sql.Statement
    │       ├── PreparedStatementHandler  → java.sql.PreparedStatement  ← 最常用
    │       └── CallableStatementHandler  → java.sql.CallableStatement
    │
    └── RoutingStatementHandler (路由器，根据 MappedStatement.statementType 分发)
实现类 对应 JDBC 类型 使用场景
SimpleStatementHandler Statement 无参数的简单 SQL，XML 中 statementType="STATEMENT"
PreparedStatementHandler PreparedStatement 带参数的 SQL（默认），支持 #{} 预编译
CallableStatementHandler CallableStatement 调用存储过程，statementType="CALLABLE"

核心方法：

  • prepare() — 创建 Statement 对象，设置超时时间和 fetchSize
  • parameterize() — 委托 ParameterHandler 设置参数
  • query() / update() — 执行 SQL 并调用 ResultSetHandler 处理结果

实际开发中，绝大多数 SQL 使用的都是 PreparedStatementHandler。只有显式指定了 statementType 或使用 ${} 拼接的场景才会用到 SimpleStatementHandler。

12.4.3 ParameterHandler —— 参数设置

ParameterHandler 接口只有一个实现类 DefaultParameterHandler，职责单一：将 Java 参数值设置到 PreparedStatement 的占位符上。

// 核心逻辑简化
public class DefaultParameterHandler implements ParameterHandler {

    public void setParameters(PreparedStatement ps) {
        // 遍历 BoundSql 中的 ParameterMapping 列表
        for (int i = 0; i < parameterMappings.size(); i++) {
            ParameterMapping mapping = parameterMappings.get(i);
            String propertyName = mapping.getProperty();

            // 从参数对象中取值
            Object value = /* 通过 MetaObject 反射取值 */;

            // 获取对应的 TypeHandler
            TypeHandler typeHandler = mapping.getTypeHandler();

            // 通过 TypeHandler 设置参数
            typeHandler.setParameter(ps, i + 1, value, mapping.getJdbcType());
        }
    }
}

参数解析链路：

  1. XML 中的 #{username} 在解析阶段被替换为 ?，同时生成 ParameterMapping（记录属性名、Java 类型、JDBC 类型、TypeHandler）
  2. 运行时，DefaultParameterHandler 遍历 ParameterMapping 列表
  3. 通过 MetaObject（MyBatis 的反射工具）从参数对象中提取属性值
  4. 调用对应的 TypeHandler.setParameter() 写入 PreparedStatement

常见问题：There is no getter for property named 'xxx' 这个错误就出在这里 —— MetaObject 找不到参数对象上的属性。

12.4.4 ResultSetHandler —— 结果集映射

ResultSetHandler 负责将 JDBC ResultSet 转换为 Java 对象，核心实现类是 DefaultResultSetHandler，也是 MyBatis 中最复杂的组件之一。

处理流程：

ResultSet 返回
    │
    ▼
DefaultResultSetHandler.handleResultSets()
    │
    ├─ 获取 ResultMap 配置
    │
    ├─ handleResultSet()
    │   │
    │   ├─ handleRowValues()
    │   │   │
    │   │   ├─ 简单映射：handleRowValuesForSimpleResultMap()
    │   │   │   ├─ 创建目标对象（ObjectFactory）
    │   │   │   ├─ 自动映射（autoMapping，列名 → 属性名）
    │   │   │   ├─ 手动映射（resultMap 中显式定义的 column → property）
    │   │   │   └─ 通过 TypeHandler 逐列转换值
    │   │   │
    │   │   └─ 嵌套映射：handleRowValuesForNestedResultMap()
    │   │       ├─ 处理 association / collection
    │   │       ├─ 基于 rowKey 做结果对象的去重与合并
    │   │       └─ 递归调用映射逻辑
    │   │
    │   └─ 将映射后的对象添加到结果列表
    │
    ▼
返回 List<Object>

核心类与方法：

  • DefaultResultSetHandler.handleResultSets() — 入口
  • DefaultResultSetHandler.getRowValue() — 单行映射
  • DefaultResultSetHandler.applyAutomaticMappings() — 自动映射
  • DefaultResultSetHandler.applyPropertyMappings() — 显式映射
  • ObjectFactory.create() — 创建结果对象实例

性能提示：autoMapping 在每次查询时都会通过反射匹配列名与属性名。如果你遇到性能敏感场景，显式定义 resultMap 可以避免这部分开销。

12.5 动态 SQL 解析原理 —— SqlNode 与 SqlSource

第四章讲了动态 SQL 的用法（<if>、<where>、<foreach> 等），这里看看它在源码层面是如何实现的。

SqlNode 树结构

MyBatis 将 Mapper XML 中的每个动态 SQL 标签解析为一个 SqlNode 节点，整条 SQL 形成一棵树：

<select id="findUsers">
    SELECT * FROM user
    <where>
        <if test="name != null">
            AND name = #{name}
        </if>
        <if test="age != null">
            AND age = #{age}
        </if>
    </where>
</select>

解析后的 SqlNode 树：

MixedSqlNode
  ├── StaticTextSqlNode ("SELECT * FROM user")
  └── WhereSqlNode
        └── MixedSqlNode
              ├── IfSqlNode (test="name != null")
              │     └── StaticTextSqlNode ("AND name = #{name}")
              └── IfSqlNode (test="age != null")
                    └── StaticTextSqlNode ("AND age = #{age}")

主要 SqlNode 实现：

SqlNode 实现类 对应 XML 标签
StaticTextSqlNode 纯文本 SQL 片段
IfSqlNode <if>
ChooseSqlNode <choose> / <when> / <otherwise>
WhereSqlNode <where>（继承自 TrimSqlNode）
SetSqlNode <set>（继承自 TrimSqlNode）
TrimSqlNode <trim>
ForEachSqlNode <foreach>
TextSqlNode 含 ${} 的文本（需运行时替换）
MixedSqlNode 容器节点，持有子 SqlNode 列表

DynamicSqlSource 与 RawSqlSource

SqlSource 是从 SqlNode 树到最终可执行 BoundSql 的桥梁，有两种核心实现：

类型 场景 解析时机 性能
RawSqlSource SQL 中没有动态标签（纯 #{} 占位符） 启动时一次性解析，#{} → ? 高，无运行时开销
DynamicSqlSource SQL 中包含动态标签（<if> / <foreach> 等）或 ${} 每次执行时动态拼接 略低，每次需遍历 SqlNode 树

运行时流程：

DynamicSqlSource.getBoundSql(parameterObject)
    │
    ├─ 创建 DynamicContext（用于拼接 SQL 文本）
    │
    ├─ rootSqlNode.apply(context)
    │   └─ 递归遍历 SqlNode 树
    │       ├─ IfSqlNode: 评估 OGNL 表达式 → 决定是否拼接
    │       ├─ ForEachSqlNode: 遍历集合 → 重复拼接
    │       └─ StaticTextSqlNode: 直接追加文本
    │
    ├─ 将拼接后的 SQL 中的 #{} 替换为 ?
    │   → 同时生成 ParameterMapping 列表
    │
    ▼
    BoundSql（包含：最终 SQL 字符串 + ParameterMapping 列表 + 参数对象）

源码入口：XMLScriptBuilder.parseScriptNode() 负责将 XML 标签解析为 SqlNode 树，并根据是否包含动态元素决定创建 DynamicSqlSource 还是 RawSqlSource。

12.6 缓存实现原理 —— 装饰器模式

MyBatis 的缓存体系采用了经典的装饰器模式（Decorator Pattern），通过层层包装为缓存添加不同能力。

核心缓存类

所有缓存实现都实现了 org.apache.ibatis.cache.Cache 接口：

public interface Cache {
    String getId();
    void putObject(Object key, Object value);
    Object getObject(Object key);
    Object removeObject(Object key);
    void clear();
    int getSize();
}

装饰器层层包装

MyBatis 内置的缓存装饰器如下，由内到外逐层增强：

PerpetualCache                ← 最内层：基础实现，HashMap 存储
    │
    └── LruCache              ← 淘汰策略：最近最少使用（默认）
        │                        (也可替换为 FifoCache / SoftCache / WeakCache)
        │
        └── SerializedCache   ← 序列化：存储时序列化，取出时反序列化
            │                    （readOnly=false 时启用，保证返回的是副本）
            │
            └── LoggingCache  ← 日志：记录缓存命中率
                │
                └── SynchronizedCache  ← 线程安全：synchronized 同步
                    │
                    └── BlockingCache   ← 防穿透：阻塞等待（可选）
装饰器 作用 对应配置
PerpetualCache 基础存储，内部使用 HashMap 始终作为最内层
LruCache LRU 淘汰策略（默认） eviction="LRU"
FifoCache FIFO 先进先出淘汰 eviction="FIFO"
SoftCache 软引用，JVM 内存不足时回收 eviction="SOFT"
WeakCache 弱引用，GC 时即回收 eviction="WEAK"
SerializedCache 序列化存储，保证线程安全的深拷贝 readOnly="false"（默认）
LoggingCache 打印缓存命中率日志 默认启用
SynchronizedCache synchronized 同步访问 默认启用
BlockingCache 阻塞式缓存，防止缓存击穿 blocking="true"
ScheduledCache 定时清空缓存 flushInterval 有值时启用

装饰器组装过程

在解析 <cache> 标签时，CacheBuilder 按照以下顺序组装：

// CacheBuilder.build() 简化逻辑
public Cache build() {
    // 1. 设置默认实现
    setDefaultImplementations();  // PerpetualCache + LruCache

    // 2. 创建基础缓存
    Cache cache = newBaseCacheInstance(implementation, id);  // PerpetualCache

    // 3. 逐层添加装饰器
    cache = newCacheDecoratorInstance(LruCache.class, cache);
    // ... 其他自定义装饰器 ...

    // 4. 标准装饰器（框架自动添加）
    cache = setStandardDecorators(cache);
    //   → SerializedCache（如果 readOnly=false）
    //   → LoggingCache
    //   → SynchronizedCache
    //   → ScheduledCache（如果 flushInterval > 0）

    return cache;
}

实战意义：理解装饰器模式后，你可以自定义 Cache 实现（如整合 Redis），只需实现 Cache 接口，并在 <cache type="com.example.RedisCache"/> 中指定，MyBatis 仍然会自动为其套上 LoggingCache、SynchronizedCache 等标准装饰器。

源码阅读推荐路径

如果你想深入阅读 MyBatis 源码，建议按以下顺序切入：

1. Configuration 类          ← 全局数据结构，了解"MyBatis 运行时知道些什么"
2. MapperProxy / MapperMethod ← 代理机制，理解"接口如何变成可执行代码"
3. SimpleExecutor.doQuery()   ← 执行入口，一路跟进 StatementHandler → ParameterHandler → ResultSetHandler
4. XMLScriptBuilder           ← 动态 SQL 解析，SqlNode 树的构建过程
5. CacheBuilder / CachingExecutor ← 缓存的装饰与使用

最后提醒：源码阅读的目的不是记住每一行代码，而是建立整体认知。当你遇到诡异问题时，能快速判断"问题出在哪个环节"，然后有针对性地打断点排查 —— 这就够了。
