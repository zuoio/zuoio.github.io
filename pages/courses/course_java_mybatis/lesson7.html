第七章 缓存机制 ⭐

缓存是性能优化的重要手段，也是常见坑点。理解 MyBatis 的一级缓存与二级缓存原理，才能在生产环境中做出正确的缓存策略决策。

7.1 一级缓存（SqlSession 级别）

7.1.1 默认行为与生效范围

一级缓存是 MyBatis 默认开启且无法关闭的缓存机制，其作用域为 SqlSession。在同一个 SqlSession 内，对同一条 SQL（相同的 Statement ID + 相同的参数）的多次查询，只有第一次会真正执行 SQL 语句，后续查询直接从缓存中获取结果。

底层实现：一级缓存本质上就是 SqlSession 内部维护的一个 HashMap（PerpetualCache），key 由 MappedStatement ID + 参数 + RowBounds + SQL 组成。

// 演示一级缓存命中
public void testFirstLevelCache() {
    SqlSession sqlSession = sqlSessionFactory.openSession();
    try {
        UserMapper mapper = sqlSession.getMapper(UserMapper.class);

        // 第一次查询 —— 执行 SQL，结果写入一级缓存
        User user1 = mapper.selectById(1L);
        System.out.println("第一次查询: " + user1);

        // 第二次查询 —— 直接命中一级缓存，不发送 SQL
        User user2 = mapper.selectById(1L);
        System.out.println("第二次查询: " + user2);

        // 验证：两次返回的是同一个对象引用
        System.out.println("是否同一对象: " + (user1 == user2)); // true
    } finally {
        sqlSession.close();
    }
}

开启日志后，控制台输出如下：

==>  Preparing: SELECT id, username, email FROM user WHERE id = ?
==> Parameters: 1(Long)
<==    Columns: id, username, email
<==        Row: 1, zhangsan, zhangsan@example.com
<==      Total: 1
第一次查询: User{id=1, username='zhangsan', email='zhangsan@example.com'}
第二次查询: User{id=1, username='zhangsan', email='zhangsan@example.com'}
是否同一对象: true

可以清晰地看到只有 一条 SQL 被执行，第二次查询直接返回了缓存中的同一个 Java 对象。

注意：一级缓存返回的是对象引用而非副本。如果修改了返回对象的属性，缓存中的数据也会被改变，这可能导致后续查询拿到脏数据。

7.1.2 一级缓存失效的四种场景

以下四种场景会导致一级缓存失效，必须重新执行 SQL：

场景一：不同的 SqlSession

一级缓存的作用域是 SqlSession，不同 SqlSession 拥有各自独立的缓存空间。

public void testDifferentSqlSession() {
    SqlSession sqlSession1 = sqlSessionFactory.openSession();
    SqlSession sqlSession2 = sqlSessionFactory.openSession();
    try {
        UserMapper mapper1 = sqlSession1.getMapper(UserMapper.class);
        UserMapper mapper2 = sqlSession2.getMapper(UserMapper.class);

        // sqlSession1 执行查询 —— 发送 SQL
        User user1 = mapper1.selectById(1L);
        // sqlSession2 执行相同查询 —— 仍然发送 SQL（不同 SqlSession，缓存不共享）
        User user2 = mapper2.selectById(1L);

        System.out.println("是否同一对象: " + (user1 == user2)); // false
    } finally {
        sqlSession1.close();
        sqlSession2.close();
    }
}

场景二：相同 SqlSession 但查询条件不同

查询条件不同意味着 CacheKey 不同，自然无法命中缓存。

public void testDifferentParams() {
    SqlSession sqlSession = sqlSessionFactory.openSession();
    try {
        UserMapper mapper = sqlSession.getMapper(UserMapper.class);

        // 查询 id=1 —— 发送 SQL
        User user1 = mapper.selectById(1L);
        // 查询 id=2 —— 发送 SQL（参数不同，缓存未命中）
        User user2 = mapper.selectById(2L);

        System.out.println("是否同一对象: " + (user1 == user2)); // false
    } finally {
        sqlSession.close();
    }
}

场景三：两次查询之间执行了增删改操作

任何 INSERT、UPDATE、DELETE 操作都会清空当前 SqlSession 的全部一级缓存，无论操作的是哪张表。

public void testCacheClearAfterUpdate() {
    SqlSession sqlSession = sqlSessionFactory.openSession();
    try {
        UserMapper mapper = sqlSession.getMapper(UserMapper.class);

        // 第一次查询 —— 发送 SQL，结果写入缓存
        User user1 = mapper.selectById(1L);
        System.out.println("第一次查询: " + user1);

        // 执行一次更新操作（哪怕更新的是其他记录）
        mapper.updateEmail(2L, "new@example.com");

        // 第二次查询 —— 重新发送 SQL（缓存已被清空）
        User user2 = mapper.selectById(1L);
        System.out.println("第二次查询: " + user2);

        System.out.println("是否同一对象: " + (user1 == user2)); // false
    } finally {
        sqlSession.close();
    }
}

场景四：手动清空缓存

调用 sqlSession.clearCache() 可以手动清空一级缓存。

public void testManualClearCache() {
    SqlSession sqlSession = sqlSessionFactory.openSession();
    try {
        UserMapper mapper = sqlSession.getMapper(UserMapper.class);

        // 第一次查询 —— 发送 SQL
        User user1 = mapper.selectById(1L);

        // 手动清空一级缓存
        sqlSession.clearCache();

        // 第二次查询 —— 重新发送 SQL
        User user2 = mapper.selectById(1L);

        System.out.println("是否同一对象: " + (user1 == user2)); // false
    } finally {
        sqlSession.close();
    }
}

四种场景速记：

# 失效场景 原因
1 不同 SqlSession 缓存作用域隔离
2 查询条件不同 CacheKey 不匹配
3 中间执行了增删改 执行 update/insert/delete 后自动清空缓存
4 手动调用 clearCache() 主动清空

7.1.3 Spring 集成下一级缓存的特殊表现

在 Spring 框架中整合 MyBatis 后，一级缓存的行为会发生显著变化。这是很多开发者不了解的一个关键点。

核心问题：Spring 中通过 SqlSessionTemplate 管理 SqlSession，默认情况下每次 Mapper 方法调用都会创建一个新的 SqlSession，方法执行完毕后立即关闭。这意味着在没有事务的情况下，一级缓存基本上形同虚设。

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    // 没有 @Transactional，每次 mapper 调用使用不同的 SqlSession
    public void queryWithoutTransaction() {
        // 第一次查询 —— SqlSession-A，发送 SQL，查询完毕后关闭 SqlSession-A
        User user1 = userMapper.selectById(1L);
        // 第二次查询 —— SqlSession-B，发送 SQL，查询完毕后关闭 SqlSession-B
        User user2 = userMapper.selectById(1L);

        // 两次都发送了 SQL，一级缓存未生效
        System.out.println("是否同一对象: " + (user1 == user2)); // false
    }

    // 加了 @Transactional，整个方法共用同一个 SqlSession
    @Transactional
    public void queryWithTransaction() {
        // 第一次查询 —— SqlSession-A，发送 SQL
        User user1 = userMapper.selectById(1L);
        // 第二次查询 —— 仍然是 SqlSession-A，命中一级缓存
        User user2 = userMapper.selectById(1L);

        // 一级缓存生效
        System.out.println("是否同一对象: " + (user1 == user2)); // true
    }
}

原理说明：

无事务场景：
  方法调用 → 创建 SqlSession → 执行查询 → 关闭 SqlSession（缓存销毁）
  方法调用 → 创建 SqlSession → 执行查询 → 关闭 SqlSession（缓存销毁）
  → 两次查询使用不同的 SqlSession，一级缓存无法命中

有事务场景（@Transactional）：
  开启事务 → 创建 SqlSession → 绑定到 ThreadLocal
    方法调用 → 复用同一个 SqlSession → 执行查询（结果进入缓存）
    方法调用 → 复用同一个 SqlSession → 命中一级缓存
  提交事务 → 关闭 SqlSession
  → 同一事务内的查询共享一级缓存

实际影响：

  • 在 Spring Boot 项目中，大部分查询操作不在同一个事务中，因此一级缓存的实际命中率非常低
  • 只有在 @Transactional 方法内部的多次相同查询，一级缓存才会生效
  • 不要依赖一级缓存来做性能优化，它更多是一个"附带福利"

结论：在 Spring 整合环境下，一级缓存的价值非常有限。如果需要缓存优化，应该使用 Redis 等外部缓存方案（见 7.3 节）。

7.2 二级缓存（namespace 级别）

二级缓存是 MyBatis 提供的跨 SqlSession 的缓存机制，其作用域为 Mapper 的 namespace。同一个 namespace 下的所有 SqlSession 共享同一块二级缓存区域。

执行流程：查询时先查二级缓存 → 未命中则查一级缓存 → 仍未命中则查数据库。

查询请求
  ↓
二级缓存（namespace 级别）—— 命中则直接返回
  ↓ （未命中）
一级缓存（SqlSession 级别）—— 命中则直接返回
  ↓ （未命中）
数据库查询 → 结果写入一级缓存
  ↓ （SqlSession 关闭或提交时）
一级缓存数据刷入二级缓存

7.2.1 开启方式 —— <cache> 标签

二级缓存默认是关闭的，需要手动开启。开启步骤如下：

步骤一：确认全局配置已开启二级缓存（默认即为 true）

<!-- mybatis-config.xml -->
<configuration>
    <settings>
        <!-- 全局开关，默认 true，一般不需要显式配置 -->
        <setting name="cacheEnabled" value="true"/>
    </settings>
</configuration>

Spring Boot 中对应的配置：

# application.yml
mybatis:
  configuration:
    cache-enabled: true  # 默认 true

步骤二：在 Mapper XML 中添加 <cache> 标签

<!-- UserMapper.xml -->
<mapper namespace="com.example.mapper.UserMapper">

    <!-- 开启当前 namespace 的二级缓存 -->
    <cache/>

    <select id="selectById" resultType="com.example.entity.User">
        SELECT id, username, email FROM user WHERE id = #{id}
    </select>

    <update id="updateEmail">
        UPDATE user SET email = #{email} WHERE id = #{id}
    </update>

</mapper>

步骤三：实体类实现 Serializable 接口（见 7.2.3）

完整使用示例：

public void testSecondLevelCache() {
    // 第一个 SqlSession
    SqlSession sqlSession1 = sqlSessionFactory.openSession();
    UserMapper mapper1 = sqlSession1.getMapper(UserMapper.class);
    User user1 = mapper1.selectById(1L);
    System.out.println("第一次查询: " + user1);
    sqlSession1.close(); // 关键：关闭 SqlSession 后，一级缓存数据才会刷入二级缓存

    // 第二个 SqlSession
    SqlSession sqlSession2 = sqlSessionFactory.openSession();
    UserMapper mapper2 = sqlSession2.getMapper(UserMapper.class);
    User user2 = mapper2.selectById(1L); // 命中二级缓存，不发送 SQL
    System.out.println("第二次查询: " + user2);
    sqlSession2.close();

    // 注意：二级缓存返回的是反序列化后的新对象（readOnly=false 时）
    System.out.println("是否同一对象: " + (user1 == user2)); // false
}

日志输出：

[sqlSession1]
==>  Preparing: SELECT id, username, email FROM user WHERE id = ?
==> Parameters: 1(Long)
<==      Total: 1
第一次查询: User{id=1, username='zhangsan', email='zhangsan@example.com'}

[sqlSession2]
Cache Hit Ratio [com.example.mapper.UserMapper]: 0.5
第二次查询: User{id=1, username='zhangsan', email='zhangsan@example.com'}
是否同一对象: false

关键点：一级缓存中的数据只有在 SqlSession 关闭（close） 或 提交（commit） 后，才会被写入二级缓存。如果 SqlSession 未关闭，二级缓存是拿不到数据的。

7.2.2 cache 属性详解 —— eviction / flushInterval / size / readOnly

<cache> 标签支持多个属性来控制缓存行为：

<cache
    eviction="LRU"
    flushInterval="60000"
    size="512"
    readOnly="false"
    blocking="false"
/>

属性详解表：

属性 含义 可选值 默认值 说明
eviction 缓存淘汰策略 LRU / FIFO / SOFT / WEAK LRU 缓存满时的回收策略
flushInterval 自动刷新间隔（毫秒） 任意正整数 无（不自动刷新） 设置后会按时间间隔自动清空缓存
size 最大缓存引用数 任意正整数 1024 缓存可以存储的对象（或列表）引用数量上限
readOnly 是否只读 true / false false true 返回同一引用（快但不安全），false 返回序列化副本（安全但慢）
blocking 是否阻塞 true / false false true 时同一缓存 key 只允许一个线程查询，防缓存击穿
type 自定义缓存实现类 实现 org.apache.ibatis.cache.Cache 接口的全类名 MyBatis 内置实现 可以指定 Redis、Ehcache 等第三方实现

淘汰策略详解：

策略 全称 说明
LRU Least Recently Used 移除最近最少使用的对象（最常用）
FIFO First In First Out 按先进先出顺序移除
SOFT Soft Reference 基于 JVM 软引用回收，内存不足时才回收
WEAK Weak Reference 基于 JVM 弱引用回收，GC 时即回收（存活时间最短）

readOnly 属性的重要区别：

// readOnly="true" 时：
// 所有调用者拿到的是同一个对象引用，性能好但不安全
// 如果一个调用者修改了对象属性，其他调用者也会受影响
User user1 = mapper1.selectById(1L);
user1.setUsername("modified"); // 会影响缓存中的数据！

// readOnly="false" 时（默认）：
// 每次从缓存中取出对象时都会通过序列化/反序列化创建一个新的副本
// 安全但有额外的序列化开销
User user1 = mapper1.selectById(1L);
user1.setUsername("modified"); // 不影响缓存中的数据

建议：生产环境中如果确定使用二级缓存，推荐保持 readOnly="false"（默认值）以避免并发修改问题。

7.2.3 实体类序列化要求

当 readOnly="false"（默认值）时，二级缓存通过 Java 序列化机制来创建对象副本，因此实体类必须实现 Serializable 接口，否则会抛出 NotSerializableException。

// 正确：实现 Serializable 接口
public class User implements Serializable {

    private static final long serialVersionUID = 1L;

    private Long id;
    private String username;
    private String email;

    // getter / setter ...
}

如果实体类包含关联对象，关联对象也必须实现 Serializable：

public class Order implements Serializable {

    private static final long serialVersionUID = 1L;

    private Long id;
    private BigDecimal amount;
    private User user; // User 也必须实现 Serializable

    // getter / setter ...
}

例外：当 readOnly="true" 时，缓存直接返回对象引用，不需要序列化，因此实体类可以不实现 Serializable。但这样做会牺牲线程安全性，不推荐在生产环境使用。

7.2.4 <cache-ref> —— 跨 namespace 共享缓存

默认情况下，每个 Mapper namespace 拥有独立的二级缓存空间。但某些场景下，多个 Mapper 操作的是同一组数据表，需要共享缓存以避免脏读问题。

<cache-ref> 可以让当前 namespace 引用另一个 namespace 的缓存空间：

<!-- OrderMapper.xml -->
<mapper namespace="com.example.mapper.OrderMapper">

    <!-- 引用 UserMapper 的缓存空间，两个 namespace 共享同一块缓存 -->
    <cache-ref namespace="com.example.mapper.UserMapper"/>

    <select id="selectOrderWithUser" resultMap="orderWithUserMap">
        SELECT o.*, u.username
        FROM orders o
        LEFT JOIN user u ON o.user_id = u.id
        WHERE o.id = #{id}
    </select>

</mapper>

使用 cache-ref 后的效果：

  • OrderMapper 和 UserMapper 共享同一个缓存区域
  • OrderMapper 中的增删改操作会清空共享缓存，UserMapper 的缓存也会被清空，反之亦然
  • 这可以在一定程度上缓解多表关联场景下的脏读问题

注意事项：

  • 使用 <cache-ref> 后，当前 namespace 不能再使用 <cache> 标签
  • 共享缓存会导致缓存粒度变粗 —— 任意一个 namespace 的写操作都会清空整个共享缓存
  • 这种方案只适用于关联度很高的少数几个 Mapper，不宜大范围使用

7.2.5 二级缓存的脏读问题与规避方案

二级缓存最大的陷阱在于多表关联场景下的脏读。这也是生产环境中通常建议关闭二级缓存的核心原因。

脏读场景复现

假设有两张表：user 和 orders，以及两个 Mapper：UserMapper 和 OrderMapper。

<!-- UserMapper.xml —— 独立的缓存空间 -->
<mapper namespace="com.example.mapper.UserMapper">
    <cache/>

    <select id="selectById" resultType="User">
        SELECT id, username, email FROM user WHERE id = #{id}
    </select>
</mapper>

<!-- OrderMapper.xml —— 独立的缓存空间 -->
<mapper namespace="com.example.mapper.OrderMapper">
    <cache/>

    <!-- 这条 SQL 关联查询了 user 表 -->
    <select id="selectOrderDetail" resultType="OrderDetail">
        SELECT o.id, o.amount, u.username AS userName
        FROM orders o
        LEFT JOIN user u ON o.user_id = u.id
        WHERE o.id = #{id}
    </select>
</mapper>

脏读过程：

时刻 T1：OrderMapper.selectOrderDetail(1)
         → 执行 SQL，查询到 userName = "zhangsan"
         → 结果写入 OrderMapper 的二级缓存

时刻 T2：UserMapper.updateUsername(1, "lisi")
         → 更新 user 表，userName 从 "zhangsan" 变为 "lisi"
         → 清空 UserMapper 的二级缓存（✓）
         → OrderMapper 的缓存不受影响！（✗ 脏读根源）

时刻 T3：OrderMapper.selectOrderDetail(1)
         → 命中 OrderMapper 的二级缓存
         → 返回 userName = "zhangsan"（脏数据！实际已经是 "lisi"）

问题本质：二级缓存以 namespace 为隔离单元。当 SQL 跨表查询时，表 A 的数据变更只会清空表 A 所在 namespace 的缓存，而包含表 A 数据的其他 namespace 缓存不会被感知到。

用代码验证脏读

public void testDirtyRead() {
    // T1: 通过 OrderMapper 查询（关联了 user 表）
    SqlSession session1 = sqlSessionFactory.openSession();
    OrderMapper orderMapper1 = session1.getMapper(OrderMapper.class);
    OrderDetail detail1 = orderMapper1.selectOrderDetail(1L);
    System.out.println("T1 查询结果: " + detail1.getUserName()); // zhangsan
    session1.close(); // 结果写入 OrderMapper 二级缓存

    // T2: 通过 UserMapper 更新 user 表
    SqlSession session2 = sqlSessionFactory.openSession();
    UserMapper userMapper = session2.getMapper(UserMapper.class);
    userMapper.updateUsername(1L, "lisi");
    session2.commit();
    session2.close(); // 清空 UserMapper 二级缓存，但 OrderMapper 缓存不受影响

    // T3: 再次通过 OrderMapper 查询 —— 脏读！
    SqlSession session3 = sqlSessionFactory.openSession();
    OrderMapper orderMapper3 = session3.getMapper(OrderMapper.class);
    OrderDetail detail3 = orderMapper3.selectOrderDetail(1L);
    System.out.println("T3 查询结果: " + detail3.getUserName()); // zhangsan（脏数据！）
    session3.close();
}

规避方案

方案 做法 优点 缺点
cache-ref OrderMapper 中使用 <cache-ref namespace="...UserMapper"/> 简单直接 缓存粒度变粗，命中率下降
关闭二级缓存 不在 Mapper XML 中添加 <cache> 标签 彻底消除脏读风险 失去二级缓存的性能收益
使用外部缓存 在 Service 层使用 Redis / Caffeine 等缓存，手动控制失效 灵活可控，粒度精细 需要额外开发和维护
避免跨表查询缓存 只对单表查询的 Mapper 开启缓存 务实折中 需要仔细甄别每个 Mapper

最佳实践：在存在多表关联查询的场景下，不要使用 MyBatis 二级缓存。用 Redis 等外部缓存在 Service 层做精确控制，是更安全、更主流的做法。

7.3 自定义缓存 —— 整合 Redis / Ehcache

MyBatis 支持通过 type 属性指定自定义缓存实现，只要实现 org.apache.ibatis.cache.Cache 接口即可。下面以整合 Redis 为例，演示完整的配置过程。

方式一：使用 mybatis-redis 官方模块

步骤一：添加 Maven 依赖

<dependencies>
    <!-- MyBatis -->
    <dependency>
        <groupId>org.mybatis.spring.boot</groupId>
        <artifactId>mybatis-spring-boot-starter</artifactId>
        <version>3.0.3</version>
    </dependency>

    <!-- Redis -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-data-redis</artifactId>
    </dependency>
</dependencies>

步骤二：自定义 Redis Cache 实现

import org.apache.ibatis.cache.Cache;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.serializer.GenericJackson2JsonRedisSerializer;
import org.springframework.data.redis.serializer.StringRedisSerializer;
import org.springframework.data.redis.connection.lettuce.LettuceConnectionFactory;

import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class MybatisRedisCache implements Cache {

    private final String id;
    private final ReadWriteLock readWriteLock = new ReentrantReadWriteLock();

    private static volatile RedisTemplate<String, Object> redisTemplate;

    public MybatisRedisCache(String id) {
        if (id == null) {
            throw new IllegalArgumentException("Cache instances require an ID");
        }
        this.id = id;
    }

    /**
     * 由于 MyBatis 自己实例化缓存对象，无法使用 Spring 注入，
     * 需要通过静态方法手动设置 RedisTemplate。
     */
    public static void setRedisTemplate(RedisTemplate<String, Object> template) {
        redisTemplate = template;
    }

    private RedisTemplate<String, Object> getRedisTemplate() {
        if (redisTemplate == null) {
            throw new IllegalStateException("RedisTemplate has not been initialized");
        }
        return redisTemplate;
    }

    @Override
    public String getId() {
        return this.id;
    }

    @Override
    public void putObject(Object key, Object value) {
        getRedisTemplate().opsForHash().put(id, key.toString(), value);
    }

    @Override
    public Object getObject(Object key) {
        return getRedisTemplate().opsForHash().get(id, key.toString());
    }

    @Override
    public Object removeObject(Object key) {
        Object value = getObject(key);
        getRedisTemplate().opsForHash().delete(id, key.toString());
        return value;
    }

    @Override
    public void clear() {
        getRedisTemplate().delete(id);
    }

    @Override
    public int getSize() {
        Long size = getRedisTemplate().opsForHash().size(id);
        return size == null ? 0 : size.intValue();
    }

    @Override
    public ReadWriteLock getReadWriteLock() {
        return readWriteLock;
    }
}

            LettuceConnectionFactory connectionFactory) {

    }

方式二：使用 Spring Cache 抽象（推荐）

相比直接整合 MyBatis 二级缓存，更推荐的做法是在 Service 层 使用 Spring Cache 抽象 + Redis：

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    @Cacheable(value = "user", key = "#id")
    public User getUserById(Long id) {
        return userMapper.selectById(id);
    }

    @CacheEvict(value = "user", key = "#user.id")
    public void updateUser(User user) {
        userMapper.updateById(user);
    }

    @CacheEvict(value = "user", allEntries = true)
    public void deleteUser(Long id) {
        userMapper.deleteById(id);
    }
}

配合 Redis 配置：

@Configuration
@EnableCaching
public class RedisCacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory connectionFactory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(30))
                .serializeKeysWith(
                    RedisSerializationContext.SerializationPair
                        .fromSerializer(new StringRedisSerializer()))
                .serializeValuesWith(
                    RedisSerializationContext.SerializationPair
                        .fromSerializer(new GenericJackson2JsonRedisSerializer()))
                .disableCachingNullValues();

        return RedisCacheManager.builder(connectionFactory)
                .cacheDefaults(config)
                .build();
    }
}

7.4 生产环境缓存策略建议

基于实际项目经验，给出以下明确的生产环境缓存策略建议：

一级缓存：保持默认，不依赖

  • 一级缓存默认开启且无法关闭，保持默认即可
  • 在 Spring 环境下，一级缓存的实际价值非常有限（仅在 @Transactional 方法内生效）
  • 不要依赖一级缓存做性能优化，把它当作一个透明的"附赠品"

二级缓存：一般建议关闭

推荐在生产环境中关闭 MyBatis 二级缓存，原因如下：

  1. 脏读风险：多表关联场景下存在无法彻底解决的脏读问题
  2. 粒度太粗：以 namespace 为单位的缓存失效策略过于粗暴，任何写操作都会清空整个 namespace 的缓存
  3. 分布式场景失效：MyBatis 内置的二级缓存是本地缓存（JVM 内存），在多实例部署的分布式环境下各节点缓存不一致
  4. 序列化开销：readOnly=false 时每次缓存读取都需要反序列化，性能收益被削减
  5. Debug 困难：缓存导致的数据不一致问题难以排查

关闭方式：

# application.yml — 全局关闭二级缓存
mybatis:
  configuration:
    cache-enabled: false

或者更简单 —— 不在任何 Mapper XML 中添加 <cache> 标签即可。

推荐方案：Service 层 + Redis 外部缓存

生产环境 Checklist

检查项 建议
一级缓存 保持默认开启，不做特殊处理
二级缓存 关闭（cache-enabled: false 或不加 <cache> 标签）
缓存方案 使用 Spring Cache + Redis
缓存粒度 Service 方法级别，按业务需求精确控制
缓存 TTL 根据业务设置合理的过期时间，避免永不过期
缓存穿透 对空结果也进行短时间缓存，或使用布隆过滤器
缓存雪崩 TTL 加随机偏移量，避免大量缓存同时过期
缓存击穿 热点数据使用分布式锁或提前预热
数据一致性 更新数据时先更新 DB 再删除缓存（Cache Aside 模式）
监控 接入缓存命中率监控，及时发现异常

总结

MyBatis 缓存决策流程：

Q: 需要缓存优化吗？
├── 否 → 保持默认，不做任何处理
└── 是
    Q: 是单体应用还是分布式部署？
    ├── 单体 + 纯单表查询 → 可以考虑 MyBatis 二级缓存（但仍不推荐）
    └── 其他场景 → 使用 Redis 等外部缓存（推荐）
        ├── 简单场景 → Spring Cache 注解（@Cacheable / @CacheEvict）
        └── 复杂场景 → 手动操作 RedisTemplate，精确控制缓存逻辑

一句话总结：MyBatis 的一级缓存了解原理即可，二级缓存在生产环境建议关闭，真正的缓存优化交给 Redis + Spring Cache 来完成。
