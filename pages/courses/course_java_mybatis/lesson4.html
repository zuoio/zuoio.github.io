第四章 动态 SQL ⭐

MyBatis 最强大的特性之一，直接影响代码质量。
动态 SQL 让你可以根据运行时参数，在 XML 中灵活拼接 SQL 语句，彻底告别手动字符串拼接的噩梦。

在实际业务开发中，绝大多数查询都不是固定 SQL —— 用户可能只填了部分搜索条件，更新时可能只修改了几个字段，批量操作的数据量也不固定。MyBatis 提供了一组 XML 标签来优雅地处理这些场景，本章将逐一讲解。

本章使用的示例实体类：

public class User {
    private Long id;
    private String username;
    private String email;
    private Integer age;
    private Integer status;     // 0-禁用 1-正常
    private String department;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
    // getter/setter 省略
}

4.1 <if> —— 条件判断

<if> 是最基础的动态 SQL 标签，当 test 表达式为 true 时，标签内的 SQL 片段才会被拼接到最终语句中。

Mapper 接口

public interface UserMapper {
    List<User> findByCondition(@Param("username") String username,
                               @Param("email") String email,
                               @Param("status") Integer status);
}

XML 映射

<select id="findByCondition" resultType="User">
    SELECT id, username, email, age, status, department, create_time, update_time
    FROM t_user
    WHERE 1=1
    <if test="username != null and username != ''">
        AND username LIKE CONCAT('%', #{username}, '%')
    </if>
    <if test="email != null and email != ''">
        AND email = #{email}
    </if>
    <if test="status != null">
        AND status = #{status}
    </if>
</select>

关键说明

要点 说明
test 属性 使用 OGNL 表达式，可以访问参数属性、调用方法
字符串判空 字符串通常需要同时判断 != null 和 != ''
数值类型 Integer / Long 等包装类型只需判断 != null，不要判断 != ''（OGNL 会将数字与空字符串比较导致异常）
WHERE 1=1 这里用了 WHERE 1=1 的"兜底"写法来保证语法正确，但这不是最佳实践——4.3 节的 <where> 标签可以彻底解决这个问题

OGNL 常用表达式速查：

  • username != null —— 非空判断
  • username != '' —— 非空字符串
  • list != null and list.size() > 0 —— 集合非空判断
  • type == 'admin' —— 字符串比较（注意用单引号）
  • age > 18 —— 数值比较

4.2 <choose> / <when> / <otherwise> —— 多分支选择

类似 Java 中的 switch-case-default，多个条件中只有第一个满足的分支会被执行。

Mapper 接口

public interface UserMapper {
    List<User> findByPriority(@Param("id") Long id,
                              @Param("username") String username,
                              @Param("email") String email);
}

XML 映射

<select id="findByPriority" resultType="User">
    SELECT id, username, email, age, status, department, create_time, update_time
    FROM t_user
    WHERE 1=1
    <choose>
        <when test="id != null">
            AND id = #{id}
        </when>
        <when test="username != null and username != ''">
            AND username = #{username}
        </when>
        <when test="email != null and email != ''">
            AND email = #{email}
        </when>
        <otherwise>
            AND status = 1
        </otherwise>
    </choose>
</select>

执行逻辑

  1. 如果传了 id，只按 id 查询（后续条件不再判断）
  2. 没传 id 但传了 username，按 username 查询
  3. 都没传，走 <otherwise> 默认查询正常状态的用户

与 <if> 的区别： <if> 各分支相互独立，可以同时生效；<choose> 是互斥的，只生效第一个匹配的分支。

4.3 <where> —— 智能 WHERE 拼接

问题：手动拼接 WHERE 1=1 的尴尬

在 4.1 节中，我们用了 WHERE 1=1 来保证无论哪个 <if> 生效，SQL 语法都正确：

<!-- 手动兜底写法 -->
<select id="findByCondition" resultType="User">
    SELECT * FROM t_user
    WHERE 1=1
    <if test="username != null and username != ''">
        AND username LIKE CONCAT('%', #{username}, '%')
    </if>
    <if test="status != null">
        AND status = #{status}
    </if>
</select>

这种写法存在两个问题：

  1. WHERE 1=1 不美观，代码审查时经常被质疑
  2. 如果第一个条件不带 AND，而后面的条件带 AND，需要仔细管理每个条件的前缀

解决方案：使用 <where> 标签

<where> 标签会智能处理以下逻辑：

  • 只有在至少一个子元素有内容时，才插入 WHERE 关键字
  • 自动去除内容开头多余的 AND 或 OR

Mapper 接口

public interface UserMapper {
    List<User> searchUsers(@Param("username") String username,
                           @Param("email") String email,
                           @Param("status") Integer status,
                           @Param("department") String department);
}

XML 映射

<select id="searchUsers" resultType="User">
    SELECT id, username, email, age, status, department, create_time, update_time
    FROM t_user
    <where>
        <if test="username != null and username != ''">
            AND username LIKE CONCAT('%', #{username}, '%')
        </if>
        <if test="email != null and email != ''">
            AND email = #{email}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
        <if test="department != null and department != ''">
            AND department = #{department}
        </if>
    </where>
</select>

对比分析

场景 手动 WHERE 1=1 使用 <where>
所有条件都为空 WHERE 1=1（返回全部数据） 无 WHERE 子句（返回全部数据）
只有 status 有值 WHERE 1=1 AND status = ? WHERE status = ?
username 和 status 都有值 WHERE 1=1 AND username LIKE ? AND status = ? WHERE username LIKE ? AND status = ?
生成的 SQL 始终带有多余的 1=1 干净、语义明确
可读性 一般 好

最佳实践： 在所有需要动态 WHERE 条件的场景中，优先使用 <where> 标签代替 WHERE 1=1。

4.4 <set> —— 智能 SET 拼接

<set> 是 <where> 在 UPDATE 语句中的对应版本，它解决的是动态更新字段时的逗号问题。

问题场景

当我们只想更新用户修改过的字段时，手动拼接 SET 子句需要处理末尾多余的逗号：

-- 如果只更新了 username，拼出来可能变成：
UPDATE t_user SET username = ?,  WHERE id = ?
--                             ↑ 多余的逗号导致语法错误

Mapper 接口

public interface UserMapper {
    int updateSelective(User user);
}

XML 映射

<update id="updateSelective">
    UPDATE t_user
    <set>
        <if test="username != null and username != ''">
            username = #{username},
        </if>
        <if test="email != null and email != ''">
            email = #{email},
        </if>
        <if test="age != null">
            age = #{age},
        </if>
        <if test="status != null">
            status = #{status},
        </if>
        <if test="department != null and department != ''">
            department = #{department},
        </if>
        <if test="updateTime != null">
            update_time = #{updateTime},
        </if>
    </set>
    WHERE id = #{id}
</update>

<set> 的行为

  1. 如果至少有一个子元素有内容，自动插入 SET 关键字
  2. 自动去除内容末尾多余的逗号
  3. 如果所有条件都为空，不会插入 SET（此时 SQL 语法错误——调用方应确保至少有一个字段需要更新）

注意： 每个 <if> 内的赋值语句末尾都带了逗号，这是故意的——<set> 会自动处理最后一个多余的逗号。

4.5 <trim> —— 自定义前缀/后缀裁剪

<trim> 是 <where> 和 <set> 的底层通用实现。理解了 <trim>，你就理解了 MyBatis 动态 SQL 拼接的核心机制。

四个核心属性

属性 说明 示例值
prefix 当 trim 元素有内容时，在内容前面插入的前缀 WHERE、SET、(
prefixOverrides 从内容开头去除的多余文本（可用 | 分隔多个值） AND |OR
suffix 当 trim 元素有内容时，在内容后面插入的后缀 )
suffixOverrides 从内容末尾去除的多余文本（可用 | 分隔多个值） ,

注意 prefixOverrides 和 suffixOverrides 中的空格： AND 后面有空格，这样才能精确匹配 AND username 中的 AND ，而不会误匹配 Android 中的 And。

<where> 本质上就是特殊的 <trim>

<!-- <where> 等价于以下 <trim> -->
<trim prefix="WHERE" prefixOverrides="AND |OR ">
    ...
</trim>

解释：

  • prefix="WHERE" —— 如果有内容，前面加上 WHERE
  • prefixOverrides="AND |OR " —— 去除内容开头多余的 AND 或 OR

<set> 本质上也是特殊的 <trim>

<!-- <set> 等价于以下 <trim> -->
<trim prefix="SET" suffixOverrides=",">
    ...
</trim>

解释：

  • prefix="SET" —— 如果有内容，前面加上 SET
  • suffixOverrides="," —— 去除内容末尾多余的逗号

用 <trim> 实现 <where> 的效果

<select id="searchUsers" resultType="User">
    SELECT id, username, email, age, status
    FROM t_user
    <trim prefix="WHERE" prefixOverrides="AND |OR ">
        <if test="username != null and username != ''">
            AND username LIKE CONCAT('%', #{username}, '%')
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
    </trim>
</select>

用 <trim> 实现 <set> 的效果

<update id="updateSelective">
    UPDATE t_user
    <trim prefix="SET" suffixOverrides=",">
        <if test="username != null and username != ''">
            username = #{username},
        </if>
        <if test="email != null and email != ''">
            email = #{email},
        </if>
    </trim>
    WHERE id = #{id}
</update>

<trim> 的高级用法：构造插入语句

<trim> 可以同时使用 prefix、suffix、suffixOverrides 来构造复杂的 SQL 片段：

<insert id="insertSelective">
    INSERT INTO t_user
    <trim prefix="(" suffix=")" suffixOverrides=",">
        <if test="username != null">username,</if>
        <if test="email != null">email,</if>
        <if test="age != null">age,</if>
        <if test="status != null">status,</if>
    </trim>
    <trim prefix="VALUES (" suffix=")" suffixOverrides=",">
        <if test="username != null">#{username},</if>
        <if test="email != null">#{email},</if>
        <if test="age != null">#{age},</if>
        <if test="status != null">#{status},</if>
    </trim>
</insert>

当只传了 username 和 email 时，生成的 SQL 为：

INSERT INTO t_user (username, email) VALUES (?, ?)

小结： <where> 和 <set> 是 <trim> 的语法糖。日常开发中优先使用 <where> 和 <set>（更简洁），当需要自定义裁剪逻辑时，使用 <trim>。

4.6 <foreach> —— 集合遍历 ⭐

<foreach> 用于遍历集合或数组，生成重复的 SQL 片段。最常见的两个场景是 IN 查询和批量 INSERT。

4.6.1 属性详解

属性 说明 是否必填
collection 要遍历的集合或数组（取值规则见下方说明） 是
item 每次迭代的元素变量名 是
index 迭代的索引变量名（List 是下标，Map 是 key） 否
open 拼接内容的开始符号 否
close 拼接内容的结束符号 否
separator 每次迭代之间的分隔符 否

collection 属性取值规则（⭐ 重点）

collection 的值取决于参数的传递方式：

参数形式 collection 的值 说明
List<Long> ids（单个 List 参数，无 @Param） list 或 collection MyBatis 对 List 类型自动注册为 list 和 collection
Long[] ids（单个数组参数，无 @Param） array MyBatis 对数组类型自动注册为 array
@Param("ids") List<Long> ids ids 使用 @Param 注解指定的名称
@Param("ids") Long[] ids ids 使用 @Param 注解指定的名称
POJO 对象中的属性，如 query.ids ids（属性名） 直接使用 POJO 中的属性名
Map<String, Object> 中的 key Map 中对应的 key 如 map.put("ids", list) 则用 ids

最佳实践： 始终使用 @Param 注解来明确参数名，避免歧义。这样 collection 的值就是注解中指定的名称，最直观。

4.6.2 List、Array、Map 的遍历写法

遍历 List —— IN 查询

Mapper 接口：

public interface UserMapper {
    List<User> findByIds(@Param("ids") List<Long> ids);
}

XML 映射：

<select id="findByIds" resultType="User">
    SELECT id, username, email, age, status
    FROM t_user
    WHERE id IN
    <foreach collection="ids" item="id" open="(" close=")" separator=",">
        #{id}
    </foreach>
</select>

生成的 SQL（假设传入 [1, 2, 3]）：

SELECT id, username, email, age, status FROM t_user WHERE id IN (1, 2, 3)

遍历 Array

Mapper 接口：

public interface UserMapper {
    List<User> findByStatusArray(@Param("statusArr") Integer[] statusArr);
}

XML 映射：

<select id="findByStatusArray" resultType="User">
    SELECT id, username, email, age, status
    FROM t_user
    WHERE status IN
    <foreach collection="statusArr" item="s" open="(" close=")" separator=",">
        #{s}
    </foreach>
</select>

遍历 Map

遍历 Map 时，index 是 key，item 是 value。常用于动态构建 SET 子句或条件：

Mapper 接口：

public interface UserMapper {
    List<User> findByMap(@Param("conditions") Map<String, Object> conditions);
}

XML 映射：

<select id="findByMap" resultType="User">
    SELECT id, username, email, age, status
    FROM t_user
    <where>
        <foreach collection="conditions" index="column" item="value">
            AND ${column} = #{value}
        </foreach>
    </where>
</select>

⚠️ 安全警告： 上面的 ${column} 使用了字符串拼接，存在 SQL 注入风险。在生产环境中，必须对 column 做白名单校验，确保只允许预定义的列名。

不使用 @Param 时的 collection 值

// 单个 List 参数，无 @Param
List<User> findByIds(List<Long> ids);
// collection="list" 或 collection="collection"

// 单个数组参数，无 @Param  
List<User> findByIds(Long[] ids);
// collection="array"

虽然可以省略 @Param，但强烈建议始终添加 @Param，提高可读性和可维护性。

4.6.3 批量插入的性能优化

场景一：MySQL 多行 VALUES 语法（推荐）

利用 MySQL 支持的 INSERT INTO ... VALUES (...), (...), (...) 语法，一条 SQL 插入多行数据：

Mapper 接口：

public interface UserMapper {
    int batchInsert(@Param("users") List<User> users);
}

XML 映射：

<insert id="batchInsert">
    INSERT INTO t_user (username, email, age, status)
    VALUES
    <foreach collection="users" item="user" separator=",">
        (#{user.username}, #{user.email}, #{user.age}, #{user.status})
    </foreach>
</insert>

生成的 SQL（假设传入 3 条数据）：

INSERT INTO t_user (username, email, age, status)
VALUES ('张三', 'zs@test.com', 25, 1),
       ('李四', 'ls@test.com', 30, 1),
       ('王五', 'ww@test.com', 28, 1)

场景二：多条 INSERT 语句（需开启 allowMultiQueries）

另一种方式是生成多条独立的 INSERT 语句，需要在 JDBC URL 中添加 allowMultiQueries=true：

<insert id="batchInsertMulti">
    <foreach collection="users" item="user" separator=";">
        INSERT INTO t_user (username, email, age, status)
        VALUES (#{user.username}, #{user.email}, #{user.age}, #{user.status})
    </foreach>
</insert>

两种方式的对比

对比项 多行 VALUES（推荐） 多条 INSERT
SQL 条数 1 条 N 条
网络开销 一次请求 一次请求（但 SQL 文本更长）
执行效率 高（MySQL 引擎优化） 较低
SQL 长度限制 受 max_allowed_packet 限制 同样受限
事务控制 天然在一个事务内 需要手动管理事务
数据库兼容性 MySQL、PostgreSQL 等支持 大多数数据库支持

批量插入的性能优化策略

1. 控制单次批量大小

MySQL 的 max_allowed_packet 默认为 4MB（可配置），当数据量很大时，一次性拼接所有 VALUES 会导致 SQL 过长。推荐在业务层分批：

public void batchInsertUsers(List<User> users) {
    int batchSize = 500;
    List<List<User>> partitions = Lists.partition(users, batchSize);
    for (List<User> batch : partitions) {
        userMapper.batchInsert(batch);
    }
}

Lists.partition() 来自 Guava，也可以手动分片。

2. 使用 BATCH 模式的 Executor

对于超大批量（万级以上），可以使用 MyBatis 的 BATCH 执行器，复用 PreparedStatement：

try (SqlSession session = sqlSessionFactory.openSession(ExecutorType.BATCH)) {
    UserMapper mapper = session.getMapper(UserMapper.class);
    for (int i = 0; i < users.size(); i++) {
        mapper.insertOne(users.get(i));
        if ((i + 1) % 500 == 0) {
            session.flushStatements();
        }
    }
    session.flushStatements();
    session.commit();
}

3. 性能建议总结

数据量 推荐方案
< 100 条 多行 VALUES，一次搞定
100 ~ 5000 条 多行 VALUES，按 500 条分批
> 5000 条 BATCH Executor + 分批 flush
> 10 万条 考虑 LOAD DATA INFILE 或数据库原生导入工具

4.7 <bind> —— 创建 OGNL 变量

<bind> 标签允许你在 OGNL 上下文中创建一个变量，然后在后续 SQL 中引用。最典型的场景是模糊查询。

问题场景

模糊查询需要在参数两边拼接 %，常见的做法是用数据库函数 CONCAT：

<!-- 使用 CONCAT 的写法 -->
<select id="findByUsername" resultType="User">
    SELECT * FROM t_user
    WHERE username LIKE CONCAT('%', #{username}, '%')
</select>

这种写法有两个小问题：

  1. CONCAT 是 MySQL 函数，换数据库可能不兼容（Oracle 用 ||，SQL Server 用 +）
  2. 如果多处使用，每处都要写 CONCAT

使用 <bind> 的写法

Mapper 接口：

public interface UserMapper {
    List<User> findByUsernameLike(@Param("username") String username);
}

XML 映射：

<select id="findByUsernameLike" resultType="User">
    <bind name="usernamePattern" value="'%' + username + '%'" />
    SELECT id, username, email, age, status
    FROM t_user
    WHERE username LIKE #{usernamePattern}
</select>

<bind> 的属性

属性 说明
name 新创建的变量名，后续可通过 #{} 或 ${} 引用
value OGNL 表达式，用于计算变量的值

多个 bind 变量

<select id="searchByKeyword" resultType="User">
    <bind name="usernamePattern" value="'%' + keyword + '%'" />
    <bind name="emailPattern" value="'%' + keyword + '%'" />
    SELECT id, username, email, age, status
    FROM t_user
    <where>
        <if test="keyword != null and keyword != ''">
            username LIKE #{usernamePattern}
            OR email LIKE #{emailPattern}
        </if>
    </where>
</select>

两种方式对比

方式 优点 缺点
CONCAT('%', #{param}, '%') 直观易懂 依赖数据库函数，不同数据库语法不同
<bind> 数据库无关，可复用变量 需要额外一行声明

选择建议： 如果项目只使用 MySQL 且查询较简单，用 CONCAT 更直观；如果需要兼容多种数据库或同一参数多处使用，用 <bind> 更优雅。

4.8 动态 SQL 实战：通用条件查询模板

下面给出一个综合使用多种动态 SQL 标签的实战示例——一个通用的用户搜索功能。

业务场景

后台管理系统的用户查询页面，支持以下搜索条件：

  • 用户名（模糊匹配）
  • 邮箱（精确匹配）
  • 年龄范围（最小/最大）
  • 状态（精确匹配）
  • 部门（多选，IN 查询）
  • 创建时间范围
  • 排序字段和排序方向

查询条件对象

public class UserQuery {
    private String username;
    private String email;
    private Integer minAge;
    private Integer maxAge;
    private Integer status;
    private List<String> departments;
    private LocalDateTime createTimeStart;
    private LocalDateTime createTimeEnd;
    private String orderBy;      // 排序字段：username / create_time / age
    private String orderDir;     // 排序方向：ASC / DESC
    // getter/setter 省略
}

Mapper 接口

public interface UserMapper {
    List<User> searchByQuery(@Param("query") UserQuery query);
}

XML 映射——完整模板

<select id="searchByQuery" resultType="User">
    <!-- 使用 bind 预处理模糊查询参数 -->
    <if test="query.username != null and query.username != ''">
        <bind name="usernamePattern" value="'%' + query.username + '%'" />
    </if>

    SELECT id, username, email, age, status, department, create_time, update_time
    FROM t_user

    <!-- 使用 where 标签自动管理 WHERE 关键字和多余的 AND -->
    <where>
        <!-- 模糊查询：使用 bind 创建的变量 -->
        <if test="query.username != null and query.username != ''">
            AND username LIKE #{usernamePattern}
        </if>

        <!-- 精确匹配 -->
        <if test="query.email != null and query.email != ''">
            AND email = #{query.email}
        </if>

        <!-- 范围查询 -->
        <if test="query.minAge != null">
            AND age &gt;= #{query.minAge}
        </if>
        <if test="query.maxAge != null">
            AND age &lt;= #{query.maxAge}
        </if>

        <!-- 精确匹配 -->
        <if test="query.status != null">
            AND status = #{query.status}
        </if>

        <!-- IN 查询：使用 foreach 遍历部门列表 -->
        <if test="query.departments != null and query.departments.size() > 0">
            AND department IN
            <foreach collection="query.departments" item="dept" open="(" close=")" separator=",">
                #{dept}
            </foreach>
        </if>

        <!-- 时间范围 -->
        <if test="query.createTimeStart != null">
            AND create_time &gt;= #{query.createTimeStart}
        </if>
        <if test="query.createTimeEnd != null">
            AND create_time &lt;= #{query.createTimeEnd}
        </if>
    </where>

    <!-- 排序：使用 choose 设置默认排序 -->
    <choose>
        <when test="query.orderBy != null and query.orderBy != ''">
            ORDER BY ${query.orderBy}
            <choose>
                <when test="query.orderDir != null and query.orderDir == 'ASC'">ASC</when>
                <otherwise>DESC</otherwise>
            </choose>
        </when>
        <otherwise>
            ORDER BY create_time DESC
        </otherwise>
    </choose>
</select>

SQL 片段复用优化

如果多个查询都使用类似的条件，可以用 <sql> + <include> 抽取公共片段：

<sql id="userQueryCondition">
    <if test="query.username != null and query.username != ''">
        <bind name="usernamePattern" value="'%' + query.username + '%'" />
        AND username LIKE #{usernamePattern}
    </if>
    <if test="query.email != null and query.email != ''">
        AND email = #{query.email}
    </if>
    <if test="query.status != null">
        AND status = #{query.status}
    </if>
    <if test="query.departments != null and query.departments.size() > 0">
        AND department IN
        <foreach collection="query.departments" item="dept" open="(" close=")" separator=",">
            #{dept}
        </foreach>
    </if>
</sql>

<!-- 查询列表 -->
<select id="searchByQuery" resultType="User">
    SELECT id, username, email, age, status, department, create_time, update_time
    FROM t_user
    <where>
        <include refid="userQueryCondition" />
    </where>
    ORDER BY create_time DESC
</select>

<!-- 统计总数 -->
<select id="countByQuery" resultType="int">
    SELECT COUNT(*)
    FROM t_user
    <where>
        <include refid="userQueryCondition" />
    </where>
</select>

安全注意事项

上面排序字段使用了 ${query.orderBy}（字符串拼接），存在 SQL 注入风险。生产环境中必须做白名单校验：

private static final Set<String> ALLOWED_ORDER_COLUMNS =
    Set.of("username", "create_time", "age", "status");

public List<User> searchUsers(UserQuery query) {
    if (query.getOrderBy() != null
        && !ALLOWED_ORDER_COLUMNS.contains(query.getOrderBy())) {
        throw new IllegalArgumentException("非法排序字段: " + query.getOrderBy());
    }
    return userMapper.searchByQuery(query);
}

调用示例

UserQuery query = new UserQuery();
query.setUsername("张");
query.setMinAge(20);
query.setMaxAge(35);
query.setDepartments(Arrays.asList("技术部", "产品部"));
query.setOrderBy("create_time");
query.setOrderDir("DESC");

List<User> users = userMapper.searchByQuery(query);

生成的 SQL：

SELECT id, username, email, age, status, department, create_time, update_time
FROM t_user
WHERE username LIKE '%张%'
  AND age >= 20
  AND age <= 35
  AND department IN ('技术部', '产品部')
ORDER BY create_time DESC

本章小结

标签 作用 典型场景
<if> 条件判断 动态过滤条件
<choose>/<when>/<otherwise> 多分支选择（互斥） 按优先级查询
<where> 智能 WHERE 拼接 替代 WHERE 1=1
<set> 智能 SET 拼接 动态更新字段
<trim> 自定义前缀/后缀裁剪 where/set 的底层实现，也可独立使用
<foreach> 集合遍历 IN 查询、批量插入
<bind> 创建 OGNL 变量 模糊查询参数预处理

核心原则： 动态 SQL 的目标是让 XML 中的 SQL 尽量声明式——描述"要什么"而不是"怎么拼"。善用这些标签，可以写出既安全又可维护的数据访问层代码。
