第十章 Spring / Spring Boot 整合 ⭐

实际项目中几乎不会单独使用 MyBatis，而是通过 MyBatis-Spring 或 mybatis-spring-boot-starter 与 Spring 生态整合。本章是日常开发的标准使用姿势，务必牢固掌握。

10.1 MyBatis-Spring 核心组件

MyBatis-Spring 是 MyBatis 社区维护的一个桥梁模块，它的核心目标是：让 MyBatis 参与到 Spring 的依赖注入和事务管理体系中。

在纯 MyBatis 时代，我们需要手动管理 SqlSessionFactory、手动打开/关闭 SqlSession、手动提交/回滚事务。MyBatis-Spring 通过三个核心组件将这些工作全部交给 Spring 容器：

组件 职责
SqlSessionFactoryBean 在 Spring 中创建并配置 SqlSessionFactory
MapperFactoryBean / MapperScannerConfigurer 将 Mapper 接口注册为 Spring Bean
SqlSessionTemplate 线程安全的 SqlSession 实现，替代原生 DefaultSqlSession

10.1.1 SqlSessionFactoryBean

在纯 MyBatis 中，我们通过 SqlSessionFactoryBuilder 读取 mybatis-config.xml 来构建 SqlSessionFactory。而在 Spring 中，这个工作由 SqlSessionFactoryBean 接管。

作用

  • 接收 DataSource，创建 SqlSessionFactory
  • 加载 MyBatis 的全局配置（可以内联配置，也可以引用外部 mybatis-config.xml）
  • 扫描并加载 Mapper XML 映射文件
  • 注册 TypeHandler、TypeAlias、Plugin 等

Spring XML 配置完整示例

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="
           http://www.springframework.org/schema/beans
           http://www.springframework.org/schema/beans/spring-beans.xsd
           http://www.springframework.org/schema/context
           http://www.springframework.org/schema/context/spring-context.xsd">

    <!-- 1. 加载数据库配置文件 -->
    <context:property-placeholder location="classpath:db.properties"/>

    <!-- 2. 配置数据源 -->
    <bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource" destroy-method="close">
        <property name="driverClassName" value="${jdbc.driver}"/>
        <property name="jdbcUrl" value="${jdbc.url}"/>
        <property name="username" value="${jdbc.username}"/>
        <property name="password" value="${jdbc.password}"/>
        <property name="maximumPoolSize" value="20"/>
    </bean>

    <!-- 3. 配置 SqlSessionFactory -->
    <bean id="sqlSessionFactory" class="org.mybatis.spring.SqlSessionFactoryBean">
        <!-- 必须：注入数据源 -->
        <property name="dataSource" ref="dataSource"/>

        <!-- 可选：指定 MyBatis 全局配置文件（如果需要保留 mybatis-config.xml） -->
        <property name="configLocation" value="classpath:mybatis-config.xml"/>

        <!-- 可选：指定 Mapper XML 文件位置（支持通配符） -->
        <property name="mapperLocations" value="classpath:mapper/**/*.xml"/>

        <!-- 可选：类型别名包扫描 -->
        <property name="typeAliasesPackage" value="com.example.entity"/>

        <!-- 可选：内联 MyBatis 配置（与 configLocation 二选一，或配合使用） -->
        <property name="configuration">
            <bean class="org.apache.ibatis.session.Configuration">
                <property name="mapUnderscoreToCamelCase" value="true"/>
                <property name="cacheEnabled" value="true"/>
                <property name="lazyLoadingEnabled" value="false"/>
            </bean>
        </property>

        <!-- 可选：注册插件 -->
        <property name="plugins">
            <array>
                <bean class="com.github.pagehelper.PageInterceptor">
                    <property name="properties">
                        <value>helperDialect=mysql</value>
                    </property>
                </bean>
            </array>
        </property>
    </bean>

</beans>

注意：configLocation（外部配置文件）与 configuration（内联配置对象）不能同时设置，否则会抛出异常。如果你已有 mybatis-config.xml 且不想改动，使用 configLocation；如果想全部由 Spring 管理，用 configuration 内联。

常用属性一览

属性 说明
dataSource 必须。注入数据源
configLocation MyBatis 全局配置文件路径
mapperLocations Mapper XML 文件路径（支持 Ant 通配符）
typeAliasesPackage 类型别名扫描包路径
typeHandlersPackage TypeHandler 扫描包路径
plugins 插件数组
configuration 内联 MyBatis Configuration 对象
databaseIdProvider 数据库厂商标识提供者

10.1.2 MapperFactoryBean / MapperScannerConfigurer

有了 SqlSessionFactory，下一步就是把 Mapper 接口变成 Spring Bean，这样才能在 Service 中通过 @Autowired 注入使用。

MapperFactoryBean —— 逐个注册

MapperFactoryBean 是最基本的注册方式，每个 Mapper 接口需要单独配置一个 Bean 定义：

<!-- 逐个注册 Mapper（Mapper 较少时可用） -->
<bean id="userMapper" class="org.mybatis.spring.mapper.MapperFactoryBean">
    <property name="mapperInterface" value="com.example.mapper.UserMapper"/>
    <property name="sqlSessionFactory" ref="sqlSessionFactory"/>
</bean>

<bean id="orderMapper" class="org.mybatis.spring.mapper.MapperFactoryBean">
    <property name="mapperInterface" value="com.example.mapper.OrderMapper"/>
    <property name="sqlSessionFactory" ref="sqlSessionFactory"/>
</bean>

这种方式的缺点很明显：Mapper 接口数量多时，配置冗长且难以维护。

MapperScannerConfigurer —— 批量扫描（推荐）

MapperScannerConfigurer 可以自动扫描指定包下的所有 Mapper 接口，一次性注册为 Spring Bean：

<!-- 批量扫描注册 Mapper（推荐方式） -->
<bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
    <!-- 指定 Mapper 接口所在的包（多个包用逗号分隔） -->
    <property name="basePackage" value="com.example.mapper"/>

    <!-- 可选：指定 SqlSessionFactory 的 Bean 名称 -->
    <property name="sqlSessionFactoryBeanName" value="sqlSessionFactory"/>

    <!-- 可选：只扫描带有特定注解的接口 -->
    <property name="annotationClass" value="org.apache.ibatis.annotations.Mapper"/>

    <!-- 可选：只扫描继承了特定接口的子接口 -->
    <!-- <property name="markerInterface" value="com.example.mapper.BaseMapper"/> -->
</bean>

完整的 Spring XML 配置（SqlSessionFactoryBean + MapperScannerConfigurer）

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xmlns:context="http://www.springframework.org/schema/context"
       xsi:schemaLocation="
           http://www.springframework.org/schema/beans
           http://www.springframework.org/schema/beans/spring-beans.xsd
           http://www.springframework.org/schema/context
           http://www.springframework.org/schema/context/spring-context.xsd">

    <context:property-placeholder location="classpath:db.properties"/>

    <!-- 数据源 -->
    <bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource" destroy-method="close">
        <property name="driverClassName" value="${jdbc.driver}"/>
        <property name="jdbcUrl" value="${jdbc.url}"/>
        <property name="username" value="${jdbc.username}"/>
        <property name="password" value="${jdbc.password}"/>
    </bean>

    <!-- SqlSessionFactory -->
    <bean id="sqlSessionFactory" class="org.mybatis.spring.SqlSessionFactoryBean">
        <property name="dataSource" ref="dataSource"/>
        <property name="mapperLocations" value="classpath:mapper/**/*.xml"/>
        <property name="typeAliasesPackage" value="com.example.entity"/>
        <property name="configuration">
            <bean class="org.apache.ibatis.session.Configuration">
                <property name="mapUnderscoreToCamelCase" value="true"/>
            </bean>
        </property>
    </bean>

    <!-- 事务管理器 -->
    <bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
        <property name="dataSource" ref="dataSource"/>
    </bean>

    <!-- Mapper 扫描 -->
    <bean class="org.mybatis.spring.mapper.MapperScannerConfigurer">
        <property name="basePackage" value="com.example.mapper"/>
        <property name="sqlSessionFactoryBeanName" value="sqlSessionFactory"/>
    </bean>

</beans>

实际效果：MapperScannerConfigurer 扫描到 com.example.mapper 包下的所有接口后，为每个接口创建一个 MapperFactoryBean 实例并注册到 Spring 容器。Service 层就可以直接 @Autowired UserMapper userMapper 使用了。

10.1.3 SqlSessionTemplate —— 线程安全的 SqlSession

问题背景

回顾第一章的知识：原生 MyBatis 的 DefaultSqlSession 不是线程安全的，每次使用都需要手动 openSession() / close()。但在 Spring 的 Web 应用中，Service 是单例 Bean，它内部注入的 Mapper 也是单例的，多个请求线程会并发访问同一个 Mapper 实例。这就需要一个线程安全的 SqlSession 实现。

SqlSessionTemplate 的角色

SqlSessionTemplate 是 MyBatis-Spring 提供的 SqlSession 实现，它是线程安全的，可以在多个线程间共享。实际上，当你通过 MapperFactoryBean 或 MapperScannerConfigurer 注册 Mapper 时，底层使用的就是 SqlSessionTemplate。

<!-- 通常不需要手动配置，MapperFactoryBean 内部会自动创建 -->
<!-- 但如果你需要在 DAO 中直接使用 SqlSession，可以显式配置 -->
<bean id="sqlSessionTemplate" class="org.mybatis.spring.SqlSessionTemplate">
    <constructor-arg index="0" ref="sqlSessionFactory"/>
    <!-- 可选：指定执行器类型（SIMPLE, REUSE, BATCH） -->
    <!-- <constructor-arg index="1" value="BATCH"/> -->
</bean>

线程安全的实现原理

SqlSessionTemplate 的线程安全性不是通过加锁实现的，而是通过 JDK 动态代理 + ThreadLocal 的方式实现的。核心机制如下：

1. 代理模式入口

SqlSessionTemplate 的构造方法中创建了一个 SqlSession 的 JDK 动态代理对象 sqlSessionProxy：

public SqlSessionTemplate(SqlSessionFactory sqlSessionFactory, ExecutorType executorType,
    PersistenceExceptionTranslator exceptionTranslator) {

    this.sqlSessionFactory = sqlSessionFactory;
    this.executorType = executorType;
    this.exceptionTranslator = exceptionTranslator;

    // 创建代理对象 —— 所有 SqlSession 方法调用都会经过 SqlSessionInterceptor
    this.sqlSessionProxy = (SqlSession) newProxyInstance(
        SqlSessionFactory.class.getClassLoader(),
        new Class[] { SqlSession.class },
        new SqlSessionInterceptor());
}

当调用 SqlSessionTemplate 的任何方法（如 selectOne()、insert() 等），实际上都委托给了 sqlSessionProxy，而代理的拦截逻辑在 SqlSessionInterceptor 中。

2. SqlSessionInterceptor 的工作流程

private class SqlSessionInterceptor implements InvocationHandler {
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 关键步骤 1：从 Spring 事务上下文中获取 SqlSession
        // 内部通过 TransactionSynchronizationManager（基于 ThreadLocal）实现
        SqlSession sqlSession = getSqlSession(
            SqlSessionTemplate.this.sqlSessionFactory,
            SqlSessionTemplate.this.executorType,
            SqlSessionTemplate.this.exceptionTranslator);

        try {
            // 关键步骤 2：用当前线程专属的 SqlSession 执行操作
            Object result = method.invoke(sqlSession, args);

            // 关键步骤 3：如果不在 Spring 事务中，立即提交
            if (!isSqlSessionTransactional(sqlSession, SqlSessionTemplate.this.sqlSessionFactory)) {
                sqlSession.commit(true);
            }
            return result;
        } catch (Throwable t) {
            // 异常处理...
            throw t;
        } finally {
            // 关键步骤 4：关闭/释放 SqlSession
            if (sqlSession != null) {
                closeSqlSession(sqlSession, SqlSessionTemplate.this.sqlSessionFactory);
            }
        }
    }
}

3. getSqlSession() 的 ThreadLocal 机制

getSqlSession() 方法内部调用了 Spring 的 TransactionSynchronizationManager，该管理器使用 ThreadLocal 维护每个线程的资源绑定：

线程 A 调用 Mapper 方法
    → SqlSessionInterceptor.invoke()
    → getSqlSession()
    → TransactionSynchronizationManager.getResource(sqlSessionFactory)
    → 从线程 A 的 ThreadLocal 中获取/创建 SqlSession A
    → 使用 SqlSession A 执行 SQL

线程 B 同时调用同一个 Mapper 方法
    → SqlSessionInterceptor.invoke()
    → getSqlSession()
    → TransactionSynchronizationManager.getResource(sqlSessionFactory)
    → 从线程 B 的 ThreadLocal 中获取/创建 SqlSession B
    → 使用 SqlSession B 执行 SQL

核心结论

要点 说明
外观 SqlSessionTemplate 是一个全局单例，可以被多个线程共享
内部 每次方法调用都通过代理拦截，从当前线程的 ThreadLocal 中获取/创建真正的 SqlSession
事务内 同一个事务中的多次 Mapper 调用使用同一个 SqlSession（由 Spring 事务管理器绑定）
事务外 每次 Mapper 调用都会创建新的 SqlSession，执行完立即关闭
开发者无感 开发者只需要 @Autowired Mapper 接口，线程安全由框架保证

一句话总结：SqlSessionTemplate 线程安全的秘密在于——它本身不持有任何 SqlSession 状态，而是每次操作都通过动态代理从 Spring 的 TransactionSynchronizationManager（ThreadLocal）中获取当前线程专属的 SqlSession 实例。

10.2 Spring Boot 自动配置（mybatis-spring-boot-starter）

Spring Boot 的出现让 MyBatis 整合变得极其简单。mybatis-spring-boot-starter 通过自动配置帮你完成了 10.1 节中所有手动配置的工作。

只需一个依赖：

<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>3.0.3</version>
</dependency>

它自动完成的事情：

  1. 检测 DataSource Bean，自动创建 SqlSessionFactory
  2. 自动创建 SqlSessionTemplate
  3. 自动扫描带 @Mapper 注解的接口并注册为 Spring Bean

10.2.2 Mapper 扫描 —— @MapperScan vs @Mapper

在 Spring Boot 中注册 Mapper 有两种方式：@Mapper（标注在每个接口上）和 @MapperScan（标注在配置类上）。

@Mapper —— 逐个标注

import org.apache.ibatis.annotations.Mapper;

@Mapper
public interface UserMapper {
    User findById(Long id);
    List<User> findAll();
}

@Mapper
public interface OrderMapper {
    Order findById(Long id);
}

每个 Mapper 接口上都加 @Mapper 注解，mybatis-spring-boot-starter 的自动配置会扫描所有带此注解的接口。

@MapperScan —— 包级别扫描

import org.mybatis.spring.annotation.MapperScan;

@SpringBootApplication
@MapperScan("com.example.mapper")
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

一个注解搞定整个包下的所有 Mapper，无需在每个接口上加 @Mapper。

对比

维度 @Mapper @MapperScan
作用位置 标注在每个 Mapper 接口上 标注在配置类（通常是启动类）上
扫描范围 仅标注了 @Mapper 的接口 指定包下的所有接口
侵入性 每个接口都需要加注解 接口无需任何注解
适用场景 Mapper 数量少；不同包的 Mapper 用不同数据源（不想全扫） 项目中绝大多数场景（推荐）
多包支持 天然支持（只要加了注解就行） @MapperScan({"com.example.mapper", "com.example.other"})
精确控制 可精确控制哪些接口是 Mapper 可能误扫非 Mapper 接口（可用 annotationClass 过滤）
与 IDE 的配合 IDE 能直接识别为 MyBatis Mapper 部分 IDE 可能出现 "Could not autowire" 误报警告

实践建议

场景 推荐方案
单数据源、Mapper 集中在一个包 @MapperScan —— 最简洁
多数据源、不同包对应不同数据源 @MapperScan 分别指定不同包（见 10.2.3）
团队规范要求每个 Mapper 显式标识 @Mapper 逐个标注
混合使用 完全可以，两者不冲突

IDE 警告处理：使用 @MapperScan 时，IntelliJ IDEA 可能在 @Autowired Mapper 处显示红色波浪线（"Could not autowire"），这是 IDE 的误报。解决方式：

  1. 在 Mapper 接口上额外加 @Repository 注解
  2. 使用 @Resource 代替 @Autowired
  3. 在 IDEA 设置中降低该检查的级别
  4. 安装 MyBatis 插件（如 MyBatisX）

10.2.3 多数据源配置

实际项目中经常需要连接多个数据库（如主库 + 从库、业务库 + 日志库）。Spring Boot 没有开箱即用的多数据源支持，需要手动配置。

核心思路

为每个数据源分别配置一套完整的组件链：

DataSource → SqlSessionFactory → SqlSessionTemplate → MapperScan

通过 @MapperScan 的 basePackages 指定不同包路径，让不同包下的 Mapper 绑定到不同的数据源。

项目结构

src/main/java/com/example/
├── config/
│   ├── PrimaryDataSourceConfig.java    # 主数据源配置
│   └── SecondaryDataSourceConfig.java  # 第二数据源配置
├── mapper/
│   ├── primary/                        # 主库 Mapper
│   │   ├── UserMapper.java
│   │   └── OrderMapper.java
│   └── secondary/                      # 第二库 Mapper
│       ├── LogMapper.java
│       └── StatMapper.java
└── Application.java

application.yml

  datasource:
    primary:
      url: jdbc:mysql://localhost:3306/main_db?useSSL=false&serverTimezone=Asia/Shanghai
      username: root
      password: 123456
      driver-class-name: com.mysql.cj.jdbc.Driver
    secondary:

主数据源配置类

package com.example.config;

import org.apache.ibatis.session.SqlSessionFactory;
import org.mybatis.spring.SqlSessionFactoryBean;
import org.mybatis.spring.SqlSessionTemplate;
import org.mybatis.spring.annotation.MapperScan;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.boot.jdbc.DataSourceBuilder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;
import org.springframework.core.io.support.PathMatchingResourcePatternResolver;
import org.springframework.jdbc.datasource.DataSourceTransactionManager;

import javax.sql.DataSource;

@Configuration
@MapperScan(
    basePackages = "com.example.mapper.primary",
    sqlSessionFactoryRef = "primarySqlSessionFactory"
)
public class PrimaryDataSourceConfig {

    @Primary
    @Bean("primaryDataSource")
    @ConfigurationProperties(prefix = "spring.datasource.primary")
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Primary
    @Bean("primarySqlSessionFactory")
    public SqlSessionFactory primarySqlSessionFactory(
            @Qualifier("primaryDataSource") DataSource dataSource) throws Exception {

        SqlSessionFactoryBean factoryBean = new SqlSessionFactoryBean();
        factoryBean.setDataSource(dataSource);
        factoryBean.setMapperLocations(
            new PathMatchingResourcePatternResolver()
                .getResources("classpath:mapper/primary/**/*.xml"));

        org.apache.ibatis.session.Configuration configuration =
            new org.apache.ibatis.session.Configuration();
        configuration.setMapUnderscoreToCamelCase(true);
        factoryBean.setConfiguration(configuration);

        return factoryBean.getObject();
    }

    @Primary
    @Bean("primarySqlSessionTemplate")
    public SqlSessionTemplate primarySqlSessionTemplate(
            @Qualifier("primarySqlSessionFactory") SqlSessionFactory sqlSessionFactory) {
        return new SqlSessionTemplate(sqlSessionFactory);
    }

    @Primary
    @Bean("primaryTransactionManager")
    public DataSourceTransactionManager primaryTransactionManager(
            @Qualifier("primaryDataSource") DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}

第二数据源配置类

import org.mybatis.spring.SqlSessionFactoryBean;

import org.mybatis.spring.annotation.MapperScan;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.core.io.support.PathMatchingResourcePatternResolver;
import org.springframework.jdbc.datasource.DataSourceTransactionManager;

import javax.sql.DataSource;

@Configuration
@MapperScan(
    basePackages = "com.example.mapper.secondary",
    sqlSessionFactoryRef = "secondarySqlSessionFactory"
)
public class SecondaryDataSourceConfig {

    @Bean("secondaryDataSource")
    @ConfigurationProperties(prefix = "spring.datasource.secondary")
    public DataSource secondaryDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean("secondarySqlSessionFactory")
    public SqlSessionFactory secondarySqlSessionFactory(
            @Qualifier("secondaryDataSource") DataSource dataSource) throws Exception {

        SqlSessionFactoryBean factoryBean = new SqlSessionFactoryBean();
        factoryBean.setDataSource(dataSource);
        factoryBean.setMapperLocations(
            new PathMatchingResourcePatternResolver()
                .getResources("classpath:mapper/secondary/**/*.xml"));

        org.apache.ibatis.session.Configuration configuration =
            new org.apache.ibatis.session.Configuration();
        configuration.setMapUnderscoreToCamelCase(true);
        factoryBean.setConfiguration(configuration);

        return factoryBean.getObject();
    }

    @Bean("secondarySqlSessionTemplate")
    public SqlSessionTemplate secondarySqlSessionTemplate(
            @Qualifier("secondarySqlSessionFactory") SqlSessionFactory sqlSessionFactory) {
        return new SqlSessionTemplate(sqlSessionFactory);
    }

    @Bean("secondaryTransactionManager")
    public DataSourceTransactionManager secondaryTransactionManager(
            @Qualifier("secondaryDataSource") DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}

启动类

@SpringBootApplication
// 注意：使用多数据源时，不要在启动类上加 @MapperScan，

使用时的注意事项

要点 说明
@Primary 主数据源的所有 Bean 都要标 @Primary，否则 Spring 无法确定默认注入哪个
@Qualifier 通过名称限定注入，避免歧义
事务管理器 每个数据源有独立的事务管理器，@Transactional 需指定 transactionManager 名称
Mapper XML 路径 每个 SqlSessionFactory 只加载自己数据源对应的 XML 文件
排除自动配置 如果遇到冲突，可排除默认的 DataSource 自动配置 
// 在使用非主数据源的 Service 中指定事务管理器
@Transactional(transactionManager = "secondaryTransactionManager")
public void saveLog(LogEntity log) {
    logMapper.insert(log);
}

替代方案：如果觉得这种配置太繁琐，可以考虑使用 dynamic-datasource-spring-boot-starter（苞米豆出品）或 AbstractRoutingDataSource（Spring 原生），它们提供了更优雅的动态数据源切换方案。

10.3 事务管理

10.3.1 Spring 声明式事务与 MyBatis 的协作

纯 MyBatis 的事务管理

在纯 MyBatis 中，事务由 SqlSession 管理：

SqlSession session = sqlSessionFactory.openSession(); // 默认不自动提交
try {
    UserMapper mapper = session.getMapper(UserMapper.class);
    mapper.insert(user);
    mapper.update(order);
    session.commit();   // 手动提交
} catch (Exception e) {
    session.rollback();  // 手动回滚
} finally {
    session.close();     // 手动关闭
}

问题很明显：事务管理代码和业务代码耦合在一起，且容易遗漏关闭或回滚。

Spring 整合后的事务管理

整合 Spring 后，事务管理交给 Spring 的 DataSourceTransactionManager，开发者只需在 Service 方法上加 @Transactional 即可：

@Service
public class OrderService {

    @Autowired
    private UserMapper userMapper;

    @Autowired
    private OrderMapper orderMapper;

    @Transactional
    public void createOrder(User user, Order order) {
        userMapper.updateBalance(user.getId(), order.getAmount());
        orderMapper.insert(order);
        // 方法正常结束 → 自动提交
        // 抛出 RuntimeException → 自动回滚
    }
}

协作原理

Spring 事务管理器与 MyBatis SqlSession 的协作关系是理解整合的关键：

@Transactional 方法调用开始
    │
    ▼
TransactionInterceptor（AOP 代理拦截）
    │
    ▼
DataSourceTransactionManager.doBegin()
    ├── 从 DataSource 获取 Connection
    ├── 设置 Connection.setAutoCommit(false)
    └── 将 Connection 绑定到 ThreadLocal
            （TransactionSynchronizationManager）
    │
    ▼
业务代码执行：userMapper.updateBalance(...)
    │
    ▼
SqlSessionTemplate → SqlSessionInterceptor.invoke()
    ├── getSqlSession()
    │       └── 发现当前线程存在 Spring 事务
    │       └── 创建 SqlSession，使用事务中已绑定的 Connection
    │       └── 将 SqlSession 也绑定到 ThreadLocal
    ├── 执行 SQL（使用事务中的 Connection）
    └── 不提交（因为在 Spring 事务中，提交权交给事务管理器）
    │
    ▼
业务代码执行：orderMapper.insert(...)
    │
    ▼
SqlSessionTemplate → SqlSessionInterceptor.invoke()
    ├── getSqlSession()
    │       └── 从 ThreadLocal 获取同一个 SqlSession（复用！）
    ├── 执行 SQL（使用同一个 Connection）
    └── 不提交
    │
    ▼
@Transactional 方法正常结束
    │
    ▼
DataSourceTransactionManager.doCommit()
    ├── Connection.commit()     ← 两条 SQL 作为一个事务提交
    ├── 关闭 SqlSession
    └── 释放 Connection

关键协作机制

环节 详细说明
Connection 共享 Spring 事务管理器在事务开始时获取 Connection 并通过 TransactionSynchronizationManager 绑定到 ThreadLocal。MyBatis 的 SpringManagedTransaction 从同一个 ThreadLocal 获取 Connection，保证了 MyBatis 使用的是 Spring 事务管理的那个连接
SqlSession 复用 同一个事务中多次 Mapper 调用共享同一个 SqlSession，这由 SqlSessionUtils.getSqlSession() 中的 ThreadLocal 绑定逻辑保证
提交权转让 在 Spring 事务内，SqlSessionTemplate 不会自行提交或回滚，而是将控制权完全交给 Spring 的 DataSourceTransactionManager。事务的提交/回滚由 Spring AOP 在方法执行结束后统一处理
异常触发回滚 默认情况下，RuntimeException 和 Error 触发回滚，checked Exception 不触发（可通过 rollbackFor 配置）

核心要点：Spring 和 MyBatis 能协作的根基是共享同一个 Connection。Spring 事务管理器和 MyBatis 的 SpringManagedTransaction 通过 TransactionSynchronizationManager（ThreadLocal）共享同一个数据库连接，从而实现统一的事务控制。

10.3.2 @Transactional 常见失效场景

@Transactional 是 Spring 中使用最广泛的注解之一，但它的失效场景也是面试和实际开发中的高频坑点。以下逐一分析：

场景一：方法不是 public 的

@Service
public class UserService {

    @Transactional
    protected void updateUser(User user) {  // ❌ 事务不生效
        userMapper.update(user);
    }
}

原因：Spring 的事务代理（无论是 JDK 动态代理还是 CGLIB 代理）默认只对 public 方法生效。protected、private、default 修饰的方法上的 @Transactional 会被忽略。

解决：将方法改为 public。

Spring Framework 6.0+ 在非 public 方法上使用 @Transactional 会打印警告日志，但仍不会生效。

场景二：同类内部方法调用（自调用问题）

@Service
public class OrderService {

    public void createOrder(Order order) {
        // 直接通过 this 调用，不会经过代理
        this.saveOrder(order);  // ❌ 事务不生效
    }

    @Transactional
    public void saveOrder(Order order) {
        orderMapper.insert(order);
        // 即使这里抛异常，也不会回滚
    }
}

原因：@Transactional 基于 AOP 代理实现。外部调用 orderService.saveOrder() 时，调用的是代理对象的方法，代理会拦截并开启事务。但 createOrder() 内部通过 this.saveOrder() 调用时，this 指向的是原始对象而非代理对象，因此绕过了事务拦截器。

解决方案：

// 方案 1：注入自身代理
@Service
public class OrderService {

    @Autowired
    private OrderService self;  // Spring 4.3+ 支持注入自身代理

    public void createOrder(Order order) {
        self.saveOrder(order);  // ✅ 通过代理调用
    }

    @Transactional
    public void saveOrder(Order order) {
        orderMapper.insert(order);
    }
}

// 方案 2：从 ApplicationContext 获取代理
@Service
public class OrderService {

    @Autowired
    private ApplicationContext applicationContext;

    public void createOrder(Order order) {
        applicationContext.getBean(OrderService.class).saveOrder(order);
    }

    @Transactional
    public void saveOrder(Order order) {
        orderMapper.insert(order);
    }
}

// 方案 3：将事务方法抽取到另一个 Service（最推荐）
@Service
public class OrderService {

    @Autowired
    private OrderTransactionService txService;

    public void createOrder(Order order) {
        txService.saveOrder(order);  // ✅ 跨类调用，走代理
    }
}

@Service
public class OrderTransactionService {

    @Transactional
    public void saveOrder(Order order) {
        orderMapper.insert(order);
    }
}

场景三：异常类型不匹配

@Service
public class UserService {

    @Transactional  // 默认只回滚 RuntimeException 和 Error
    public void importUsers(List<User> users) throws Exception {
        for (User user : users) {
            userMapper.insert(user);
        }
        // 抛出 checked exception
        throw new IOException("文件读取失败");  // ❌ 事务不会回滚！
    }
}

原因：@Transactional 默认的回滚规则是：

  • RuntimeException（及其子类）→ 回滚
  • Error（及其子类）→ 回滚
  • checked Exception（如 IOException、SQLException）→ 不回滚，正常提交

解决：

// 方案 1：指定 rollbackFor（推荐）
@Transactional(rollbackFor = Exception.class)
public void importUsers(List<User> users) throws Exception {
    // 现在任何异常都会回滚
}

// 方案 2：将 checked exception 包装为 RuntimeException
@Transactional
public void importUsers(List<User> users) {
    try {
        // ...
    } catch (IOException e) {
        throw new RuntimeException("导入失败", e);
    }
}

最佳实践：团队规范中建议统一使用 @Transactional(rollbackFor = Exception.class)，避免遗漏 checked exception 导致数据不一致。

场景四：异常被 catch 吞掉

@Service
public class UserService {

    @Transactional
    public void batchUpdate(List<User> users) {
        try {
            for (User user : users) {
                userMapper.update(user);
            }
            int x = 1 / 0;  // 会抛 ArithmeticException
        } catch (Exception e) {
            log.error("更新失败", e);
            // ❌ 异常被 catch 了，事务管理器感知不到异常，不会回滚
        }
    }
}

原因：Spring 事务管理器通过捕获方法抛出的异常来决定是否回滚。如果异常在方法内部被 catch 吞掉了，方法正常返回，事务管理器认为执行成功，于是提交事务。

解决：

@Transactional
public void batchUpdate(List<User> users) {
    try {
        for (User user : users) {
            userMapper.update(user);
        }
        int x = 1 / 0;
    } catch (Exception e) {
        log.error("更新失败", e);
        throw e;  // ✅ 重新抛出，让事务管理器感知
    }
}

// 或者使用编程式事务回滚
@Transactional
public void batchUpdate(List<User> users) {
    try {
        // ...
    } catch (Exception e) {
        log.error("更新失败", e);
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();  // ✅ 手动标记回滚
    }
}

场景五：propagation 传播行为配置不当

@Service
public class OuterService {

    @Autowired
    private InnerService innerService;

    @Transactional
    public void outerMethod() {
        // 外层事务
        userMapper.insert(user);

        try {
            innerService.innerMethod();  // 内层方法抛异常
        } catch (Exception e) {
            log.error("内层失败，但外层想继续提交");
            // ❌ 外层事务仍然会回滚！
        }
    }
}

@Service
public class InnerService {

    @Transactional  // 默认 REQUIRED，加入外层事务
    public void innerMethod() {
        orderMapper.insert(order);
        throw new RuntimeException("内层失败");
    }
}

原因：默认传播行为 REQUIRED 会让内层方法加入外层事务。内层抛异常时，事务已被标记为 rollback-only。即使外层 catch 了异常，在提交时事务管理器发现 rollback-only 标记，仍然会回滚整个事务，并抛出 UnexpectedRollbackException。

解决：如果内层失败不应影响外层，使用 REQUIRES_NEW 让内层开启独立事务：

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void innerMethod() {
    orderMapper.insert(order);
    throw new RuntimeException("内层失败");
    // 只回滚内层事务，外层事务不受影响
}

场景六：数据库引擎不支持事务

@Transactional
public void updateUser(User user) {
    userMapper.update(user);  // ❌ 如果表使用 MyISAM 引擎，事务无效
}

原因：事务是数据库层面的能力。MySQL 的 MyISAM 引擎不支持事务，即使 Spring 发起了事务管理，数据库也不会执行真正的事务操作（每条 SQL 立即生效，无法回滚）。

解决：确保使用支持事务的存储引擎。MySQL 5.5+ 的默认引擎是 InnoDB（支持事务），如果是老项目需要检查：

-- 查看表的存储引擎
SHOW TABLE STATUS WHERE Name = 'user';

-- 修改为 InnoDB
ALTER TABLE user ENGINE = InnoDB;

场景七：未被 Spring 管理的类

// ❌ 没有 @Service 或 @Component，不是 Spring Bean
public class UserService {

    @Transactional
    public void updateUser(User user) {
        // 事务不生效，因为这个类不在 Spring 容器中
    }
}

原因：@Transactional 依赖 Spring AOP 代理。如果类不是 Spring Bean（没有被 @Service、@Component 等注解标注，或者是通过 new 创建的实例），Spring 无法为其创建代理，@Transactional 自然不生效。

解决：确保类被 Spring 管理（加上 @Service、@Component 等注解）。

场景八：多线程环境

@Service
public class AsyncService {

    @Transactional
    public void asyncBatchInsert(List<User> users) {
        // 主线程开启事务
        users.parallelStream().forEach(user -> {
            userMapper.insert(user);  // ❌ 子线程拿不到主线程的事务
        });
    }
}

原因：Spring 事务基于 ThreadLocal 实现，事务上下文（Connection、SqlSession）绑定在当前线程。子线程无法获取主线程的 ThreadLocal 数据，因此子线程中的数据库操作在事务管理之外。

解决：避免在 @Transactional 方法中使用多线程操作数据库。如果确实需要并发，每个子线程应独立管理自己的事务，或使用分布式事务方案。

失效场景速查表

# 失效场景 根因 解决方案
1 非 public 方法 Spring AOP 只代理 public 方法 改为 public
2 同类内部调用 绕过了代理对象 注入自身 / 抽取到另一个 Service
3 异常类型不匹配 默认只回滚 RuntimeException rollbackFor = Exception.class
4 异常被 catch 吞掉 事务管理器感知不到异常 重新 throw 或手动 setRollbackOnly()
5 propagation 不当 REQUIRED 下内层异常标记整个事务回滚 按需使用 REQUIRES_NEW
6 数据库不支持事务 底层引擎（如 MyISAM）无事务能力 使用 InnoDB
7 类未被 Spring 管理 非 Spring Bean，无法生成代理 加 @Service / @Component
8 多线程 事务基于 ThreadLocal，子线程不共享 避免跨线程事务操作

10.4 与 Spring 整合后缓存行为的变化

MyBatis 的一级缓存和二级缓存在与 Spring 整合后，行为会发生显著变化。这是一个容易被忽略但非常重要的知识点。

一级缓存基本失效

回顾：一级缓存的前提

一级缓存是 SqlSession 级别的缓存。它的生效前提是：在同一个 SqlSession 中执行相同的查询。

在纯 MyBatis 环境中，开发者手动控制 SqlSession 的生命周期：

try (SqlSession session = sqlSessionFactory.openSession()) {
    UserMapper mapper = session.getMapper(UserMapper.class);
    User user1 = mapper.findById(1L);  // 第一次查询，走数据库
    User user2 = mapper.findById(1L);  // 第二次查询，命中一级缓存 ✅
    // user1 == user2 （同一个对象引用）
}

同一个 SqlSession 内两次查询命中缓存，只执行了一次 SQL。

Spring 整合后的行为

整合 Spring 后，SqlSession 的生命周期由 SqlSessionTemplate 管理：

情况一：没有 @Transactional（最常见的场景）

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    // 没有 @Transactional
    public void queryTwice() {
        User user1 = userMapper.findById(1L);  // 创建 SqlSession A，执行 SQL，关闭 SqlSession A
        User user2 = userMapper.findById(1L);  // 创建 SqlSession B，执行 SQL，关闭 SqlSession B
        // 两次查询都走了数据库，一级缓存没有生效 ❌
    }
}

原因：没有事务的情况下，SqlSessionTemplate 每次 Mapper 调用都会创建一个新的 SqlSession，执行完立即关闭。两次查询使用的是不同的 SqlSession，一级缓存自然无法命中。

情况二：有 @Transactional

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    @Transactional
    public void queryTwice() {
        User user1 = userMapper.findById(1L);  // 使用事务绑定的 SqlSession，执行 SQL
        User user2 = userMapper.findById(1L);  // 复用同一个 SqlSession，命中一级缓存 ✅
        // 只执行了一次 SQL
    }
}

原因：在 @Transactional 范围内，Spring 事务管理器将 SqlSession 绑定到 ThreadLocal，事务内的所有 Mapper 调用复用同一个 SqlSession，因此一级缓存生效。

总结：一级缓存在 Spring 整合后的表现

场景 SqlSession 生命周期 一级缓存
纯 MyBatis 开发者手动管理（try-with-resources） 在 SqlSession 关闭前有效
Spring 无事务 每次 Mapper 调用创建/关闭一个 SqlSession 基本失效
Spring 有事务 事务范围内共享同一个 SqlSession 在事务范围内有效

为什么说"基本失效"：在实际项目中，大部分查询操作（特别是 Controller 直接调 Service 的简单查询）不会加 @Transactional，因此一级缓存在大多数场景下不起作用。这也是为什么很多团队直接将一级缓存的作用域设置为 STATEMENT（每次执行完就清空），明确放弃一级缓存：

mybatis:
  configuration:
    local-cache-scope: STATEMENT  # 每次查询后立即清空一级缓存

二级缓存不受影响

二级缓存是 namespace（Mapper）级别的，不依赖特定的 SqlSession 实例。它存储在 SqlSessionFactory 的全局空间中，跨 SqlSession 共享。因此，二级缓存在 Spring 整合后行为不变。

但需要注意的是：

  • 在 @Transactional 事务提交后，事务中的查询结果才会写入二级缓存
  • 事务回滚时，不会写入二级缓存
  • 生产环境是否使用二级缓存需要谨慎评估（参见第七章 7.4 节）

实践建议

建议 说明
不要依赖一级缓存 Spring 整合后一级缓存在无事务场景下失效，不要基于它做性能优化假设
设置 local-cache-scope: STATEMENT 在配置中明确关闭一级缓存，避免在事务内出现脏读的意外行为
谨慎使用二级缓存 适合读多写少、对实时性要求不高的数据。对一致性要求高的场景，使用分布式缓存（如 Redis）
缓存策略外置 推荐在 Service 层或专门的缓存层使用 Spring Cache（@Cacheable）配合 Redis，而不是依赖 MyBatis 内置缓存
