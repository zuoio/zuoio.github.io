第三章 Mapper XML 映射文件 ⭐

核心必读 —— 这是日常开发的核心战场，必须烂熟于心。
Mapper XML 是 MyBatis 中编写 SQL 的主要载体，掌握它就掌握了 MyBatis 80% 的日常工作。

本章我们将以一个统一的示例贯穿始终。假设数据库中有如下表：

CREATE TABLE sys_user (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    username    VARCHAR(50)  NOT NULL,
    email       VARCHAR(100),
    age         INT,
    dept_id     BIGINT,
    status      TINYINT DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

对应的 Java 实体类：

public class User {
    private Long id;
    private String username;
    private String email;
    private Integer age;
    private Long deptId;
    private Integer status;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;

    // getter / setter / toString 省略
}

3.1 namespace 与 Mapper 接口的绑定规则

MyBatis 通过 namespace 将 XML 映射文件与 Java 的 Mapper 接口进行绑定。这是整个 Mapper 机制能够工作的基础。

绑定规则

  1. namespace 的值必须是 Mapper 接口的全限定类名（包名 + 类名）
  2. XML 中每个 SQL 语句的 id 必须与 Mapper 接口中的方法名完全一致
  3. SQL 语句的 parameterType 要与方法的参数类型兼容
  4. SQL 语句的 resultType / resultMap 要与方法的返回值类型兼容

Mapper 接口

package com.example.mapper;

public interface UserMapper {
    User selectById(Long id);
}

XML 映射文件

<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper
  PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
  "http://mybatis.org/dtd/mybatis-3-mapper.dtd">

<mapper namespace="com.example.mapper.UserMapper">

    <select id="selectById" parameterType="long" resultType="com.example.entity.User">
        SELECT id, username, email, age, dept_id, status, create_time, update_time
        FROM sys_user
        WHERE id = #{id}
    </select>

</mapper>

常见错误

如果 namespace 与接口全限定名不一致，启动时会抛出：

org.apache.ibatis.binding.BindingException: Invalid bound statement (not found): 
    com.example.mapper.UserMapper.selectById

排查这个错误时，检查以下三点：

  1. namespace 拼写是否与接口全限定名完全一致（区分大小写）
  2. XML 文件是否被正确扫描到（检查 mybatis-config.xml 的 <mappers> 配置或 Spring Boot 的 mapper-locations）
  3. SQL 语句的 id 是否与接口方法名一致

XML 文件放置位置

方式 位置 说明
Maven 默认 src/main/resources 下与接口同包名的目录 例如接口在 com.example.mapper，XML 放在 resources/com/example/mapper/
Spring Boot 配置 mapper-locations 指定 例如 mybatis.mapper-locations=classpath:mapper/*.xml
与接口同目录 src/main/java 下与接口同包 需要在 pom.xml 中配置资源过滤，不推荐

推荐做法：在 resources/mapper/ 目录下统一存放 XML 文件，通过 Spring Boot 配置指定路径：

mybatis:
  mapper-locations: classpath:mapper/**/*.xml

3.2 基础 CRUD 标签

MyBatis 提供了四个核心标签来处理增删改查操作：<select>、<insert>、<update>、<delete>。

3.2.1 <select> —— 查询

<select> 是使用频率最高的标签，用于执行查询语句。

常用属性

属性 说明 是否必须
id 对应 Mapper 接口的方法名 是
parameterType 参数类型（可省略，MyBatis 会自动推断） 否
resultType 返回值类型的全限定名或别名 与 resultMap 二选一
resultMap 引用已定义的 resultMap 与 resultType 二选一
flushCache 执行后是否清空本地缓存和二级缓存，默认 false 否
useCache 是否使用二级缓存，默认 true 否
timeout 超时时间（秒） 否
fetchSize 驱动程序每次批量返回的行数提示 否
statementType STATEMENT / PREPARED(默认) / CALLABLE 否

Mapper 接口

public interface UserMapper {

    // 根据 ID 查询
    User selectById(Long id);

    // 查询所有用户
    List<User> selectAll();

    // 根据用户名模糊查询
    List<User> selectByUsername(String username);

    // 统计用户数
    int countAll();
}

XML 映射

<mapper namespace="com.example.mapper.UserMapper">

    <!-- 根据 ID 查询单个用户 -->
    <select id="selectById" resultType="com.example.entity.User">
        SELECT id, username, email, age, dept_id AS deptId, status, 
               create_time AS createTime, update_time AS updateTime
        FROM sys_user
        WHERE id = #{id}
    </select>

    <!-- 查询所有用户（返回 List 时 resultType 写的是集合中元素的类型） -->
    <select id="selectAll" resultType="com.example.entity.User">
        SELECT id, username, email, age, dept_id AS deptId, status,
               create_time AS createTime, update_time AS updateTime
        FROM sys_user
    </select>

    <!-- 模糊查询 -->
    <select id="selectByUsername" resultType="com.example.entity.User">
        SELECT id, username, email, age, dept_id AS deptId, status,
               create_time AS createTime, update_time AS updateTime
        FROM sys_user
        WHERE username LIKE CONCAT('%', #{username}, '%')
    </select>

    <!-- 统计总数 -->
    <select id="countAll" resultType="int">
        SELECT COUNT(*) FROM sys_user
    </select>

</mapper>

提示：上面示例中使用了列别名（dept_id AS deptId）来解决数据库下划线命名与 Java 驼峰命名的映射问题。更优雅的方式是开启全局驼峰映射配置 mapUnderscoreToCamelCase=true（见第二章 2.3.1），开启后就不需要手动写别名了。

3.2.2 <insert> —— 插入（含 useGeneratedKeys / selectKey）

<insert> 用于执行插入操作。在实际开发中，插入后获取自增主键是一个非常常见的需求，MyBatis 提供了两种方式来实现。

常用属性

属性 说明 默认值
id 对应 Mapper 接口方法名 —
parameterType 参数类型 可省略
useGeneratedKeys 是否使用 JDBC 的 getGeneratedKeys 获取自增主键 false
keyProperty 将获取到的主键值赋给参数对象的哪个属性 —
keyColumn 数据库中自增主键的列名（多数情况可省略） —

基础插入

Mapper 接口：

public interface UserMapper {
    int insert(User user);
}

XML 映射：

<insert id="insert" parameterType="com.example.entity.User">
    INSERT INTO sys_user (username, email, age, dept_id, status)
    VALUES (#{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

调用 insert 方法后，返回值是受影响的行数（通常为 1），但此时 user.getId() 仍然是 null，因为我们还没有告诉 MyBatis 要回填主键。

方式一：useGeneratedKeys（推荐）

这是获取自增主键最简洁、最推荐的方式。它依赖 JDBC 驱动的 Statement.getGeneratedKeys() 方法。

Mapper 接口：

public interface UserMapper {
    int insertAndGetId(User user);
}

XML 映射：

<insert id="insertAndGetId" parameterType="com.example.entity.User"
        useGeneratedKeys="true" keyProperty="id">
    INSERT INTO sys_user (username, email, age, dept_id, status)
    VALUES (#{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

使用示例：

User user = new User();
user.setUsername("zhangsan");
user.setEmail("zhangsan@example.com");
user.setAge(25);
user.setDeptId(1L);
user.setStatus(1);

int rows = userMapper.insertAndGetId(user);
System.out.println("影响行数: " + rows);      // 1
System.out.println("自增主键: " + user.getId()); // 自增的 ID 值，如 1001

关键点：

  • useGeneratedKeys="true" 告诉 MyBatis 使用 JDBC 原生的 getGeneratedKeys() 来获取自增主键
  • keyProperty="id" 指定将获取到的主键值设置到参数对象（User）的 id 属性上
  • 方法返回值仍然是影响行数，主键值是通过修改入参对象来传递的

方式二：<selectKey>

<selectKey> 是 MyBatis 提供的更通用的主键获取方式。它通过执行一条额外的 SQL 来获取主键值，适用于以下场景：

  • 数据库不支持自增主键（如 Oracle 使用 Sequence）
  • 需要在插入前生成主键（如使用 UUID）
  • 使用自定义的主键生成策略

Mapper 接口：

public interface UserMapper {
    int insertWithSelectKey(User user);
}

MySQL 示例（插入后获取）：

<insert id="insertWithSelectKey" parameterType="com.example.entity.User">
    <selectKey keyProperty="id" resultType="long" order="AFTER">
        SELECT LAST_INSERT_ID()
    </selectKey>
    INSERT INTO sys_user (username, email, age, dept_id, status)
    VALUES (#{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

Oracle 示例（插入前获取 Sequence）：

<insert id="insertWithSelectKey" parameterType="com.example.entity.User">
    <selectKey keyProperty="id" resultType="long" order="BEFORE">
        SELECT user_seq.NEXTVAL FROM DUAL
    </selectKey>
    INSERT INTO sys_user (id, username, email, age, dept_id, status)
    VALUES (#{id}, #{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

UUID 示例（插入前生成）：

<insert id="insertWithUUID" parameterType="com.example.entity.User">
    <selectKey keyProperty="id" resultType="string" order="BEFORE">
        SELECT REPLACE(UUID(), '-', '')
    </selectKey>
    INSERT INTO sys_user (id, username, email, age, dept_id, status)
    VALUES (#{id}, #{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

<selectKey> 属性说明

属性 说明
keyProperty 将结果设置到参数对象的哪个属性
resultType 返回值类型
order BEFORE：在 INSERT 之前执行；AFTER：在 INSERT 之后执行
statementType 默认 PREPARED

两种方式对比

对比维度 useGeneratedKeys <selectKey>
实现原理 依赖 JDBC 驱动的 getGeneratedKeys() 执行额外的 SQL 查询
执行时机 INSERT 之后自动获取 可以在 INSERT 之前或之后
适用数据库 支持自增主键的数据库（MySQL、SQL Server 等） 所有数据库（Oracle Sequence、UUID 等）
性能 更好（无需额外 SQL） 多一次 SQL 查询
灵活性 低，仅支持自增主键 高，支持任意主键生成策略
推荐场景 MySQL / PostgreSQL 自增主键 Oracle Sequence、UUID、自定义主键策略

最佳实践：MySQL 环境下优先使用 useGeneratedKeys，简洁高效；Oracle 或需要自定义主键策略时使用 <selectKey>。

3.2.3 <update> —— 更新

<update> 用于执行更新操作，返回值为受影响的行数。

Mapper 接口

public interface UserMapper {

    // 根据 ID 更新用户信息
    int updateById(User user);

    // 根据 ID 更新状态
    int updateStatus(@Param("id") Long id, @Param("status") Integer status);
}

XML 映射

<!-- 根据 ID 更新全部字段 -->
<update id="updateById" parameterType="com.example.entity.User">
    UPDATE sys_user
    SET username = #{username},
        email = #{email},
        age = #{age},
        dept_id = #{deptId},
        status = #{status},
        update_time = NOW()
    WHERE id = #{id}
</update>

<!-- 根据 ID 更新状态 -->
<update id="updateStatus">
    UPDATE sys_user
    SET status = #{status}, update_time = NOW()
    WHERE id = #{id}
</update>

使用示例：

User user = userMapper.selectById(1L);
user.setEmail("new_email@example.com");

int rows = userMapper.updateById(user);
System.out.println("更新了 " + rows + " 行");

注意：上面的 updateById 会把所有字段都更新一遍，即使某些字段没有变化。如果只想更新非空字段，可以结合第四章的动态 SQL <set> + <if> 标签实现。

3.2.4 <delete> —— 删除

<delete> 用于执行删除操作，返回值为受影响的行数。

Mapper 接口

public interface UserMapper {

    // 根据 ID 删除
    int deleteById(Long id);

    // 根据 ID 列表批量删除
    int deleteBatchIds(@Param("ids") List<Long> ids);
}

XML 映射

<!-- 根据 ID 删除 -->
<delete id="deleteById">
    DELETE FROM sys_user WHERE id = #{id}
</delete>

<!-- 批量删除（使用动态 SQL foreach，详见第四章） -->
<delete id="deleteBatchIds">
    DELETE FROM sys_user
    WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</delete>

使用示例：

// 单条删除
int rows = userMapper.deleteById(1L);

// 批量删除
List<Long> ids = Arrays.asList(2L, 3L, 4L);
int batchRows = userMapper.deleteBatchIds(ids);

CRUD 返回值小结

操作 返回值类型 含义
<select> 实体对象 / List / Map 等 查询结果
<insert> int 插入的行数（通常为 1）
<update> int 更新影响的行数
<delete> int 删除影响的行数

MyBatis 中 insert / update / delete 的返回值也可以声明为 long、boolean（大于 0 为 true）、void。

3.3 参数传递（⭐ 重点）

参数传递是使用 MyBatis 时最容易出错的地方之一。理解参数的传递规则，能让你在编写 SQL 时游刃有余。

3.3.1 单个基本类型参数

当 Mapper 方法只有一个参数且为基本类型（或 String、包装类型）时，XML 中可以用任意名称引用该参数。

Mapper 接口

public interface UserMapper {
    User selectById(Long id);
    User selectByUsername(String username);
}

XML 映射

<!-- 参数名可以随意写，但建议与方法参数名保持一致以增强可读性 -->
<select id="selectById" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE id = #{id}
    <!-- 下面这样写也可以，但不推荐 -->
    <!-- SELECT * FROM sys_user WHERE id = #{whatever} -->
</select>

<select id="selectByUsername" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE username = #{username}
</select>

原理：当只有一个参数时，MyBatis 不关心 #{} 中的名称，直接将该参数的值设置到 SQL 中。但为了代码可读性，强烈建议使用与方法参数名一致的名称。

3.3.2 多个参数 —— @Param 注解

当 Mapper 方法有多个参数时，必须使用 @Param 注解来为每个参数指定名称，否则 MyBatis 会使用默认的 arg0, arg1... 或 param1, param2... 来引用参数，可读性极差且容易出错。

Mapper 接口

public interface UserMapper {

    // 推荐写法：使用 @Param 注解
    List<User> selectByAgeRange(@Param("minAge") Integer minAge,
                                 @Param("maxAge") Integer maxAge);

    // 不推荐：不使用 @Param
    List<User> selectByDeptAndStatus(Long deptId, Integer status);
}

XML 映射

<!-- 使用 @Param 注解后，XML 中用注解指定的名称引用 -->
<select id="selectByAgeRange" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE age BETWEEN #{minAge} AND #{maxAge}
</select>

<!-- 不使用 @Param 时，只能用默认名称（不推荐！） -->
<select id="selectByDeptAndStatus" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE dept_id = #{param1} AND status = #{param2}
    <!-- 或者用 #{arg0} #{arg1}，同样不推荐 -->
</select>

最佳实践：只要方法有两个及以上参数，一律加 @Param 注解。虽然某些编译器配置（-parameters）可以让 MyBatis 自动获取参数名，但显式标注 @Param 更可靠、更清晰。

参数名解析规则详解

当 Mapper 方法有多个参数时，MyBatis 会将参数封装到一个 Map 中。以下是可用的 key：

条件 可用的 key 示例
使用 @Param("name") 注解值 + param1, param2... #{name} 或 #{param1}
未使用 @Param arg0, arg1... + param1, param2... #{arg0} 或 #{param1}
编译时开启 -parameters 形参名 + param1, param2... #{deptId} 或 #{param1}

3.3.3 POJO / Map / List 作为参数

POJO 作为参数

当参数是一个 POJO（Java Bean）时，MyBatis 会通过 getter 方法获取属性值，#{} 中直接写属性名即可。

Mapper 接口：

public interface UserMapper {
    int insert(User user);
    List<User> selectByCondition(User user);
}

XML 映射：

<insert id="insert" parameterType="com.example.entity.User"
        useGeneratedKeys="true" keyProperty="id">
    INSERT INTO sys_user (username, email, age, dept_id, status)
    VALUES (#{username}, #{email}, #{age}, #{deptId}, #{status})
</insert>

<select id="selectByCondition" parameterType="com.example.entity.User"
        resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE dept_id = #{deptId} AND status = #{status}
</select>

Map 作为参数

当查询条件不固定、不想为此创建 POJO 时，可以使用 Map 传参。#{} 中写的是 Map 的 key。

Mapper 接口：

public interface UserMapper {
    List<User> selectByMap(Map<String, Object> params);
}

XML 映射：

<select id="selectByMap" parameterType="map" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE dept_id = #{deptId} AND status = #{status}
</select>

使用示例：

Map<String, Object> params = new HashMap<>();
params.put("deptId", 1L);
params.put("status", 1);
List<User> users = userMapper.selectByMap(params);

注意：Map 传参虽然灵活，但缺乏类型检查和语义表达。在业务代码中，推荐使用 POJO（或专门的查询条件对象 DTO），Map 仅在非常灵活的查询场景下使用。

List / Array 作为参数

集合参数主要用于 IN 查询和批量操作，通常需要配合 <foreach> 标签使用。

Mapper 接口：

public interface UserMapper {
    List<User> selectByIds(@Param("ids") List<Long> ids);
}

XML 映射：

<select id="selectByIds" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE id IN
    <foreach collection="ids" item="id" open="(" separator="," close=")">
        #{id}
    </foreach>
</select>

使用示例：

List<Long> ids = Arrays.asList(1L, 2L, 3L);
List<User> users = userMapper.selectByIds(ids);
// 生成的 SQL: SELECT * FROM sys_user WHERE id IN (1, 2, 3)

关于 collection 属性：当使用 @Param("ids") 注解时，collection 的值就是注解指定的名称。如果不使用 @Param，则 List 类型默认用 list、数组类型默认用 array。强烈建议始终使用 @Param，避免记忆默认规则。

POJO + @Param 混合使用

在复杂场景下，可以同时传递 POJO 和其他参数：

Mapper 接口：

public interface UserMapper {
    List<User> selectByPage(@Param("user") User user,
                            @Param("offset") int offset,
                            @Param("limit") int limit);
}

XML 映射：

<select id="selectByPage" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    WHERE dept_id = #{user.deptId} AND status = #{user.status}
    LIMIT #{offset}, #{limit}
</select>

注意 POJO 属性的引用方式：#{user.deptId}，需要加上 @Param 指定的前缀。

3.3.4 #{} vs ${} —— 预编译与字符串拼接的本质区别（SQL 注入防范）

这是本章乃至整个 MyBatis 课程最重要的知识点之一。 搞不清 #{} 和 ${} 的区别，轻则功能异常，重则引发 SQL 注入漏洞。

核心区别对比

对比维度 #{}（井号） ${}（美元符）
本质 占位符（Parameter Marker） 字符串拼接（String Substitution）
底层实现 PreparedStatement，参数用 ? 占位 Statement，参数直接拼入 SQL 字符串
SQL 注入 安全 —— 参数不参与 SQL 编译 不安全 —— 参数直接拼入 SQL，有注入风险
参数类型处理 自动调用 TypeHandler 进行类型转换 原样拼接，不做类型转换
字符串参数 自动加单引号 不会自动加引号
性能 可利用 PreparedStatement 预编译缓存 每次都需要重新编译 SQL
适用场景 绝大多数场景（WHERE 条件值、INSERT 值等） 表名、列名、ORDER BY 等不能用占位符的场景

#{} 的工作原理

<select id="selectById" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE id = #{id}
</select>

MyBatis 处理过程：

1. 解析 SQL → SELECT * FROM sys_user WHERE id = ?
2. 创建 PreparedStatement
3. 调用 ps.setLong(1, 1001)  ← 通过 TypeHandler 设置参数
4. 执行查询

等价的 JDBC 代码：

String sql = "SELECT * FROM sys_user WHERE id = ?";
PreparedStatement ps = connection.prepareStatement(sql);
ps.setLong(1, 1001L);
ResultSet rs = ps.executeQuery();

${} 的工作原理

<select id="selectByColumn" resultType="com.example.entity.User">
    SELECT * FROM sys_user ORDER BY ${orderColumn}
</select>

MyBatis 处理过程：

1. 将参数值直接拼接到 SQL 字符串中
2. 假设 orderColumn = "username"
3. 最终 SQL → SELECT * FROM sys_user ORDER BY username
4. 创建 Statement 并执行

等价的 JDBC 代码：

String sql = "SELECT * FROM sys_user ORDER BY " + orderColumn;
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery(sql);

SQL 注入演示

假设有一个用错了 ${} 的查询：

<!-- ❌ 错误示范：WHERE 条件值使用 ${} -->
<select id="selectByUsername" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE username = '${username}'
</select>

正常调用时一切正常：

输入: username = "zhangsan"
SQL:  SELECT * FROM sys_user WHERE username = 'zhangsan'   ✅ 正常

但恶意用户可以构造如下输入：

输入: username = "' OR '1'='1"
SQL:  SELECT * FROM sys_user WHERE username = '' OR '1'='1'

这条 SQL 会返回所有用户数据！ 因为 '1'='1' 永远为真。

更危险的注入：

输入: username = "'; DROP TABLE sys_user; --"
SQL:  SELECT * FROM sys_user WHERE username = ''; DROP TABLE sys_user; --'

这将删除整个用户表！

如果使用 #{} 则完全不会有这个问题：

<!-- ✅ 正确写法：使用 #{} -->
<select id="selectByUsername" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE username = #{username}
</select>

输入: username = "' OR '1'='1"
SQL:  SELECT * FROM sys_user WHERE username = ?
参数: ps.setString(1, "' OR '1'='1'")   ← 整个字符串被当作普通文本值
结果: 查询 username 等于 "' OR '1'='1'" 的用户（不存在），返回空  ✅ 安全

什么时候必须用 ${}？

SQL 语法中有些位置不允许使用 ? 占位符，此时只能用 ${}：

1. 动态表名：

<select id="selectFromTable" resultType="map">
    SELECT * FROM ${tableName}
</select>

2. 动态列名（ORDER BY / GROUP BY）：

<select id="selectAllOrdered" resultType="com.example.entity.User">
    SELECT * FROM sys_user ORDER BY ${orderColumn} ${orderDir}
</select>

3. 动态 SQL 关键字：

<select id="selectWithSchema" resultType="com.example.entity.User">
    SELECT * FROM ${schema}.sys_user WHERE id = #{id}
</select>

使用 ${} 时的安全防范措施

既然有些场景不得不用 ${}，那如何防范 SQL 注入？

方式一：Java 代码中校验白名单

private static final Set<String> ALLOWED_COLUMNS = 
    Set.of("id", "username", "email", "age", "create_time");

public List<User> selectAllOrdered(String orderColumn) {
    if (!ALLOWED_COLUMNS.contains(orderColumn)) {
        throw new IllegalArgumentException("非法的排序字段: " + orderColumn);
    }
    return userMapper.selectAllOrdered(orderColumn);
}

方式二：使用枚举限制合法值

public enum OrderColumn {
    ID("id"), USERNAME("username"), CREATE_TIME("create_time");
    
    private final String column;
    OrderColumn(String column) { this.column = column; }
    public String getColumn() { return column; }
}

方式三：在 SQL 中使用 <choose> 代替 ${}

<select id="selectAllOrdered" resultType="com.example.entity.User">
    SELECT * FROM sys_user
    ORDER BY
    <choose>
        <when test="orderColumn == 'username'">username</when>
        <when test="orderColumn == 'age'">age</when>
        <when test="orderColumn == 'createTime'">create_time</when>
        <otherwise>id</otherwise>
    </choose>
</select>

这种方式最安全——排序字段直接写死在 XML 中，用户输入只是用来做条件判断，完全杜绝了注入可能。

总结：选择原则

默认用 #{} ——→ 安全、高效、自动类型转换

只在以下场景用 ${}：
  ├── 动态表名
  ├── 动态列名（ORDER BY / GROUP BY）
  ├── 动态 Schema
  └── 其他不能使用占位符的语法位置

使用 ${} 时必须：
  ├── 后端白名单校验
  ├── 绝不允许用户输入直接传入
  └── 或用 <choose> 等方式替代

3.4 返回值映射

MyBatis 的强大之处在于能将 SQL 查询结果自动映射为 Java 对象。根据映射复杂度的不同，提供了 resultType 和 resultMap 两种机制。

3.4.1 resultType —— 简单映射

resultType 适用于查询结果列与 Java 对象属性可以直接对应的场景。

映射规则

MyBatis 将查询结果的列名与 Java 对象的属性名进行匹配（通过 setter 方法）：

  • 列名与属性名完全一致时自动映射（不区分大小写）
  • 开启 mapUnderscoreToCamelCase 后，dept_id 自动映射到 deptId
  • 不匹配的列会被忽略，不匹配的属性保持默认值

常见的 resultType 类型

1. 映射到实体类：

<select id="selectById" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE id = #{id}
</select>

2. 映射到基本类型 / 包装类型：

<!-- 返回 int -->
<select id="countAll" resultType="int">
    SELECT COUNT(*) FROM sys_user
</select>

<!-- 返回 String -->
<select id="selectUsernameById" resultType="string">
    SELECT username FROM sys_user WHERE id = #{id}
</select>

<!-- 返回 BigDecimal -->
<select id="selectAvgAge" resultType="java.math.BigDecimal">
    SELECT AVG(age) FROM sys_user
</select>

3. 映射到 Map：

当你不想（或不需要）创建实体类时，可以将单行结果映射为 Map<String, Object>，其中 key 是列名，value 是列值。

public interface UserMapper {
    Map<String, Object> selectMapById(Long id);
}

<select id="selectMapById" resultType="map">
    SELECT id, username, email, age FROM sys_user WHERE id = #{id}
</select>

Map<String, Object> userMap = userMapper.selectMapById(1L);
// {id=1, username=zhangsan, email=zhangsan@example.com, age=25}

使用类型别名简化 resultType

在 mybatis-config.xml 中配置别名后，可以使用短名称：

<typeAliases>
    <package name="com.example.entity"/>
</typeAliases>

配置后 XML 中可以简写：

<!-- 配置别名前 -->
<select id="selectById" resultType="com.example.entity.User">

<!-- 配置别名后（类名首字母小写） -->
<select id="selectById" resultType="user">

MyBatis 还内置了常用类型的别名：int、long、string、map、list 等，可直接使用。

3.4.2 resultMap —— 复杂映射（详见第五章）

当列名与属性名无法直接对应，或者需要处理关联关系（一对一、一对多）时，需要使用 resultMap。

本小节先介绍基础用法，高级映射（association、collection）在第五章详细展开。

基本结构

<resultMap id="userResultMap" type="com.example.entity.User">
    <!-- 主键列映射（id 标签有特殊的性能优化意义） -->
    <id property="id" column="id"/>
    
    <!-- 普通列映射 -->
    <result property="username" column="username"/>
    <result property="email" column="email"/>
    <result property="age" column="age"/>
    <result property="deptId" column="dept_id"/>
    <result property="status" column="status"/>
    <result property="createTime" column="create_time"/>
    <result property="updateTime" column="update_time"/>
</resultMap>

<select id="selectById" resultMap="userResultMap">
    SELECT * FROM sys_user WHERE id = #{id}
</select>

<id> 与 <result> 的区别

两者功能基本相同，但 <id> 标签有额外作用：

  • 标记主键列，MyBatis 在处理嵌套结果映射（association / collection）时，会用 <id> 来判断行的唯一性
  • 在启用缓存时，<id> 用于构建缓存的 key
  • 建议：主键列始终使用 <id> 标签

什么时候用 resultType，什么时候用 resultMap？

场景 推荐
列名与属性名一致（或已开启驼峰映射） resultType
只查询少量字段，映射到基本类型或 Map resultType
列名与属性名不一致，且无法通过别名或驼峰映射解决 resultMap
需要一对一（association）或一对多（collection）关联 resultMap
需要鉴别器（discriminator） resultMap

实际开发建议：开启 mapUnderscoreToCamelCase=true 后，大部分简单查询都可以用 resultType。只有在处理复杂关联关系时才需要 resultMap。

3.4.3 返回 List、Map、单个对象的区别

MyBatis 根据 Mapper 接口方法的返回值类型 来决定如何处理查询结果。理解这些差异对于正确使用 MyBatis 至关重要。

返回单个对象

public interface UserMapper {
    User selectById(Long id);
}

<select id="selectById" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE id = #{id}
</select>
  • 查询返回 0 行 → 返回 null
  • 查询返回 1 行 → 返回对应的 User 对象
  • 查询返回多行 → 抛出 TooManyResultsException

返回 List<Entity>

public interface UserMapper {
    List<User> selectByDeptId(Long deptId);
}

<!-- resultType 写的是集合元素的类型，不是 List！ -->
<select id="selectByDeptId" resultType="com.example.entity.User">
    SELECT * FROM sys_user WHERE dept_id = #{deptId}
</select>
  • 查询返回 0 行 → 返回空 List（不是 null）
  • 查询返回 N 行 → 返回包含 N 个 User 的 List

重要：resultType 中写的是集合元素的类型（User），而不是 List。MyBatis 根据接口方法的返回值类型来决定是返回单个对象还是 List。

返回 Map<String, Object> —— 单行结果

将单行查询结果映射为 Map，key 是列名，value 是列值：

public interface UserMapper {
    Map<String, Object> selectMapById(Long id);
}

<select id="selectMapById" resultType="map">
    SELECT id, username, email, age FROM sys_user WHERE id = #{id}
</select>

Map<String, Object> result = userMapper.selectMapById(1L);
// {id=1, username="zhangsan", email="zhangsan@example.com", age=25}

返回 List<Map<String, Object>> —— 多行结果

将多行结果映射为 Map 的 List：

public interface UserMapper {
    List<Map<String, Object>> selectAllAsMap();
}

<select id="selectAllAsMap" resultType="map">
    SELECT id, username, email FROM sys_user
</select>

List<Map<String, Object>> list = userMapper.selectAllAsMap();
// [{id=1, username="zhangsan", email="..."}, {id=2, username="lisi", email="..."}, ...]

返回 Map<K, V> + @MapKey —— 以某列为 key 的 Map

这是一个非常实用但容易被忽略的特性。@MapKey 注解可以将查询结果组织成一个以指定列为 key 的 Map，方便后续按 key 快速查找。

public interface UserMapper {
    @MapKey("id")
    Map<Long, User> selectAllAsIdMap();

    @MapKey("username")
    Map<String, User> selectAllAsUsernameMap();
}

<!-- resultType 仍然写元素类型 -->
<select id="selectAllAsIdMap" resultType="com.example.entity.User">
    SELECT * FROM sys_user
</select>

<select id="selectAllAsUsernameMap" resultType="com.example.entity.User">
    SELECT * FROM sys_user
</select>

使用示例：

// 以 id 为 key
Map<Long, User> idMap = userMapper.selectAllAsIdMap();
User user = idMap.get(1001L);  // 直接用 ID 获取，无需遍历

// 以 username 为 key
Map<String, User> nameMap = userMapper.selectAllAsUsernameMap();
User zhangsan = nameMap.get("zhangsan");

@MapKey 的典型使用场景：

  • 批量查询后需要按 ID 快速查找（避免 N+1 查询）
  • 构建缓存 Map
  • 需要对结果进行去重或分组（注意：相同 key 的后面会覆盖前面）

返回值类型完整速查

Mapper 方法返回值 resultType 写什么 查询结果行为
User com.example.entity.User 0行→null；1行→对象；多行→异常
List<User> com.example.entity.User 0行→空List；N行→N个元素的List
Map<String, Object> map 单行→{列名:值}；多行→异常
List<Map<String, Object>> map 多行→[{列名:值}, ...]
@MapKey("id") Map<Long, User> com.example.entity.User 以id为key的Map
int / long int / long 返回数值（如 COUNT）
String string 返回单个字符串值

3.5 <sql> 与 <include> —— SQL 片段复用

在实际项目中，很多 SQL 语句都有重复的部分，比如查询的列列表、通用的 WHERE 条件等。MyBatis 提供了 <sql> 和 <include> 标签来实现 SQL 片段的复用，避免重复编写、便于统一维护。

基本用法

定义 SQL 片段：

<mapper namespace="com.example.mapper.UserMapper">

    <!-- 定义通用的列列表片段 -->
    <sql id="baseColumns">
        id, username, email, age, dept_id, status, create_time, update_time
    </sql>

    <!-- 定义带表别名的列列表 -->
    <sql id="baseColumnsWithAlias">
        ${alias}.id, ${alias}.username, ${alias}.email, ${alias}.age,
        ${alias}.dept_id, ${alias}.status, ${alias}.create_time, ${alias}.update_time
    </sql>

    <!-- 定义通用的 WHERE 条件片段 -->
    <sql id="activeCondition">
        status = 1
    </sql>

</mapper>

引用 SQL 片段：

<select id="selectById" resultType="com.example.entity.User">
    SELECT
    <include refid="baseColumns"/>
    FROM sys_user
    WHERE id = #{id}
</select>

<select id="selectAll" resultType="com.example.entity.User">
    SELECT
    <include refid="baseColumns"/>
    FROM sys_user
    WHERE <include refid="activeCondition"/>
</select>

<select id="selectByDeptId" resultType="com.example.entity.User">
    SELECT
    <include refid="baseColumns"/>
    FROM sys_user
    WHERE dept_id = #{deptId} AND <include refid="activeCondition"/>
</select>

最终生成的 SQL：

-- selectById
SELECT id, username, email, age, dept_id, status, create_time, update_time
FROM sys_user WHERE id = ?

-- selectAll
SELECT id, username, email, age, dept_id, status, create_time, update_time
FROM sys_user WHERE status = 1

-- selectByDeptId
SELECT id, username, email, age, dept_id, status, create_time, update_time
FROM sys_user WHERE dept_id = ? AND status = 1

带参数的 SQL 片段

<include> 可以通过 <property> 子标签向 SQL 片段传递参数：

<!-- 定义带参数的片段 -->
<sql id="baseColumnsWithAlias">
    ${alias}.id, ${alias}.username, ${alias}.email, ${alias}.age,
    ${alias}.dept_id, ${alias}.status, ${alias}.create_time, ${alias}.update_time
</sql>

<!-- 使用时传递参数 -->
<select id="selectUserWithDept" resultType="map">
    SELECT
    <include refid="baseColumnsWithAlias">
        <property name="alias" value="u"/>
    </include>,
    d.dept_name
    FROM sys_user u
    LEFT JOIN sys_dept d ON u.dept_id = d.id
    WHERE u.id = #{id}
</select>

生成的 SQL：

SELECT u.id, u.username, u.email, u.age, u.dept_id, u.status, 
       u.create_time, u.update_time, d.dept_name
FROM sys_user u
LEFT JOIN sys_dept d ON u.dept_id = d.id
WHERE u.id = ?

注意：<sql> 片段中使用 <property> 传入的参数时，必须用 ${}（因为这是在 SQL 解析阶段进行的文本替换，不是运行时参数），这里的 ${} 是安全的，因为值来自 XML 中的硬编码而非用户输入。

跨 Mapper 引用 SQL 片段

在大型项目中，可以将通用的 SQL 片段定义在一个公共的 Mapper XML 中，然后在其他 Mapper 中通过 namespace.id 的形式引用：

CommonMapper.xml：

<mapper namespace="com.example.mapper.CommonMapper">

    <sql id="paginationSuffix">
        LIMIT #{offset}, #{limit}
    </sql>

    <sql id="orderBy">
        ORDER BY ${orderColumn} ${orderDir}
    </sql>

</mapper>

UserMapper.xml 中引用：

<select id="selectPage" resultType="com.example.entity.User">
    SELECT
    <include refid="baseColumns"/>
    FROM sys_user
    WHERE <include refid="activeCondition"/>
    <include refid="com.example.mapper.CommonMapper.orderBy">
        <property name="orderColumn" value="create_time"/>
        <property name="orderDir" value="DESC"/>
    </include>
    <include refid="com.example.mapper.CommonMapper.paginationSuffix"/>
</select>

实际项目中的完整复用示例

下面是一个贴近实际开发的完整示例，展示如何通过 <sql> + <include> 减少重复代码：

<mapper namespace="com.example.mapper.UserMapper">

    <!-- ==================== SQL 片段定义 ==================== -->

    <sql id="tableName">sys_user</sql>

    <sql id="baseColumns">
        id, username, email, age, dept_id, status, create_time, update_time
    </sql>

    <sql id="whereActive">
        <where>
            status = 1
            <if test="username != null and username != ''">
                AND username LIKE CONCAT('%', #{username}, '%')
            </if>
            <if test="deptId != null">
                AND dept_id = #{deptId}
            </if>
            <if test="minAge != null">
                AND age &gt;= #{minAge}
            </if>
            <if test="maxAge != null">
                AND age &lt;= #{maxAge}
            </if>
        </where>
    </sql>

    <!-- ==================== CRUD 语句 ==================== -->

    <select id="selectById" resultType="com.example.entity.User">
        SELECT <include refid="baseColumns"/>
        FROM <include refid="tableName"/>
        WHERE id = #{id}
    </select>

    <select id="selectList" resultType="com.example.entity.User">
        SELECT <include refid="baseColumns"/>
        FROM <include refid="tableName"/>
        <include refid="whereActive"/>
        ORDER BY create_time DESC
    </select>

    <select id="countByCondition" resultType="int">
        SELECT COUNT(*)
        FROM <include refid="tableName"/>
        <include refid="whereActive"/>
    </select>

    <insert id="insert" useGeneratedKeys="true" keyProperty="id">
        INSERT INTO <include refid="tableName"/>
        (username, email, age, dept_id, status)
        VALUES (#{username}, #{email}, #{age}, #{deptId}, #{status})
    </insert>

    <update id="updateById">
        UPDATE <include refid="tableName"/>
        SET username = #{username}, email = #{email}, age = #{age},
            dept_id = #{deptId}, update_time = NOW()
        WHERE id = #{id}
    </update>

    <delete id="deleteById">
        DELETE FROM <include refid="tableName"/> WHERE id = #{id}
    </delete>

</mapper>

<sql> / <include> 使用建议

建议 说明
列列表必须抽取 这是最高频的重复代码，一旦加字段只需改一处
通用条件可以抽取 多个查询共享相同的 WHERE 条件时
不要过度抽取 如果片段只被用了一次，直接写在 SQL 中更清晰
命名要语义化 baseColumns、detailColumns、activeCondition 比 sql1、sql2 好得多
跨 Mapper 引用要谨慎 过多跨文件引用会增加理解成本，仅对真正通用的片段使用

本章完整介绍了 Mapper XML 映射文件的核心内容，这些知识是日常 MyBatis 开发的基石：

知识点 掌握程度要求
namespace 绑定规则 必须理解，否则连基本的 Mapper 都无法使用
CRUD 四个标签 烂熟于心，这是每天都要写的
useGeneratedKeys vs selectKey 理解原理和适用场景，根据数据库选择
参数传递规则 必须掌握，尤其是 @Param 的使用时机
#{} vs ${} 最重要的知识点，直接关系到系统安全
resultType vs resultMap 理解各自适用场景
返回 List / Map / @MapKey 灵活运用不同的返回值类型
<sql> + <include> 养成复用习惯，保持代码整洁
