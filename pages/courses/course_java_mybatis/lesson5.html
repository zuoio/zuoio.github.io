第五章 ResultMap 高级映射 ⭐

处理复杂业务模型的关键能力。当数据库表与 Java 对象之间不再是简单的一一对应，而是涉及关联对象、集合嵌套、继承多态时，resultMap 是 MyBatis 最核心的武器。

本章数据模型与建表 SQL

本章使用一个贯穿始终的电商场景数据模型：用户（User）→ 订单（Order）→ 订单项（OrderItem）→ 商品（Product）。

ER 关系

User (1) ──── (N) Order (1) ──── (N) OrderItem (N) ──── (1) Product
  • 一个用户可以有多个订单（一对多）
  • 一个订单属于一个用户（一对一 / 多对一）
  • 一个订单包含多个订单项（一对多）
  • 一个订单项对应一个商品（一对一 / 多对一）

建表 SQL（MySQL）

-- 用户表
CREATE TABLE t_user (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    username    VARCHAR(50)  NOT NULL,
    email       VARCHAR(100) NOT NULL,
    phone       VARCHAR(20),
    create_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 商品表
CREATE TABLE t_product (
    id             BIGINT PRIMARY KEY AUTO_INCREMENT,
    product_name   VARCHAR(200) NOT NULL,
    price          DECIMAL(10, 2) NOT NULL,
    stock          INT          NOT NULL DEFAULT 0,
    category       VARCHAR(50),
    create_time    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单表
CREATE TABLE t_order (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no    VARCHAR(64)    NOT NULL UNIQUE,
    user_id     BIGINT         NOT NULL,
    total_amount DECIMAL(12, 2) NOT NULL,
    status      TINYINT        NOT NULL DEFAULT 0 COMMENT '0-待支付 1-已支付 2-已发货 3-已完成 4-已取消',
    create_time DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES t_user(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单项表
CREATE TABLE t_order_item (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id    BIGINT         NOT NULL,
    product_id  BIGINT         NOT NULL,
    quantity    INT            NOT NULL,
    unit_price  DECIMAL(10, 2) NOT NULL,
    FOREIGN KEY (order_id)   REFERENCES t_order(id),
    FOREIGN KEY (product_id) REFERENCES t_product(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

5.1 resultMap 基础结构 —— id、result、constructor

5.1.1 为什么需要 resultMap

resultType 只能处理列名与属性名完全匹配（或在开启 mapUnderscoreToCamelCase 后下划线到驼峰自动转换）的简单情况。当出现以下场景时，必须使用 resultMap：

  1. 列名与属性名不匹配，且无法通过驼峰规则转换
  2. 需要映射关联对象（一对一、一对多）
  3. 需要根据某个列的值映射到不同的 Java 子类（鉴别器）
  4. 需要通过构造器创建对象

5.1.2 resultMap 完整结构

<resultMap id="唯一标识" type="映射的Java全限定类名或别名"
           autoMapping="true|false" extends="继承另一个resultMap的id">

    <!-- 构造器映射（可选，与 id/result 二选一来设置主键和普通列） -->
    <constructor>
        <idArg column="列名" javaType="Java类型"/>
        <arg   column="列名" javaType="Java类型"/>
    </constructor>

    <!-- 主键列映射（推荐始终声明，影响性能优化和嵌套映射去重） -->
    <id property="Java属性名" column="列名" javaType="" jdbcType="" typeHandler=""/>

    <!-- 普通列映射 -->
    <result property="Java属性名" column="列名" javaType="" jdbcType="" typeHandler=""/>

    <!-- 一对一关联映射 -->
    <association property="属性名" javaType="关联对象类型" .../>

    <!-- 一对多关联映射 -->
    <collection property="属性名" ofType="集合元素类型" .../>

    <!-- 鉴别器（多态映射） -->
    <discriminator javaType="" column="">
        <case value="值" resultMap="" | resultType="">
            <!-- 嵌套映射 -->
        </case>
    </discriminator>

</resultMap>

5.1.3 各子标签详解

<id> —— 主键映射

<id property="id" column="id"/>

与 <result> 功能相同，但有两个重要区别：

  1. 性能优化：MyBatis 内部使用 <id> 标记的列值作为缓存的 key，在嵌套映射中用于判断对象是否相同（去重合并）
  2. 语义明确：标识哪一列是主键，提升可读性

最佳实践：即使所有列名和属性名都能自动匹配，也建议显式声明 <id>，尤其在使用 association / collection 嵌套结果映射时——不声明 <id> 会导致 MyBatis 使用所有列的组合值来判断对象唯一性，既慢又容易出错。

<result> —— 普通列映射

<result property="username" column="username"/>
<result property="createTime" column="create_time"/>

常用属性：

属性 说明
property Java 对象的属性名
column 查询结果集的列名（或别名）
javaType Java 属性类型，大多数情况 MyBatis 可自动推断，无需显式指定
jdbcType JDBC 类型，通常在插入 null 值时才需要指定（如 Oracle）
typeHandler 指定自定义类型处理器的全限定类名

<constructor> —— 构造器映射

当实体类没有无参构造器，或你希望创建不可变对象（immutable object）时使用：

public class User {
    private final Long id;
    private final String username;

    public User(Long id, String username) {
        this.id = id;
        this.username = username;
    }

    // 只有 getter，没有 setter
}

<resultMap id="userMap" type="User">
    <constructor>
        <idArg column="id" javaType="long"/>
        <arg column="username" javaType="string"/>
    </constructor>
    <!-- 其余列仍可用 result 映射 -->
    <result property="email" column="email"/>
</resultMap>

注意：<constructor> 中 <arg> 的顺序必须与构造器参数的顺序严格一致。从 MyBatis 3.4.3 开始，也可以使用 name 属性指定参数名来避免顺序依赖（需配合编译参数 -parameters 或 @Param 注解）。

5.1.4 基础示例：User 的完整 resultMap

<resultMap id="baseUserMap" type="com.example.entity.User">
    <id     property="id"         column="id"/>
    <result property="username"   column="username"/>
    <result property="email"      column="email"/>
    <result property="phone"      column="phone"/>
    <result property="createTime" column="create_time"/>
</resultMap>

<select id="selectUserById" resultMap="baseUserMap">
    SELECT id, username, email, phone, create_time
    FROM t_user
    WHERE id = #{id}
</select>

5.1.5 resultMap 的继承（extends）

当多个 resultMap 有共同的基础映射时，可以用 extends 避免重复：

<!-- 基础映射：只包含 User 自身的列 -->
<resultMap id="baseUserMap" type="User">
    <id     property="id"         column="id"/>
    <result property="username"   column="username"/>
    <result property="email"      column="email"/>
    <result property="phone"      column="phone"/>
    <result property="createTime" column="create_time"/>
</resultMap>

<!-- 扩展映射：继承基础映射，额外加上关联 -->
<resultMap id="userWithOrdersMap" type="User" extends="baseUserMap">
    <collection property="orders" ofType="Order">
        <!-- ... -->
    </collection>
</resultMap>

5.2 association —— 一对一关联映射

<association> 用于映射 Java 对象中的关联对象属性，即属性类型是另一个 POJO 的场景。

在我们的数据模型中，Order 对象包含一个 User user 属性——这就是典型的一对一（多对一）关联。MyBatis 提供两种方式来实现这种映射：嵌套查询和嵌套结果。

5.2.1 嵌套查询（Nested Select / N+1 问题）

原理

嵌套查询的工作方式是：先执行主查询获取 Order 数据，然后对每一条 Order 记录，再单独发起一次查询去获取关联的 User 对象。

实体类

public class Order {
    private Long id;
    private String orderNo;
    private Long userId;
    private BigDecimal totalAmount;
    private Integer status;
    private LocalDateTime createTime;

    private User user; // 关联的用户对象
    // getter / setter 省略
}

Mapper 接口

public interface OrderMapper {
    Order selectOrderById(Long id);
    List<Order> selectAllOrders();
}

public interface UserMapper {
    User selectUserById(Long id);
}

Mapper XML

UserMapper.xml

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.UserMapper">

    <resultMap id="baseUserMap" type="com.example.entity.User">
        <id     property="id"         column="id"/>
        <result property="username"   column="username"/>
        <result property="email"      column="email"/>
        <result property="phone"      column="phone"/>
        <result property="createTime" column="create_time"/>
    </resultMap>

    <select id="selectUserById" resultMap="baseUserMap">
        SELECT id, username, email, phone, create_time
        FROM t_user
        WHERE id = #{id}
    </select>

</mapper>

OrderMapper.xml

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
        "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.OrderMapper">

    <!-- 嵌套查询方式的 resultMap -->
    <resultMap id="orderWithUserNestedSelectMap" type="com.example.entity.Order">
        <id     property="id"          column="id"/>
        <result property="orderNo"     column="order_no"/>
        <result property="userId"      column="user_id"/>
        <result property="totalAmount" column="total_amount"/>
        <result property="status"      column="status"/>
        <result property="createTime"  column="create_time"/>
        <!--
            association 嵌套查询：
            - property: Order 中的关联属性名
            - column:   传递给嵌套查询的参数（主查询结果集中的列名）
            - select:   嵌套查询的 statement id（namespace.id）
        -->
        <association property="user"
                     column="user_id"
                     select="com.example.mapper.UserMapper.selectUserById"/>
    </resultMap>

    <select id="selectOrderById" resultMap="orderWithUserNestedSelectMap">
        SELECT id, order_no, user_id, total_amount, status, create_time
        FROM t_order
        WHERE id = #{id}
    </select>

    <select id="selectAllOrders" resultMap="orderWithUserNestedSelectMap">
        SELECT id, order_no, user_id, total_amount, status, create_time
        FROM t_order
    </select>

</mapper>

执行效果

当调用 selectOrderById(1) 时，MyBatis 实际执行了两条 SQL：

-- 第1条：主查询
SELECT id, order_no, user_id, total_amount, status, create_time
FROM t_order WHERE id = 1;

-- 第2条：嵌套查询（根据主查询结果中的 user_id 发起）
SELECT id, username, email, phone, create_time
FROM t_user WHERE id = 1;

N+1 问题

N+1 问题是嵌套查询最大的隐患。 当调用 selectAllOrders() 查询所有订单时：

第 1 条 SQL：SELECT ... FROM t_order               → 返回 N 条订单
第 2 条 SQL：SELECT ... FROM t_user WHERE id = ?    → 订单1 的用户
第 3 条 SQL：SELECT ... FROM t_user WHERE id = ?    → 订单2 的用户
...
第 N+1 条 SQL：SELECT ... FROM t_user WHERE id = ?  → 订单N 的用户

一共执行了 1 + N 条 SQL。如果有 1000 条订单，就会触发 1001 次数据库查询。即使有些用户重复（如同一个用户的多个订单），默认情况下仍然会重复查询。

N+1 问题产生的根本原因：嵌套查询是「逐行触发」的——MyBatis 遍历主查询的每一行结果，对每一行都独立执行一次嵌套子查询。这是一种典型的用多次网络往返换取代码简洁性的方式，在数据量小的时候可以接受，但数据量一大就会成为严重的性能瓶颈。

缓解手段：

  1. 开启延迟加载（Lazy Loading）—— 只有真正访问 order.getUser() 时才触发嵌套查询，不访问则不查
  2. MyBatis 的一级缓存可以避免同一个 SqlSession 内对相同 user_id 的重复查询
  3. 从根本上解决——改用嵌套结果映射（JOIN 查询），见下节

多列参数传递

如果嵌套查询需要多个参数，使用 column 属性的多列语法：

<association property="user"
             column="{id=user_id, type=user_type}"
             select="com.example.mapper.UserMapper.selectUserByIdAndType"/>

格式为 {嵌套查询的参数名=主查询的列名, ...}。

5.2.2 嵌套结果（Nested ResultMap / JOIN 查询）

原理

嵌套结果的方式是：通过一条 JOIN SQL 一次性查出所有需要的数据，然后由 MyBatis 在内存中根据 resultMap 的映射规则将扁平的结果集"拆分"并组装成嵌套的对象结构。

Mapper 接口

public interface OrderMapper {
    Order selectOrderWithUserById(Long id);
    List<Order> selectAllOrdersWithUser();
}

Mapper XML

<mapper namespace="com.example.mapper.OrderMapper">

    <!-- 方式一：在 association 内部直接写映射 -->
    <resultMap id="orderWithUserNestedResultMap" type="com.example.entity.Order">
        <id     property="id"          column="o_id"/>
        <result property="orderNo"     column="order_no"/>
        <result property="userId"      column="user_id"/>
        <result property="totalAmount" column="total_amount"/>
        <result property="status"      column="status"/>
        <result property="createTime"  column="o_create_time"/>
        <association property="user" javaType="com.example.entity.User">
            <id     property="id"         column="u_id"/>
            <result property="username"   column="username"/>
            <result property="email"      column="email"/>
            <result property="phone"      column="phone"/>
            <result property="createTime" column="u_create_time"/>
        </association>
    </resultMap>

    <!-- 方式二：引用外部 resultMap（推荐，可复用） -->
    <resultMap id="orderWithUserNestedResultMap2" type="com.example.entity.Order">
        <id     property="id"          column="o_id"/>
        <result property="orderNo"     column="order_no"/>
        <result property="userId"      column="user_id"/>
        <result property="totalAmount" column="total_amount"/>
        <result property="status"      column="status"/>
        <result property="createTime"  column="o_create_time"/>
        <association property="user"
                     resultMap="com.example.mapper.UserMapper.baseUserMap"
                     columnPrefix="u_"/>
    </resultMap>

    <select id="selectOrderWithUserById" resultMap="orderWithUserNestedResultMap">
        SELECT
            o.id            AS o_id,
            o.order_no,
            o.user_id,
            o.total_amount,
            o.status,
            o.create_time   AS o_create_time,
            u.id            AS u_id,
            u.username,
            u.email,
            u.phone,
            u.create_time   AS u_create_time
        FROM t_order o
        INNER JOIN t_user u ON o.user_id = u.id
        WHERE o.id = #{id}
    </select>

    <select id="selectAllOrdersWithUser" resultMap="orderWithUserNestedResultMap">
        SELECT
            o.id            AS o_id,
            o.order_no,
            o.user_id,
            o.total_amount,
            o.status,
            o.create_time   AS o_create_time,
            u.id            AS u_id,
            u.username,
            u.email,
            u.phone,
            u.create_time   AS u_create_time
        FROM t_order o
        INNER JOIN t_user u ON o.user_id = u.id
    </select>

</mapper>

columnPrefix 的妙用

当使用 方式二 引用外部 resultMap 时，columnPrefix 属性非常实用。它的作用是：被引用的 resultMap 中的每个 column 值都会自动加上这个前缀去匹配结果集列名。

例如 baseUserMap 中的 <id column="id"/> 在加上 columnPrefix="u_" 之后，实际匹配的列名就变成了 u_id。这样做的好处：

  1. 不需要修改被引用的 resultMap，保持其通用性
  2. 避免多表 JOIN 时的列名冲突（t_order.id 和 t_user.id 都叫 id）
  3. SQL 中只需给列加上对应前缀的别名即可

执行效果

无论查询一条还是查全部，只执行一条 SQL：

SELECT o.id AS o_id, o.order_no, o.user_id, o.total_amount, o.status,
       o.create_time AS o_create_time,
       u.id AS u_id, u.username, u.email, u.phone, u.create_time AS u_create_time
FROM t_order o
INNER JOIN t_user u ON o.user_id = u.id;

MyBatis 从每一行结果中提取 o_ 前缀的列组装 Order 对象，提取 u_ 前缀的列组装 User 对象，然后将 User 设置到 Order.user 属性上。

5.2.3 两种方式的选择策略

维度 嵌套查询（Nested Select） 嵌套结果（Nested ResultMap）
SQL 条数 1 + N 条（有 N+1 问题） 1 条（JOIN 查询）
网络往返 多次 一次
是否支持延迟加载 ✅ 支持（核心优势） ❌ 不支持
SQL 复杂度 每条 SQL 简单 JOIN SQL 可能较复杂
resultMap 复杂度 简单（只需声明 select 引用） 较复杂（需要处理列别名冲突）
数据重复 无重复 JOIN 可能导致主表数据重复传输
跨 namespace 复用 天然支持（引用其他 Mapper 的 select） 支持（通过 resultMap 引用 + columnPrefix）
缓存友好度 子查询可被一级/二级缓存命中 无法利用子查询缓存
性能（少量数据） 可接受 优秀
性能（大量数据） 差（N+1 问题严重） 优秀
可维护性 SQL 独立，职责清晰 SQL 耦合度高

选择建议

┌───────────────────────────────────────────────────────────────┐
│  默认优先使用嵌套结果（JOIN）                                     │
│  ├── 查询数据量可能较大时 → 嵌套结果（避免 N+1）                   │
│  ├── 关联数据不一定每次都用到 → 嵌套查询 + 延迟加载                 │
│  ├── 关联关系很深（3层以上嵌套）→ 考虑在业务层分步查询再组装           │
│  └── 分页查询的主表 + 关联 → 主表分页用嵌套结果，深层关联用延迟加载    │
└───────────────────────────────────────────────────────────────┘

核心原则：

  1. 大多数场景选嵌套结果：一条 SQL 解决，性能有保障
  2. 需要延迟加载时选嵌套查询：配合 fetchType="lazy" 使用，按需加载
  3. 复杂场景混合使用：主关联用 JOIN，深层关联用嵌套查询 + 延迟加载

5.3 collection —— 一对多关联映射

<collection> 用于映射 Java 对象中的集合属性，即属性类型是 List<T> / Set<T> 等集合的场景。

在我们的数据模型中，Order 对象包含 List<OrderItem> orderItems —— 一个订单包含多个订单项，这就是典型的一对多关联。

5.3.1 嵌套查询

Mapper 接口

public interface OrderItemMapper {
    List<OrderItem> selectByOrderId(Long orderId);
}

Mapper XML

OrderItemMapper.xml

<mapper namespace="com.example.mapper.OrderItemMapper">

    <resultMap id="baseOrderItemMap" type="com.example.entity.OrderItem">
        <id     property="id"        column="id"/>
        <result property="orderId"   column="order_id"/>
        <result property="productId" column="product_id"/>
        <result property="quantity"  column="quantity"/>
        <result property="unitPrice" column="unit_price"/>
    </resultMap>

    <select id="selectByOrderId" resultMap="baseOrderItemMap">
        SELECT id, order_id, product_id, quantity, unit_price
        FROM t_order_item
        WHERE order_id = #{orderId}
    </select>

</mapper>

OrderMapper.xml（新增嵌套查询 resultMap）

<resultMap id="orderWithItemsNestedSelectMap" type="com.example.entity.Order">
    <id     property="id"          column="id"/>
    <result property="orderNo"     column="order_no"/>
    <result property="userId"      column="user_id"/>
    <result property="totalAmount" column="total_amount"/>
    <result property="status"      column="status"/>
    <result property="createTime"  column="create_time"/>
    <!--
        collection 嵌套查询：
        - property: Order 中的集合属性名
        - ofType:   集合中元素的类型
        - column:   传递给嵌套查询的参数
        - select:   嵌套查询的 statement id
    -->
    <collection property="orderItems"
                ofType="com.example.entity.OrderItem"
                column="id"
                select="com.example.mapper.OrderItemMapper.selectByOrderId"/>
</resultMap>

<select id="selectOrderWithItemsById" resultMap="orderWithItemsNestedSelectMap">
    SELECT id, order_no, user_id, total_amount, status, create_time
    FROM t_order
    WHERE id = #{id}
</select>

执行 selectOrderWithItemsById(1) 时的 SQL：

-- 主查询
SELECT id, order_no, user_id, total_amount, status, create_time
FROM t_order WHERE id = 1;

-- 嵌套查询（使用主查询结果的 id 作为参数）
SELECT id, order_id, product_id, quantity, unit_price
FROM t_order_item WHERE order_id = 1;

注意：collection 的嵌套查询同样存在 N+1 问题。如果查询所有订单，每个订单都会触发一次订单项子查询。

5.3.2 嵌套结果

Mapper XML

<resultMap id="orderWithItemsNestedResultMap" type="com.example.entity.Order">
    <id     property="id"          column="o_id"/>
    <result property="orderNo"     column="order_no"/>
    <result property="userId"      column="user_id"/>
    <result property="totalAmount" column="total_amount"/>
    <result property="status"      column="status"/>
    <result property="createTime"  column="o_create_time"/>
    <collection property="orderItems" ofType="com.example.entity.OrderItem">
        <id     property="id"        column="oi_id"/>
        <result property="orderId"   column="oi_order_id"/>
        <result property="productId" column="oi_product_id"/>
        <result property="quantity"  column="quantity"/>
        <result property="unitPrice" column="unit_price"/>
    </collection>
</resultMap>

<select id="selectOrderWithItemsById" resultMap="orderWithItemsNestedResultMap">
    SELECT
        o.id            AS o_id,
        o.order_no,
        o.user_id,
        o.total_amount,
        o.status,
        o.create_time   AS o_create_time,
        oi.id           AS oi_id,
        oi.order_id     AS oi_order_id,
        oi.product_id   AS oi_product_id,
        oi.quantity,
        oi.unit_price
    FROM t_order o
    LEFT JOIN t_order_item oi ON o.id = oi.order_id
    WHERE o.id = #{id}
</select>

嵌套结果的去重原理

这条 JOIN 查询对于 订单1 会返回 3 行数据（因为它有 3 个订单项）：

o_id order_no ... oi_id quantity unit_price
1 ORD20250101001 ... 1 1 108.00
1 ORD20250101001 ... 2 1 14999.00
1 ORD20250101001 ... 3 1 79.00

MyBatis 是怎么知道这 3 行应该合并成 1 个 Order + 3 个 OrderItem 的？

关键在于 <id> 标签。MyBatis 用 <id column="o_id"/> 标记的列值来判断是否是同一个 Order 对象——这 3 行的 o_id 都是 1，所以被识别为同一个 Order。而 <id column="oi_id"/> 的值分别是 1、2、3，所以被识别为 3 个不同的 OrderItem，收集到 orderItems 集合中。

重要：在使用 collection 嵌套结果映射时，必须正确声明 <id> 标签。如果遗漏，MyBatis 可能无法正确去重，导致结果集中出现重复的对象。

5.4 discriminator —— 鉴别器（类似 Java 多态映射）

<discriminator> 类似 Java 中的 switch 语句或多态的 instanceof 判断——根据某一列的值，决定使用哪个 resultMap 来映射结果。这在需要将查询结果映射到不同子类时非常有用。

场景设计：通知系统

假设系统中有一个通知表，通知类型不同，携带的额外信息也不同：

CREATE TABLE t_notification (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    type        VARCHAR(20) NOT NULL COMMENT 'SMS-短信, EMAIL-邮件, PUSH-推送',
    title       VARCHAR(200) NOT NULL,
    content     TEXT,
    recipient   VARCHAR(100) NOT NULL,
    -- 短信通知特有
    phone       VARCHAR(20),
    -- 邮件通知特有
    email_addr  VARCHAR(100),
    cc_list     VARCHAR(500),
    -- 推送通知特有
    device_token VARCHAR(200),
    badge_count  INT,
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

Java 类结构

public class Notification {
    private Long id;
    private String type;
    private String title;
    private String content;
    private String recipient;
    private LocalDateTime createTime;
    // getter / setter
}

public class SmsNotification extends Notification {
    private String phone;
    // getter / setter
}

public class EmailNotification extends Notification {
    private String emailAddr;
    private String ccList;
    // getter / setter
}

public class PushNotification extends Notification {
    private String deviceToken;
    private Integer badgeCount;
    // getter / setter
}

Mapper XML

<mapper namespace="com.example.mapper.NotificationMapper">

    <resultMap id="notificationMap" type="com.example.entity.Notification">
        <id     property="id"         column="id"/>
        <result property="type"       column="type"/>
        <result property="title"      column="title"/>
        <result property="content"    column="content"/>
        <result property="recipient"  column="recipient"/>
        <result property="createTime" column="create_time"/>
        <discriminator javaType="string" column="type">
            <case value="SMS"   resultType="com.example.entity.SmsNotification">
                <result property="phone" column="phone"/>
            </case>
            <case value="EMAIL" resultType="com.example.entity.EmailNotification">
                <result property="emailAddr" column="email_addr"/>
                <result property="ccList"    column="cc_list"/>
            </case>
            <case value="PUSH"  resultType="com.example.entity.PushNotification">
                <result property="deviceToken" column="device_token"/>
                <result property="badgeCount"  column="badge_count"/>
            </case>
        </discriminator>
    </resultMap>

    <select id="selectAll" resultMap="notificationMap">
        SELECT * FROM t_notification ORDER BY create_time DESC
    </select>

    <select id="selectById" resultMap="notificationMap">
        SELECT * FROM t_notification WHERE id = #{id}
    </select>

</mapper>

Mapper 接口

public interface NotificationMapper {
    List<Notification> selectAll();
    Notification selectById(Long id);
}

执行效果

List<Notification> notifications = notificationMapper.selectAll();
for (Notification n : notifications) {
    if (n instanceof SmsNotification sms) {
        System.out.println("短信通知 → " + sms.getPhone());
    } else if (n instanceof EmailNotification email) {
        System.out.println("邮件通知 → " + email.getEmailAddr());
    } else if (n instanceof PushNotification push) {
        System.out.println("推送通知 → " + push.getDeviceToken());
    }
}

MyBatis 会根据每行数据的 type 列值自动选择映射逻辑：

  • type = 'SMS' → 创建 SmsNotification 对象，映射 phone 字段
  • type = 'EMAIL' → 创建 EmailNotification 对象，映射 emailAddr 和 ccList
  • type = 'PUSH' → 创建 PushNotification 对象，映射 deviceToken 和 badgeCount
  • 其他值 → 使用默认的 Notification 类型

注意：<case> 标签中可以使用 resultType（内联映射）或 resultMap（引用外部 resultMap），后者适合映射较复杂的子类。<case> 中定义的映射会覆盖父级 resultMap 中同名 property 的映射（如果有的话），而不是追加。如果你希望子类同时拥有父级的映射，需要在 <case> 中引用一个 extends 了父级映射的 resultMap。

5.5 自动映射（autoMapping）与映射策略

5.5.1 什么是自动映射

MyBatis 具备自动将结果集列映射到 Java 属性的能力——当列名（或转换后的列名）与属性名匹配时，无需显式声明 <result> 标签。

5.5.2 全局映射策略（autoMappingBehavior）

在 mybatis-config.xml 的 <settings> 中配置：

<settings>
    <!-- NONE: 禁用自动映射 | PARTIAL: 部分自动映射（默认） | FULL: 完全自动映射 -->
    <setting name="autoMappingBehavior" value="PARTIAL"/>
</settings>
策略 说明
NONE 完全禁用自动映射，所有列都必须在 resultMap 中显式声明
PARTIAL（默认） 对没有嵌套映射（association/collection）的 resultMap 启用自动映射。一旦 resultMap 中出现了嵌套映射，未显式声明的列就不会自动映射
FULL 无条件启用自动映射，即使存在嵌套映射也会自动映射未声明的列

5.5.3 resultMap 级别的 autoMapping 属性

可以在单个 resultMap 上覆盖全局策略：

<!-- 对这个 resultMap 强制启用自动映射，即使全局策略是 NONE 或有嵌套映射 -->
<resultMap id="orderMap" type="Order" autoMapping="true">
    <id property="id" column="o_id"/>
    <!-- username、email 等同名列会自动映射，无需显式声明 -->
    <association property="user" javaType="User" autoMapping="true">
        <id property="id" column="u_id"/>
        <!-- user 的其他属性如果列名匹配也会自动映射 -->
    </association>
</resultMap>

5.5.4 配合 mapUnderscoreToCamelCase

自动映射能力与驼峰转换配合使用效果最佳：

<settings>
    <setting name="mapUnderscoreToCamelCase" value="true"/>
</settings>

开启后，数据库列名 create_time 会自动匹配 Java 属性 createTime，product_name 匹配 productName，以此类推。

5.5.5 最佳实践

推荐配置：
  autoMappingBehavior = PARTIAL（默认值，不需要改）
  mapUnderscoreToCamelCase = true（强烈推荐开启）

编码建议：
  1. 即使开启自动映射，<id> 标签始终显式声明
  2. 嵌套映射中的列建议显式声明，不依赖 FULL 模式
  3. 简单的 resultMap 可以放心依赖自动映射减少样板代码

5.5.6 autoMappingUnknownColumnBehavior

当自动映射遇到无法匹配的列时的行为：

<settings>
    <!-- NONE: 静默忽略（默认） | WARNING: 日志警告 | FAILING: 抛出异常 -->
    <setting name="autoMappingUnknownColumnBehavior" value="NONE"/>
</settings>

开发阶段建议设为 WARNING，便于及时发现映射遗漏。

5.6 延迟加载（Lazy Loading）的配置与原理

5.6.1 什么是延迟加载

延迟加载（也叫懒加载）是指：关联对象不在主查询时立即加载，而是推迟到第一次访问该属性时才触发查询。它只能配合嵌套查询（Nested Select）使用，对嵌套结果（Nested ResultMap）无效。

5.6.2 配置方式

全局配置

<!-- mybatis-config.xml -->
<settings>
    <!-- 全局开关：是否启用延迟加载（默认 false） -->
    <setting name="lazyLoadingEnabled" value="true"/>

    <!--
        积极/侵入式加载：访问对象的任意属性时是否触发所有延迟加载属性的加载
        MyBatis 3.4.1 之前默认 true，之后默认 false
        推荐 false——只有访问具体的延迟加载属性时才触发该属性的加载
    -->
    <setting name="aggressiveLazyLoading" value="false"/>

    <!--
        触发延迟加载的方法列表
        调用这些方法时会强制加载所有延迟属性
        默认值："equals,clone,hashCode,toString"
    -->
    <setting name="lazyLoadTriggerMethods" value="equals,clone,hashCode,toString"/>
</settings>

局部配置（推荐）

在 <association> 或 <collection> 上使用 fetchType 属性，优先级高于全局配置：

<association property="user"
             column="user_id"
             select="com.example.mapper.UserMapper.selectUserById"
             fetchType="lazy"/>

<collection property="orderItems"
            ofType="com.example.entity.OrderItem"
            column="id"
            select="com.example.mapper.OrderItemMapper.selectByOrderId"
            fetchType="eager"/>
fetchType 值 说明
lazy 延迟加载，访问时才查询
eager 立即加载，主查询完成后立刻查询

推荐做法：不开启全局延迟加载，而是在需要的 association / collection 上逐个指定 fetchType="lazy"，做到精确控制。

5.6.3 底层原理：基于动态代理

延迟加载的核心实现依赖于动态代理：

                      ┌──────────────────────────────────────┐
                      │     MyBatis 延迟加载原理               │
                      └──────────────┬───────────────────────┘
                                     │
                      ┌──────────────▼───────────────────────┐
                      │  1. 执行主查询，获取 Order 数据         │
                      └──────────────┬───────────────────────┘
                                     │
                      ┌──────────────▼───────────────────────┐
                      │  2. 创建 Order 对象时，发现 user 属性   │
                      │     配置了延迟加载                      │
                      └──────────────┬───────────────────────┘
                                     │
                      ┌──────────────▼───────────────────────┐
                      │  3. 不直接创建 Order 实例，而是创建     │
                      │     Order 的代理对象（Proxy）            │
                      │     - 优先使用 Javassist               │
                      │     - 备选使用 Cglib                    │
                      └──────────────┬───────────────────────┘
                                     │
                      ┌──────────────▼───────────────────────┐
                      │  4. 代理对象拦截 getUser() 方法调用     │
                      │     → 检查 user 属性是否已加载          │
                      │     → 未加载则执行嵌套 SQL 查询         │
                      │     → 将查询结果设置到 user 属性        │
                      │     → 返回 user 对象                   │
                      └──────────────────────────────────────┘

代理技术选择

MyBatis 通过 proxyFactory 设置选择代理实现：

<settings>
    <!-- JAVASSIST（默认）或 CGLIB -->
    <setting name="proxyFactory" value="JAVASSIST"/>
</settings>
代理技术 原理 特点
Javassist（默认） 在运行时直接操作字节码生成子类 MyBatis 内置依赖，无需额外引入 jar
Cglib 基于 ASM 框架在运行时生成目标类的子类 需额外引入 cglib 依赖

两者的共同机制：

  1. 生成目标实体类的子类（所以实体类不能是 final 的）
  2. 重写 getter 方法，在方法入口处插入拦截逻辑
  3. 首次调用 getter 时执行延迟加载 SQL，后续调用直接返回已加载的值

源码层面的关键类

ResultLoader       : 持有嵌套查询的信息（MappedStatement、参数等），负责实际执行延迟加载SQL
ResultLoaderMap    : 管理一个对象上所有待延迟加载的属性
JavassistProxyFactory / CglibProxyFactory : 创建代理对象的工厂
EnhancedResultObjectProxyImpl             : 代理对象的方法拦截器，核心逻辑所在

5.6.4 延迟加载的注意事项

  1. 序列化问题：代理对象在序列化（如 JSON 序列化、RPC 传输）时可能触发所有延迟加载，或者因为代理类型导致序列化失败。建议在跨层传输前转为 DTO

  2. SqlSession 关闭后访问：延迟加载需要通过 SqlSession 执行 SQL。如果 SqlSession 已经关闭后再访问延迟属性，会抛出异常。在 Spring 环境中，默认的 SqlSessionTemplate 会在每次操作后关闭 Session，这时需要配合 OpenSessionInViewFilter/Interceptor 或改用立即加载

  3. 调试时意外触发：在 IDE 的 Debug 模式下，鼠标悬浮查看对象属性或 Watch 窗口会触发 toString()，从而触发所有延迟加载。可以在 lazyLoadTriggerMethods 中移除 toString 来缓解

  4. 不能和嵌套结果混用：延迟加载只在嵌套查询（select 属性）模式下生效，嵌套结果（JOIN 查询 + 内联 resultMap）不支持延迟加载

5.6.5 完整配置示例

<!-- Order 的 resultMap：user 延迟加载，orderItems 延迟加载 -->
<resultMap id="orderLazyLoadMap" type="com.example.entity.Order">
    <id     property="id"          column="id"/>
    <result property="orderNo"     column="order_no"/>
    <result property="userId"      column="user_id"/>
    <result property="totalAmount" column="total_amount"/>
    <result property="status"      column="status"/>
    <result property="createTime"  column="create_time"/>
    <association property="user"
                 column="user_id"
                 select="com.example.mapper.UserMapper.selectUserById"
                 fetchType="lazy"/>
    <collection property="orderItems"
                ofType="com.example.entity.OrderItem"
                column="id"
                select="com.example.mapper.OrderItemMapper.selectByOrderId"
                fetchType="lazy"/>
</resultMap>

<select id="selectOrderLazyById" resultMap="orderLazyLoadMap">
    SELECT id, order_no, user_id, total_amount, status, create_time
    FROM t_order WHERE id = #{id}
</select>

Order order = orderMapper.selectOrderLazyById(1L);
// 此时只执行了 1 条 SQL：查询 t_order

System.out.println(order.getOrderNo());
// 不涉及关联属性，不触发额外查询

System.out.println(order.getUser().getUsername());
// 首次访问 user 属性，触发延迟加载
// → 执行 SELECT ... FROM t_user WHERE id = ?

System.out.println(order.getOrderItems().size());

5.7 复杂映射实战：订单 - 用户 - 商品三表关联

本节将把前面学到的所有知识综合运用，实现一个完整的多表关联映射：查询订单详情，包含用户信息、订单项列表，每个订单项还关联了商品信息。

5.7.1 目标对象结构

Order
├── id, orderNo, totalAmount, status, createTime
├── user: User                      ← 一对一（association）
│   └── id, username, email, phone
└── orderItems: List<OrderItem>     ← 一对多（collection）
    ├── OrderItem
    │   ├── id, quantity, unitPrice
    │   └── product: Product        ← 一对一（association）
    │       └── id, productName, price, category
    ├── OrderItem
    │   └── product: Product
    └── ...

5.7.2 方案一：嵌套结果（一条 JOIN SQL，推荐）

Mapper 接口

public interface OrderMapper {
    Order selectOrderDetailById(Long id);
    List<Order> selectAllOrderDetails();
}

Mapper XML

<mapper namespace="com.example.mapper.OrderMapper">

    <!-- ==================== 基础 ResultMap（可复用） ==================== -->

    <resultMap id="baseOrderMap" type="com.example.entity.Order">
        <id     property="id"          column="o_id"/>
        <result property="orderNo"     column="o_order_no"/>
        <result property="userId"      column="o_user_id"/>
        <result property="totalAmount" column="o_total_amount"/>
        <result property="status"      column="o_status"/>
        <result property="createTime"  column="o_create_time"/>
    </resultMap>

    <!-- ==================== 完整订单详情 ResultMap ==================== -->

    <resultMap id="orderDetailMap" type="com.example.entity.Order" extends="baseOrderMap">

        <!-- 一对一：Order → User -->
        <association property="user" javaType="com.example.entity.User">
            <id     property="id"         column="u_id"/>
            <result property="username"   column="u_username"/>
            <result property="email"      column="u_email"/>
            <result property="phone"      column="u_phone"/>
            <result property="createTime" column="u_create_time"/>
        </association>

        <!-- 一对多：Order → List<OrderItem> -->
        <collection property="orderItems" ofType="com.example.entity.OrderItem">
            <id     property="id"        column="oi_id"/>
            <result property="orderId"   column="oi_order_id"/>
            <result property="productId" column="oi_product_id"/>
            <result property="quantity"  column="oi_quantity"/>
            <result property="unitPrice" column="oi_unit_price"/>

            <!-- 一对一：OrderItem → Product（嵌套在 collection 内部） -->
            <association property="product" javaType="com.example.entity.Product">
                <id     property="id"          column="p_id"/>
                <result property="productName" column="p_product_name"/>
                <result property="price"       column="p_price"/>
                <result property="stock"       column="p_stock"/>
                <result property="category"    column="p_category"/>
            </association>
        </collection>

    </resultMap>

    <!-- ==================== SQL 片段 ==================== -->

    <sql id="orderDetailColumns">
        o.id            AS o_id,
        o.order_no      AS o_order_no,
        o.user_id       AS o_user_id,
        o.total_amount  AS o_total_amount,
        o.status        AS o_status,
        o.create_time   AS o_create_time,
        u.id            AS u_id,
        u.username      AS u_username,
        u.email         AS u_email,
        u.phone         AS u_phone,
        u.create_time   AS u_create_time,
        oi.id           AS oi_id,
        oi.order_id     AS oi_order_id,
        oi.product_id   AS oi_product_id,
        oi.quantity      AS oi_quantity,
        oi.unit_price   AS oi_unit_price,
        p.id            AS p_id,
        p.product_name  AS p_product_name,
        p.price         AS p_price,
        p.stock         AS p_stock,
        p.category      AS p_category
    </sql>

    <!-- ==================== 查询语句 ==================== -->

    <select id="selectOrderDetailById" resultMap="orderDetailMap">
        SELECT
            <include refid="orderDetailColumns"/>
        FROM t_order o
        INNER JOIN t_user u        ON o.user_id    = u.id
        LEFT  JOIN t_order_item oi ON o.id          = oi.order_id
        LEFT  JOIN t_product p     ON oi.product_id = p.id
        WHERE o.id = #{id}
    </select>

    <select id="selectAllOrderDetails" resultMap="orderDetailMap">
        SELECT
            <include refid="orderDetailColumns"/>
        FROM t_order o
        INNER JOIN t_user u        ON o.user_id    = u.id
        LEFT  JOIN t_order_item oi ON o.id          = oi.order_id
        LEFT  JOIN t_product p     ON oi.product_id = p.id
        ORDER BY o.id, oi.id
    </select>

</mapper>

执行效果分析

调用 selectOrderDetailById(1) 只执行一条 SQL，数据库返回 3 行（因为订单 1 有 3 个订单项）：

o_id o_order_no u_id u_username oi_id oi_quantity p_id p_product_name
1 ORD20250101001 1 张三 1 1 1 Java编程思想
1 ORD20250101001 1 张三 2 1 2 MacBook Pro 14
1 ORD20250101001 1 张三 3 1 3 MyBatis实战指南

MyBatis 映射过程：

  1. 读取第 1 行，o_id = 1 创建 Order 对象，u_id = 1 创建 User 设置到 order.user
  2. oi_id = 1 创建 OrderItem，p_id = 1 创建 Product 设置到 orderItem.product，将 OrderItem 加入 orderItems 列表
  3. 读取第 2 行，o_id = 1 与已有 Order 匹配（同一个对象），u_id = 1 同理复用
  4. oi_id = 2 是新的 OrderItem，创建并加入列表
  5. 第 3 行同理

最终结果：1 个 Order 对象，内含 1 个 User + 3 个 OrderItem，每个 OrderItem 内含 1 个 Product。

使用示例

Order order = orderMapper.selectOrderDetailById(1L);

System.out.println("订单号：" + order.getOrderNo());
System.out.println("用户：" + order.getUser().getUsername());
System.out.println("订单项数量：" + order.getOrderItems().size());

for (OrderItem item : order.getOrderItems()) {
    System.out.printf("  商品：%s × %d = ¥%.2f%n",
        item.getProduct().getProductName(),
        item.getQuantity(),

}

5.7.3 方案二：混合策略（主关联 JOIN + 深层关联延迟加载）

在实际项目中，有时并不总是需要加载所有层级的关联数据。这时可以采用混合策略：

<!-- 混合策略 ResultMap：User 用 JOIN，OrderItems 和 Product 用延迟加载 -->
<resultMap id="orderMixedMap" type="com.example.entity.Order">
    <id     property="id"          column="o_id"/>
    <result property="orderNo"     column="order_no"/>
    <result property="userId"      column="user_id"/>
    <result property="totalAmount" column="total_amount"/>
    <result property="status"      column="status"/>
    <result property="createTime"  column="o_create_time"/>

    <!-- User 用嵌套结果：一次 JOIN 搞定 -->
    <association property="user" javaType="com.example.entity.User">
        <id     property="id"         column="u_id"/>
        <result property="username"   column="username"/>
        <result property="email"      column="email"/>
        <result property="phone"      column="phone"/>
        <result property="createTime" column="u_create_time"/>
    </association>

    <!-- OrderItems 用嵌套查询 + 延迟加载：按需加载 -->
    <collection property="orderItems"
                ofType="com.example.entity.OrderItem"
                column="o_id"
                select="com.example.mapper.OrderItemMapper.selectWithProductByOrderId"
                fetchType="lazy"/>
</resultMap>

<select id="selectOrderMixedById" resultMap="orderMixedMap">
    SELECT
        o.id AS o_id, o.order_no, o.user_id, o.total_amount, o.status,
        o.create_time AS o_create_time,
        u.id AS u_id, u.username, u.email, u.phone,
        u.create_time AS u_create_time
    FROM t_order o
    INNER JOIN t_user u ON o.user_id = u.id
    WHERE o.id = #{id}
</select>

OrderItemMapper.xml（带 Product 关联的查询）

<resultMap id="orderItemWithProductMap" type="com.example.entity.OrderItem">
    <id     property="id"        column="oi_id"/>
    <result property="orderId"   column="order_id"/>
    <result property="productId" column="product_id"/>
    <result property="quantity"  column="quantity"/>
    <result property="unitPrice" column="unit_price"/>
    <association property="product" javaType="com.example.entity.Product">
        <id     property="id"          column="p_id"/>
        <result property="productName" column="product_name"/>
        <result property="price"       column="price"/>
        <result property="stock"       column="stock"/>
        <result property="category"    column="category"/>
    </association>
</resultMap>

<select id="selectWithProductByOrderId" resultMap="orderItemWithProductMap">
    SELECT
        oi.id AS oi_id, oi.order_id, oi.product_id, oi.quantity, oi.unit_price,
        p.id AS p_id, p.product_name, p.price, p.stock, p.category
    FROM t_order_item oi
    INNER JOIN t_product p ON oi.product_id = p.id
    WHERE oi.order_id = #{orderId}
</select>

执行效果

Order order = orderMapper.selectOrderMixedById(1L);
// SQL 1: SELECT ... FROM t_order o JOIN t_user u ... WHERE o.id = 1
// 此时 user 已加载，orderItems 未加载

System.out.println(order.getUser().getUsername()); // → "张三"（无额外 SQL）

// 只有在真正需要时才加载订单项
List<OrderItem> items = order.getOrderItems();
// SQL 2: SELECT ... FROM t_order_item oi JOIN t_product p ... WHERE oi.order_id = 1
// 此时才执行第二条 SQL，一次性加载所有订单项及关联商品

这种混合策略的优势：

  • 用户信息作为订单页面必然展示的数据，通过 JOIN 一次性获取
  • 订单项列表可能需要用户点击"展开详情"才看到，通过延迟加载按需获取
  • 总共最多执行 2 条 SQL（而非 1 + N 条），兼顾性能和灵活性

5.7.4 方案对比总结

方案 SQL 条数 优势 劣势 适用场景
嵌套结果（全 JOIN） 1 条 性能最优，一次网络往返 SQL 复杂，数据冗余传输 确定需要所有关联数据
嵌套查询（全延迟） 1 + N 条 代码简洁，按需加载 潜在的 N+1 问题 关联数据很少被访问
混合策略 2 条左右 灵活平衡 设计稍复杂 部分关联必需，部分按需
业务层组装 多条独立查询 完全可控，利于缓存 代码量大，需要手动组装 跨服务调用、需要独立缓存

本章小结

知识点 核心要点
resultMap 基础 <id> 必须声明（影响去重和缓存）；<constructor> 用于不可变对象
association 一对一映射；嵌套查询简单但有 N+1 风险；嵌套结果高效但 SQL 复杂
collection 一对多映射；ofType 指定集合元素类型；<id> 是去重关键
discriminator 多态映射；根据列值决定实例化哪个子类
autoMapping PARTIAL 模式 + mapUnderscoreToCamelCase 是最佳实践
延迟加载 仅适用于嵌套查询；基于 Javassist/Cglib 代理；注意序列化和 Session 关闭问题
复杂映射实战 优先使用嵌套结果（JOIN）；深层关联考虑混合策略（JOIN + 延迟加载）
