第八章 插件（拦截器）机制 📖

理解原理，掌握常用插件的使用，能自定义简单插件。

MyBatis 提供了一套强大的插件机制，允许在 SQL 执行的关键节点进行拦截和增强。本质上，MyBatis 插件就是拦截器（Interceptor），基于 JDK 动态代理实现，可以在不修改框架源码的情况下对 MyBatis 的核心行为进行扩展。

8.1 MyBatis 四大组件

MyBatis 的 SQL 执行流程由四个核心组件协同完成，插件只能拦截这四大组件中的方法：

组件 职责 可拦截的方法
Executor SQL 执行的总调度器，负责一级缓存、事务提交/回滚，委派给 StatementHandler 执行具体 SQL update(MappedStatement, Object)
query(MappedStatement, Object, RowBounds, ResultHandler)
query(MappedStatement, Object, RowBounds, ResultHandler, CacheKey, BoundSql)
queryCursor(MappedStatement, Object, RowBounds)
flushStatements()
commit(boolean)
rollback(boolean)
close(boolean)
getTransaction()
createCacheKey(MappedStatement, Object, RowBounds, BoundSql)
isCached(MappedStatement, CacheKey)
StatementHandler 负责创建 JDBC Statement 对象，设置参数，执行 SQL，处理结果集 prepare(Connection, Integer)
parameterize(Statement)
batch(Statement)
update(Statement)
query(Statement, ResultHandler)
queryCursor(Statement)
getBoundSql()
getParameterHandler()
ParameterHandler 负责将用户传入的参数设置到 PreparedStatement 中 getParameterObject()
setParameters(PreparedStatement)
ResultSetHandler 负责将 JDBC ResultSet 映射为 Java 对象 handleResultSets(Statement)
handleCursorResultSets(Statement)
handleOutputParameters(CallableStatement)

实践指导：最常拦截的组件是 Executor（适合做缓存、事务等全局性增强）和 StatementHandler（适合做 SQL 改写、分页等操作）。ParameterHandler 和 ResultSetHandler 较少直接拦截。

四大组件在执行链路中的关系：

SqlSession
  └── Executor（调度 + 缓存）
        └── StatementHandler（SQL 执行）
              ├── ParameterHandler（参数设置）
              └── ResultSetHandler（结果映射）

8.2 插件原理 —— 基于 JDK 动态代理的责任链

8.2.1 Interceptor 接口

所有 MyBatis 插件都必须实现 org.apache.ibatis.plugin.Interceptor 接口：

public interface Interceptor {
    // 拦截逻辑的核心方法，Invocation 封装了目标对象、方法和参数
    Object intercept(Invocation invocation) throws Throwable;

    // 决定是否对 target 生成代理对象（默认实现使用 Plugin.wrap）
    default Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }

    // 接收配置文件中传入的属性参数
    default void setProperties(Properties properties) {
        // NOP
    }
}

8.2.2 Plugin.wrap() 方法详解

Plugin.wrap() 是代理链构建的核心，其工作机制如下：

public class Plugin implements InvocationHandler {

    public static Object wrap(Object target, Interceptor interceptor) {
        // 1. 解析 @Intercepts 注解，获取该拦截器关心的 类型 → 方法集合
        Map<Class<?>, Set<Method>> signatureMap = getSignatureMap(interceptor);

        // 2. 获取目标对象的实际类型
        Class<?> type = target.getClass();

        // 3. 找出 target 实现的接口中，哪些在 signatureMap 里有对应的拦截方法
        Class<?>[] interfaces = getAllInterfaces(type, signatureMap);

        // 4. 如果有匹配的接口 → 生成 JDK 动态代理；否则 → 原样返回
        if (interfaces.length > 0) {
            return Proxy.newProxyInstance(
                type.getClassLoader(),
                interfaces,
                new Plugin(target, interceptor, signatureMap)  // this 作为 InvocationHandler
            );
        }
        return target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        Set<Method> methods = signatureMap.get(method.getDeclaringClass());
        // 只有在签名匹配时才走拦截逻辑
        if (methods != null && methods.contains(method)) {
            return interceptor.intercept(new Invocation(target, method, args));
        }
        // 否则直接放行
        return method.invoke(target, args);
    }
}

关键点：

  1. Plugin.wrap() 并非无条件创建代理，而是先检查拦截器的 @Signature 是否匹配目标对象的接口方法。
  2. 如果不匹配，直接返回原对象，不会产生额外开销。
  3. 如果匹配，会创建一个 JDK 动态代理，将拦截逻辑织入。

8.2.3 代理链的构建过程

当配置了多个插件时，MyBatis 会按照配置顺序依次调用每个插件的 plugin() 方法，形成嵌套代理（责任链）：

// Configuration 类中的逻辑（伪代码）
public ParameterHandler newParameterHandler(...) {
    ParameterHandler handler = new DefaultParameterHandler(...);
    // interceptorChain 按配置顺序保存所有拦截器
    handler = (ParameterHandler) interceptorChain.pluginAll(handler);
    return handler;
}

// InterceptorChain.pluginAll()
public Object pluginAll(Object target) {
    for (Interceptor interceptor : interceptors) {
        target = interceptor.plugin(target);  // 逐层包装代理
    }
    return target;
}

代理层叠示意（假设有 PluginA、PluginB、PluginC 三个插件，配置顺序为 A → B → C）：

调用方
  → Proxy(PluginC)        ← 最后包装的在最外层，最先执行
    → Proxy(PluginB)
      → Proxy(PluginA)
        → 原始目标对象

注意：配置顺序靠后的插件包裹在最外层，因此最先被执行。如果多个插件拦截同一个方法，执行顺序与配置顺序相反。

8.3 @Intercepts / @Signature 注解详解

8.3.1 注解定义

@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface Intercepts {
    Signature[] value();   // 一个拦截器可以拦截多个方法签名
}

@Documented
@Retention(RetentionPolicy.RUNTIME)
@Target({})
public @interface Signature {
    Class<?> type();       // 要拦截的组件类型（四大接口之一）
    String method();       // 要拦截的方法名
    Class<?>[] args();     // 方法的参数类型列表（用于精确匹配重载方法）
}

8.3.2 使用方式

@Intercepts 注解标注在实现了 Interceptor 接口的类上，通过一个或多个 @Signature 指定要拦截的目标方法：

@Intercepts({
    @Signature(
        type = Executor.class,                    // 拦截 Executor
        method = "query",                         // 拦截 query 方法
        args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}
    ),
    @Signature(
        type = Executor.class,
        method = "update",                        // 同时拦截 update 方法
        args = {MappedStatement.class, Object.class}
    )
})
public class MyPlugin implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 拦截逻辑
        return invocation.proceed();  // 放行，执行原方法
    }
}

8.3.3 常见拦截签名速查

目标 type method args
拦截查询 Executor.class "query" {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}
拦截更新（增/删/改） Executor.class "update" {MappedStatement.class, Object.class}
拦截 SQL 预编译 StatementHandler.class "prepare" {Connection.class, Integer.class}
拦截参数设置 ParameterHandler.class "setParameters" {PreparedStatement.class}
拦截结果集处理 ResultSetHandler.class "handleResultSets" {Statement.class}

8.3.4 插件注册方式

方式一：mybatis-config.xml 中注册

<plugins>
    <plugin interceptor="com.example.plugin.MyPlugin">
        <property name="someProperty" value="100"/>
    </plugin>
</plugins>

方式二：Spring Boot 中注册（推荐）

将插件类声明为 Spring Bean 即可，MyBatis-Spring-Boot-Starter 会自动注册：

    // ...
}

或通过 @Configuration 类显式注册：

@Configuration

    @Bean
    public MyPlugin myPlugin() {
        return new MyPlugin();
    }
}

8.4 自定义插件实战

8.4.1 SQL 执行耗时统计插件

这是最经典的 MyBatis 插件场景之一，用于监控慢 SQL。

import org.apache.ibatis.executor.statement.StatementHandler;
import org.apache.ibatis.mapping.BoundSql;
import org.apache.ibatis.mapping.MappedStatement;
import org.apache.ibatis.plugin.*;
import org.apache.ibatis.reflection.MetaObject;
import org.apache.ibatis.reflection.SystemMetaObject;
import org.apache.ibatis.session.ResultHandler;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.sql.Statement;
import java.util.Properties;

@Intercepts({
    @Signature(type = StatementHandler.class, method = "query",  args = {Statement.class, ResultHandler.class}),
    @Signature(type = StatementHandler.class, method = "update", args = {Statement.class}),
    @Signature(type = StatementHandler.class, method = "batch",  args = {Statement.class})
})
public class SqlExecutionTimePlugin implements Interceptor {

    private static final Logger log = LoggerFactory.getLogger(SqlExecutionTimePlugin.class);

    /**
     * 慢 SQL 阈值（毫秒），超过此值会以 WARN 级别输出
     */
    private long slowSqlThreshold = 1000L;

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();

        // 通过 MetaObject 反射获取内部的 MappedStatement，从而得到 SQL ID
        MetaObject metaObject = SystemMetaObject.forObject(handler);
        MappedStatement ms = (MappedStatement) metaObject.getValue("delegate.mappedStatement");
        String sqlId = ms.getId();

        BoundSql boundSql = handler.getBoundSql();
        String sql = boundSql.getSql().replaceAll("\\s+", " ").trim();

        long startTime = System.currentTimeMillis();
        try {
            return invocation.proceed();
        } finally {
            long elapsed = System.currentTimeMillis() - startTime;
            if (elapsed >= slowSqlThreshold) {
                log.warn("[慢SQL] {}ms | ID: {} | SQL: {}", elapsed, sqlId, sql);
            } else {
                log.debug("[SQL] {}ms | ID: {} | SQL: {}", elapsed, sqlId, sql);
            }
        }
    }

    @Override
    public void setProperties(Properties properties) {
        String threshold = properties.getProperty("slowSqlThreshold");
        if (threshold != null) {
            this.slowSqlThreshold = Long.parseLong(threshold);
        }
    }
}

Spring Boot 注册：

@Configuration
public class MyBatisPluginConfig {

    @Bean
    public SqlExecutionTimePlugin sqlExecutionTimePlugin() {
        SqlExecutionTimePlugin plugin = new SqlExecutionTimePlugin();
        Properties props = new Properties();
        props.setProperty("slowSqlThreshold", "500");  // 设置慢 SQL 阈值为 500ms
        plugin.setProperties(props);
        return plugin;
    }
}

XML 注册方式：

<plugins>
    <plugin interceptor="com.example.plugin.SqlExecutionTimePlugin">
        <property name="slowSqlThreshold" value="500"/>
    </plugin>
</plugins>

输出效果：

[WARN ] [慢SQL] 1523ms | ID: com.example.mapper.UserMapper.selectByCondition | SQL: SELECT * FROM user WHERE status = ? AND create_time > ?
[DEBUG] [SQL] 12ms | ID: com.example.mapper.UserMapper.selectById | SQL: SELECT * FROM user WHERE id = ?

8.4.2 通用字段自动填充插件（createTime / updateTime）

在插入和更新操作时，自动为实体对象填充 createTime 和 updateTime 字段，避免业务代码中重复手动设置。

import org.apache.ibatis.executor.Executor;
import org.apache.ibatis.mapping.MappedStatement;
import org.apache.ibatis.mapping.SqlCommandType;
import org.apache.ibatis.plugin.*;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.lang.reflect.Field;
import java.time.LocalDateTime;
import java.util.*;

@Intercepts({
    @Signature(
        type = Executor.class,
        method = "update",
        args = {MappedStatement.class, Object.class}
    )
})
public class AutoFillTimePlugin implements Interceptor {

    private static final Logger log = LoggerFactory.getLogger(AutoFillTimePlugin.class);

    private static final String CREATE_TIME_FIELD = "createTime";
    private static final String UPDATE_TIME_FIELD = "updateTime";

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        Object[] args = invocation.getArgs();
        MappedStatement ms = (MappedStatement) args[0];
        Object parameter = args[1];

        if (parameter == null) {
            return invocation.proceed();
        }

        SqlCommandType sqlCommandType = ms.getSqlCommandType();

        if (SqlCommandType.INSERT == sqlCommandType) {
            fillFieldValue(parameter, CREATE_TIME_FIELD, LocalDateTime.now());
            fillFieldValue(parameter, UPDATE_TIME_FIELD, LocalDateTime.now());
        } else if (SqlCommandType.UPDATE == sqlCommandType) {
            fillFieldValue(parameter, UPDATE_TIME_FIELD, LocalDateTime.now());
        }

        return invocation.proceed();
    }

    /**
     * 通过反射设置字段值。
     * 仅在字段存在且当前值为 null 时才填充，避免覆盖业务代码显式设置的值。
     */
    private void fillFieldValue(Object obj, String fieldName, Object value) {
        if (obj == null) {
            return;
        }

        // 处理 MyBatis 包装的 Map 类型参数（多参数场景）
        if (obj instanceof Map) {
            Map<?, ?> map = (Map<?, ?>) obj;
            for (Object val : map.values()) {
                if (val != null && !(val instanceof String) && !(val instanceof Number)) {
                    fillFieldValue(val, fieldName, value);
                }
            }
            return;
        }

        // 处理集合类型（批量插入场景）
        if (obj instanceof Collection) {
            for (Object item : (Collection<?>) obj) {
                fillFieldValue(item, fieldName, value);
            }
            return;
        }

        // 处理普通 POJO
        try {
            Field field = getField(obj.getClass(), fieldName);
            if (field != null) {
                field.setAccessible(true);
                Object currentValue = field.get(obj);
                if (currentValue == null) {
                    field.set(obj, value);
                    log.debug("自动填充字段: {}.{} = {}", obj.getClass().getSimpleName(), fieldName, value);
                }
            }
        } catch (IllegalAccessException e) {
            log.warn("自动填充字段失败: {}.{}", obj.getClass().getSimpleName(), fieldName, e);
        }
    }

    /**
     * 递归查找字段（包括父类）
     */
    private Field getField(Class<?> clazz, String fieldName) {
        while (clazz != null && clazz != Object.class) {
            try {
                return clazz.getDeclaredField(fieldName);
            } catch (NoSuchFieldException e) {
                clazz = clazz.getSuperclass();
            }
        }
        return null;
    }
}

实体类示例：

public class User {
    private Long id;
    private String name;
    private String email;
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
    // getter / setter 省略
}

Spring Boot 注册：

@Configuration
public class MyBatisPluginConfig {

    @Bean
    public AutoFillTimePlugin autoFillTimePlugin() {
        return new AutoFillTimePlugin();
    }
}

使用效果：

user.setEmail("zhangsan@example.com");
// 无需手动设置 createTime 和 updateTime

// 插件自动填充: createTime = 2025-01-15T10:30:00, updateTime = 2025-01-15T10:30:00

8.5 常用第三方插件

8.5.1 PageHelper —— 分页插件 ⭐

PageHelper 是 MyBatis 生态中最流行的分页插件，通过拦截 Executor 的 query 方法，在原始 SQL 外层自动添加 LIMIT（或对应数据库的分页语法），实现物理分页。

Maven 依赖

Spring Boot 项目（推荐使用 starter）：

<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper-spring-boot-starter</artifactId>
    <version>2.1.0</version>
</dependency>

starter 会自动注册 PageInterceptor，无需手动配置拦截器。

非 Spring Boot 项目：

<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper</artifactId>
    <version>6.1.0</version>
</dependency>

Spring Boot 配置

在 application.yml 中配置：

pagehelper:
  # 指定数据库方言，PageHelper 会根据方言生成对应的分页 SQL
  helper-dialect: mysql
  # 合理化分页：pageNum <= 0 时查询第一页，pageNum > 总页数时查询最后一页
  reasonable: true
  # 支持通过 Mapper 接口参数传递分页参数
  support-methods-arguments: true
  # 分页参数配置
  params: count=countSql

常用配置项说明：

参数 说明 默认值
helperDialect 数据库方言（mysql / oracle / postgresql / sqlserver 等） 自动检测
reasonable 分页合理化，防止越界查询 false
supportMethodsArguments 是否支持接口参数传递分页参数 false
pageSizeZero pageSize = 0 时是否查询全部结果 false
offsetAsPageNum 是否将 RowBounds 的 offset 作为 pageNum 使用 false

使用示例

基本用法 —— PageHelper.startPage()

import com.github.pagehelper.PageHelper;
import com.github.pagehelper.PageInfo;

@Service
public class UserService {

    @Autowired
    private UserMapper userMapper;

    public PageInfo<User> getUsersByPage(int pageNum, int pageSize) {
        // 1. 调用 startPage，设置分页参数（必须紧跟在查询方法之前）
        PageHelper.startPage(pageNum, pageSize);

        // 2. 紧跟的第一个查询会被自动分页
        List<User> users = userMapper.selectAll();

        // 3. 用 PageInfo 包装结果，获取完整分页信息
        PageInfo<User> pageInfo = new PageInfo<>(users);
        return pageInfo;
    }
}

PageInfo 包含的分页信息：

pageInfo.isHasPreviousPage(); // 是否有上一页

pageInfo.isIsLastPage();    // 是否为最后一页

带条件的分页查询：

public PageInfo<User> searchUsers(String keyword, int pageNum, int pageSize) {
    PageHelper.startPage(pageNum, pageSize);
    // 带条件查询同样被分页
    List<User> users = userMapper.selectByKeyword(keyword);
    return new PageInfo<>(users);
}

PageHelper 注意事项

  1. startPage() 必须紧跟查询方法：中间不能有其他 SQL 查询语句，否则分页会作用到错误的查询上。
// ❌ 错误用法：startPage 和目标查询之间插入了其他查询
PageHelper.startPage(1, 10);
User admin = userMapper.selectById(1L);  // 分页会作用到这条查询上！
List<User> users = userMapper.selectAll(); // 这条不会被分页

// ✅ 正确用法
User admin = userMapper.selectById(1L);
PageHelper.startPage(1, 10);
List<User> users = userMapper.selectAll(); // 分页正确作用到这条查询
  1. 线程安全：PageHelper.startPage() 使用 ThreadLocal 保存分页参数，查询执行后会自动清除。但如果 startPage() 之后发生异常导致查询没有执行，需要手动清除：
try {
    PageHelper.startPage(pageNum, pageSize);
    List<User> users = userMapper.selectAll();
    return new PageInfo<>(users);
} finally {
    PageHelper.clearPage();  // 确保 ThreadLocal 被清除
}
  1. 只对第一个查询生效：startPage() 只对紧跟其后的第一个 MyBatis 查询方法生效，后续查询不会被分页。

知识点 要点
四大组件 Executor（调度）、StatementHandler（SQL 执行）、ParameterHandler（参数设置）、ResultSetHandler（结果映射）
