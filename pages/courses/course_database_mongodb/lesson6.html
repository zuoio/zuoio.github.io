第06章 索引基础

索引是数据库性能优化的基石。在 MongoDB 中，合理的索引设计可以将查询速度从秒级提升到毫秒级。本章将系统讲解索引的原理、常用索引类型、在 Spring Boot 中创建索引的方式，以及如何使用 explain() 分析和优化查询性能。

6.1 索引的作用与原理简述

6.1.1 为什么需要索引

想象你有一本 500 页的技术书籍，要查找"聚合管道"这个词出现在哪些页面。你有两种方式：

  1. 逐页翻阅：从第 1 页开始，一页一页翻到第 500 页。虽然一定能找到，但效率极低。
  2. 查看目录/索引：翻到书末的索引页，直接找到"聚合管道 → 第 127、203、315 页"，然后精确跳转。

MongoDB 中的索引就像书末的索引页。没有索引时，MongoDB 必须执行全集合扫描（Collection Scan，简称 COLLSCAN） —— 逐个检查集合中的每个文档来匹配查询条件。当集合包含数十万甚至数百万文档时，全集合扫描的代价极其高昂。

有索引 vs 无索引的性能对比：

假设 users 集合有 100 万条文档，查询 email = "zhangsan@example.com"：

对比项 无索引（COLLSCAN） 有索引（IXSCAN）
扫描方式 逐个扫描所有文档 通过索引定位，直接跳转
扫描文档数 1,000,000 1（精确匹配）
时间复杂度 O(n) O(log n)
查询耗时 数百毫秒甚至数秒 通常 < 1 毫秒
CPU & I/O 开销 极高 极低 
// 在 mongosh 中验证：查看无索引时的查询计划
db.users.find({ email: "zhangsan@example.com" }).explain("executionStats")

// 关键输出（无索引）：
// "winningPlan": { "stage": "COLLSCAN" }        ← 全集合扫描
// "totalDocsExamined": 1000000                   ← 扫描了全部100万文档
// "executionTimeMillis": 850                     ← 耗时850毫秒

// 创建索引后：
db.users.createIndex({ email: 1 })
db.users.find({ email: "zhangsan@example.com" }).explain("executionStats")

// 关键输出（有索引）：
// "winningPlan": { "stage": "FETCH", "inputStage": { "stage": "IXSCAN" } }  ← 索引扫描
// "totalDocsExamined": 1                         ← 只扫描了1个文档
// "totalKeysExamined": 1                         ← 只检查了1个索引键
// "executionTimeMillis": 0                       ← 几乎瞬间完成

6.1.2 B-Tree 索引结构简述

MongoDB 默认使用 B-Tree（B树） 作为索引的底层数据结构（准确来说，WiredTiger 引擎使用的是 B+ Tree 的变体）。

B-Tree 的核心特点：

  1. 平衡多路搜索树：所有叶子节点在同一层，保证查询路径长度一致
  2. 有序存储：索引键按照指定顺序排列，支持范围查询和排序
  3. 对数时间复杂度：查找任意一个键的时间复杂度为 O(log n)

索引结构示意图（简化）：

假设对 age 字段建立升序索引，集合中有以下文档：

文档数据:
  doc1: {age: 18, name: "小明"}
  doc2: {age: 22, name: "小红"}
  doc3: {age: 25, name: "张三"}
  doc4: {age: 30, name: "李四"}
  doc5: {age: 35, name: "王五"}
  doc6: {age: 40, name: "赵六"}

索引 B-Tree 结构:
                    [25, 35]
                   /    |    \
              [18,22]  [30]  [40]
               ↓  ↓     ↓     ↓
             doc1 doc2  doc4  doc6
                  doc3  doc5

查找 age=30 的过程:
  根节点 [25, 35] → 25 < 30 < 35 → 走中间分支
  到达 [30] → 找到！→ 指向 doc4
  总共只需要 2 次比较，而非扫描 6 个文档

B-Tree 索引支持的操作：

  • 精确匹配：{ age: 25 } → 直接定位
  • 范围查询：{ age: { $gt: 20, $lt: 35 } } → 定位起点后顺序遍历
  • 排序：索引本身有序，可以避免额外的内存排序
  • 前缀匹配：对于复合索引 { a: 1, b: 1 }，可以只用 a 字段查询

6.1.3 索引的代价

索引并非免费的，它有以下代价：

代价 说明
存储空间 每个索引都需要额外的磁盘空间，索引数据量通常为原始数据的 10%~30%
写入性能 每次插入、更新、删除文档时，相关索引也需要同步更新
内存占用 索引数据需要尽量保持在内存中才能发挥最大效果

最佳实践：不要创建过多不必要的索引。每个集合建议索引数量不超过 10 个。只为频繁查询的字段创建索引。

6.2 单字段索引与复合索引

6.2.1 单字段索引

单字段索引是最基本的索引类型，对集合中的一个字段创建索引。

创建语法：

// 语法：db.collection.createIndex({ field: direction })
// direction: 1 表示升序，-1 表示降序

// 对 email 字段创建升序索引
db.users.createIndex({ email: 1 })

// 对 createdAt 字段创建降序索引
db.users.createIndex({ createdAt: -1 })

// 创建索引时指定名称（建议为索引起有意义的名称）
db.users.createIndex(
    { email: 1 },
    { name: "idx_users_email" }
)

升序(1)和降序(-1)的区别：

对于单字段索引，升序和降序在查询时效果完全相同 —— MongoDB 可以双向遍历索引。区别主要体现在复合索引的排序场景中。

// 以下两个索引在单字段查询时效果一样
db.users.createIndex({ age: 1 })   // 升序
db.users.createIndex({ age: -1 })  // 降序

// 无论用哪个索引，以下查询都能受益
db.users.find({ age: { $gt: 25 } }).sort({ age: 1 })   // 升序排序 ✓
db.users.find({ age: { $gt: 25 } }).sort({ age: -1 })  // 降序排序 ✓

查看集合的所有索引：

db.users.getIndexes()

// 输出示例：
// [
//   { "v": 2, "key": { "_id": 1 }, "name": "_id_" },              ← 默认索引
//   { "v": 2, "key": { "email": 1 }, "name": "idx_users_email" }  ← 自定义索引
// ]

注意：_id 字段的索引是 MongoDB 自动创建的，不能删除。

删除索引：

// 根据索引名称删除
db.users.dropIndex("idx_users_email")

// 根据索引定义删除
db.users.dropIndex({ email: 1 })

// 删除除 _id 外的所有索引
db.users.dropIndexes()

6.2.2 复合索引

复合索引（Compound Index）是对多个字段创建的索引。当查询条件涉及多个字段时，复合索引比多个单字段索引更高效。

创建复合索引：

// 对 status 和 createdAt 创建复合索引
db.users.createIndex({ status: 1, createdAt: -1 })

// 对 status、age、createdAt 三个字段创建复合索引
db.users.createIndex(
    { status: 1, age: 1, createdAt: -1 },
    { name: "idx_users_status_age_createdAt" }
)

6.2.3 字段顺序的重要性

复合索引中字段的顺序至关重要，它决定了索引的使用效率和适用范围。

// 创建复合索引：{ status: 1, age: 1, createdAt: -1 }

// 可以理解为索引按照以下方式组织数据：
//
//   status    age    createdAt          文档
//   ─────────────────────────────────────────
//   ACTIVE    18     2024-03-15         → doc1
//   ACTIVE    22     2024-03-10         → doc2
//   ACTIVE    25     2024-03-01         → doc3
//   ACTIVE    30     2024-02-15         → doc4
//   INACTIVE  20     2024-03-12         → doc5
//   INACTIVE  35     2024-01-20         → doc6
//
// 先按 status 排序，status 相同的按 age 排序，age 相同的按 createdAt 降序排序

6.2.4 前缀匹配原则

复合索引遵循前缀匹配原则（Prefix Rule）：MongoDB 可以使用复合索引来支持对索引字段前缀的查询。

对于复合索引 { a: 1, b: 1, c: 1 }，其可用的前缀有：

  • { a: 1 }
  • { a: 1, b: 1 }
  • { a: 1, b: 1, c: 1 }
// 假设索引：{ status: 1, age: 1, createdAt: -1 }

// ✅ 可以使用索引的查询（匹配前缀）
db.users.find({ status: "ACTIVE" })                          // 前缀: status
db.users.find({ status: "ACTIVE", age: 25 })                 // 前缀: status + age
db.users.find({ status: "ACTIVE", age: { $gt: 20 }, createdAt: { $lt: new Date() } })
                                                              // 完整索引

// ❌ 无法使用索引的查询（不匹配前缀）
db.users.find({ age: 25 })                                   // 跳过了 status
db.users.find({ createdAt: { $gt: new Date("2024-01-01") } }) // 跳过了 status 和 age
db.users.find({ age: 25, createdAt: { $lt: new Date() } })   // 跳过了 status

前缀匹配示例汇总：

索引定义 查询条件 是否命中索引 说明
{a:1, b:1, c:1} {a: "x"} ✅ 命中 匹配前缀 {a}
{a:1, b:1, c:1} {a: "x", b: "y"} ✅ 命中 匹配前缀 {a, b}
{a:1, b:1, c:1} {a: "x", b: "y", c: "z"} ✅ 命中 完整匹配
{a:1, b:1, c:1} {b: "y"} ❌ 不命中 跳过了 a
{a:1, b:1, c:1} {c: "z"} ❌ 不命中 跳过了 a 和 b
{a:1, b:1, c:1} {a: "x", c: "z"} ⚠️ 部分命中 只能用 {a} 前缀，c 无法走索引

6.2.5 复合索引的排序方向

在复合索引中，字段的排序方向（升序/降序）会影响排序查询的效率：

// 索引：{ status: 1, createdAt: -1 }

// ✅ 能利用索引排序（方向匹配或完全相反）
db.users.find({ status: "ACTIVE" }).sort({ status: 1, createdAt: -1 })   // 完全匹配
db.users.find({ status: "ACTIVE" }).sort({ status: -1, createdAt: 1 })   // 完全相反也行

// ❌ 无法利用索引排序（方向不一致）
db.users.find({ status: "ACTIVE" }).sort({ status: 1, createdAt: 1 })    // 方向冲突
db.users.find({ status: "ACTIVE" }).sort({ status: -1, createdAt: -1 })  // 方向冲突

规则：复合索引的排序方向必须与查询的排序方向完全一致或完全相反，才能利用索引排序。

6.2.6 索引设计建议

// 场景：管理后台的用户列表，支持按状态筛选，按创建时间倒序排列
// 推荐索引：
db.users.createIndex({ status: 1, createdAt: -1 })

// 场景：根据用户名模糊搜索 + 状态筛选 + 时间范围
// 推荐索引：（将等值查询字段放前面，范围查询字段放后面）
db.users.createIndex({ status: 1, createdAt: -1, name: 1 })

// 场景：商品列表，按分类筛选 + 价格范围 + 销量倒序
db.products.createIndex({ category: 1, price: 1, sales: -1 })

6.3 唯一索引与 TTL 索引

6.3.1 唯一索引

唯一索引（Unique Index）确保索引字段的值在集合中是唯一的，类似于关系型数据库的 UNIQUE 约束。

创建唯一索引：

// 对 email 字段创建唯一索引
db.users.createIndex(
    { email: 1 },
    { unique: true, name: "idx_users_email_unique" }
)

// 复合唯一索引：确保 userId + deviceId 组合唯一
db.user_devices.createIndex(
    { userId: 1, deviceId: 1 },
    { unique: true }
)

唯一索引的行为：

// 插入第一条：成功
db.users.insertOne({ email: "zhangsan@example.com", name: "张三" })

// 插入重复的 email：失败！
db.users.insertOne({ email: "zhangsan@example.com", name: "李四" })
// 报错：E11000 duplicate key error collection: mydb.users
//       index: idx_users_email_unique dup key: { email: "zhangsan@example.com" }

// 注意：null 也算一个值，多个文档的 email 为 null 也会冲突
db.users.insertOne({ name: "王五" })  // email 字段不存在，视为 null
db.users.insertOne({ name: "赵六" })  // 报错！已经有一个文档的 email 为 null

部分唯一索引（Partial Unique Index）：

有时候我们希望唯一约束只针对满足特定条件的文档。例如，只对已激活用户的 email 做唯一约束：

// 部分唯一索引：只对 status 为 "ACTIVE" 的文档检查 email 唯一性
db.users.createIndex(
    { email: 1 },
    {
        unique: true,
        partialFilterExpression: { status: "ACTIVE" },
        name: "idx_users_email_unique_active"
    }
)

// 以下操作合法：
db.users.insertOne({ email: "test@example.com", status: "ACTIVE" })  // ✅
db.users.insertOne({ email: "test@example.com", status: "DELETED" }) // ✅ status 不是 ACTIVE，不受约束

// 以下操作非法：
db.users.insertOne({ email: "test@example.com", status: "ACTIVE" })  // ❌ 重复！

稀疏唯一索引：

如果允许多个文档的索引字段为 null（即字段不存在），可以使用稀疏索引：

// 稀疏索引：只为字段存在的文档建立索引
db.users.createIndex(
    { email: 1 },
    { unique: true, sparse: true, name: "idx_users_email_unique_sparse" }
)

// 稀疏索引下，多个文档可以没有 email 字段
db.users.insertOne({ name: "王五" })   // ✅ 没有 email 字段，稀疏索引忽略它
db.users.insertOne({ name: "赵六" })   // ✅ 同样没有 email 字段，不冲突
db.users.insertOne({ email: null })    // ❌ 注意：email: null 仍会被索引！

6.3.2 TTL 索引

TTL（Time-To-Live）索引是一种特殊的单字段索引，它可以让 MongoDB 自动删除过期文档。TTL 索引只能建立在日期类型的字段上。

创建 TTL 索引：

// 文档在 expireAt 指定的时间点过期
db.sessions.createIndex(
    { expireAt: 1 },
    { expireAfterSeconds: 0, name: "idx_sessions_ttl" }
)

// 插入文档时指定过期时间：30分钟后过期
db.sessions.insertOne({
    sessionId: "sess_abc123",
    userId: "user001",
    data: { cart: ["item1", "item2"] },
    expireAt: new Date(Date.now() + 30 * 60 * 1000)  // 当前时间 + 30分钟
})

// 也可以让文档在创建后固定时间过期
// expireAfterSeconds: 3600 表示文档在 createdAt 之后 3600 秒（1小时）过期
db.verificationCodes.createIndex(
    { createdAt: 1 },
    { expireAfterSeconds: 3600, name: "idx_codes_ttl_1h" }
)

db.verificationCodes.insertOne({
    phone: "13800138000",
    code: "123456",
    createdAt: new Date()  // 该文档将在 1 小时后自动删除
})

TTL 索引的工作机制：

  • MongoDB 后台有一个专用的清理线程，每 60 秒运行一次
  • 它检查 TTL 索引中过期的文档并删除
  • 因此文档不会在精确的过期时间点被删除，可能有最多 60 秒的延迟
  • TTL 索引只能建立在单字段上，且该字段必须是 Date 类型

TTL 索引的应用场景：

场景 过期策略 说明
会话管理 30 分钟无操作后过期 用户 Session 自动清理
验证码 5 分钟后过期 短信/邮箱验证码自动失效
临时令牌 24 小时后过期 密码重置链接、邀请链接
日志数据 30 天后过期 自动清理旧日志，节省存储空间
缓存数据 自定义过期时间 类似 Redis 的 TTL 功能

完整的验证码管理示例：

// 1. 创建验证码集合的 TTL 索引（5分钟过期）
db.verificationCodes.createIndex(
    { createdAt: 1 },
    { expireAfterSeconds: 300 }  // 300秒 = 5分钟
)

// 2. 发送验证码时插入文档
db.verificationCodes.insertOne({
    phone: "13800138000",
    code: "582943",
    type: "LOGIN",
    createdAt: new Date(),
    used: false
})

// 3. 验证时查询
db.verificationCodes.findOne({
    phone: "13800138000",
    code: "582943",
    used: false
})
// 如果文档还没过期，能查到；过期后自动被删除，查不到

// 4. 验证成功后标记为已使用（防止重复使用）
db.verificationCodes.updateOne(
    { phone: "13800138000", code: "582943" },
    { $set: { used: true } }
)

修改 TTL 过期时间：

// 使用 collMod 命令修改 TTL 时间
db.runCommand({
    collMod: "verificationCodes",
    index: {
        keyPattern: { createdAt: 1 },
        expireAfterSeconds: 600  // 改为 10 分钟
    }
})

6.4 在 Spring Boot 中通过注解和代码创建索引

6.4.1 @Indexed 注解 —— 单字段索引

@Indexed 是 Spring Data MongoDB 提供的注解，用于在实体类的字段上声明索引。

package com.example.mongo.entity;

import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.index.Indexed;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.mapping.Field;
import java.time.LocalDateTime;
import java.util.List;

@Document(collection = "users")
public class User {

    @Id
    private String id;

    private String name;

    private Integer age;

    @Indexed(unique = true)
    private String email;

    @Indexed
    private String status;

    private List<String> tags;

    @Indexed(direction = org.springframework.data.mongodb.core.index.IndexDirection.DESCENDING)
    @Field("created_at")
    private LocalDateTime createdAt;

    // 构造方法、getter、setter 省略
}

@Indexed 注解的常用属性：

import org.springframework.data.mongodb.core.index.Indexed;
import org.springframework.data.mongodb.core.index.IndexDirection;

public class IndexedExamples {

    // 基本索引（升序）
    @Indexed
    private String username;

    // 唯一索引
    @Indexed(unique = true)
    private String email;

    // 降序索引
    @Indexed(direction = IndexDirection.DESCENDING)
    private LocalDateTime createdAt;

    // 自定义索引名称
    @Indexed(name = "idx_phone")
    private String phone;

    // 稀疏索引
    @Indexed(sparse = true)
    private String nickname;

    // TTL 索引：文档在该字段值之后 3600 秒过期
    @Indexed(expireAfterSeconds = 3600)
    private LocalDateTime expireAt;

    // TTL 索引：验证码 5 分钟后过期
    @Indexed(expireAfterSeconds = 300)
    private LocalDateTime codeCreatedAt;

    // 组合多个属性
    @Indexed(unique = true, sparse = true, name = "idx_wechat_unique")
    private String wechatOpenId;
}

TTL 索引的实体类示例 —— 验证码：

package com.example.mongo.entity;

import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.index.Indexed;
import org.springframework.data.mongodb.core.mapping.Document;
import java.time.LocalDateTime;

@Document(collection = "verification_codes")
public class VerificationCode {

    @Id
    private String id;

    @Indexed
    private String phone;

    private String code;

    private String type;

    private boolean used;

    @Indexed(expireAfterSeconds = 300)
    private LocalDateTime createdAt;

    public VerificationCode() {}

    public VerificationCode(String phone, String code, String type) {
        this.phone = phone;
        this.code = code;
        this.type = type;
        this.used = false;
        this.createdAt = LocalDateTime.now();
    }

    // getter / setter 省略
}

6.4.2 @CompoundIndex 注解 —— 复合索引

@CompoundIndex 用于在实体类级别声明复合索引，因为复合索引涉及多个字段，无法直接在单个字段上标注。

package com.example.mongo.entity;

import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.index.CompoundIndex;
import org.springframework.data.mongodb.core.index.CompoundIndexes;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.mapping.Field;
import java.time.LocalDateTime;
import java.util.List;

@Document(collection = "articles")
@CompoundIndexes({
    @CompoundIndex(
        name = "idx_status_createdAt",
        def = "{'status': 1, 'created_at': -1}"
    ),
    @CompoundIndex(
        name = "idx_category_status",
        def = "{'category': 1, 'status': 1}"
    ),
    @CompoundIndex(
        name = "idx_author_createdAt",

6.4.3 @TextIndexed 注解 —— 文本索引

文本索引用于支持全文搜索。MongoDB 的文本索引支持对字符串内容进行分词和搜索。

package com.example.mongo.entity;

import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.index.TextIndexed;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.mapping.Field;
import java.time.LocalDateTime;

@Document(collection = "articles")
public class Article {

    @Id
    private String id;

    @TextIndexed(weight = 3)
    private String title;

    @TextIndexed(weight = 1)
    private String content;

    @TextIndexed(weight = 2)
    private String summary;

    private String category;

    private String status;

    @Field("created_at")
    private LocalDateTime createdAt;

    // 构造方法、getter、setter 省略
}

说明：weight 属性设置字段在全文搜索中的权重。上面的例子中 title 权重为 3，summary 权重为 2，content 权重为 1，意味着标题中的匹配会获得更高的相关性分数。

使用文本索引进行全文搜索：

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.TextCriteria;
import org.springframework.data.mongodb.core.query.TextQuery;
import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class ArticleSearchService {

    private final MongoTemplate mongoTemplate;

    public ArticleSearchService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    public List<Article> fullTextSearch(String keyword) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
            .matching(keyword);

        Query query = TextQuery.queryText(textCriteria)
            .sortByScore();

        return mongoTemplate.find(query, Article.class);
    }

    public List<Article> fullTextSearchWithFilter(String keyword, String category) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
            .matching(keyword);

        Query query = TextQuery.queryText(textCriteria)
            .sortByScore();
        query.addCriteria(Criteria.where("category").is(category));

        return mongoTemplate.find(query, Article.class);
    }
}

注意：每个集合只能有一个文本索引。MongoDB 的内置文本搜索对中文的分词支持有限，生产环境如需高质量中文全文搜索，建议结合 Elasticsearch 或 MongoDB Atlas Search 使用。

6.4.4 使用 MongoTemplate 编程式创建索引

除了注解方式，还可以通过 MongoTemplate 的 IndexOperations 在代码中动态创建索引。这种方式在以下场景非常有用：

  • 需要根据配置动态创建索引
  • 索引定义较复杂，注解难以表达
  • 需要在应用启动时统一管理索引
package com.example.mongo.config;

import org.springframework.boot.CommandLineRunner;
import org.springframework.data.domain.Sort;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.index.Index;
import org.springframework.data.mongodb.core.index.IndexOperations;
import org.springframework.data.mongodb.core.index.IndexInfo;
import org.springframework.stereotype.Component;
import java.time.Duration;
import java.util.List;

@Component
public class MongoIndexInitializer implements CommandLineRunner {

    private final MongoTemplate mongoTemplate;

    public MongoIndexInitializer(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    @Override
    public void run(String... args) {
        createUserIndexes();
        createArticleIndexes();
        createSessionIndexes();
        System.out.println("MongoDB 索引初始化完成");
    }

    private void createUserIndexes() {
        IndexOperations indexOps = mongoTemplate.indexOps("users");

        // 单字段唯一索引
        indexOps.ensureIndex(
            new Index()
                .on("email", Sort.Direction.ASC)
                .unique()
                .named("idx_users_email_unique")
        );

        // 单字段普通索引
        indexOps.ensureIndex(
            new Index()
                .on("status", Sort.Direction.ASC)
                .named("idx_users_status")
        );

        // 复合索引
        indexOps.ensureIndex(
            new Index()
                .on("status", Sort.Direction.ASC)
                .on("createdAt", Sort.Direction.DESC)
                .named("idx_users_status_createdAt")
        );

        // 稀疏唯一索引
        indexOps.ensureIndex(
            new Index()
                .on("phone", Sort.Direction.ASC)
                .unique()
                .sparse()
                .named("idx_users_phone_unique_sparse")
        );
    }

    private void createArticleIndexes() {
        IndexOperations indexOps = mongoTemplate.indexOps("articles");

        // 复合索引
        indexOps.ensureIndex(
            new Index()
                .on("category", Sort.Direction.ASC)
                .on("status", Sort.Direction.ASC)
                .on("createdAt", Sort.Direction.DESC)
                .named("idx_articles_category_status_createdAt")
        );

        // 作者索引
        indexOps.ensureIndex(
            new Index()
                .on("authorId", Sort.Direction.ASC)
                .on("createdAt", Sort.Direction.DESC)
                .named("idx_articles_author_createdAt")
        );
    }

    private void createSessionIndexes() {
        IndexOperations indexOps = mongoTemplate.indexOps("sessions");

        // TTL 索引：30分钟后过期
        indexOps.ensureIndex(
            new Index()
                .on("createdAt", Sort.Direction.ASC)
                .expire(Duration.ofMinutes(30))
                .named("idx_sessions_ttl")
        );

        // 会话 ID 唯一索引
        indexOps.ensureIndex(
            new Index()
                .on("sessionId", Sort.Direction.ASC)
                .unique()
                .named("idx_sessions_sessionId_unique")
        );
    }
}

6.4.5 使用 IndexOperations 管理索引

IndexOperations 不仅能创建索引，还能查询和删除索引：

package com.example.mongo.service;

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.index.IndexInfo;
import org.springframework.data.mongodb.core.index.IndexOperations;
import org.springframework.stereotype.Service;
import java.util.List;
import java.util.stream.Collectors;

@Service
public class IndexManagementService {

    private final MongoTemplate mongoTemplate;

    public IndexManagementService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 获取指定集合的所有索引信息
     */
    public List<IndexInfo> getIndexes(String collectionName) {
        IndexOperations indexOps = mongoTemplate.indexOps(collectionName);
        return indexOps.getIndexInfo();
    }

    /**
     * 打印集合的索引详情
     */
    public void printIndexes(String collectionName) {
        List<IndexInfo> indexes = getIndexes(collectionName);
        System.out.println("集合 [" + collectionName + "] 的索引列表：");
        System.out.println("─".repeat(60));
        for (IndexInfo index : indexes) {
            System.out.println("名称: " + index.getName());
            System.out.println("字段: " + index.getIndexFields());
            System.out.println("唯一: " + index.isUnique());
            System.out.println("稀疏: " + index.isSparse());
            index.getExpireAfter().ifPresent(
                duration -> System.out.println("TTL:  " + duration.getSeconds() + " 秒")
            );
            System.out.println("─".repeat(60));
        }
    }

    /**
     * 删除指定索引
     */
    public void dropIndex(String collectionName, String indexName) {
        IndexOperations indexOps = mongoTemplate.indexOps(collectionName);
        indexOps.dropIndex(indexName);
        System.out.println("已删除索引: " + collectionName + "." + indexName);
    }

    /**
     * 删除集合中除 _id 外的所有索引
     */
    public void dropAllIndexes(String collectionName) {
        IndexOperations indexOps = mongoTemplate.indexOps(collectionName);
        indexOps.dropAllIndexes();
        System.out.println("已删除集合 [" + collectionName + "] 的所有自定义索引");
    }

    /**
     * 获取索引名称列表
     */
    public List<String> getIndexNames(String collectionName) {
        return getIndexes(collectionName).stream()
            .map(IndexInfo::getName)
            .collect(Collectors.toList());
    }
}

在 Controller 中暴露索引管理接口（仅供管理后台使用）：

package com.example.mongo.controller;

import com.example.mongo.service.IndexManagementService;

6.4.6 自动索引创建的配置

Spring Data MongoDB 默认会在应用启动时根据注解自动创建索引。可以通过配置控制此行为：

# application.yml
spring:
  data:
    mongodb:
      uri: mongodb://localhost:27017/mydb
      auto-index-creation: true   # 默认为 true，设为 false 禁用自动创建

在生产环境中，建议将 auto-index-creation 设置为 false，改用编程式方式（如上面的 MongoIndexInitializer）管理索引，原因如下：

  1. 可控性：避免意外创建或修改索引影响线上性能
  2. 可审计：索引变更集中管理，便于 Code Review
  3. 性能影响：在大集合上创建索引可能耗时很长，自动创建可能导致应用启动缓慢
// 也可以通过 Java 配置关闭自动索引创建
package com.example.mongo.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.data.mongodb.config.AbstractMongoClientConfiguration;

@Configuration
public class MongoConfig extends AbstractMongoClientConfiguration {

    @Override
    protected String getDatabaseName() {
        return "mydb";
    }

    @Override
    protected boolean autoIndexCreation() {
        return false;
    }
}

6.5 使用 explain() 分析查询性能

6.5.1 explain() 概述

explain() 是 MongoDB 提供的查询分析工具，它不执行实际查询（或在特定模式下执行），而是返回查询计划的详细信息。通过 explain() 可以了解 MongoDB 如何执行查询、是否使用了索引、扫描了多少文档等关键信息。

6.5.2 explain() 的三种模式

模式 说明 是否执行查询
queryPlanner（默认） 显示查询优化器选择的执行计划 ❌ 不执行
executionStats 执行查询并返回详细的执行统计信息 ✅ 执行
allPlansExecution 执行查询并返回所有候选计划的执行信息 ✅ 执行所有计划 
// queryPlanner 模式（默认）—— 只看执行计划，不实际执行
db.users.find({ status: "ACTIVE" }).explain()
db.users.find({ status: "ACTIVE" }).explain("queryPlanner")

// executionStats 模式 —— 最常用，可以看到实际的执行统计
db.users.find({ status: "ACTIVE" }).explain("executionStats")

// allPlansExecution 模式 —— 看所有候选计划的执行情况
db.users.find({ status: "ACTIVE" }).explain("allPlansExecution")

6.5.3 关键指标解读

以 executionStats 模式的输出为例，详细解读各个关键指标：

db.users.find({ status: "ACTIVE", age: { $gt: 25 } })
    .sort({ createdAt: -1 })
    .explain("executionStats")

输出结构（关键字段）：

{
  "queryPlanner": {
    "winningPlan": {
      "stage": "SORT",                    // 最外层操作阶段
      "sortPattern": { "createdAt": -1 },
      "inputStage": {
        "stage": "FETCH",                 // 获取完整文档
        "inputStage": {
          "stage": "IXSCAN",             // 索引扫描 ← 关键！
          "keyPattern": { "status": 1, "age": 1 },
          "indexName": "idx_status_age",
          "direction": "forward"
        }
      }
    },
    "rejectedPlans": [ ... ]             // 被拒绝的候选计划
  },
  "executionStats": {
    "executionSuccess": true,
    "nReturned": 150,                    // 最终返回的文档数
    "executionTimeMillis": 12,           // 总执行时间（毫秒）
    "totalKeysExamined": 420,            // 扫描的索引键数量
    "totalDocsExamined": 420,            // 扫描的文档数量
    "executionStages": {
      "stage": "SORT",
      "nReturned": 150,
      "executionTimeMillisEstimate": 5,
      "memUsage": 102400,               // 内存使用（字节）
      "inputStage": {
        "stage": "FETCH",
        "nReturned": 420,
        "docsExamined": 420,
        "inputStage": {
          "stage": "IXSCAN",
          "nReturned": 420,
          "keysExamined": 420
        }
      }
    }
  }
}

关键指标速查表：

指标 含义 优化目标
winningPlan.stage 查询执行的阶段类型 期望看到 IXSCAN，避免 COLLSCAN
nReturned 最终返回的文档数 —
executionTimeMillis 查询总执行时间（毫秒） 越小越好
totalKeysExamined 扫描的索引键数量 越接近 nReturned 越好
totalDocsExamined 扫描的文档数量 越接近 nReturned 越好

常见的执行阶段（stage）类型：

Stage 含义 是否期望
COLLSCAN 全集合扫描（无索引） ❌ 性能杀手，需要优化
IXSCAN 索引扫描 ✅ 期望看到
FETCH 根据索引定位后获取完整文档 ✅ 正常阶段
SORT 内存中排序 ⚠️ 如果数据量大需注意
SORT_KEY_GENERATOR 生成排序键 ⚠️ 配合 SORT 出现
PROJECTION 字段投影 ✅ 正常阶段
LIMIT 限制返回数量 ✅ 正常阶段
SKIP 跳过文档 ✅ 正常阶段
COUNT_SCAN 使用索引进行计数 ✅ 高效计数
COVERED_QUERY 覆盖查询（无需 FETCH） ✅ 最高效

性能判断经验法则：

理想情况：
  totalKeysExamined ≈ totalDocsExamined ≈ nReturned

需要优化的情况：
  1. totalDocsExamined >> nReturned  → 扫描了大量文档但返回很少，索引不够精确
  2. totalKeysExamined >> nReturned  → 索引扫描范围过大
  3. stage 出现 COLLSCAN            → 没有使用索引，需要创建索引
  4. stage 出现 SORT 且数据量大      → 排序无法利用索引，考虑调整索引

6.5.4 完整的索引优化案例

下面通过一个完整的案例，演示如何从 COLLSCAN 优化到 IXSCAN。

场景：文章列表页，需要查询指定分类下已发布的文章，按创建时间倒序排列。

// 初始数据：articles 集合有 50 万条数据，目前没有自定义索引

// 查询语句
db.articles.find({
    category: "技术",
    status: "PUBLISHED"
}).sort({ created_at: -1 }).limit(20)

第一步：分析当前查询性能

db.articles.find({
    category: "技术",
    status: "PUBLISHED"
}).sort({ created_at: -1 }).limit(20).explain("executionStats")

// 结果（无索引）：
// {
//   "queryPlanner": {
//     "winningPlan": {
//       "stage": "SORT",
//       "inputStage": {
//         "stage": "COLLSCAN"           ← 全集合扫描！
//       }
//     }
//   },
//   "executionStats": {
//     "nReturned": 20,
//     "executionTimeMillis": 1250,       ← 耗时 1.25 秒！
//     "totalKeysExamined": 0,            ← 没有使用索引
//     "totalDocsExamined": 500000        ← 扫描了全部 50万 文档
//   }
// }

问题分析：

  • COLLSCAN：没有使用任何索引，进行了全集合扫描
  • 扫描了 50 万文档，但只返回 20 条
  • 内存中排序 50 万文档后再取前 20 条，极其低效

第二步：创建索引

// 根据查询分析，创建复合索引
// 字段顺序：等值条件字段在前，排序字段在后
db.articles.createIndex(
    { category: 1, status: 1, created_at: -1 },
    { name: "idx_articles_category_status_createdAt" }
)

第三步：验证优化效果

db.articles.find({
    category: "技术",
    status: "PUBLISHED"
}).sort({ created_at: -1 }).limit(20).explain("executionStats")

// 结果（有索引）：
// {
//   "queryPlanner": {
//     "winningPlan": {
//       "stage": "LIMIT",
//       "inputStage": {
//         "stage": "FETCH",
//         "inputStage": {
//           "stage": "IXSCAN",           ← 使用了索引扫描！
//           "indexName": "idx_articles_category_status_createdAt",
//           "keyPattern": { "category": 1, "status": 1, "created_at": -1 }
//         }
//       }
//     }
//   },
//   "executionStats": {
//     "nReturned": 20,
//     "executionTimeMillis": 0,           ← 不到 1 毫秒！
//     "totalKeysExamined": 20,            ← 只扫描了 20 个索引键
//     "totalDocsExamined": 20             ← 只获取了 20 个文档
//   }
// }

优化效果对比：

指标 优化前（COLLSCAN） 优化后（IXSCAN） 提升倍数
执行时间 1250ms < 1ms 1250x+
扫描文档数 500,000 20 25,000x
扫描索引键数 0 20 —
排序方式 内存排序 索引有序 避免内存排序

为什么这个索引有效？

索引 { category: 1, status: 1, created_at: -1 }

查询过程：
1. 通过索引快速定位 category = "技术" 的区间
2. 在该区间内进一步定位 status = "PUBLISHED" 的子区间
3. 该子区间内的数据已经按 created_at 降序排列 → 无需额外排序！
4. 直接取前 20 条即可 → 只扫描 20 个索引键和 20 个文档

6.5.5 在 Java 代码中获取 explain 结果

在 Spring Boot 中，可以通过 MongoTemplate 执行 explain 操作：

package com.example.mongo.service;

import org.bson.Document;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.stereotype.Service;

@Service
public class QueryAnalysisService {

    private final MongoTemplate mongoTemplate;

    public QueryAnalysisService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 分析查询的执行计划
     */
    public Document explainQuery(Query query, String collectionName) {
        return mongoTemplate.getCollection(collectionName)
            .find(query.getQueryObject())
            .sort(query.getSortObject())
            .explain();
    }

    /**
     * 获取详细的执行统计（executionStats 模式）
     */
    public Document explainWithStats(Query query, String collectionName) {
        return mongoTemplate.getCollection(collectionName)
            .find(query.getQueryObject())
            .sort(query.getSortObject())
            .explain(com.mongodb.ExplainVerbosity.EXECUTION_STATS);
    }

    /**
     * 分析并打印查询性能摘要
     */
    public void analyzeQuery(Query query, String collectionName) {
        Document explanation = explainWithStats(query, collectionName);

        Document executionStats = explanation.get("executionStats", Document.class);
        Document queryPlanner = explanation.get("queryPlanner", Document.class);
        Document winningPlan = queryPlanner.get("winningPlan", Document.class);

        System.out.println("═".repeat(60));
        System.out.println("查询性能分析报告");
        System.out.println("═".repeat(60));
        System.out.println("集合:           " + collectionName);
        System.out.println("查询条件:       " + query.getQueryObject().toJson());
        System.out.println("排序条件:       " + query.getSortObject().toJson());
        System.out.println("─".repeat(60));
        System.out.println("执行计划:       " + getStageChain(winningPlan));
        System.out.println("返回文档数:     " + executionStats.getInteger("nReturned"));
        System.out.println("扫描索引键数:   " + executionStats.getInteger("totalKeysExamined"));
        System.out.println("扫描文档数:     " + executionStats.getInteger("totalDocsExamined"));
        System.out.println("执行时间:       " + executionStats.getInteger("executionTimeMillis") + " ms");
        System.out.println("═".repeat(60));

        int nReturned = executionStats.getInteger("nReturned");
        int docsExamined = executionStats.getInteger("totalDocsExamined");
        if (docsExamined > nReturned * 10 && docsExamined > 100) {
            System.out.println("⚠ 警告: 扫描文档数远大于返回文档数，建议优化索引！");
        }

        String stage = getTopStage(winningPlan);
        if ("COLLSCAN".equals(stage)) {
            System.out.println("⚠ 警告: 使用了全集合扫描(COLLSCAN)，强烈建议创建索引！");
        }
    }

    private String getStageChain(Document plan) {
        StringBuilder chain = new StringBuilder();
        Document current = plan;
        while (current != null) {
            if (!chain.isEmpty()) {
                chain.append(" → ");
            }
            chain.append(current.getString("stage"));
            current = current.get("inputStage", Document.class);
        }
        return chain.toString();
    }

    private String getTopStage(Document plan) {
        Document current = plan;
        Document last = current;
        while (current != null) {
            last = current;
            current = current.get("inputStage", Document.class);
        }
        return last.getString("stage");
    }
}

package com.example.mongo.controller;

import com.example.mongo.service.QueryAnalysisService;
import org.springframework.data.domain.Sort;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/admin/analyze")
public class QueryAnalysisController {

    private final QueryAnalysisService analysisService;

    public QueryAnalysisController(QueryAnalysisService analysisService) {
        this.analysisService = analysisService;
    }

    @GetMapping("/articles")
    public String analyzeArticleQuery(
            @RequestParam(required = false) String category,
            @RequestParam(required = false) String status) {

        Query query = new Query();
        if (category != null) {
            query.addCriteria(Criteria.where("category").is(category));
        }
        if (status != null) {
            query.addCriteria(Criteria.where("status").is(status));
        }
        query.with(Sort.by(Sort.Direction.DESC, "created_at"));
        query.limit(20);

        analysisService.analyzeQuery(query, "articles");
        return "查询分析已打印到控制台";
    }
}

6.5.6 explain 分析的最佳实践

  1. 开发阶段：对所有核心查询都执行 explain 分析，确保使用了索引
  2. 上线前：用接近真实的数据量做性能测试，小数据量下即使 COLLSCAN 也很快
  3. 关注比率：totalDocsExamined / nReturned 比值越接近 1 越好
  4. 避免内存排序：如果看到 SORT stage 且数据量大，考虑调整索引让排序走索引
  5. 定期巡检：定期检查慢查询日志，对新出现的慢查询进行 explain 分析

本章小结

知识点 核心内容
索引的作用 避免全集合扫描，将 O(n) 查询优化为 O(log n)
B-Tree 结构 平衡多路搜索树，支持精确匹配、范围查询、排序
单字段索引 最基本的索引类型，createIndex({ field: 1 })
复合索引 多字段索引，字段顺序影响使用效率，遵循前缀匹配原则
唯一索引 保证字段值唯一，支持部分唯一索引和稀疏唯一索引
TTL 索引 自动过期删除文档，适用于会话、验证码、临时数据
@Indexed Spring Data 注解方式创建单字段索引
@CompoundIndex Spring Data 注解方式创建复合索引
@TextIndexed Spring Data 注解方式创建文本索引
IndexOperations 编程式创建和管理索引，适合生产环境
explain() 查询性能分析工具，关注 COLLSCAN vs IXSCAN、扫描文档数、执行时间
索引优化 等值条件字段在前，排序字段在后；避免过多索引；确保索引在内存中
