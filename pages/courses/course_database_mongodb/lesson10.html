第10章 高级索引与查询优化

第06章介绍了索引的基本概念和常用类型。本章将深入探讨 MongoDB 提供的高级索引类型——文本索引、地理空间索引、通配符索引，以及索引设计的核心策略 ESR 规则和慢查询分析调优方法。掌握这些内容，你将能够为复杂业务场景设计高效的索引策略。

10.1 文本索引与全文搜索

文本索引（Text Index）允许你对字符串字段进行全文搜索，类似于 Elasticsearch 的基本功能。适用于搜索文章内容、商品描述、用户评论等场景。

10.1.1 创建文本索引

单字段文本索引：

// 对文章的 content 字段创建文本索引
db.articles.createIndex({ content: "text" })

多字段文本索引：

// 同时对 title 和 content 创建文本索引
db.articles.createIndex({ title: "text", content: "text" })

注意：一个集合只能有一个文本索引，但这个索引可以包含多个字段。

10.1.2 权重配置

可以为不同字段设置不同的权重，影响搜索结果的相关性评分：

db.articles.createIndex(
    { title: "text", content: "text", tags: "text" },
    {
        weights: {
            title: 10,    // 标题匹配权重最高
            tags: 5,      // 标签匹配权重次之
            content: 1    // 内容匹配权重最低（默认值）
        },
        name: "article_text_index"
    }
)

10.1.4 $meta: "textScore" 排序

文本搜索会为每个匹配结果计算一个相关性分数，可以用来排序：

// 按相关性分数排序，分数越高越靠前
db.articles.find(
    { $text: { $search: "MongoDB 数据库" } },
    { score: { $meta: "textScore" } }
).sort({ score: { $meta: "textScore" } })

// 返回结果会包含 score 字段：
// { title: "MongoDB 入门教程", ..., score: 2.5 }
// { title: "Spring Boot 集成 MongoDB 实战", ..., score: 1.8 }
// { title: "MySQL 索引优化指南", ..., score: 0.75 }

10.1.5 语言配置

MongoDB 的文本索引支持多种语言的分词和停用词过滤：

// 指定语言
db.articles.createIndex(
    { content: "text" },
    { default_language: "english" }  // 默认英文分词
)

// 对于中文，可以使用 "none" 禁用语言特定的分词（按空格分词）
db.articles.createIndex(
    { title: "text", content: "text" },
    { default_language: "none" }  // 不使用特定语言的分词器
)

// 也可以在文档中指定语言字段
db.articles.createIndex(
    { content: "text" },
    { language_override: "lang" }  // 使用文档中的 lang 字段指定语言
)

中文全文搜索的局限：MongoDB 原生文本索引对中文的支持有限（按空格分词），如果需要高级中文分词能力，建议使用 MongoDB Atlas Search 或配合 Elasticsearch。

10.1.6 完整的 Java 代码示例

使用 TextCriteria 进行全文搜索：

import org.springframework.data.domain.Sort;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.TextCriteria;
import org.springframework.data.mongodb.core.query.TextQuery;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
public class ArticleSearchService {

    private final MongoTemplate mongoTemplate;

    public ArticleSearchService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 基本全文搜索
     */
    public List<Article> searchByText(String keyword) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
                .matching(keyword);

        Query query = TextQuery.queryText(textCriteria)
                .sortByScore();  // 按相关性分数排序

        return mongoTemplate.find(query, Article.class);
    }

    /**
     * 全文搜索 + 额外过滤条件
     */
    public List<Article> searchByTextWithFilter(String keyword, String author) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
                .matching(keyword);

        Query query = TextQuery.queryText(textCriteria)
                .sortByScore()
                .addCriteria(Criteria.where("author").is(author));

        return mongoTemplate.find(query, Article.class);
    }

    /**
     * 短语搜索
     */
    public List<Article> searchByPhrase(String phrase) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
                .matchingPhrase(phrase);  // 精确短语匹配

        Query query = TextQuery.queryText(textCriteria)
                .sortByScore();

        return mongoTemplate.find(query, Article.class);
    }

    /**
     * 搜索并包含相关性分数
     */
    public List<Article> searchWithScore(String keyword) {
        TextCriteria textCriteria = TextCriteria.forDefaultLanguage()
                .matching(keyword);

        Query query = TextQuery.queryText(textCriteria)
                .sortByScore()
                .includeScore();  // 将 textScore 映射到实体的 score 字段

        return mongoTemplate.find(query, Article.class);
    }
}

10.2 地理空间索引（2dsphere）与位置查询

地理空间索引是 MongoDB 的一大特色功能，非常适合"查找附近的XXX"这类基于位置的查询场景。

10.2.1 GeoJSON 数据格式

MongoDB 使用 GeoJSON 标准格式存储地理空间数据：

Point（点）：

{
    type: "Point",
    coordinates: [116.397428, 39.90923]  // [经度, 纬度]
}

LineString（线）：

{
    type: "LineString",
    coordinates: [
        [116.397428, 39.90923],
        [116.407428, 39.91923],
        [116.417428, 39.92923]
    ]
}

Polygon（多边形）：

{
    type: "Polygon",
    coordinates: [[
        [116.38, 39.90],
        [116.42, 39.90],
        [116.42, 39.93],
        [116.38, 39.93],
        [116.38, 39.90]   // 首尾点相同，闭合多边形
    ]]
}

注意：GeoJSON 中坐标的顺序是 [经度, 纬度]（longitude, latitude），而不是我们日常说的"纬度, 经度"。

10.2.2 创建 2dsphere 索引

// 对 location 字段创建地理空间索引
db.restaurants.createIndex({ location: "2dsphere" })

// 复合索引：地理空间 + 其他字段
db.restaurants.createIndex({ location: "2dsphere", cuisine: 1, rating: -1 })

10.2.7 距离计算：使用聚合管道

使用 $geoNear 聚合阶段可以获取精确的距离值：

// 查找天安门附近5公里内的餐厅，并返回距离
db.restaurants.aggregate([
    {
        $geoNear: {
            near: { type: "Point", coordinates: [116.397477, 39.908692] },
            distanceField: "distance",          // 将距离值存入此字段（单位：米）
            maxDistance: 5000,                   // 最大距离5公里
            spherical: true,                     // 使用球面计算
            query: { rating: { $gte: 4.0 } }   // 附加过滤条件
        }
    },
    {
        $project: {
            name: 1,
            cuisine: 1,
            rating: 1,
            address: 1,
            distanceInKm: {
                $round: [{ $divide: ["$distance", 1000] }, 2]  // 转为公里，保留2位小数
            }
        }
    },
    { $sort: { distanceInKm: 1 } }
])

Service 层：

import org.springframework.data.geo.*;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.geo.GeoJsonPoint;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.NearQuery;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
public class RestaurantService {

    private final MongoTemplate mongoTemplate;

    public RestaurantService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 查找附近的餐厅（使用 NearQuery）
     *
     * @param longitude 经度
     * @param latitude  纬度
     * @param maxDistanceKm 最大距离（公里）
     * @return 按距离排序的餐厅列表（包含距离信息）
     */
    public GeoResults<Restaurant> findNearbyRestaurants(
            double longitude, double latitude, double maxDistanceKm) {

        Point point = new Point(longitude, latitude);

        NearQuery nearQuery = NearQuery.near(point)
                .maxDistance(new Distance(maxDistanceKm, Metrics.KILOMETERS))
                .spherical(true);

        return mongoTemplate.geoNear(nearQuery, Restaurant.class);
    }

    /**
     * 查找附近的特定类型餐厅
     */
    public GeoResults<Restaurant> findNearbyByCuisine(
            double longitude, double latitude, double maxDistanceKm, String cuisine) {

        Point point = new Point(longitude, latitude);

        NearQuery nearQuery = NearQuery.near(point)
                .maxDistance(new Distance(maxDistanceKm, Metrics.KILOMETERS))
                .spherical(true)
                .query(Query.query(Criteria.where("cuisine").is(cuisine)));

        return mongoTemplate.geoNear(nearQuery, Restaurant.class);
    }

    /**
     * 查找附近高评分餐厅（带分页）
     */
    public GeoResults<Restaurant> findNearbyHighRated(
            double longitude, double latitude, double maxDistanceKm,
            double minRating, int page, int size) {

        Point point = new Point(longitude, latitude);

        NearQuery nearQuery = NearQuery.near(point)
                .maxDistance(new Distance(maxDistanceKm, Metrics.KILOMETERS))
                .spherical(true)
                .query(Query.query(Criteria.where("rating").gte(minRating)))
                .skip((long) page * size)
                .limit(size);

        return mongoTemplate.geoNear(nearQuery, Restaurant.class);
    }

    /**
     * 查找多边形区域内的餐厅
     */
    public List<Restaurant> findWithinArea(List<Point> polygonPoints) {
        GeoJsonPolygon polygon = new GeoJsonPolygon(polygonPoints);

        Query query = Query.query(
                Criteria.where("location").within(polygon)
        );

        return mongoTemplate.find(query, Restaurant.class);
    }
}

Controller 层：

import org.springframework.data.geo.GeoResult;
import org.springframework.data.geo.GeoResults;
import org.springframework.web.bind.annotation.*;

import java.util.List;
import java.util.stream.Collectors;

@RestController
@RequestMapping("/api/restaurants")
public class RestaurantController {

    private final RestaurantService restaurantService;

    public RestaurantController(RestaurantService restaurantService) {
        this.restaurantService = restaurantService;
    }

    @GetMapping("/nearby")
    public List<NearbyRestaurantDTO> findNearby(
            @RequestParam double longitude,
            @RequestParam double latitude,
            @RequestParam(defaultValue = "3") double maxDistanceKm,
            @RequestParam(required = false) String cuisine) {

        GeoResults<Restaurant> results;
        if (cuisine != null) {
            results = restaurantService.findNearbyByCuisine(
                    longitude, latitude, maxDistanceKm, cuisine);
        } else {
            results = restaurantService.findNearbyRestaurants(
                    longitude, latitude, maxDistanceKm);
        }

        return results.getContent().stream()
                .map(this::toDTO)
                .collect(Collectors.toList());
    }

    private NearbyRestaurantDTO toDTO(GeoResult<Restaurant> geoResult) {
        Restaurant restaurant = geoResult.getContent();
        double distanceKm = geoResult.getDistance().in(Metrics.KILOMETERS).getValue();

        NearbyRestaurantDTO dto = new NearbyRestaurantDTO();
        dto.setId(restaurant.getId());
        dto.setName(restaurant.getName());
        dto.setCuisine(restaurant.getCuisine());
        dto.setRating(restaurant.getRating());
        dto.setAddress(restaurant.getAddress());
        dto.setDistanceKm(Math.round(distanceKm * 100.0) / 100.0);
        return dto;
    }
}

    public void setDistanceKm(double distanceKm) { this.distanceKm = distanceKm; }
}

使用 MongoRepository 的方式：

import org.springframework.data.geo.*;
import org.springframework.data.mongodb.repository.MongoRepository;

import java.util.List;

    // 查找指定点附近、指定距离内的餐厅
    GeoResults<Restaurant> findByLocationNear(Point point, Distance maxDistance);

    GeoResults<Restaurant> findByCuisineAndLocationNear(String cuisine, Point point, Distance maxDistance);
}

10.3 通配符索引（Wildcard Index）

通配符索引（Wildcard Index）是 MongoDB 4.2 引入的特性，允许你为文档中动态或不可预测的字段创建索引。

10.3.1 什么时候使用通配符索引

通配符索引适用于以下场景：

  • 文档结构不固定：不同文档可能有不同的字段
  • 字段名不可预知：如用户自定义的属性（metadata、tags 等）
  • 嵌套文档结构多变：查询可能涉及任意嵌套路径

典型场景：物联网设备数据

// 不同设备上报的数据字段不同
{ deviceId: "D001", type: "温度传感器", data: { temperature: 25.5, humidity: 60 } }
{ deviceId: "D002", type: "GPS追踪器",  data: { latitude: 39.9, longitude: 116.4, speed: 30 } }
{ deviceId: "D003", type: "空气监测器", data: { pm25: 35, co2: 400, temperature: 22 } }

如果要为 data 下的所有可能字段创建索引，提前无法确定字段名，此时通配符索引就很合适。

10.3.2 创建通配符索引

全字段通配符索引：

// 为集合中所有字段创建通配符索引
db.deviceData.createIndex({ "$**": 1 })

注意：全字段通配符索引会为文档中的每个字段（包括嵌套字段）建立索引，可能会导致索引体积非常大、写入性能下降。

指定路径的通配符索引：

// 仅为 data 子文档中的所有字段创建通配符索引
db.deviceData.createIndex({ "data.$**": 1 })

使用 wildcardProjection 限定范围：

// 只对 data.temperature 和 data.humidity 创建索引
db.deviceData.createIndex(
    { "$**": 1 },
    {
        wildcardProjection: {
            "data.temperature": 1,
            "data.humidity": 1
        }
    }
)

// 排除某些字段
db.deviceData.createIndex(
    { "$**": 1 },
    {
        wildcardProjection: {
            "data.rawPayload": 0  // 排除 rawPayload 字段
        }
    }
)

10.3.4 限制和注意事项

限制 说明
不支持复合查询 通配符索引不能同时优化多个字段的查询条件
不支持排序 除非排序字段是查询中精确匹配的字段
不索引 _id 字段 $** 不会为 _id 字段创建冗余索引
不索引数组元素的多键组合 不能用于跨数组元素的查询
写入开销大 每个字段都要维护索引，写入性能影响明显
不能用于 TTL、文本、地理空间 通配符索引只支持普通的升序/降序索引 
// ❌ 通配符索引不能优化的查询
db.products.find({
    "attributes.brand": "Apple",
    "attributes.color": "黑色"    // 不能同时使用通配符索引优化两个条件
})

// ✅ 如果经常需要按 brand + color 查询，应该创建精确的复合索引
db.products.createIndex({ "attributes.brand": 1, "attributes.color": 1 })

10.3.5 Java 代码示例

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.index.IndexOperations;
import org.springframework.data.mongodb.core.index.IndexDefinition;
import org.springframework.data.mongodb.core.index.WildcardIndex;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.Map;

@Service
public class ProductService {

    private final MongoTemplate mongoTemplate;

    public ProductService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 通过代码创建通配符索引
     */
    public void createWildcardIndex() {
        IndexOperations indexOps = mongoTemplate.indexOps("products");
        IndexDefinition wildcardIndex = new WildcardIndex("attributes");
        indexOps.ensureIndex(wildcardIndex);
    }

    /**
     * 按动态属性查询商品
     */
    public List<Product> findByAttribute(String attributeName, Object attributeValue) {
        Query query = Query.query(
                Criteria.where("attributes." + attributeName).is(attributeValue)
        );
        return mongoTemplate.find(query, Product.class);
    }

    /**
     * 按多个动态属性查询
     */
    public List<Product> findByAttributes(Map<String, Object> attributes) {
        Criteria criteria = new Criteria();
        for (Map.Entry<String, Object> entry : attributes.entrySet()) {
            criteria = criteria.and("attributes." + entry.getKey()).is(entry.getValue());
        }
        return mongoTemplate.find(Query.query(criteria), Product.class);
    }
}

10.4 索引策略：ESR 规则（Equality, Sort, Range）

ESR 规则是设计复合索引时最重要的指导原则，它决定了索引中字段的排列顺序。

10.4.1 ESR 规则详解

复合索引中字段的排列顺序应遵循以下规则：

索引字段顺序：Equality 字段（最前）→ Sort 字段（中间）→ Range 字段（最后）
类型 含义 常见操作符 放置位置
Equality（等值） 精确匹配的字段 $eq、is 最前面
Sort（排序） 排序使用的字段 sort() 中间
Range（范围） 范围查询的字段 $gt、$lt、$gte、$lte、$in、$ne 最后面

为什么这个顺序最优？

  • Equality 放最前：等值条件能最快地缩小扫描范围，直接定位到 B-Tree 的特定分支
  • Sort 放中间：排序字段紧跟等值字段，可以利用索引的有序性避免内存排序（SORT_KEY_GENERATOR）
  • Range 放最后：范围条件匹配的是一段连续的索引范围，放在最后不影响等值定位和排序

10.4.2 案例一：电商订单查询

场景：查询某个用户的订单，按创建时间降序排列，筛选金额大于100的订单。

// 查询语句
db.orders.find({
    userId: "U001",                    // Equality
    totalAmount: { $gt: 100 }          // Range
}).sort({ createdAt: -1 })             // Sort

ESR 分析：

  • E：userId（等值匹配）
  • S：createdAt（排序字段）
  • R：totalAmount（范围查询）

最佳索引：

// ✅ 遵循 ESR 规则
db.orders.createIndex({ userId: 1, createdAt: -1, totalAmount: 1 })

验证：

// 准备测试数据（10万条）
for (let i = 0; i < 100000; i++) {
    db.orders.insertOne({
        userId: "U" + String(Math.floor(Math.random() * 1000)).padStart(4, "0"),
        totalAmount: Math.floor(Math.random() * 1000),
        status: ["PENDING", "PAID", "SHIPPED", "COMPLETED"][Math.floor(Math.random() * 4)],
        createdAt: new Date(2025, 0, 1 + Math.floor(Math.random() * 365))
    });
}

// 使用 ESR 索引
db.orders.createIndex({ userId: 1, createdAt: -1, totalAmount: 1 });

db.orders.find({
    userId: "U0001",
    totalAmount: { $gt: 100 }
}).sort({ createdAt: -1 }).explain("executionStats");
// totalDocsExamined 约等于 nReturned
// 不需要 SORT_KEY_GENERATOR（内存排序）

10.4.3 案例二：文章列表页

场景：按状态筛选已发布的文章，按发布时间降序，筛选某个时间范围内的文章。

// 查询语句
db.articles.find({
    status: "published",                                    // Equality
    publishedAt: {
        $gte: ISODate("2025-01-01"),                       // Range
        $lte: ISODate("2025-06-30")                        // Range
    }
}).sort({ publishedAt: -1 })                               // Sort

ESR 分析：

  • E：status（等值匹配）
  • S：publishedAt（排序字段）
  • R：publishedAt（范围查询）

此处 Sort 和 Range 使用的是同一个字段 publishedAt，此时将 publishedAt 放在 status 之后即可：

// ✅ publishedAt 同时满足排序和范围
db.articles.createIndex({ status: 1, publishedAt: -1 })

10.4.4 案例三：用户搜索

场景：搜索特定城市中年龄在20-30之间的活跃用户，按注册时间排序。

db.users.find({
    city: "北京",                      // Equality
    isActive: true,                    // Equality
    age: { $gte: 20, $lte: 30 }       // Range
}).sort({ registeredAt: -1 })          // Sort

ESR 分析：

  • E：city，isActive（两个等值条件，顺序不影响，按选择性高的放前面）
  • S：registeredAt（排序字段）
  • R：age（范围查询）
// ✅ ESR 索引
db.users.createIndex({ city: 1, isActive: 1, registeredAt: -1, age: 1 })

// 也可以（如果 isActive 的选择性更低）
db.users.createIndex({ isActive: 1, city: 1, registeredAt: -1, age: 1 })

提示：多个 Equality 字段的内部顺序理论上不影响性能，但通常建议把选择性高的字段放前面（即不同值更多的字段）。比如 city 可能有几百个取值，而 isActive 只有 true/false 两个值，所以 city 放前面更好。

10.4.5 反面案例：不遵循 ESR 的性能对比

以案例一的订单查询为例：

db.orders.find({
    userId: "U0001",
    totalAmount: { $gt: 100 }
}).sort({ createdAt: -1 })

对比不同索引顺序的性能：

// ❌ 索引1：Range 放在 Sort 前面（RSE 而非 ESR）
db.orders.createIndex({ totalAmount: 1, createdAt: -1, userId: 1 });

// explain 结果：
// - stage: SORT（内存排序！）
// - totalDocsExamined: 很大（扫描大量不属于 U0001 的文档）
// - executionTimeMillis: 较高

// ❌ 索引2：Sort 放在 Equality 前面（SER 顺序）
db.orders.createIndex({ createdAt: -1, userId: 1, totalAmount: 1 });

// explain 结果：
// - 虽然避免了内存排序，但扫描的文档数远多于 ESR 索引
// - totalDocsExamined: 较大

// ❌ 索引3：Range 在中间（ERS 而非 ESR）
db.orders.createIndex({ userId: 1, totalAmount: 1, createdAt: -1 });

// explain 结果：
// - stage: SORT（内存排序！）
// - 虽然 userId 定位很快，但 totalAmount 范围查询后 createdAt 不再有序

// ✅ 索引4：ESR 顺序
db.orders.createIndex({ userId: 1, createdAt: -1, totalAmount: 1 });

// explain 结果：
// - 无 SORT stage（利用索引排序）
// - totalDocsExamined ≈ nReturned（最少的文档扫描）
// - executionTimeMillis: 最低

性能对比总结：

索引顺序 内存排序 文档扫描数 相对耗时
{ totalAmount, createdAt, userId } (RSE) 是 很多 最慢
{ createdAt, userId, totalAmount } (SER) 否 较多 较慢
{ userId, totalAmount, createdAt } (ERS) 是 较少 中等
{ userId, createdAt, totalAmount } (ESR) 否 最少 最快

10.5 慢查询日志分析与性能调优

10.5.1 开启 Profiler

MongoDB 的 Database Profiler 可以记录超过指定阈值的慢查询：

// 查看当前 profiling 级别
db.getProfilingStatus()

// 级别说明：
// 0 - 关闭（默认）
// 1 - 只记录慢查询（超过 slowms 阈值）
// 2 - 记录所有操作

// 开启 profiler，记录超过 100ms 的查询
db.setProfilingLevel(1, { slowms: 100 })

// 记录所有操作（调试时使用，生产环境不建议）
db.setProfilingLevel(2)

// 关闭 profiler
db.setProfilingLevel(0)

10.5.2 system.profile 集合

慢查询日志记录在 system.profile 集合中：

// 查看最近的5条慢查询
db.system.profile.find().sort({ ts: -1 }).limit(5).pretty()

// 查看最耗时的查询
db.system.profile.find().sort({ millis: -1 }).limit(5).pretty()

// 查看特定集合的慢查询
db.system.profile.find({ ns: "mydb.orders" }).sort({ ts: -1 }).limit(10)

// 查看全表扫描的查询（没有使用索引）
db.system.profile.find({
    "planSummary": "COLLSCAN"
}).sort({ ts: -1 })

// 查看超过1秒的查询
db.system.profile.find({
    millis: { $gt: 1000 }
}).sort({ millis: -1 })

profile 文档的关键字段：

{
    "op": "query",                           // 操作类型
    "ns": "mydb.orders",                     // 命名空间（数据库.集合）
    "command": {                              // 执行的命令
        "find": "orders",
        "filter": { "userId": "U0001" },
        "sort": { "createdAt": -1 }
    },
    "keysExamined": 150,                     // 扫描的索引键数
    "docsExamined": 150,                     // 扫描的文档数
    "nreturned": 50,                         // 返回的文档数
    "millis": 250,                           // 执行时间（毫秒）
    "planSummary": "IXSCAN { userId: 1 }",  // 执行计划摘要
    "ts": ISODate("2025-03-15T10:30:00Z")   // 时间戳
}

10.5.3 慢查询分析流程

按照以下流程系统地分析和优化慢查询：

1. 发现慢查询
   ↓  （通过 profiler、监控告警、应用日志）
2. 使用 explain() 分析执行计划
   ↓  （查看 queryPlanner 和 executionStats）
3. 识别问题根因
   ↓  （COLLSCAN？内存排序？大量文档扫描？）
4. 制定优化方案
   ↓  （创建索引？修改查询？调整数据模型？）
5. 实施并验证
   ↓  （创建索引后重新 explain()，对比性能指标）
6. 持续监控

使用 explain() 的三个级别：

// 级别1：queryPlanner - 只看查询计划（不执行查询）
db.orders.find({ userId: "U0001" }).explain("queryPlanner")

// 级别2：executionStats - 执行查询并统计信息（最常用）
db.orders.find({ userId: "U0001" }).explain("executionStats")

// 级别3：allPlansExecution - 所有候选计划的执行信息
db.orders.find({ userId: "U0001" }).explain("allPlansExecution")

explain 结果的关键指标：

{
    "executionStats": {
        "executionSuccess": true,
        "nReturned": 50,              // ① 返回文档数
        "executionTimeMillis": 5,     // ② 执行时间
        "totalKeysExamined": 50,      // ③ 索引键扫描数
        "totalDocsExamined": 50,      // ④ 文档扫描数
        "executionStages": {
            "stage": "FETCH",         // ⑤ 执行阶段
            "inputStage": {
                "stage": "IXSCAN",    // 使用了索引扫描
                "indexName": "userId_1_createdAt_-1"
            }
        }
    }
}

性能评估原则：

指标 理想状态 问题信号
totalDocsExamined / nReturned 接近 1:1 比值 >> 1，说明扫描了大量无用文档
totalKeysExamined / nReturned 接近 1:1 比值 >> 1，说明索引选择性不够
executionStages.stage IXSCAN COLLSCAN（全集合扫描）
是否有 SORT 阶段 无 有 SORT 说明无法利用索引排序，需要内存排序

10.5.4 常见性能问题及解决方案

问题一：全集合扫描（COLLSCAN）

// 问题查询
db.orders.find({ status: "PAID", totalAmount: { $gt: 500 } })
// explain 显示：stage: "COLLSCAN"

// 解决方案：创建合适的索引
db.orders.createIndex({ status: 1, totalAmount: 1 })

问题二：内存排序（SORT 阶段）

// 问题查询
db.orders.find({ userId: "U0001" }).sort({ createdAt: -1 })
// 现有索引：{ userId: 1 }
// explain 显示有 SORT_KEY_GENERATOR 阶段

// 解决方案：在索引中包含排序字段
db.orders.createIndex({ userId: 1, createdAt: -1 })

问题三：索引选择性低

// 问题：isActive 只有 true/false 两个值，选择性很低
db.users.find({ isActive: true }).sort({ createdAt: -1 }).limit(20)
// 即使有 { isActive: 1, createdAt: -1 } 索引，仍然要扫描大量文档

// 解决方案：结合业务添加更有选择性的条件
db.users.find({ isActive: true, city: "北京" })
    .sort({ createdAt: -1 }).limit(20)
db.users.createIndex({ isActive: 1, city: 1, createdAt: -1 })

问题四：返回不必要的字段

// 问题：只需要 name 和 price，却返回了整个文档（包括大型 description 字段）
db.products.find({ category: "电子产品" })

// 解决方案：使用 projection 只返回需要的字段
db.products.find(
    { category: "电子产品" },
    { name: 1, price: 1, _id: 0 }
)

10.5.5 索引覆盖查询（Covered Query）

如果一个查询只需要索引中包含的字段，MongoDB 可以直接从索引返回结果，完全不需要访问文档。这被称为覆盖查询。

// 创建索引
db.orders.createIndex({ userId: 1, status: 1, totalAmount: 1 })

// ✅ 覆盖查询：查询条件和返回字段都在索引中
db.orders.find(
    { userId: "U0001", status: "PAID" },
    { totalAmount: 1, _id: 0 }           // _id: 0 是关键，_id 不在索引中
)

// explain 结果中会看到：
// totalDocsExamined: 0  ← 零文档扫描！
// 因为所有需要的数据都在索引中

// ❌ 非覆盖查询：返回了不在索引中的字段
db.orders.find(
    { userId: "U0001", status: "PAID" },
    { totalAmount: 1, createdAt: 1, _id: 0 }  // createdAt 不在索引中
)
// totalDocsExamined > 0（需要回表查文档）

覆盖查询的条件：

  1. 查询的所有字段都在索引中
  2. 返回的所有字段都在索引中
  3. 没有返回 _id（除非 _id 在索引中），或显式排除 _id: 0

Java 示例：

public List<BigDecimal> getOrderAmounts(String userId, String status) {
    Query query = Query.query(
        Criteria.where("userId").is(userId)
                .and("status").is(status)
    );
    query.fields().include("totalAmount").exclude("id");

    List<Order> orders = mongoTemplate.find(query, Order.class);
    return orders.stream()
            .map(Order::getTotalAmount)
            .collect(Collectors.toList());
}

10.5.6 索引交叉（Index Intersection）

MongoDB 可以在某些情况下使用多个单字段索引的交叉来满足查询，而不需要一个复合索引：

// 假设有两个单字段索引
db.orders.createIndex({ userId: 1 });
db.orders.createIndex({ status: 1 });

// MongoDB 可能会使用索引交叉来处理这个查询
db.orders.find({ userId: "U0001", status: "PAID" })

但是，索引交叉通常性能不如复合索引：

// ✅ 更好的方案：使用复合索引
db.orders.createIndex({ userId: 1, status: 1 })

// 复合索引 vs 索引交叉的性能对比
db.orders.find({ userId: "U0001", status: "PAID" }).explain("executionStats")
// 复合索引：totalKeysExamined ≈ nReturned
// 索引交叉：totalKeysExamined >> nReturned（需要合并两个索引的结果）

建议：不要依赖索引交叉。如果一个查询频繁使用多个字段的条件，应该创建相应的复合索引。索引交叉只是一种"备选机制"，不是优化手段。

10.5.7 完整的性能调优案例

场景：一个电商平台的订单列表页，用户反馈查询速度很慢。

步骤一：收集慢查询信息

// 开启 profiler
db.setProfilingLevel(1, { slowms: 50 });

// 等待一段时间后查看慢查询
db.system.profile.find({
    ns: "shop.orders"
}).sort({ millis: -1 }).limit(5).pretty();

发现以下查询频繁出现且耗时严重：

// 慢查询：按用户ID + 状态筛选 + 时间范围 + 时间排序
db.orders.find({
    userId: "U0001",
    status: { $in: ["PAID", "SHIPPED"] },
    createdAt: { $gte: ISODate("2025-01-01"), $lte: ISODate("2025-03-31") }
}).sort({ createdAt: -1 }).limit(20)

步骤二：分析当前执行计划

问题诊断：全集合扫描 + 内存排序，没有可用的索引。

步骤三：设计索引

使用 ESR 规则分析：

  • Equality：userId（精确匹配）
  • Sort：createdAt（排序字段）
  • Range：status（$in 是范围查询）、createdAt（日期范围）

注意 createdAt 同时用于排序和范围查询。status 使用了 $in，虽然看似是几个离散值，但在索引中 $in 被视为范围查询。

// 方案一：userId + createdAt + status（ESR）
db.orders.createIndex({ userId: 1, createdAt: -1, status: 1 })

步骤六：在 Java 中实现优化后的查询

@Service
public class OrderQueryService {

    private final MongoTemplate mongoTemplate;

    public OrderQueryService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 分页查询用户订单（已优化索引）
     */
    public List<Order> findUserOrders(
            String userId, List<String> statuses,
            Instant startDate, Instant endDate,
            int page, int size) {

        Criteria criteria = Criteria.where("userId").is(userId);

        if (statuses != null && !statuses.isEmpty()) {
            criteria = criteria.and("status").in(statuses);
        }

        if (startDate != null && endDate != null) {
            criteria = criteria.and("createdAt").gte(startDate).lte(endDate);
        }

        Query query = Query.query(criteria)
                .with(Sort.by(Sort.Direction.DESC, "createdAt"))
                .skip((long) page * size)
                .limit(size);

        return mongoTemplate.find(query, Order.class);
    }

    /**
     * 程序化创建索引（ESR 顺序）
     */
    public void createOptimizedIndexes() {
        IndexOperations indexOps = mongoTemplate.indexOps(Order.class);

        indexOps.ensureIndex(
            new Index()
                .on("userId", Sort.Direction.ASC)
                .on("createdAt", Sort.Direction.DESC)
                .on("status", Sort.Direction.ASC)
                .named("idx_user_orders_esr")
        );
    }
}

本章小结

知识点 关键要点
文本索引 一个集合只能有一个；支持权重配置和 $text 查询
地理空间索引 使用 GeoJSON 格式；$near、$geoWithin、$geoNear
通配符索引 适用于动态字段；不支持复合查询和排序优化
ESR 规则 Equality 最前 → Sort 中间 → Range 最后
慢查询分析 profiler + explain() + 性能指标对比
覆盖查询 查询和返回字段都在索引中，零文档扫描
索引交叉 不如复合索引高效，不应作为优化策略
