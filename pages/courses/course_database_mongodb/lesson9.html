第09章 事务与数据一致性

MongoDB 并非一开始就支持多文档事务，但它对单文档操作天然具有原子性。随着 4.0 和 4.2 版本的迭代，MongoDB 已经具备了完整的 ACID 事务能力。本章将从单文档原子性出发，逐步深入多文档事务，并重点讲解如何在 Spring Boot 中优雅地管理事务。

9.1 MongoDB 事务的发展历程与支持范围

9.1.1 事务支持的演进

MongoDB 的事务支持经历了三个关键阶段：

版本 事务能力 说明
MongoDB 3.x 及之前 仅单文档原子性 单个文档的增删改操作是原子的
MongoDB 4.0（2018） 副本集多文档事务 首次支持跨多个文档的 ACID 事务，但仅限副本集部署
MongoDB 4.2（2019） 分片集群多文档事务 事务能力扩展到分片集群，支持跨分片的分布式事务

9.1.2 MongoDB 4.0：副本集多文档事务

MongoDB 4.0 是一个里程碑版本，它在副本集（Replica Set）环境中引入了多文档事务。这意味着：

  • 可以在一个事务中对同一个副本集内的多个集合、多个文档进行读写操作
  • 事务中的所有操作要么全部提交，要么全部回滚
  • 需要注意：独立模式（Standalone）不支持事务，至少需要一个单节点副本集

9.1.3 MongoDB 4.2：分片集群事务

MongoDB 4.2 将事务能力扩展到了分片集群：

  • 支持跨多个分片的分布式事务
  • 使用两阶段提交（Two-Phase Commit）协议确保一致性
  • 事务可以跨越多个分片上的多个集合

9.1.4 事务的 ACID 保证

MongoDB 的多文档事务提供完整的 ACID 保证：

特性 说明
Atomicity（原子性） 事务中的所有操作要么全部成功提交，要么全部回滚，不存在部分成功的情况
Consistency（一致性） 事务执行前后，数据库保持一致状态，所有约束（如唯一索引）都会被校验
Isolation（隔离性） 使用快照隔离（Snapshot Isolation），事务内读取的是事务开始时的数据快照
Durability（持久性） 事务提交后，数据会持久化到磁盘（通过写关注 writeConcern: "majority" 保证）

9.1.5 与关系型数据库事务的对比

对比维度 关系型数据库（如 MySQL） MongoDB
事务支持时间 从诞生起就支持 4.0 版本开始支持多文档事务
单行/单文档原子性 支持 天然支持（无需事务）
多行/多文档事务 核心特性，使用频繁 支持但建议谨慎使用
隔离级别 多种级别（RU/RC/RR/Serializable） 快照隔离（Snapshot Isolation）
分布式事务 需要 XA 等协议 4.2 起原生支持分片集群事务
设计哲学 依赖事务保证一致性 优先通过文档模型设计减少事务需求
性能影响 事务是常规操作 事务有额外开销，建议控制使用范围

核心理念：在 MongoDB 中，良好的文档模型设计可以大幅减少对多文档事务的依赖。MongoDB 鼓励将相关数据内嵌在同一个文档中，利用单文档原子性来满足大部分一致性需求。多文档事务应作为"最后手段"使用。

9.2 单文档原子性操作

9.2.1 天然的单文档原子性

MongoDB 对单个文档的任何写操作都是原子的，无论该文档有多少嵌套字段或数组元素。这意味着：

  • 一次 updateOne 中使用 $set 修改多个字段，这些修改要么全部生效，要么全部不生效
  • 一次 $push 向数组添加元素的同时 $inc 某个计数器，这两个操作是原子的
  • 其他客户端不会看到"修改了一半"的文档

这是 MongoDB 与关系型数据库最大的区别之一——在关系型数据库中，修改多个列虽然也是原子的，但修改多张表的关联数据就必须使用事务。而在 MongoDB 中，如果你把关联数据内嵌在同一个文档里，就天然具有原子性。

9.2.2 findOneAndUpdate

findOneAndUpdate 在一次原子操作中查找并更新文档，同时返回更新前或更新后的文档。

Shell 示例：

// 场景：商品库存扣减（原子操作，避免超卖）
// 只有库存 >= 1 时才扣减，并返回更新后的文档
db.products.findOneAndUpdate(
    { _id: ObjectId("665a1b2c3d4e5f6a7b8c9d0e"), stock: { $gte: 1 } },
    {
        $inc: { stock: -1, soldCount: 1 },
        $set: { lastSoldAt: new Date() }
    },
    {
        returnDocument: "after",  // 返回更新后的文档
        projection: { name: 1, stock: 1, soldCount: 1 }
    }
)

// 返回结果示例：
// {
//     _id: ObjectId("665a1b2c3d4e5f6a7b8c9d0e"),
//     name: "MongoDB实战指南",
//     stock: 49,
//     soldCount: 151
// }
// 如果没有匹配的文档（库存不足），返回 null

Java 示例（MongoTemplate）：

import org.springframework.data.mongodb.core.FindAndModifyOptions;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

import java.time.Instant;

@Service
public class ProductService {

    private final MongoTemplate mongoTemplate;

    public ProductService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 原子性扣减库存，返回更新后的商品信息。
     * 如果库存不足，返回 null。
     */
    public Product decrementStock(String productId) {
        Query query = Query.query(
            Criteria.where("_id").is(productId)
                    .and("stock").gte(1)
        );

        Update update = new Update()
                .inc("stock", -1)
                .inc("soldCount", 1)
                .set("lastSoldAt", Instant.now());

        FindAndModifyOptions options = FindAndModifyOptions.options()
                .returnNew(true);  // 返回更新后的文档

        return mongoTemplate.findAndModify(query, update, options, Product.class);
    }
}

9.2.3 findOneAndDelete

findOneAndDelete 在一次原子操作中查找并删除文档，返回被删除的文档。

Shell 示例：

// 场景：从任务队列中取出一个待处理的任务（取出即删除）
db.taskQueue.findOneAndDelete(
    { status: "pending" },
    {
        sort: { priority: -1, createdAt: 1 }  // 优先级最高、最早创建的优先
    }
)

// 返回被删除的文档：
// {
//     _id: ObjectId("..."),
//     taskType: "sendEmail",
//     payload: { to: "user@example.com", subject: "Welcome" },
//     status: "pending",
//     priority: 10,
//     createdAt: ISODate("2025-03-01T10:00:00Z")
// }

Java 示例：

public Task pollTask() {
    Query query = Query.query(Criteria.where("status").is("pending"))
            .with(Sort.by(Sort.Direction.DESC, "priority")
                       .and(Sort.by(Sort.Direction.ASC, "createdAt")));

    return mongoTemplate.findAndRemove(query, Task.class);
}

9.2.4 findOneAndReplace

findOneAndReplace 在一次原子操作中查找并替换整个文档。

Shell 示例：

// 场景：用新版本的配置替换旧版本（保留 _id，替换其他所有字段）
db.configs.findOneAndReplace(
    { key: "app_settings", version: 2 },
    {
        key: "app_settings",
        version: 3,
        theme: "dark",
        language: "zh-CN",
        maxUploadSize: 10485760,
        updatedAt: new Date()
    },
    { returnDocument: "after" }
)

Java 示例：

public AppConfig replaceConfig(AppConfig newConfig) {
    Query query = Query.query(
        Criteria.where("key").is(newConfig.getKey())
                .and("version").is(newConfig.getVersion() - 1)
    );

    return mongoTemplate.findAndReplace(query, newConfig,
            FindAndReplaceOptions.options().returnNew());
}

9.2.5 $set 多个字段的原子性

在单次更新操作中使用 $set 修改多个字段，所有修改都是原子的：

Shell 示例：

// 场景：用户完成实名认证，同时更新多个字段
db.users.updateOne(
    { _id: ObjectId("665a1b2c3d4e5f6a7b8c9d0e") },
    {
        $set: {
            "profile.realName": "张三",
            "profile.idCard": "110101199001011234",
            "profile.verified": true,
            "profile.verifiedAt": new Date(),
            status: "verified"
        },
        $inc: { version: 1 }
    }
)
// 上述所有字段的修改在一次原子操作中完成
// 其他客户端不会看到"realName已更新但verified还是false"的中间状态

Java 示例：

public UpdateResult verifyUser(String userId, String realName, String idCard) {
    Query query = Query.query(Criteria.where("_id").is(userId));

    Update update = new Update()
            .set("profile.realName", realName)
            .set("profile.idCard", idCard)
            .set("profile.verified", true)
            .set("profile.verifiedAt", Instant.now())
            .set("status", "verified")
            .inc("version", 1);

    return mongoTemplate.updateFirst(query, update, User.class);
}

小结：在设计数据模型时，尽量将需要一起修改的数据放在同一个文档中。这样利用单文档原子性，无需事务即可保证数据一致性。

9.3 多文档事务（Multi-document Transaction）

当业务逻辑确实需要跨多个文档或多个集合保证一致性时，就需要使用多文档事务。

9.3.1 事务的基本语法（Shell）

// 事务的基本结构
const session = db.getMongo().startSession();
session.startTransaction({
    readConcern: { level: "snapshot" },
    writeConcern: { w: "majority" },
    readPreference: "primary"
});

try {
    const accountsColl = session.getDatabase("bank").getCollection("accounts");

    // 在事务中执行操作...
    accountsColl.updateOne(
        { accountNo: "A001" },
        { $inc: { balance: -500 } },
        { session }   // 注意：Shell 中需要传递 session（MongoDB 6.0+ 写法）
    );

    accountsColl.updateOne(
        { accountNo: "A002" },
        { $inc: { balance: 500 } },
        { session }
    );

    // 提交事务
    session.commitTransaction();
    print("事务提交成功");
} catch (error) {
    // 回滚事务
    session.abortTransaction();
    print("事务回滚：" + error.message);
} finally {
    session.endSession();
}

核心 API：

方法 说明
startSession() 创建一个会话（Session）
session.startTransaction(options) 在会话中开始一个事务
session.commitTransaction() 提交事务，使所有变更生效
session.abortTransaction() 中止事务，回滚所有变更
session.endSession() 结束会话，释放资源

9.3.2 完整的 Shell 事务示例：银行转账

这是一个经典的转账场景，需要保证"转出扣款"和"转入加款"要么同时成功，要么同时失败。

准备数据：

// 创建测试数据
use bank;
db.accounts.drop();
db.accounts.insertMany([
    { accountNo: "A001", owner: "张三", balance: 10000 },
    { accountNo: "A002", owner: "李四", balance: 5000 }
]);

// 转账记录集合
db.transferLogs.drop();

执行转账事务：

// 转账函数：从 fromAccount 转 amount 元到 toAccount
function transfer(fromAccountNo, toAccountNo, amount) {
    const session = db.getMongo().startSession();

    session.startTransaction({
        readConcern: { level: "snapshot" },
        writeConcern: { w: "majority" }
    });

    try {
        const accounts = session.getDatabase("bank").getCollection("accounts");
        const transferLogs = session.getDatabase("bank").getCollection("transferLogs");

        // 1. 检查转出账户余额
        const fromAccount = accounts.findOne(
            { accountNo: fromAccountNo },
            { session }
        );

        if (!fromAccount) {
            throw new Error("转出账户不存在: " + fromAccountNo);
        }
        if (fromAccount.balance < amount) {
            throw new Error("余额不足，当前余额: " + fromAccount.balance);
        }

        // 2. 转出方扣款
        accounts.updateOne(
            { accountNo: fromAccountNo },
            {
                $inc: { balance: -amount },
                $set: { lastTransferAt: new Date() }
            },
            { session }
        );

        // 3. 转入方加款
        const toResult = accounts.updateOne(
            { accountNo: toAccountNo },
            {
                $inc: { balance: amount },
                $set: { lastTransferAt: new Date() }
            },
            { session }
        );

        if (toResult.matchedCount === 0) {
            throw new Error("转入账户不存在: " + toAccountNo);
        }

        // 4. 记录转账日志
        transferLogs.insertOne({
            from: fromAccountNo,
            to: toAccountNo,
            amount: amount,
            status: "success",
            createdAt: new Date()
        }, { session });

        // 5. 提交事务
        session.commitTransaction();
        print("转账成功：" + fromAccountNo + " -> " + toAccountNo + "，金额：" + amount);

    } catch (error) {
        session.abortTransaction();
        print("转账失败，事务已回滚：" + error.message);
    } finally {
        session.endSession();
    }
}

// 执行转账
transfer("A001", "A002", 2000);

// 验证结果
db.accounts.find().pretty();
// 张三余额：8000，李四余额：7000

db.transferLogs.find().pretty();
// 有一条转账成功的记录

// 测试失败场景：余额不足
transfer("A001", "A002", 99999);
// 输出：转账失败，事务已回滚：余额不足，当前余额: 8000

9.3.3 使用 MongoTemplate 进行手动事务管理

在 Spring Boot 中，可以使用 MongoTemplate 配合 ClientSession 手动管理事务。

Java 代码：

import com.mongodb.ClientSessionOptions;
import com.mongodb.TransactionOptions;
import com.mongodb.ReadConcern;
import com.mongodb.WriteConcern;
import com.mongodb.client.ClientSession;
import com.mongodb.client.MongoClient;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

import java.math.BigDecimal;
import java.time.Instant;

@Service
public class TransferService {

    private final MongoClient mongoClient;
    private final MongoTemplate mongoTemplate;

    public TransferService(MongoClient mongoClient, MongoTemplate mongoTemplate) {
        this.mongoClient = mongoClient;
        this.mongoTemplate = mongoTemplate;
    }

    public void transfer(String fromAccountNo, String toAccountNo, BigDecimal amount) {
        // 创建 Session
        ClientSession session = mongoClient.startSession(
                ClientSessionOptions.builder()
                        .causallyConsistent(true)
                        .build()
        );

        // 配置事务选项
        TransactionOptions txnOptions = TransactionOptions.builder()
                .readConcern(ReadConcern.SNAPSHOT)
                .writeConcern(WriteConcern.MAJORITY)
                .build();

        try {
            session.startTransaction(txnOptions);

            // 1. 查询转出账户
            Query fromQuery = Query.query(Criteria.where("accountNo").is(fromAccountNo));
            Account fromAccount = mongoTemplate.withSession(() -> session)
                    .execute(template -> template.findOne(fromQuery, Account.class));

            if (fromAccount == null) {
                throw new RuntimeException("转出账户不存在: " + fromAccountNo);
            }
            if (fromAccount.getBalance().compareTo(amount) < 0) {
                throw new RuntimeException("余额不足，当前余额: " + fromAccount.getBalance());
            }

            // 2. 扣减转出方余额
            mongoTemplate.withSession(() -> session).execute(template -> {
                template.updateFirst(
                        Query.query(Criteria.where("accountNo").is(fromAccountNo)),
                        new Update().inc("balance", amount.negate().doubleValue())
                                    .set("lastTransferAt", Instant.now()),
                        Account.class
                );
                return null;
            });

            // 3. 增加转入方余额
            mongoTemplate.withSession(() -> session).execute(template -> {
                var result = template.updateFirst(
                        Query.query(Criteria.where("accountNo").is(toAccountNo)),
                        new Update().inc("balance", amount.doubleValue())
                                    .set("lastTransferAt", Instant.now()),
                        Account.class
                );
                if (result.getMatchedCount() == 0) {
                    throw new RuntimeException("转入账户不存在: " + toAccountNo);
                }
                return null;
            });

            // 4. 记录转账日志
            mongoTemplate.withSession(() -> session).execute(template -> {
                TransferLog log = new TransferLog();
                log.setFromAccount(fromAccountNo);
                log.setToAccount(toAccountNo);
                log.setAmount(amount);
                log.setStatus("success");
                log.setCreatedAt(Instant.now());
                template.insert(log);
                return null;
            });

            // 5. 提交事务
            session.commitTransaction();

        } catch (Exception e) {
            session.abortTransaction();
            throw new RuntimeException("转账失败: " + e.getMessage(), e);
        } finally {
            session.close();
        }
    }
}

9.3.4 事务的超时和重试

事务超时

MongoDB 事务有默认的超时时间：

  • 事务生命周期超时：默认 60 秒（transactionLifetimeLimitSeconds），可在服务端配置
  • 锁等待超时：默认 5 毫秒（maxTransactionLockRequestTimeoutMillis）

如果事务在超时时间内没有提交或中止，MongoDB 会自动中止该事务。

事务重试

MongoDB 驱动支持两种重试标签：

  • RetryableWriteError：可重试的写错误
  • TransientTransactionError：瞬时事务错误（如写冲突），可以安全地重试整个事务
  • UnknownTransactionCommitResult：提交结果未知，可以安全地重试提交操作

带重试逻辑的事务模板（Shell）：

function runTransactionWithRetry(txnFunc, session) {
    while (true) {
        try {
            txnFunc(session);
            break;
        } catch (error) {
            if (error.hasOwnProperty("errorLabels") &&
                error.errorLabels.includes("TransientTransactionError")) {
                print("遇到 TransientTransactionError，重试事务...");
                continue;
            }
            throw error;
        }
    }
}

function commitWithRetry(session) {
    while (true) {
        try {
            session.commitTransaction();
            print("事务提交成功");
            break;
        } catch (error) {
            if (error.hasOwnProperty("errorLabels") &&
                error.errorLabels.includes("UnknownTransactionCommitResult")) {
                print("提交结果未知，重试提交...");
                continue;
            }
            throw error;
        }
    }
}

Java 中的重试逻辑：

public void transferWithRetry(String from, String to, BigDecimal amount) {
    int maxRetries = 3;
    for (int attempt = 1; attempt <= maxRetries; attempt++) {
        try {
            transfer(from, to, amount);
            return; // 成功则退出
        } catch (Exception e) {
            if (isTransientError(e) && attempt < maxRetries) {
                System.out.println("事务瞬时错误，第 " + attempt + " 次重试...");
                try {
                    Thread.sleep(100L * attempt); // 指数退避
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException(ie);
                }
            } else {
                throw e;
            }
        }
    }
}

private boolean isTransientError(Exception e) {
    String message = e.getMessage();
    return message != null && (
        message.contains("TransientTransactionError") ||
        message.contains("WriteConflict")
    );
}

9.4 在 Spring Boot 中使用 @Transactional 管理 MongoDB 事务

手动管理事务（如 9.3 节所示）虽然灵活，但代码冗长。Spring Boot 提供了声明式事务管理，通过 @Transactional 注解即可自动管理事务的开始、提交和回滚。

9.4.1 配置 MongoTransactionManager Bean

要使用 @Transactional，需要配置 MongoTransactionManager：

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.mongodb.MongoDatabaseFactory;
import org.springframework.data.mongodb.MongoTransactionManager;

@Configuration
public class MongoTransactionConfig {

    @Bean
    public MongoTransactionManager transactionManager(MongoDatabaseFactory dbFactory) {
        return new MongoTransactionManager(dbFactory);
    }
}

这就是全部配置！Spring Boot 会自动将这个 MongoTransactionManager 与 @Transactional 注解关联。

9.4.2 使用 @Transactional 注解

有了上面的配置，就可以像操作关系型数据库一样使用 @Transactional 了：

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;

import java.math.BigDecimal;
import java.time.Instant;

@Service
public class AccountService {

    private final MongoTemplate mongoTemplate;

    public AccountService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 使用 @Transactional 声明式事务进行转账。
     * 如果方法中抛出运行时异常，事务会自动回滚。
     */
    @Transactional
    public void transfer(String fromAccountNo, String toAccountNo, BigDecimal amount) {
        // 1. 查询转出账户
        Account fromAccount = mongoTemplate.findOne(
                Query.query(Criteria.where("accountNo").is(fromAccountNo)),
                Account.class
        );

        if (fromAccount == null) {
            throw new RuntimeException("转出账户不存在: " + fromAccountNo);
        }
        if (fromAccount.getBalance().compareTo(amount) < 0) {
            throw new RuntimeException("余额不足，当前余额: " + fromAccount.getBalance());
        }

        // 2. 扣减转出方余额
        mongoTemplate.updateFirst(
                Query.query(Criteria.where("accountNo").is(fromAccountNo)),
                new Update().inc("balance", amount.negate().doubleValue())
                            .set("lastTransferAt", Instant.now()),
                Account.class
        );

        // 3. 增加转入方余额
        var result = mongoTemplate.updateFirst(
                Query.query(Criteria.where("accountNo").is(toAccountNo)),
                new Update().inc("balance", amount.doubleValue())
                            .set("lastTransferAt", Instant.now()),
                Account.class
        );

        if (result.getMatchedCount() == 0) {
            throw new RuntimeException("转入账户不存在: " + toAccountNo);
        }

        // 4. 记录转账日志
        TransferLog log = new TransferLog();
        log.setFromAccount(fromAccountNo);
        log.setToAccount(toAccountNo);
        log.setAmount(amount);
        log.setStatus("success");
        log.setCreatedAt(Instant.now());
        mongoTemplate.insert(log);
    }

    /**
     * 删除文章并清理所有评论（事务保证一致性）
     */
    @Transactional
    public void deleteArticleWithComments(String articleId) {
        // 删除文章
        mongoTemplate.remove(
                Query.query(Criteria.where("_id").is(articleId)),
                "articles"
        );

        // 删除该文章的所有评论
        mongoTemplate.remove(
                Query.query(Criteria.where("articleId").is(articleId)),
                "comments"
        );
    }
}

9.4.3 重要注意事项：MongoDB 事务要求副本集部署

这是初学者最容易踩的坑：MongoDB 事务（包括 @Transactional）必须在副本集或分片集群环境下才能使用，单独的 Standalone 实例不支持事务。

如果你在 Standalone 模式下使用 @Transactional，会报错：

com.mongodb.MongoCommandException: Command failed with error 263 (NoReplicationEnabled):
'Transaction numbers are only allowed on a replica set member or mongos'

9.4.4 Docker 单节点副本集配置方法

在开发环境中，可以使用 Docker 快速启动一个单节点副本集来支持事务：

docker-compose.yml：

version: '3.8'
services:
  mongodb:
    image: mongo:7.0
    container_name: mongo-rs
    ports:
      - "27017:27017"
    command: ["mongod", "--replSet", "rs0", "--bind_ip_all"]
    volumes:
      - mongo-data:/data/db

volumes:
  mongo-data:

启动并初始化副本集：

docker-compose up -d

# 进入容器并初始化副本集
docker exec -it mongo-rs mongosh --eval "
rs.initiate({
    _id: 'rs0',
    members: [{ _id: 0, host: 'localhost:27017' }]
})
"

# 验证副本集状态
docker exec -it mongo-rs mongosh --eval "rs.status().ok"
# 输出 1 表示成功

application.yml 配置：

spring:
  data:
    mongodb:
      uri: mongodb://localhost:27017/mydb?replicaSet=rs0

提示：使用 replicaSet=rs0 连接参数告诉驱动这是一个副本集。即使只有一个节点，也要这样配置才能使用事务。

9.4.5 完整的 Java 示例（Service 层事务方法）

下面是一个完整的订单创建场景，涉及多个集合的操作：

实体类：

@Document(collection = "orders")
public class Order {
    @Id
    private String id;
    private String userId;
    private List<OrderItem> items;
    private BigDecimal totalAmount;
    private String status;  // PENDING, PAID, CANCELLED
    private Instant createdAt;

    // getters / setters ...
}

public class OrderItem {
    private String productId;
    private String productName;
    private int quantity;
    private BigDecimal price;

    // getters / setters ...
}

Service 层：

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;

import java.math.BigDecimal;
import java.time.Instant;
import java.util.List;

@Service
public class OrderService {

    private final MongoTemplate mongoTemplate;

    public OrderService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 创建订单（事务操作）：
     * 1. 检查并扣减库存
     * 2. 创建订单文档
     * 3. 更新用户购买统计
     */
    @Transactional
    public Order createOrder(String userId, List<OrderItem> items) {
        BigDecimal totalAmount = BigDecimal.ZERO;

        // 1. 逐个检查并扣减库存
        for (OrderItem item : items) {
            Query stockQuery = Query.query(
                Criteria.where("_id").is(item.getProductId())
                        .and("stock").gte(item.getQuantity())
            );
            Update stockUpdate = new Update()
                    .inc("stock", -item.getQuantity())
                    .inc("soldCount", item.getQuantity());

            var result = mongoTemplate.updateFirst(stockQuery, stockUpdate, Product.class);

            if (result.getMatchedCount() == 0) {
                // 抛出异常会触发事务回滚，之前扣减的库存也会恢复
                throw new RuntimeException("商品库存不足: " + item.getProductName());
            }

            totalAmount = totalAmount.add(
                item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity()))
            );
        }

        // 2. 创建订单
        Order order = new Order();
        order.setUserId(userId);
        order.setItems(items);
        order.setTotalAmount(totalAmount);
        order.setStatus("PENDING");
        order.setCreatedAt(Instant.now());

        mongoTemplate.insert(order);

        // 3. 更新用户购买统计
        mongoTemplate.updateFirst(
                Query.query(Criteria.where("_id").is(userId)),
                new Update()
                        .inc("stats.orderCount", 1)
                        .inc("stats.totalSpent", totalAmount.doubleValue())
                        .set("stats.lastOrderAt", Instant.now()),
                User.class
        );

        return order;
    }

    /**
     * 取消订单（事务操作）：
     * 1. 将订单状态改为 CANCELLED
     * 2. 恢复库存
     */
    @Transactional
    public void cancelOrder(String orderId) {
        Order order = mongoTemplate.findById(orderId, Order.class);
        if (order == null) {
            throw new RuntimeException("订单不存在: " + orderId);
        }
        if (!"PENDING".equals(order.getStatus())) {
            throw new RuntimeException("只能取消待支付的订单，当前状态: " + order.getStatus());
        }

        // 1. 更新订单状态
        mongoTemplate.updateFirst(
                Query.query(Criteria.where("_id").is(orderId)),
                new Update().set("status", "CANCELLED")
                            .set("cancelledAt", Instant.now()),
                Order.class
        );

        // 2. 恢复库存
        for (OrderItem item : order.getItems()) {
            mongoTemplate.updateFirst(
                    Query.query(Criteria.where("_id").is(item.getProductId())),
                    new Update()
                            .inc("stock", item.getQuantity())
                            .inc("soldCount", -item.getQuantity()),
                    Product.class
            );
        }

        // 3. 更新用户统计
        mongoTemplate.updateFirst(
                Query.query(Criteria.where("_id").is(order.getUserId())),
                new Update()
                        .inc("stats.orderCount", -1)
                        .inc("stats.totalSpent", order.getTotalAmount().negate().doubleValue()),
                User.class
        );
    }
}

9.4.6 事务回滚测试

编写测试验证事务回滚是否正常工作：

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;

import java.math.BigDecimal;
import java.util.List;

import static org.assertj.core.api.Assertions.*;

@SpringBootTest
class OrderServiceTest {

    @Autowired
    private OrderService orderService;

    @Autowired
    private MongoTemplate mongoTemplate;

    @Test
    void testCreateOrder_shouldRollbackWhenStockInsufficient() {
        // 准备数据：商品A库存10，商品B库存1
        Product productA = new Product("P001", "商品A", 10, BigDecimal.valueOf(100));
        Product productB = new Product("P002", "商品B", 1, BigDecimal.valueOf(200));
        mongoTemplate.save(productA);
        mongoTemplate.save(productB);

        // 下单：商品A买2件，商品B买5件（超过库存）
        List<OrderItem> items = List.of(
                new OrderItem("P001", "商品A", 2, BigDecimal.valueOf(100)),
                new OrderItem("P002", "商品B", 5, BigDecimal.valueOf(200))  // 库存不足
        );

        // 应该抛出异常
        assertThatThrownBy(() -> orderService.createOrder("U001", items))
                .isInstanceOf(RuntimeException.class)
                .hasMessageContaining("库存不足");

        // 验证事务回滚：商品A的库存应该恢复为10（不是8）
        Product productAAfter = mongoTemplate.findById("P001", Product.class);
        assertThat(productAAfter.getStock()).isEqualTo(10);

        // 验证没有订单被创建
        long orderCount = mongoTemplate.count(
                Query.query(Criteria.where("userId").is("U001")),
                Order.class
        );
        assertThat(orderCount).isEqualTo(0);
    }

    @Test
    void testCreateOrder_shouldCommitWhenAllSucceed() {
        Product productA = new Product("P001", "商品A", 10, BigDecimal.valueOf(100));
        Product productB = new Product("P002", "商品B", 10, BigDecimal.valueOf(200));
        mongoTemplate.save(productA);
        mongoTemplate.save(productB);

        List<OrderItem> items = List.of(
                new OrderItem("P001", "商品A", 2, BigDecimal.valueOf(100)),
                new OrderItem("P002", "商品B", 3, BigDecimal.valueOf(200))
        );

        Order order = orderService.createOrder("U001", items);

        // 验证订单创建成功
        assertThat(order.getId()).isNotNull();
        assertThat(order.getTotalAmount()).isEqualByComparingTo(BigDecimal.valueOf(800));

        // 验证库存已扣减
        Product productAAfter = mongoTemplate.findById("P001", Product.class);
        assertThat(productAAfter.getStock()).isEqualTo(8);

        Product productBAfter = mongoTemplate.findById("P002", Product.class);
        assertThat(productBAfter.getStock()).isEqualTo(7);
    }
}

9.5 事务的限制与最佳实践

9.5.1 性能影响

事务相比普通操作有额外的性能开销：

  • 延迟增加：事务需要协调多个操作，涉及锁获取、快照管理和提交协议
  • 资源消耗：事务会持有锁资源和 WiredTiger 快照，占用更多内存
  • 吞吐量下降：长时间运行的事务可能阻塞其他操作
  • 写冲突：多个事务修改同一文档时会产生写冲突，导致其中一个需要重试

性能对比示意：

普通操作（无事务）：延迟 ~1ms
单文档原子操作：     延迟 ~1ms
多文档事务（同节点）：延迟 ~5-10ms
跨分片事务：         延迟 ~20-50ms（取决于网络和分片数量）

9.5.2 时间限制

限制 默认值 说明
transactionLifetimeLimitSeconds 60 秒 事务的最大持续时间，超时自动中止
maxTransactionLockRequestTimeoutMillis 5 毫秒 事务获取锁的等待时间

可以在服务端修改：

// 修改事务最大持续时间为 120 秒（生产环境慎用）
db.adminCommand({
    setParameter: 1,
    transactionLifetimeLimitSeconds: 120
})

9.5.3 大小限制

  • oplog 条目大小：单个事务产生的 oplog 条目不能超过 16 MB
  • 如果事务涉及大量写操作，可能会超过此限制导致事务失败
  • 每个事务中的操作数没有硬性限制，但受 oplog 大小约束

9.5.4 其他限制

  • 事务中不能创建集合（MongoDB 4.4 起部分支持，但有限制）
  • 事务中不能创建索引
  • 事务不能跨数据库使用 $graphLookup 等受限操作
  • 只读事务不会获取锁但仍消耗快照资源
  • 游标在事务外部不可使用

9.5.5 最佳实践

1. 优先通过文档设计避免事务

✅ 好的设计：把订单和订单项放在同一个文档中
{
    _id: "order-001",
    userId: "U001",
    items: [
        { productId: "P001", quantity: 2, price: 100 },
        { productId: "P002", quantity: 1, price: 200 }
    ],
    totalAmount: 400,
    status: "PENDING"
}

❌ 不好的设计：订单和订单项分成两个集合，每次操作都需要事务
orders: { _id: "order-001", userId: "U001", totalAmount: 400 }
orderItems: [
    { orderId: "order-001", productId: "P001", quantity: 2 },
    { orderId: "order-001", productId: "P002", quantity: 1 }
]

2. 事务要短小精悍

// ✅ 好：事务内只做必要的数据库操作
@Transactional
public void transfer(String from, String to, BigDecimal amount) {
    deductBalance(from, amount);
    addBalance(to, amount);
    recordLog(from, to, amount);
}

// ❌ 坏：事务内做了大量非数据库操作
@Transactional
public void transfer(String from, String to, BigDecimal amount) {
    deductBalance(from, amount);
    sendEmailNotification(from);  // 网络调用，可能很慢！
    callExternalAPI();            // 外部API调用！
    addBalance(to, amount);
    generatePDFReceipt();         // CPU密集型操作！
    recordLog(from, to, amount);
}

3. 使用重试逻辑

事务可能因为瞬时错误（如写冲突）而失败，应实现重试机制：

@Service
public class ResilientTransferService {

    private final AccountService accountService;

    public ResilientTransferService(AccountService accountService) {
        this.accountService = accountService;
    }

    @Retryable(
        retryFor = { RuntimeException.class },
        maxAttempts = 3,
        backoff = @Backoff(delay = 100, multiplier = 2)
    )
    public void transferWithRetry(String from, String to, BigDecimal amount) {
        accountService.transfer(from, to, amount);
    }
}

需要在启动类上启用重试功能：

@SpringBootApplication
@EnableRetry
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

并添加 Maven 依赖：

<dependency>
    <groupId>org.springframework.retry</groupId>
    <artifactId>spring-retry</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-aspects</artifactId>
</dependency>

9.5.6 什么时候该用事务，什么时候应该通过设计规避

场景 建议
修改单个文档的多个字段 不需要事务，单文档操作天然原子
订单和订单项一起保存 不需要事务，将订单项内嵌到订单文档中
银行转账（A扣款B加款） 需要事务，涉及两个不同账户（文档）
删除文章同时删除评论 需要事务，涉及两个集合的操作
计数器递增 + 记录日志 视情况而定，如果日志不写也可接受，则不需要事务
注册用户时同时创建默认配置 需要事务，或者将默认配置内嵌到用户文档中
批量导入数据 不建议事务，使用 bulkWrite + 业务层补偿机制
修改文档并更新缓存 不能用事务，MongoDB 事务不能管理外部系统

总结：MongoDB 的事务是一个强大的工具，但不应该像使用关系型数据库那样频繁地依赖事务。在 MongoDB 的世界里，好的文档设计才是保证数据一致性的第一选择，事务是最后的保障手段。

本章小结

知识点 关键要点
事务发展历程 4.0 副本集事务 → 4.2 分片集群事务
单文档原子性 天然保证，利用 findOneAndUpdate 等操作
多文档事务 startSession → startTransaction → commit/abort
Spring Boot 集成 配置 MongoTransactionManager + @Transactional
副本集要求 事务必须在副本集环境，可用 Docker 搭建单节点副本集
最佳实践 优先设计规避 → 事务要短 → 加重试逻辑
