第13章 安全与运维基础

MongoDB 默认安装后没有开启认证，任何人都可以连接并操作数据库——这在生产环境中是极其危险的。 本章将系统讲解 MongoDB 的安全配置（用户认证、角色授权、连接安全）和运维基础（副本集高可用、 分片扩展、备份恢复），帮助你构建安全可靠的 MongoDB 部署方案。

13.1 用户认证与角色授权（RBAC）

MongoDB 使用 RBAC（Role-Based Access Control，基于角色的访问控制） 模型来管理权限。核心概念：

  • 用户（User）：关联到特定数据库的账号
  • 角色（Role）：一组权限的集合
  • 权限（Privilege）：对特定资源执行特定操作的能力

13.1.1 创建管理员用户

首先需要创建一个拥有最高权限的管理员账号。

// 连接到 MongoDB（未开启认证时）
// mongosh

// 切换到 admin 数据库
use admin;

// 创建超级管理员（root 角色）
db.createUser({
    user: "admin",
    pwd: "Admin@2024!Secure",   // 生产环境请使用强密码
    roles: [
        { role: "root", db: "admin" }
    ]
});

// 创建用户管理员（只能管理用户，不能操作业务数据）
db.createUser({
    user: "userAdmin",
    pwd: "UserAdmin@2024!",
    roles: [
        { role: "userAdminAnyDatabase", db: "admin" }
    ]
});

// 验证用户创建成功
db.getUsers();

开启认证后重启 MongoDB：

# 方式1：命令行参数
mongod --auth --dbpath /data/db --port 27017

# 方式2：配置文件 mongod.conf
# security:
#   authorization: enabled

# 方式3：Docker 启动时添加 --auth
docker run -d --name mongo \
  -p 27017:27017 \
  -v mongo-data:/data/db \
  mongo:7.0 --auth

使用认证连接：

# 命令行认证
mongosh -u admin -p "Admin@2024!Secure" --authenticationDatabase admin

# 或者连接后再认证
mongosh
use admin;
db.auth("admin", "Admin@2024!Secure");

13.1.2 创建普通数据库用户

// 以管理员身份连接
use admin;
db.auth("admin", "Admin@2024!Secure");

// 为 shop 数据库创建读写用户
use shop;
db.createUser({
    user: "shopApp",
    pwd: "ShopApp@2024!",
    roles: [
        { role: "readWrite", db: "shop" }
    ]
});

// 为 shop 数据库创建只读用户（用于报表查询）
db.createUser({
    user: "shopReader",
    pwd: "ShopReader@2024!",
    roles: [
        { role: "read", db: "shop" }
    ]
});

// 创建可以读写多个数据库的用户
use admin;
db.createUser({
    user: "multiDbUser",
    pwd: "MultiDb@2024!",
    roles: [
        { role: "readWrite", db: "shop" },
        { role: "readWrite", db: "cms" },
        { role: "read", db: "analytics" }
    ]
});

13.1.3 内置角色

MongoDB 提供了丰富的内置角色：

数据库级别角色：

角色 权限 说明
read 读取数据 可以执行 find、aggregate 等读操作
readWrite 读写数据 在 read 基础上增加 insert、update、delete 等写操作
dbAdmin 数据库管理 索引管理、统计信息、compact 等，但不能读写用户数据
dbOwner 数据库所有者 等同于 readWrite + dbAdmin + userAdmin
userAdmin 用户管理 创建/修改/删除用户和角色

全局角色（在 admin 数据库上授予）：

角色 权限 说明
readAnyDatabase 读取所有数据库 对所有数据库有 read 权限
readWriteAnyDatabase 读写所有数据库 对所有数据库有 readWrite 权限
dbAdminAnyDatabase 管理所有数据库 对所有数据库有 dbAdmin 权限
userAdminAnyDatabase 管理所有用户 可以管理任何数据库的用户
clusterAdmin 集群管理 副本集管理、分片管理、集群监控
clusterManager 集群管理（受限） 集群监控和管理，但不能删除数据
clusterMonitor 集群监控 只能查看集群状态，不能修改
root 超级管理员 拥有所有权限

特殊角色：

角色 说明
backup 可以使用 mongodump 备份数据
restore 可以使用 mongorestore 恢复数据

13.1.4 自定义角色

当内置角色不满足需求时，可以创建自定义角色。

use admin;

// 创建自定义角色：只能对 orders 集合进行读操作和对 products 集合进行读写
db.createRole({
    role: "orderReaderProductWriter",
    privileges: [
        {
            resource: { db: "shop", collection: "orders" },
            actions: ["find", "aggregate"]
        },
        {
            resource: { db: "shop", collection: "products" },
            actions: ["find", "insert", "update", "remove"]
        }
    ],
    roles: []   // 不继承其他角色
});

// 创建只能查看日志的角色
db.createRole({
    role: "auditLogReader",
    privileges: [
        {
            resource: { db: "shop", collection: "auditLogs" },
            actions: ["find"]
        }
    ],
    roles: []
});

// 创建继承自内置角色的自定义角色
db.createRole({
    role: "shopDeveloper",
    privileges: [
        // 额外允许创建索引
        {
            resource: { db: "shop", collection: "" },
            actions: ["createIndex", "dropIndex", "listIndexes"]
        }
    ],
    roles: [
        { role: "readWrite", db: "shop" }   // 继承 readWrite 权限
    ]
});

// 使用自定义角色创建用户
use shop;
db.createUser({
    user: "developer",
    pwd: "Dev@2024!",
    roles: [
        { role: "shopDeveloper", db: "admin" }
    ]
});

13.1.5 用户管理常用命令

// ===== 查看用户 =====
use shop;
db.getUsers();                        // 查看当前数据库的所有用户
db.getUser("shopApp");                // 查看指定用户

// ===== 修改用户密码 =====
db.changeUserPassword("shopApp", "NewPassword@2024!");

// ===== 授予角色 =====
db.grantRolesToUser("shopApp", [
    { role: "dbAdmin", db: "shop" }
]);

// ===== 回收角色 =====
db.revokeRolesFromUser("shopApp", [
    { role: "dbAdmin", db: "shop" }
]);

// ===== 删除用户 =====
db.dropUser("shopReader");

// ===== 查看当前连接的用户信息 =====
db.runCommand({ connectionStatus: 1 });

// ===== 查看角色的详细权限 =====
use admin;
db.getRole("readWrite", { showPrivileges: true });

// ===== 查看自定义角色 =====
db.getRoles({ showBuiltinRoles: false });

13.2 连接安全：认证机制与连接串配置

13.2.1 SCRAM 认证机制

MongoDB 默认使用 SCRAM（Salted Challenge Response Authentication Mechanism） 认证机制：

认证机制 说明 MongoDB 版本
SCRAM-SHA-1 基于 SHA-1 的 SCRAM 3.0+
SCRAM-SHA-256 基于 SHA-256 的 SCRAM（推荐） 4.0+
x.509 基于证书的认证 2.6+
LDAP 企业版，对接 LDAP/AD 企业版
Kerberos 企业版，对接 Kerberos 企业版

SCRAM 的认证流程：

客户端                                    服务端
  │                                        │
  │─── 1. 发送用户名 ───────────────────>  │
  │                                        │
  │<── 2. 返回盐值(salt)+迭代次数 ────────  │
  │                                        │
  │─── 3. 发送加密后的认证证明 ──────────>  │
  │                                        │
  │<── 4. 验证通过/失败 ─────────────────  │
  │                                        │

密码在传输过程中不会以明文形式出现，安全性较高。

13.2.2 连接字符串格式详解

MongoDB 的标准连接字符串格式：

mongodb://[username:password@]host1[:port1][,host2[:port2],...][/database][?options]

各部分说明：

部分 说明 示例
mongodb:// 协议前缀（必须） mongodb:// 或 mongodb+srv://
username:password 认证凭据（可选） admin:Admin%402024
host:port 主机和端口 localhost:27017
/database 默认数据库 /shop
?options 连接选项 ?authSource=admin&replicaSet=rs0

完整连接字符串示例：

# 最简单的连接（无认证）
mongodb://localhost:27017/mydb

# 带认证的连接
mongodb://shopApp:ShopApp%402024!@localhost:27017/shop?authSource=shop

# 连接副本集（多节点）
mongodb://shopApp:password@host1:27017,host2:27017,host3:27017/shop?replicaSet=rs0&authSource=admin

# 使用 SRV 记录连接（MongoDB Atlas 等云服务常用）
mongodb+srv://shopApp:password@cluster0.example.mongodb.net/shop?authSource=admin

注意：密码中的特殊字符需要进行 URL 编码。例如 @ 编码为 %40，# 编码为 %23。

13.2.3 常用连接选项

选项 默认值 说明
authSource 连接的数据库 存储用户凭据的数据库，通常是 admin
authMechanism SCRAM-SHA-256 认证机制
replicaSet 无 副本集名称
directConnection false 直接连接到指定节点（不做副本集发现）
readPreference primary 读偏好：primary / secondary / nearest 等
w 1 写关注级别：1 / majority / 0
wtimeoutMS 0 写关注超时时间
connectTimeoutMS 10000 连接超时（毫秒）
socketTimeoutMS 0 Socket 超时
maxPoolSize 100 连接池最大连接数
minPoolSize 0 连接池最小连接数
maxIdleTimeMS 0 连接最大空闲时间
retryWrites true 是否自动重试写操作
retryReads true 是否自动重试读操作
ssl / tls false 是否启用 TLS/SSL
compressors 无 压缩算法：snappy / zlib / zstd

生产环境推荐的连接字符串：

mongodb://shopApp:password@node1:27017,node2:27017,node3:27017/shop?authSource=admin&replicaSet=rs0&w=majority&retryWrites=true&readPreference=secondaryPreferred&maxPoolSize=50&connectTimeoutMS=5000&compressors=zstd

13.2.4 TLS/SSL 配置简述

生产环境中，建议启用 TLS 加密传输。

MongoDB 服务端启用 TLS：

# mongod.conf
net:
  tls:
    mode: requireTLS
    certificateKeyFile: /etc/ssl/mongodb.pem
    CAFile: /etc/ssl/ca.pem

连接字符串中启用 TLS：

mongodb://user:pass@host:27017/db?tls=true&tlsCAFile=/path/to/ca.pem&tlsCertificateKeyFile=/path/to/client.pem

通过 Java 配置自定义 MongoClient：

import com.mongodb.ConnectionString;
import com.mongodb.MongoClientSettings;
import com.mongodb.client.MongoClient;
import com.mongodb.client.MongoClients;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.mongodb.config.AbstractMongoClientConfiguration;

import java.util.concurrent.TimeUnit;

@Configuration
public class MongoConfig extends AbstractMongoClientConfiguration {

    @Override
    protected String getDatabaseName() {
        return "shop";
    }

    @Override
    @Bean
    public MongoClient mongoClient() {
        ConnectionString connStr = new ConnectionString(
                "mongodb://shopApp:ShopApp%402024!@localhost:27017/shop?authSource=admin"
        );

        MongoClientSettings settings = MongoClientSettings.builder()
                .applyConnectionString(connStr)
                .applyToConnectionPoolSettings(pool -> pool
                        .maxSize(50)
                        .minSize(5)
                        .maxConnectionIdleTime(30, TimeUnit.SECONDS)
                        .maxWaitTime(5, TimeUnit.SECONDS))
                .applyToSocketSettings(socket -> socket
                        .connectTimeout(5, TimeUnit.SECONDS)
                        .readTimeout(10, TimeUnit.SECONDS))
                .retryWrites(true)
                .retryReads(true)
                .build();

        return MongoClients.create(settings);
    }
}

13.3 副本集（Replica Set）概念与高可用

13.3.1 副本集的架构

副本集是 MongoDB 提供高可用性的核心机制。一个副本集包含多个 MongoDB 实例，数据在各节点之间自动同步。

                    ┌──────────────────────┐
                    │   Primary（主节点）    │ ← 接受所有写操作
                    │   读 + 写              │
                    └──────────┬───────────┘
                               │ oplog 同步
                    ┌──────────┴───────────┐
              ┌─────┴─────┐          ┌─────┴─────┐
              │ Secondary │          │ Secondary │ ← 数据副本，可分担读
              │  （从节点） │          │  （从节点） │
              └───────────┘          └───────────┘

三种节点类型：

节点类型 说明 是否存储数据 是否参与选举
Primary 主节点，接受所有写操作 ✅ ✅
Secondary 从节点，复制主节点数据 ✅ ✅
Arbiter 仲裁节点，只参与投票 ❌ ✅

常见的副本集拓扑：

拓扑 节点数 说明
PSS 1 Primary + 2 Secondary 最常用，3个数据副本
PSA 1 Primary + 1 Secondary + 1 Arbiter 节省资源，但只有2个数据副本
PSSSS 1 Primary + 4 Secondary 更高的读扩展能力

推荐：生产环境至少使用 3 个数据节点（PSS），避免使用仲裁节点，因为仲裁节点不存储数据， 当 Primary 故障时只剩一个数据副本，风险较高。

13.3.2 自动故障转移机制

当 Primary 节点故障时，副本集会自动进行选举，选出新的 Primary：

正常状态:
  Primary ←── Secondary1
          ←── Secondary2

Primary 故障:
  Primary ✗    Secondary1  ←→  Secondary2
                    │                │
                    └──── 选举 ──────┘
                           │
                           ▼
               Secondary1 成为新的 Primary

故障恢复:
  新Primary ←── Secondary2
            ←── 原Primary（恢复后变成 Secondary）

选举规则：

  • 需要多数节点（majority）同意才能选出新 Primary
  • 3 节点副本集：至少 2 个节点存活才能选举
  • 5 节点副本集：至少 3 个节点存活才能选举
  • 选举通常在 10-12 秒内完成
  • 选举期间无法写入，但配置了 readPreference 的读操作可以继续

13.3.3 读写分离：ReadPreference

通过 ReadPreference 可以将读操作分散到从节点，减轻主节点压力。

ReadPreference 说明 适用场景
primary 默认，所有读写都走主节点 对数据实时性要求高
primaryPreferred 优先读主节点，主不可用时读从节点 大部分读需要实时数据
secondary 只从从节点读取 报表查询、数据分析
secondaryPreferred 优先读从节点，从不可用时读主节点 最常用的读写分离策略
nearest 从网络延迟最低的节点读取 跨地域部署

注意：从 Secondary 读取的数据可能有短暂的延迟（通常毫秒级），因为 oplog 同步是异步的。 对数据一致性要求极高的操作（如支付后查询余额）应使用 primary。

13.3.5 初始化副本集

# 启动所有容器
docker-compose up -d

# 等待几秒，让 MongoDB 完全启动
sleep 5

# 连接到第一个节点
docker exec -it mongo-rs-1 mongosh

// 初始化副本集
rs.initiate({
    _id: "rs0",
    members: [
        { _id: 0, host: "mongo1:27017", priority: 2 },   // 优先级高，优先选为 Primary
        { _id: 1, host: "mongo2:27017", priority: 1 },
        { _id: 2, host: "mongo3:27017", priority: 1 }
    ]
});

// 等待选举完成
sleep(3000);

// 查看副本集状态
rs.status();

// 查看副本集配置
rs.conf();

// 查看简要信息
rs.isMaster();

rs.status() 关键字段说明：

{
    "set": "rs0",                    // 副本集名称
    "myState": 1,                    // 当前节点状态（1=PRIMARY, 2=SECONDARY）
    "members": [
        {
            "name": "mongo1:27017",
            "stateStr": "PRIMARY",   // 节点角色
            "health": 1,             // 健康状态（1=正常）
            "uptime": 3600,          // 运行时间（秒）
            "optime": { ... },       // 最后一次操作的时间戳
            "optimeDate": ISODate("..."),
            "syncSourceHost": "",    // 同步源（Primary 为空）
            "electionDate": ISODate("...")
        },
        {
            "name": "mongo2:27017",
            "stateStr": "SECONDARY",
            "health": 1,
            "syncSourceHost": "mongo1:27017",
            "secondsBehind": 0       // 同步延迟（秒）
        }
        // ...
    ]
}

      uri: >-
        mongodb://shopApp:ShopApp%402024!@
        localhost:27017,localhost:27018,localhost:27019
        /shop
        ?authSource=shop
        &replicaSet=rs0
        &readPreference=secondaryPreferred
        &w=majority
        &retryWrites=true

注意：如果使用 Docker 内部网络名称（如 mongo1:27017），需要确保 Spring Boot 应用 也在同一个 Docker 网络中，或者在宿主机的 hosts 文件中配置映射。 本地开发时建议使用 localhost:27017,localhost:27018,localhost:27019。

13.4 分片（Sharding）概念简介

当单个副本集无法满足存储容量或吞吐量需求时，可以通过分片将数据水平分布到多个副本集上。

13.4.1 分片架构

           客户端应用
               │
               ▼
        ┌──────────────┐
        │    mongos     │   ← 路由器，接受客户端请求
        │  （路由进程）   │     将请求转发到正确的分片
        └──────┬───────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐
│ Shard1 │ │ Shard2 │ │ Shard3 │   ← 数据分片，每个分片是一个副本集
│ (RS)   │ │ (RS)   │ │ (RS)   │
└────────┘ └────────┘ └────────┘

        ┌──────────────┐
        │ Config Server│   ← 配置服务器（也是副本集）
        │    (RS)      │     存储分片的元数据和路由规则
        └──────────────┘

三个核心组件：

组件 说明 部署建议
mongos 路由进程，客户端的访问入口 可部署多个（无状态），建议 2-3 个
Shard 数据分片，存储实际数据 每个分片是一个副本集
Config Server 配置服务器，存储路由表和分片元数据 必须是 3 节点副本集

13.4.2 分片键的选择原则

分片键（Shard Key）决定了数据如何在各分片之间分布，一旦设定无法更改（MongoDB 5.0+ 支持 refine），是分片设计中最关键的决策。

好的分片键应具备：

特性 说明 反例
高基数（Cardinality） 取值种类多，数据能均匀分布 ❌ status（只有几个值）
低频率（Frequency） 没有某个值特别多 ❌ country（某个国家占 80%）
非单调变化 不是递增/递减的 ❌ _id（ObjectId 递增）
查询友好 常用查询条件包含分片键 ❌ 几乎不作为查询条件的字段

分片键选择示例：

场景 好的分片键 原因
用户数据 { userId: "hashed" } userId 基数高，哈希后分布均匀
订单数据 { userId: 1, createdAt: 1 } 复合键，兼顾查询和分布
日志数据 { appName: 1, timestamp: 1 } 按应用分片，时间排序
IoT 数据 { deviceId: "hashed" } 设备数量大，哈希均匀分布

13.4.3 范围分片 vs 哈希分片

对比 范围分片（Range） 哈希分片（Hashed）
分布方式 按分片键的值范围划分 按分片键的哈希值划分
数据分布 可能不均匀（热点问题） 均匀分布
范围查询 ✅ 高效（数据连续存储） ❌ 低效（数据分散在多个分片）
写入分布 单调递增键会导致写入集中 写入均匀分散
适用场景 经常按范围查询的数据 需要均匀写入分布的数据 
// 范围分片
sh.shardCollection("shop.orders", { userId: 1, createdAt: 1 });

// 哈希分片
sh.shardCollection("shop.logs", { deviceId: "hashed" });

13.4.4 何时需要分片

指标 参考阈值 说明
数据量 > 200GB ~ 500GB 单个副本集的数据量接近磁盘容量
文档数量 > 10 亿条 索引变大，查询性能下降
写入吞吐量 > 10,000 次/秒 单个 Primary 的写能力不足
读取吞吐量 > 50,000 次/秒 即使读写分离也无法满足
工作集大小 超过可用内存 频繁的磁盘 IO 导致性能急剧下降

决策建议：

  1. 不要过早分片——分片会大幅增加架构复杂度和运维成本
  2. 先尝试垂直扩展（升级硬件：更多内存、更快的 SSD）
  3. 先尝试优化索引和查询
  4. 先尝试读写分离（副本集 + ReadPreference）
  5. 以上方案都无法满足时，再考虑分片

13.5 备份与恢复：mongodump / mongorestore

13.5.1 mongodump 基本用法

mongodump 用于将 MongoDB 数据导出为 BSON 格式的备份文件。

常用参数：

参数 说明 示例
--uri 连接字符串 --uri="mongodb://user:pass@host:27017"
--host / -h 主机地址 --host=localhost:27017
--db / -d 指定数据库 --db=shop
--collection / -c 指定集合 --collection=orders
--out / -o 输出目录 --out=/backup/2024-01-15
--gzip 压缩输出 --gzip
--archive 输出为单个归档文件 --archive=/backup/shop.archive
--query / -q 按条件导出 --query='{"status":"COMPLETED"}'
--oplog 备份 oplog（支持时间点恢复） --oplog
--authenticationDatabase 认证数据库 --authenticationDatabase=admin
--numParallelCollections 并行导出集合数 --numParallelCollections=4

示例：

# 备份所有数据库
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --out=/backup/full-$(date +%Y%m%d)

# 备份单个数据库
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --db=shop \
          --out=/backup/shop-$(date +%Y%m%d)

# 备份单个集合
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --db=shop \
          --collection=orders \
          --out=/backup/orders-$(date +%Y%m%d)

# 压缩备份（节省磁盘空间）
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --db=shop \
          --gzip \
          --out=/backup/shop-gz-$(date +%Y%m%d)

# 输出为单个归档文件
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --db=shop \
          --archive=/backup/shop-$(date +%Y%m%d).archive \
          --gzip

# 按条件备份（只导出已完成的订单）
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --db=shop \
          --collection=orders \
          --query='{"status":"COMPLETED","createdAt":{"$gte":{"$date":"2024-01-01T00:00:00Z"}}}' \
          --out=/backup/completed-orders

# 备份副本集（包含 oplog，支持时间点恢复）
mongodump --uri="mongodb://admin:password@localhost:27017" \
          --authenticationDatabase=admin \
          --oplog \
          --out=/backup/full-with-oplog-$(date +%Y%m%d)

备份文件结构：

/backup/shop-20240115/
├── shop/
│   ├── orders.bson          # orders 集合的数据
│   ├── orders.metadata.json # orders 集合的元数据（索引等）
│   ├── products.bson
│   ├── products.metadata.json
│   ├── users.bson
│   └── users.metadata.json
└── oplog.bson               # oplog（使用 --oplog 时）

13.5.2 mongorestore 基本用法

mongorestore 用于从 mongodump 的备份文件中恢复数据。

常用参数：

参数 说明 示例
--uri 连接字符串 --uri="mongodb://user:pass@host:27017"
--db / -d 恢复到指定数据库 --db=shop
--collection / -c 恢复指定集合 --collection=orders
--drop 恢复前先删除已有集合 --drop
--gzip 从压缩备份恢复 --gzip
--archive 从归档文件恢复 --archive=/backup/shop.archive
--nsInclude 指定命名空间 --nsInclude="shop.orders"
--nsExclude 排除命名空间 --nsExclude="shop.logs"
--nsFrom / --nsTo 重命名命名空间 恢复到不同库/集合
--oplogReplay 回放 oplog --oplogReplay
--numParallelCollections 并行恢复集合数 --numParallelCollections=4
--noIndexRestore 不恢复索引 --noIndexRestore

示例：

# 恢复整个备份
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             /backup/full-20240115

# 恢复单个数据库
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --db=shop \
             /backup/shop-20240115/shop

# 恢复单个集合
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --db=shop \
             --collection=orders \
             /backup/shop-20240115/shop/orders.bson

# 恢复前删除已有数据（谨慎使用！）
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --drop \
             /backup/full-20240115

# 从压缩备份恢复
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --gzip \
             /backup/shop-gz-20240115

# 从归档文件恢复
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --archive=/backup/shop-20240115.archive \
             --gzip

# 恢复到不同的数据库（重命名）
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --nsFrom="shop.*" \
             --nsTo="shop_backup.*" \
             /backup/shop-20240115

# 恢复时回放 oplog（时间点恢复）
mongorestore --uri="mongodb://admin:password@localhost:27017" \
             --authenticationDatabase=admin \
             --oplogReplay \
             /backup/full-with-oplog-20240115

13.5.3 定时备份脚本

Linux/Mac 定时备份脚本（backup-mongodb.sh）：

#!/bin/bash

# MongoDB 备份脚本
# 用法: ./backup-mongodb.sh
# 建议通过 crontab 定时执行

# ========== 配置区 ==========
MONGO_URI="mongodb://admin:password@localhost:27017"
AUTH_DB="admin"
BACKUP_DIR="/data/backup/mongodb"
RETAIN_DAYS=7                    # 保留最近7天的备份
DATE=$(date +%Y%m%d_%H%M%S)
LOG_FILE="${BACKUP_DIR}/backup.log"
# ============================

# 创建备份目录
BACKUP_PATH="${BACKUP_DIR}/${DATE}"
mkdir -p "${BACKUP_PATH}"

echo "[$(date)] 开始备份..." >> "${LOG_FILE}"

# 执行备份（压缩 + oplog）
mongodump \
    --uri="${MONGO_URI}" \
    --authenticationDatabase="${AUTH_DB}" \
    --gzip \
    --oplog \
    --out="${BACKUP_PATH}" \
    2>> "${LOG_FILE}"

# 检查备份结果
if [ $? -eq 0 ]; then
    # 计算备份大小
    BACKUP_SIZE=$(du -sh "${BACKUP_PATH}" | awk '{print $1}')
    echo "[$(date)] 备份成功: ${BACKUP_PATH} (${BACKUP_SIZE})" >> "${LOG_FILE}"
else
    echo "[$(date)] 备份失败!" >> "${LOG_FILE}"
    # 可以在这里发送告警通知
    exit 1
fi

# 清理过期备份
echo "[$(date)] 清理 ${RETAIN_DAYS} 天前的备份..." >> "${LOG_FILE}"
find "${BACKUP_DIR}" -maxdepth 1 -type d -name "20*" -mtime +${RETAIN_DAYS} -exec rm -rf {} \;

echo "[$(date)] 备份任务完成" >> "${LOG_FILE}"
echo "---" >> "${LOG_FILE}"

设置定时任务（crontab）：

# 编辑 crontab
crontab -e

# 每天凌晨 2 点执行备份
0 2 * * * /opt/scripts/backup-mongodb.sh

# 每 6 小时备份一次
0 */6 * * * /opt/scripts/backup-mongodb.sh

Windows 定时备份脚本（backup-mongodb.bat）：

@echo off
setlocal

set MONGO_URI=mongodb://admin:password@localhost:27017
set AUTH_DB=admin
set BACKUP_DIR=D:\backup\mongodb
set DATE=%date:~0,4%%date:~5,2%%date:~8,2%_%time:~0,2%%time:~3,2%%time:~6,2%
set DATE=%DATE: =0%

set BACKUP_PATH=%BACKUP_DIR%\%DATE%

echo [%date% %time%] 开始备份...

mongodump --uri="%MONGO_URI%" --authenticationDatabase=%AUTH_DB% --gzip --out="%BACKUP_PATH%"

if %ERRORLEVEL% EQU 0 (
    echo [%date% %time%] 备份成功: %BACKUP_PATH%
) else (
    echo [%date% %time%] 备份失败!
)

REM 清理7天前的备份
forfiles /p "%BACKUP_DIR%" /d -7 /c "cmd /c if @isdir==TRUE rd /s /q @path" 2>nul

endlocal

13.5.4 mongoexport / mongoimport（JSON/CSV 导入导出）

mongoexport 和 mongoimport 用于将数据导出为人类可读的格式（JSON/CSV），适合数据交换和分析。

与 mongodump/mongorestore 的区别：

  • mongodump/mongorestore 使用 BSON 格式，保留所有数据类型，适合备份恢复
  • mongoexport/mongoimport 使用 JSON/CSV 格式，可能丢失部分类型信息，适合数据交换

mongoexport 示例：

# 导出为 JSON（默认格式）
mongoexport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --out=products.json

# 导出为 CSV
mongoexport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --type=csv \
            --fields=name,price,stock,category \
            --out=products.csv

# 按条件导出
mongoexport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=orders \
            --query='{"status":"COMPLETED"}' \
            --out=completed-orders.json

# 导出指定字段
mongoexport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=users \
            --fields=name,email,createdAt \
            --out=users-basic.json

# 导出为 JSON 数组格式（而非每行一个 JSON）
mongoexport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --jsonArray \
            --out=products-array.json

mongoimport 示例：

# 从 JSON 文件导入
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --file=products.json

# 从 CSV 文件导入
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --type=csv \
            --headerline \
            --file=products.csv

# 从 CSV 导入（手动指定列名）
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --type=csv \
            --fields=name,price,stock,category \
            --file=products-noheader.csv

# 导入 JSON 数组格式
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --jsonArray \
            --file=products-array.json

# 导入时更新已有文档（upsert 模式）
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --mode=upsert \
            --upsertFields=name \
            --file=products-update.json

# 导入前删除已有数据
mongoimport --uri="mongodb://admin:password@localhost:27017" \
            --authenticationDatabase=admin \
            --db=shop \
            --collection=products \
            --drop \
            --file=products.json

mongoimport 的导入模式：

模式 说明
insert（默认） 只插入新文档，遇到 _id 冲突会报错
upsert 匹配到则更新，未匹配到则插入
merge 匹配到则合并字段，未匹配到则插入
delete 匹配到则删除

实用场景：从 MySQL 迁移数据到 MongoDB

# 1. 从 MySQL 导出 CSV
mysql -u root -p -e "SELECT id, name, email, created_at FROM users" shop > users.csv

# 2. 将 CSV 导入 MongoDB
mongoimport --uri="mongodb://localhost:27017" \
            --db=shop \
            --collection=users \
            --type=csv \
            --headerline \
            --file=users.csv

知识点 要点
RBAC 权限模型 用户 → 角色 → 权限，先创建 admin 用户再开启认证
内置角色 read / readWrite / dbAdmin / userAdmin / root 等
连接字符串 mongodb://user:pass@host:port/db?options

副本集架构 Primary + Secondary，自动故障转移
ReadPreference primary / secondaryPreferred / nearest 等
分片架构 mongos + Shard + Config Server

mongodump BSON 格式备份，支持 --gzip、--oplog
mongorestore 从 BSON 备份恢复，支持 --drop、--oplogReplay
mongoexport 导出为 JSON/CSV，适合数据交换
mongoimport 从 JSON/CSV 导入，支持 upsert/merge 模式
