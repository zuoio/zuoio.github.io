第08章 文档模型设计

文档模型设计是 MongoDB 开发中最关键的环节。不同于关系型数据库遵循范式化设计，MongoDB 的核心理念是**"以查询需求驱动数据结构"**。好的文档模型可以大幅提升读取性能和开发效率，而糟糕的设计则会让系统陷入性能泥潭。本章将系统讲解文档模型设计的原则、策略和常见模式。

8.1 内嵌文档 vs 引用文档 —— 设计原则

MongoDB 中有两种主要的数据组织方式：内嵌文档（Embedding） 和引用文档（Referencing）。选择哪种方式直接影响到应用的读写性能和数据维护复杂度。

8.1.1 内嵌文档（Embedding）

内嵌文档是将相关数据直接嵌入到父文档中，形成一个自包含的文档结构。

JSON 示例：将用户地址内嵌到用户文档

{
  "_id": "U001",
  "name": "张三",
  "email": "zhangsan@example.com",
  "address": {
    "province": "北京市",
    "city": "北京",
    "district": "朝阳区",
    "street": "建国路88号",
    "zipCode": "100022"
  },
  "orders": [
    {
      "orderNo": "ORD001",
      "totalAmount": 599,
      "status": "completed",
      "orderDate": "2024-03-15T10:30:00Z"
    },
    {
      "orderNo": "ORD002",
      "totalAmount": 1299,
      "status": "shipped",
      "orderDate": "2024-04-01T14:20:00Z"
    }
  ]
}

优点：

优势 说明
读取性能好 一次查询即可获取所有相关数据，无需关联查询
原子更新 对单个文档的操作是原子性的，无需事务
数据局部性 相关数据存储在一起，减少磁盘 I/O
查询简单 不需要写 JOIN（$lookup），代码更简洁

缺点：

缺点 说明
文档大小限制 单个文档最大 16MB，内嵌数据过多时可能超限
数据冗余 相同数据可能在多个文档中重复存储
更新复杂 被嵌入的数据更新时，所有包含该数据的文档都需要更新
数组增长问题 内嵌数组不断增长会导致文档频繁迁移，影响性能

8.1.2 引用文档（Referencing）

引用文档通过存储另一个文档的 _id 来建立关联，类似于关系型数据库的外键。

JSON 示例：用户和订单分别存储，通过 userId 关联

用户文档（users 集合）：

{
  "_id": "U001",
  "name": "张三",
  "email": "zhangsan@example.com",
  "address": {
    "province": "北京市",
    "city": "北京",
    "district": "朝阳区"
  }
}

订单文档（orders 集合）：

{
  "_id": "ORD001",
  "userId": "U001",
  "totalAmount": 599,
  "status": "completed",
  "orderDate": "2024-03-15T10:30:00Z",
  "items": [
    { "productId": "P001", "quantity": 1, "price": 399 },
    { "productId": "P002", "quantity": 2, "price": 100 }
  ]
}

优点：

优势 说明
数据规范化 没有冗余数据，更新一处即可
文档精简 每个文档独立且大小可控
灵活性好 可以独立查询和更新各个集合
适合大量数据 不受 16MB 文档大小限制

缺点：

缺点 说明
需要多次查询 获取完整数据需要 $lookup 或多次查询
没有外键约束 MongoDB 不强制引用完整性，需要应用层保证
读取性能低 关联查询比内嵌文档慢
无法原子更新 跨集合操作需要事务支持

8.1.3 选择原则：访问模式决定设计

核心原则：根据应用的读写模式来决定数据结构，而不是根据数据之间的逻辑关系。

选择内嵌的场景：

  • 数据之间存在"包含"关系（整体与部分）
  • 子数据总是随父数据一起被读取
  • 子数据数量有限且不会无限增长
  • 子数据不需要被独立查询
  • 读多写少

选择引用的场景：

  • 子数据会被独立查询和使用
  • 子数据数量会持续增长（如评论、日志）
  • 子数据被多个父文档共享
  • 数据更新频繁且需要保持一致性
  • 文档大小可能超过 16MB

8.1.4 决策流程

以下是选择内嵌还是引用的决策参考路径：

开始：数据 A 和数据 B 之间有关联关系
  │
  ├── 问题1：B 是否总是和 A 一起被读取？
  │   ├── 是 → 倾向内嵌
  │   └── 否 → 问题2
  │
  ├── 问题2：B 的数量是否会持续增长且无上限？
  │   ├── 是 → 使用引用
  │   └── 否 → 问题3
  │
  ├── 问题3：B 是否需要被独立查询？
  │   ├── 是 → 使用引用
  │   └── 否 → 问题4
  │
  ├── 问题4：B 是否被多个 A 共享？
  │   ├── 是 → 使用引用
  │   └── 否 → 问题5
  │
  └── 问题5：A + B 的文档大小是否可能超过 16MB？
      ├── 是 → 使用引用
      └── 否 → 使用内嵌

8.2 一对一、一对多、多对多的建模策略

8.2.1 一对一（One-to-One）

一对一关系通常优先使用内嵌，除非子文档特别大或需要独立访问。

8.2.3 一对多 —— 大量（One-to-Many）

当"多"的一方数量很大或会持续增长时，使用引用。

场景：作者（Author）与文章（可能有上千篇）

作者文档（authors 集合）：

{
  "_id": "A001",
  "name": "张三",
  "email": "zhangsan@example.com",
  "bio": "技术博主，专注Java和MongoDB",
  "followerCount": 5600,
  "articleCount": 128
}

文章文档（articles 集合）：

{
  "_id": "ART001",
  "title": "MongoDB 聚合管道深入解析",
  "content": "文章正文...",
  "authorId": "A001",
  "authorName": "张三",
  "category": "数据库",

查询示例：

// 查询某作者的所有文章
public List<Article> findArticlesByAuthor(String authorId) {
    Query query = new Query(Criteria.where("authorId").is(authorId));
    query.with(Sort.by(Sort.Direction.DESC, "createdAt"));
    return mongoTemplate.find(query, Article.class);
}

// 使用 $lookup 关联查询作者详情
public List<Map> getArticlesWithAuthorDetail() {
    Aggregation aggregation = Aggregation.newAggregation(
        Aggregation.lookup("authors", "authorId", "_id", "authorInfo"),
        Aggregation.unwind("authorInfo"),
        Aggregation.project("title", "viewCount", "createdAt")
            .and("authorInfo.name").as("authorName")
            .and("authorInfo.bio").as("authorBio")
    );
    return mongoTemplate.aggregate(aggregation, "articles", Map.class).getMappedResults();
}

package com.example.mongo.service;

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

@Service
public class EnrollmentService {

    private final MongoTemplate mongoTemplate;

    public EnrollmentService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 学生选课：双向更新
     */
    public void enrollCourse(String studentId, String courseId) {
        // 将课程ID添加到学生的 courseIds 数组
        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(studentId)),
            new Update().addToSet("courseIds", courseId).inc("courseCount", 1),
            "students"
        );

        // 将学生ID添加到课程的 studentIds 数组
        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(courseId)),
            new Update().addToSet("studentIds", studentId).inc("studentCount", 1),
            "courses"
        );
    }

    /**
     * 学生退选：双向移除
     */
    public void dropCourse(String studentId, String courseId) {
        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(studentId)),
            new Update().pull("courseIds", courseId).inc("courseCount", -1),
            "students"
        );

        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(courseId)),
            new Update().pull("studentIds", studentId).inc("studentCount", -1),
            "courses"
        );
    }
}

注意：当多对多关系的引用数组可能增长到非常大（如数万条）时，应该考虑使用中间集合（类似关系数据库的中间表）来代替引用数组。

8.3 常见设计模式

8.3.1 Bucket 模式（分桶模式）

适用场景：时间序列数据、IoT 传感器数据、日志数据等。

核心思想：将一段时间内的多条记录聚合到同一个文档中，减少文档数量，提升查询效率。

场景：IoT 温度传感器数据

不用 Bucket 模式（每条记录一个文档）：

{ "sensorId": "SENSOR_001", "temperature": 22.5, "humidity": 65, "timestamp": "2024-03-15T10:00:00Z" }
{ "sensorId": "SENSOR_001", "temperature": 22.8, "humidity": 64, "timestamp": "2024-03-15T10:01:00Z" }
{ "sensorId": "SENSOR_001", "temperature": 23.1, "humidity": 63, "timestamp": "2024-03-15T10:02:00Z" }
// ... 每分钟一条，一天1440条，一个月43200条

这种方式会产生海量文档，每个文档还需要存储重复的 sensorId 等元数据。

使用 Bucket 模式（按小时分桶）：

{
  "_id": ObjectId("..."),
  "sensorId": "SENSOR_001",
  "date": "2024-03-15",
  "hour": 10,
  "count": 60,
  "sumTemperature": 1380.5,
  "sumHumidity": 3840,
  "minTemperature": 22.1,
  "maxTemperature": 24.3,
  "readings": [
    { "minute": 0, "temperature": 22.5, "humidity": 65 },
    { "minute": 1, "temperature": 22.8, "humidity": 64 },
    { "minute": 2, "temperature": 23.1, "humidity": 63 },
    // ... 最多60条（一小时内每分钟一条）
  ]
}

对应的 Java 实体类：

package com.example.mongo.entity;

import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.index.CompoundIndex;
import java.util.List;

@Document(collection = "sensor_readings")
@CompoundIndex(name = "sensor_date_hour_idx", def = "{'sensorId': 1, 'date': 1, 'hour': 1}", unique = true)
public class SensorBucket {

    @Id
    private String id;
    private String sensorId;
    private String date;
    private int hour;
    private int count;
    private double sumTemperature;
    private double sumHumidity;
    private double minTemperature;
    private double maxTemperature;
    private List<Reading> readings;

    // getter / setter 省略
}

public class Reading {
    private int minute;
    private double temperature;
    private double humidity;

    // getter / setter 省略
}

写入数据的逻辑：

package com.example.mongo.service;

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

@Service
public class SensorDataService {

    private final MongoTemplate mongoTemplate;

    public SensorDataService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    public void recordReading(String sensorId, double temperature, double humidity) {
        LocalDateTime now = LocalDateTime.now();
        String date = now.format(DateTimeFormatter.ofPattern("yyyy-MM-dd"));
        int hour = now.getHour();
        int minute = now.getMinute();

        Query query = new Query(
            Criteria.where("sensorId").is(sensorId)
                .and("date").is(date)
                .and("hour").is(hour)
        );

        Update update = new Update()
            .inc("count", 1)
            .inc("sumTemperature", temperature)
            .inc("sumHumidity", humidity)
            .min("minTemperature", temperature)
            .max("maxTemperature", temperature)
            .push("readings", new Reading(minute, temperature, humidity));

        mongoTemplate.upsert(query, update, "sensor_readings");
    }
}

Bucket 模式的优势：

对比项 不分桶（每条记录） 按小时分桶
每日文档数 1440 24
每月文档数 ~43,200 ~720
索引条目 43,200 720
重复元数据 每条都重复 每小时存一次
查询"10点的数据" 扫描60个文档 读取1个文档

8.3.3 Attribute 模式（属性模式）

适用场景：不同文档拥有不同数量和名称的动态属性。

核心思想：将不确定数量的键值对属性统一存储为数组，方便建立索引和查询。

场景：商品规格参数（不同商品的规格项不同）

反面示例 —— 直接用顶层字段：

// 笔记本
{ "name": "ThinkPad X1", "screen": "14英寸", "cpu": "i7-1365U", "ram": "16GB", "weight": "1.12kg" }

package com.example.mongo.service;

import com.example.mongo.entity.Comment;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

@Service
public class CommentService {

    private final MongoTemplate mongoTemplate;

    public CommentService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    public Comment addComment(String articleId, String userId, String userName, String content) {
        // 1. 插入评论文档
        Comment comment = new Comment();
        comment.setArticleId(articleId);
        comment.setUserId(userId);
        comment.setUserName(userName);
        comment.setContent(content);
        comment.setCreatedAt(java.time.LocalDateTime.now());
        mongoTemplate.insert(comment);

        // 2. 更新文章的 commentCount（预计算字段）
        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(articleId)),
            new Update().inc("stats.commentCount", 1),
            "articles"
        );

        return comment;
    }

    public void deleteComment(String commentId, String articleId) {
        mongoTemplate.remove(
            new Query(Criteria.where("_id").is(commentId)),
            "comments"
        );

        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(articleId)),
            new Update().inc("stats.commentCount", -1),
            "articles"
        );
    }
}

优势：查询文章列表时，commentCount 直接可用，无需每次都对 comments 集合执行 count 聚合。

package com.example.mongo.service;

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

@Service
public class ReviewService {

    private static final int RECENT_REVIEW_LIMIT = 3;
    private final MongoTemplate mongoTemplate;

    public ReviewService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    public void addReview(String productId, String userId, String userName,
                          int rating, String content) {
        // 1. 插入完整评价到 reviews 集合
        Review review = new Review(productId, userId, userName, rating, content);
        mongoTemplate.insert(review);

        // 2. 更新商品的 recentReviews 子集（保留最近3条）
        RecentReview recentReview = new RecentReview(userId, userName, rating, content);

        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(productId)),
            new Update()

                        org.springframework.data.domain.Sort.Direction.DESC, "createdAt"))

                    .value(recentReview)
                .inc("reviewCount", 1),

8.4 反范式化设计的取舍与冗余数据维护

8.4.1 什么时候反范式化

MongoDB 鼓励反范式化设计（冗余数据），但这不意味着所有数据都应该冗余。

适合反范式化的场景：

场景特征 示例
读多写少 文章作者名、商品分类名
关联数据变化极少 国家名、省份名
读性能要求高 首页推荐列表、搜索结果
数据变更不频繁 用户昵称（偶尔修改）

不适合反范式化的场景：

场景特征 示例
写多读少 实时库存数量
关联数据频繁变更 实时价格、实时评分
数据一致性要求极高 金融交易金额
冗余范围非常广 一处修改需要更新数万文档

8.4.2 冗余数据的同步策略

当进行反范式化设计后，冗余数据的同步是一个重要问题。以下是几种常见的同步策略：

策略一：应用层双写

在更新源数据时，同时更新所有冗余副本。

package com.example.mongo.service;

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;

@Service
public class AuthorService {

    private final MongoTemplate mongoTemplate;

    public AuthorService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 修改作者姓名 —— 需要同步更新所有冗余位置
     */
    public void updateAuthorName(String authorId, String newName) {
        // 1. 更新作者集合中的姓名
        mongoTemplate.updateFirst(
            new Query(Criteria.where("_id").is(authorId)),
            new Update().set("name", newName),
            "authors"
        );

        // 2. 同步更新所有文章中的冗余作者名
        mongoTemplate.updateMulti(
            new Query(Criteria.where("authorId").is(authorId)),
            new Update().set("authorName", newName),
            "articles"
        );

        // 3. 同步更新评论中的冗余作者名（如果有）
        mongoTemplate.updateMulti(
            new Query(Criteria.where("userId").is(authorId)),
            new Update().set("userName", newName),
            "comments"
        );
    }
}

优点：实现简单，一致性有保证（在同一请求中完成）。

缺点：如果冗余位置很多，更新操作会很慢。如果某个更新失败，数据不一致（可以使用事务解决）。

策略二：异步消息同步

通过消息队列异步处理冗余数据的更新：

// 伪代码：修改姓名时发送消息
public void updateAuthorName(String authorId, String newName) {
    // 1. 更新主数据
    mongoTemplate.updateFirst(
        new Query(Criteria.where("_id").is(authorId)),
        new Update().set("name", newName),
        "authors"
    );

    // 2. 发送异步消息，由消费者负责更新冗余数据
    messageQueue.send("author-name-changed", Map.of(
        "authorId", authorId,
        "newName", newName
    ));
}

优点：主操作快速完成，不阻塞用户请求。

缺点：存在短暂的数据不一致窗口（最终一致性）。

策略三：Change Streams 自动同步

利用 MongoDB 的 Change Streams 监听源数据的变更，自动更新冗余数据。这是最优雅的方案，将在第11章详细讲解。

// 简要示例：监听 authors 集合的更新事件
@Component
public class AuthorChangeListener {

    private final MongoTemplate mongoTemplate;

    public AuthorChangeListener(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    // 监听到 authors 集合的 update 事件后，自动更新文章中的冗余作者名
    public void onAuthorChanged(ChangeStreamDocument<Document> change) {
        if (change.getOperationType() == OperationType.UPDATE) {
            Document updateDesc = change.getUpdateDescription().getUpdatedFields();
            if (updateDesc.containsKey("name")) {
                String authorId = change.getDocumentKey().getString("_id").getValue();
                String newName = updateDesc.getString("name");

                mongoTemplate.updateMulti(
                    new Query(Criteria.where("authorId").is(authorId)),
                    new Update().set("authorName", newName),
                    "articles"
                );
            }
        }
    }
}

8.4.3 各策略对比

策略 一致性 实现复杂度 性能影响 适用场景
应用层双写 强一致（可配合事务） 低 同步更新慢 冗余位置少，更新频率低
异步消息 最终一致 中 主操作快 冗余位置多，容忍短暂不一致
Change Streams 最终一致 中高 主操作不受影响 需要解耦，多消费者场景

8.5 从关系型数据库迁移到 MongoDB 的思维转换

8.5.1 关键思维转换

不要直接把表映射为集合。 这是从关系型数据库迁移到 MongoDB 最常见的错误。

关系型思维 MongoDB 思维
先设计表结构，再写查询 先分析查询需求，再设计文档结构
数据规范化，消除冗余 适度冗余，为读取优化
用 JOIN 组合多表数据 将常一起读取的数据内嵌在同一文档
通用的表结构 为具体的访问模式定制文档结构
外键约束保证完整性 应用层保证数据完整性
固定的列结构 灵活的文档结构（Schema-less）

8.5.2 以查询需求驱动设计

设计文档结构的正确流程：

1. 列出所有的查询场景（UI 页面 / API 接口）
2. 分析每个查询需要哪些数据
3. 确定数据之间的关系和访问模式
4. 决定内嵌还是引用
5. 考虑写入模式和更新频率
6. 设计索引策略

设计决策：

  • 用户基本信息（ID、用户名、手机号）冗余内嵌：订单列表页需要显示。
  • 收货地址快照内嵌：下单时的地址不应随用户修改地址而改变。
  • 商品信息（名称、图片、价格）快照内嵌：订单中的价格是下单时的价格，不随商品调价变化。
  • 物流信息内嵌：总是和订单一起读取。

转换前后对比

对比维度 关系型数据库（转换前） MongoDB（转换后）
表 / 集合数 6 张表 3 个集合
查询订单详情 JOIN 5 张表 单次查询 1 个文档
查询性能 多表 JOIN，性能随数据量下降 单文档读取，稳定高效
数据冗余 无 有（用户名、商品名、分类名等）
修改用户名 更新 1 条记录 更新用户集合 + 异步更新冗余
修改商品价格 更新 1 条记录 只更新商品集合（历史订单保留旧价格）
收货地址变更 历史订单被影响（地址引用） 历史订单不受影响（地址快照）
数据一致性 外键约束保证 应用层保证
扩展灵活性 加字段需要 ALTER TABLE 直接写入即可

迁移最佳实践总结

  1. 从查询场景出发，而不是从数据关系出发。列出你的 API 接口，分析每个接口需要的数据。

  2. 拥抱冗余，但要有策略。对于读多写少的数据大胆冗余，对于频繁变更的数据谨慎冗余。

  3. 使用快照保存关键时刻的数据状态。订单中的商品价格、收货地址应该是下单时的快照。

  4. 逐步迁移，先从读密集的模块开始，用 MongoDB 优化读取性能。

  5. 建立冗余数据的维护机制，不管是双写、消息队列还是 Change Streams。

  6. 不要过度内嵌。如果内嵌导致文档过大或更新过于复杂，就应该考虑拆分为独立集合。

本章小结

知识点 核心内容
内嵌 vs 引用 内嵌适合读取密集、数据量小的场景；引用适合数据量大、需要独立查询的场景
一对一 通常内嵌，除非子文档特别大
一对少 内嵌数组，适合数量有限（< 100）的子数据
一对多 使用引用，在引用处适度冗余常用字段
多对多 双向引用数组，数量极大时使用中间集合
Bucket 模式 时间序列数据分桶存储，减少文档数量和索引开销
Polymorphic 模式 用 type 字段区分类型，同一集合存储不同结构的文档
Attribute 模式 动态属性使用键值对数组存储，便于索引和查询
Computed 模式 预计算并冗余存储统计数据（计数、汇总），提升读取性能
Subset 模式 主文档中只保留热数据子集，完整数据存独立集合
反范式化取舍 读多写少适合冗余，冗余数据需要同步策略（双写 / 消息 / Change Streams）
迁移思维 不要直接映射表为集合，以查询需求驱动设计，拥抱冗余但有策略
