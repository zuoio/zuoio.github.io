第01章 初识 MongoDB

本章将带你从零认识 MongoDB —— 一个面向文档的 NoSQL 数据库。我们会将它与你熟悉的关系型数据库（如 MySQL）进行对比，帮助你快速建立认知映射，并了解 MongoDB 的核心概念、数据格式和适用场景。

1.1 什么是 MongoDB —— 与关系型数据库的对比

1.1.1 MongoDB 简介

MongoDB 是一个开源的、面向文档的 NoSQL 数据库，由 MongoDB Inc. 开发和维护。它使用类似 JSON 的 BSON（Binary JSON） 格式来存储数据，天然支持灵活的文档模型，无需预先定义表结构（Schema-free）。

MongoDB 的名字来源于英文单词 "humongous"（巨大的），寓意它能够处理海量数据。自 2009 年首次发布以来，MongoDB 已经成为全球最受欢迎的 NoSQL 数据库之一。

核心特点：

  • 文档模型：数据以 JSON/BSON 文档形式存储，结构灵活
  • 无固定 Schema：同一集合中的文档可以拥有不同的字段
  • 水平扩展：原生支持分片（Sharding），轻松应对数据增长
  • 高性能：内置索引、内存映射、聚合管道等高性能特性
  • 丰富的查询语言：支持丰富的查询操作符和聚合框架
  • 高可用：内置副本集（Replica Set）机制，自动故障转移

1.1.2 MongoDB vs MySQL 核心对比

作为 Java 开发者，你很可能对 MySQL 非常熟悉。下表从多个维度对比 MongoDB 和 MySQL，帮助你快速理解两者的差异：

对比维度 MySQL MongoDB
数据库类型 关系型数据库（RDBMS） 文档型 NoSQL 数据库
数据模型 行和列（表格） JSON/BSON 文档
Schema 固定 Schema，需要预定义表结构 灵活 Schema，文档结构可随时变化
存储单位 数据库 → 表 → 行 → 列 数据库 → 集合 → 文档 → 字段
主键 自增 ID 或自定义主键 自动生成 _id（ObjectId）
关联关系 外键 + JOIN 操作 嵌入文档或引用（手动关联）
查询语言 SQL MongoDB Query Language（MQL）
事务支持 完整的 ACID 事务 单文档原子性 + 多文档事务（4.0+）
扩展方式 主要垂直扩展（升级硬件） 水平扩展（分片）+ 垂直扩展
索引 B-Tree 索引 B-Tree 索引 + 多种特殊索引
存储引擎 InnoDB / MyISAM 等 WiredTiger（默认）
适用场景 强事务、复杂关联查询 灵活 Schema、海量数据、快速迭代

1.1.3 文档数据库的灵活性

在 MySQL 中，一张表的所有行必须遵循相同的列定义。而在 MongoDB 中，同一个集合中的文档可以有完全不同的结构：

MySQL 中的用户表 —— 所有行结构必须一致：

CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(200) NOT NULL,
    age INT
);

-- 每行都必须有 id, name, email, age 这四列
INSERT INTO users (name, email, age) VALUES ('张三', 'zhangsan@example.com', 25);
INSERT INTO users (name, email, age) VALUES ('李四', 'lisi@example.com', 30);

MongoDB 中的用户集合 —— 文档结构可以灵活变化：

// 文档1：普通用户，只有基本信息
{
    "_id": ObjectId("64a1b2c3d4e5f6a7b8c9d0e1"),
    "name": "张三",
    "email": "zhangsan@example.com",
    "age": 25
}

// 文档2：用户有额外的地址信息（嵌入文档）
{
    "_id": ObjectId("64a1b2c3d4e5f6a7b8c9d0e2"),
    "name": "李四",
    "email": "lisi@example.com",
    "age": 30,
    "address": {
        "city": "北京",
        "district": "朝阳区",
        "street": "建国路88号"
    }
}

// 文档3：用户有兴趣标签数组，但没有 age 字段
{
    "_id": ObjectId("64a1b2c3d4e5f6a7b8c9d0e3"),
    "name": "王五",
    "email": "wangwu@example.com",
    "tags": ["Java", "MongoDB", "Spring Boot"],
    "registeredAt": ISODate("2024-01-15T08:30:00Z")
}

关键启示：MongoDB 的灵活 Schema 使得你可以根据业务需求随时向文档中添加新字段，而不需要执行 ALTER TABLE 操作。这在敏捷开发和快速迭代的场景中非常有价值。

1.1.4 查询语言对比

MySQL（SQL）：

-- 查找年龄大于 25 的用户
SELECT * FROM users WHERE age > 25;

-- 查找北京的用户，按年龄排序
SELECT * FROM users WHERE city = '北京' ORDER BY age DESC;

-- 统计每个城市的用户数量
SELECT city, COUNT(*) as user_count FROM users GROUP BY city;

MongoDB（MQL）：

// 查找年龄大于 25 的用户
db.users.find({ age: { $gt: 25 } })

// 查找北京的用户，按年龄排序
db.users.find({ "address.city": "北京" }).sort({ age: -1 })

// 统计每个城市的用户数量
db.users.aggregate([
    { $group: { _id: "$address.city", user_count: { $sum: 1 } } }
])

1.2 核心概念映射：Database / Collection / Document vs 数据库 / 表 / 行

1.2.1 概念映射表

如果你已经熟悉关系型数据库，以下映射表可以帮助你快速理解 MongoDB 的术语：

MySQL 概念 MongoDB 概念 说明
Database（数据库） Database（数据库） 概念一致，都是数据的最高层容器
Table（表） Collection（集合） 集合是文档的容器，类似表但无固定结构
Row（行） Document（文档） 文档是数据的基本单位，以 BSON 格式存储
Column（列） Field（字段） 文档中的键值对，键即字段名
Primary Key（主键） _id 字段 每个文档自动拥有唯一的 _id
Index（索引） Index（索引） 概念一致，用于加速查询
JOIN（连接） $lookup / 嵌入文档 MongoDB 通过嵌入文档减少 JOIN 需求
Transaction（事务） Transaction（事务） MongoDB 4.0+ 支持多文档事务

1.2.2 Database（数据库）

MongoDB 中的数据库与 MySQL 中的数据库概念一致，是数据组织的最高层容器。一个 MongoDB 实例可以包含多个数据库，每个数据库拥有独立的集合和权限。

// 查看所有数据库
show dbs

// 输出示例：
// admin    0.000GB    （系统数据库，用于管理权限）
// config   0.000GB    （分片配置数据库）
// local    0.000GB    （本地数据库，副本集信息）
// myapp    0.125GB    （用户创建的数据库）

// 切换到（或创建）数据库
use myapp

// 查看当前数据库
db
// 输出：myapp

注意：MongoDB 中的数据库是懒创建的。执行 use myapp 时，如果 myapp 不存在，MongoDB 不会立刻创建它，而是等到你第一次向其中写入数据时才真正创建。

1.2.3 Collection（集合）

集合（Collection）是 MongoDB 中文档的容器，类似于关系型数据库中的表（Table）。但与表不同的是，集合不强制要求固定的 Schema。

// 查看当前数据库中的所有集合
show collections

// 创建集合（通常不需要显式创建，插入文档时会自动创建）
db.createCollection("users")

// 直接插入文档，集合会自动创建
db.orders.insertOne({
    orderId: "ORD-20240101-001",
    customer: "张三",
    items: [
        { product: "MongoDB实战", price: 79.00, quantity: 1 },
        { product: "Spring Boot入门", price: 59.00, quantity: 2 }
    ],
    totalAmount: 197.00,
    status: "已支付",
    createdAt: new Date()
})

1.2.4 Document（文档）

文档（Document）是 MongoDB 中数据的基本单位，以 BSON 格式存储。你可以将文档理解为一个 JSON 对象。

一个完整的用户文档示例：

{
    "_id": ObjectId("65f2a1b3c4d5e6f7a8b9c0d1"),
    "username": "zhangsan",
    "profile": {
        "realName": "张三",
        "gender": "男",
        "birthday": ISODate("1995-06-15T00:00:00Z"),
        "avatar": "https://example.com/avatars/zhangsan.jpg"
    },
    "contact": {
        "email": "zhangsan@example.com",
        "phone": "13800138000",
        "addresses": [
            {
                "label": "家",
                "city": "北京",
                "detail": "朝阳区建国路88号"
            },
            {
                "label": "公司",
                "city": "北京",
                "detail": "海淀区中关村大街1号"
            }
        ]
    },
    "roles": ["user", "vip"],
    "loginCount": 128,
    "lastLoginAt": ISODate("2024-03-15T09:30:00Z"),
    "createdAt": ISODate("2023-01-10T14:20:00Z"),
    "isActive": true
}

观察要点：

  • 文档可以包含嵌入文档（如 profile、contact）
  • 文档可以包含数组（如 roles、addresses）
  • 数组中的元素可以是简单值（如 roles）或嵌入文档（如 addresses）
  • 这种嵌套结构使得 MongoDB 能够在单个文档中表达丰富的数据关系

1.2.5 _id 字段与 ObjectId

每个 MongoDB 文档都必须有一个 _id 字段作为主键，用于唯一标识文档。如果插入文档时没有指定 _id，MongoDB 会自动生成一个 ObjectId。

ObjectId 的结构：

ObjectId 是一个 12 字节（24 位十六进制字符）的值，由以下部分组成：

|  4字节时间戳  |  5字节随机值  |  3字节递增计数器  |
|  65f2a1b3    |  c4d5e6f7a8  |  b9c0d1          |
  • 4 字节时间戳：文档创建时的 Unix 时间戳（秒级）
  • 5 字节随机值：每个进程启动时生成的随机值（包含机器标识和进程ID的信息）
  • 3 字节递增计数器：同一秒内同一进程中自增，初始值随机
// 自动生成 _id
db.users.insertOne({ name: "张三", age: 25 })
// 结果：{ "_id": ObjectId("65f2a1b3c4d5e6f7a8b9c0d1"), "name": "张三", "age": 25 }

// 手动指定 _id（可以是任意类型）
db.users.insertOne({ _id: "user_001", name: "李四", age: 30 })
// 结果：{ "_id": "user_001", "name": "李四", "age": 30 }

db.users.insertOne({ _id: 10001, name: "王五", age: 28 })
// 结果：{ "_id": 10001, "name": "王五", "age": 28 }

// 从 ObjectId 中提取时间戳
var id = ObjectId("65f2a1b3c4d5e6f7a8b9c0d1")
id.getTimestamp()
// 输出：ISODate("2024-03-14T07:23:47Z")

最佳实践：在大多数场景下，推荐使用自动生成的 ObjectId。它天然具备唯一性、有序性（按时间递增），并且不需要依赖数据库的自增机制。

1.3 BSON 数据格式与常用数据类型

1.3.1 什么是 BSON

BSON（Binary JSON）是 MongoDB 用来存储文档和进行网络传输的二进制编码格式。你可以把它理解为 JSON 的二进制扩展版本。

BSON vs JSON 的区别：

特性 JSON BSON
格式 文本格式（人类可读） 二进制格式（机器优化）
数据类型 6 种（String, Number, Boolean, null, Array, Object） 20+ 种（增加了 Date, ObjectId, Int32, Int64, Decimal128 等）
存储效率 较低（文本占空间大） 较高（二进制编码更紧凑，尤其对数字类型）
遍历速度 需要解析文本 每个元素有长度前缀，可快速跳过
适用场景 数据交互和配置文件 数据库存储和网络传输

BSON 的优势：

  1. 丰富的数据类型：支持日期、正则表达式、二进制数据等 JSON 不支持的类型
  2. 高效遍历：BSON 中每个元素都包含长度信息，可以快速跳过不需要的字段
  3. 高效编码/解码：二进制格式的编解码速度优于文本格式的 JSON 解析

简单理解：当你向 MongoDB 写入一个 JSON 格式的文档时，MongoDB 驱动会自动将其转换为 BSON 格式存储；当你读取数据时，BSON 会被自动转换回 JSON 呈现给你。这个过程对开发者是透明的。

1.3.2 常用数据类型详解

MongoDB 支持丰富的数据类型，以下是常用类型的详细说明和示例：

1. String（字符串）

最常用的数据类型，MongoDB 中的字符串必须是 UTF-8 编码。

{
    "name": "张三",
    "description": "这是一段中文描述",
    "url": "https://www.mongodb.com"
}

2. Int32（32位整数）

存储 32 位有符号整数，范围：-2,147,483,648 ~ 2,147,483,647。

// 在 mongosh 中，使用 NumberInt() 明确指定 Int32
db.products.insertOne({
    name: "MongoDB实战",
    stock: NumberInt(100),
    rating: NumberInt(5)
})

3. Int64（64位整数）/ Long

存储 64 位有符号整数，适合存储大整数。

// 使用 NumberLong() 指定 Int64
db.stats.insertOne({
    metric: "page_views",
    totalCount: NumberLong("9007199254740993"),
    timestamp: new Date()
})

4. Double（双精度浮点数）

64 位 IEEE 754 浮点数，mongosh 中数字的默认类型。

// mongosh 中的数字默认就是 Double
db.products.insertOne({
    name: "Spring Boot入门",
    price: 59.90,
    weight: 0.35
})

5. Decimal128（高精度小数）

128 位十进制浮点数，适合金融等对精度要求极高的场景。

db.accounts.insertOne({
    accountId: "ACC-001",
    balance: NumberDecimal("12345.6789"),
    interestRate: NumberDecimal("0.0325")
})

重要提示：在处理金额、汇率等对精度有严格要求的场景时，务必使用 Decimal128 而非 Double，否则可能出现浮点数精度丢失问题。

6. Boolean（布尔值）

只有 true 和 false 两个值。

{
    "username": "zhangsan",
    "isActive": true,
    "isAdmin": false,
    "emailVerified": true
}

7. Date（日期）

存储 UTC 格式的日期时间，精确到毫秒。内部以 64 位整数存储自 Unix 纪元（1970-01-01）以来的毫秒数。

db.events.insertOne({
    title: "技术分享会",
    startTime: new Date("2024-06-15T14:00:00Z"),
    endTime: new Date("2024-06-15T16:00:00Z"),
    createdAt: new Date()  // 当前时间
})

// ISODate 是 mongosh 中 Date 的等价写法
db.events.insertOne({
    title: "团队建设",
    startTime: ISODate("2024-07-01T09:00:00+08:00")
})

8. ObjectId

12 字节的唯一标识符，通常用作 _id 字段的值（详见 1.2.5 节）。

db.logs.insertOne({
    _id: new ObjectId(),
    userId: ObjectId("65f2a1b3c4d5e6f7a8b9c0d1"),
    action: "login",
    timestamp: new Date()
})

9. Array（数组）

可以包含任意类型的元素，包括不同类型的混合。

{
    "name": "张三",
    "tags": ["Java", "MongoDB", "Spring Boot"],
    "scores": [95, 87, 92, 78],
    "history": [
        { "action": "login", "time": "2024-03-15T09:00:00Z" },
        { "action": "purchase", "time": "2024-03-15T10:30:00Z" }
    ]
}

10. Object / 嵌入文档（Embedded Document）

文档中可以嵌套另一个文档，这是 MongoDB 最强大的特性之一。

{
    "orderId": "ORD-2024-001",
    "customer": {
        "name": "张三",
        "phone": "13800138000",
        "address": {
            "province": "北京市",
            "city": "北京",
            "district": "朝阳区",
            "detail": "建国路88号"
        }
    },
    "payment": {
        "method": "微信支付",
        "amount": 299.00,
        "paidAt": "2024-03-15T10:30:00Z"
    }
}

11. Null（空值）

表示空值或字段不存在。

{
    "name": "李四",
    "email": "lisi@example.com",
    "phone": null,
    "deletedAt": null
}

12. Binary Data（二进制数据）

用于存储二进制内容，如小文件、加密数据、UUID 等。

// 存储 UUID
db.sessions.insertOne({
    _id: UUID("a1b2c3d4-e5f6-7890-abcd-ef1234567890"),
    userId: "zhangsan",
    token: BinData(0, "c8edabc3f9834f76..."),
    expireAt: new Date("2024-03-16T09:00:00Z")
})

13. Timestamp（时间戳）

MongoDB 内部使用的特殊时间戳类型，主要用于副本集的 oplog（操作日志）。一般开发中请使用 Date 类型，而非 Timestamp。

// Timestamp 由两部分组成：秒级时间戳 + 序号
db.internal.insertOne({
    ts: Timestamp(1710403200, 1)
})

1.4 适用场景分析：什么时候该选 MongoDB

1.4.1 适合使用 MongoDB 的场景

1. 内容管理系统（CMS）

文章、博客、新闻等内容天然适合文档模型 —— 每篇文章可能有不同的字段（有的有视频，有的有图片gallery，有的只有文本）。

{
    "title": "MongoDB入门教程",
    "author": "张三",
    "content": "这是一篇关于MongoDB的入门教程...",
    "category": "数据库",
    "tags": ["MongoDB", "NoSQL", "数据库"],
    "coverImage": "https://example.com/images/mongodb.jpg",
    "viewCount": 1520,
    "comments": [
        { "user": "李四", "content": "写得很好！", "createdAt": "2024-03-15" }
    ],
    "publishedAt": "2024-03-14",
    "status": "published"
}

2. 用户画像与个性化

不同用户的画像数据结构差异很大，MongoDB 灵活的 Schema 完美适配。

{
    "userId": "U10001",
    "demographics": { "age": 28, "gender": "M", "city": "上海" },
    "preferences": {
        "categories": ["科技", "编程"],
        "priceRange": { "min": 50, "max": 200 }
    },
    "behavior": {
        "lastActive": "2024-03-15T20:00:00Z",
        "avgSessionDuration": 1800,
        "purchaseHistory": ["P001", "P002", "P003"]
    },
    "deviceInfo": {
        "os": "Android",
        "browser": "Chrome",
        "screenSize": "1080x2400"
    }
}

3. 物联网（IoT）数据

传感器数据量大、写入频繁、结构可能随设备型号不同而变化。

{
    "deviceId": "SENSOR-T-001",
    "type": "temperature",
    "location": { "type": "Point", "coordinates": [121.4737, 31.2304] },
    "readings": [
        { "value": 23.5, "unit": "°C", "timestamp": "2024-03-15T10:00:00Z" },
        { "value": 23.8, "unit": "°C", "timestamp": "2024-03-15T10:05:00Z" },
        { "value": 24.1, "unit": "°C", "timestamp": "2024-03-15T10:10:00Z" }
    ],
    "battery": 85,
    "firmware": "v2.1.0"
}

4. 实时分析与日志存储

日志数据写入量大、结构多变，MongoDB 的高吞吐写入和灵活 Schema 非常适合。

{
    "level": "ERROR",
    "service": "order-service",
    "instance": "order-service-pod-3",
    "message": "Failed to process payment",
    "stackTrace": "java.lang.RuntimeException: Payment gateway timeout...",
    "context": {
        "orderId": "ORD-2024-001",
        "userId": "U10001",
        "amount": 299.00,
        "retryCount": 3
    },
    "timestamp": "2024-03-15T10:30:45.123Z",
    "traceId": "abc123def456"
}

5. 商品目录 / 产品信息管理

不同品类的商品属性差异极大（手机有内存、屏幕尺寸；服装有颜色、尺码），MongoDB 的灵活文档模型能优雅地处理这种多态数据。

// 手机商品
{
    "name": "iPhone 15 Pro",
    "category": "手机",
    "price": 8999,
    "specs": {
        "screen": "6.1英寸",
        "storage": "256GB",
        "chip": "A17 Pro",
        "camera": "4800万像素"
    }
}

// 服装商品 —— 属性完全不同
{
    "name": "春季休闲夹克",
    "category": "服装",
    "price": 399,
    "specs": {
        "material": "100%棉",
        "color": ["黑色", "深蓝", "卡其"],
        "size": ["S", "M", "L", "XL", "XXL"],
        "season": "春秋"
    }
}

6. 游戏数据

玩家数据、游戏状态、排行榜等场景，读写频繁且结构灵活。

1.4.2 不适合使用 MongoDB 的场景

1. 需要强事务保证的金融核心系统

虽然 MongoDB 4.0+ 支持多文档事务，但在银行核心交易、对账等需要严格 ACID 事务保证的场景中，关系型数据库仍是首选。

示例场景：银行转账
- 账户 A 扣款 1000 元
- 账户 B 入账 1000 元
- 记录转账流水

→ 这三个操作必须要么全部成功，要么全部回滚，不容许任何不一致。
→ 关系型数据库在这类场景经过了数十年的验证，更加可靠。

2. 高度关联的复杂数据模型

如果数据之间存在大量多对多关系，需要频繁执行复杂的 JOIN 查询（如 ERP 系统、传统关系密集的业务系统），关系型数据库的 JOIN 能力更加成熟高效。

示例场景：企业 ERP 系统
- 订单关联客户、产品、仓库、物流、发票...
- 一次查询可能需要 JOIN 5-6 张表
- 数据关系固定且复杂

→ MongoDB 中虽然可以通过 $lookup 实现类似 JOIN 的操作，
   但性能和便利性不如关系型数据库。

3. 需要严格 Schema 约束的场景

某些业务对数据格式有严格要求（如医疗记录、法律文书），需要数据库层面强制约束每条数据的完整性。

1.4.3 技术选型决策参考表

考量因素 选 MongoDB 选关系型数据库（MySQL/PostgreSQL）
数据结构 不固定、经常变化 固定、很少变化
数据关联 关联少，可通过嵌入文档解决 关联复杂，需要频繁 JOIN
数据量 海量数据（TB/PB级别） 中小规模数据
写入吞吐 高吞吐写入 中等写入量
事务需求 单文档原子性已够用 需要跨表强事务
扩展需求 需要水平扩展 垂直扩展可满足
开发速度 快速迭代，Schema 频繁变化 需求稳定，Schema 变化少
查询模式 以文档为中心的查询 复杂的多表关联查询
团队经验 团队熟悉文档数据库 团队熟悉 SQL

1.4.4 现实中的混合使用

在实际项目中，MongoDB 和关系型数据库往往并不是非此即彼的关系，而是可以组合使用：

典型的混合架构：
┌──────────────────────────────────────────────────┐
│                   应用层                          │
├──────────────────────────────────────────────────┤
│                                                  │
│   MySQL / PostgreSQL          MongoDB            │
│   ┌──────────────┐       ┌──────────────┐       │
│   │ 用户账户     │       │ 商品目录     │       │
│   │ 订单交易     │       │ 用户画像     │       │
│   │ 财务对账     │       │ 日志数据     │       │
│   │ 权限管理     │       │ 内容管理     │       │
│   └──────────────┘       └──────────────┘       │
│                                                  │
│   → 强事务、复杂关联       → 灵活Schema、高吞吐  │
└──────────────────────────────────────────────────┘

总结：选择数据库不应该只看"流行度"或"新旧"，而应该根据具体业务场景的需求做出判断。理解每种数据库的优势和局限，才能做出最合适的技术选型。

本章小结

本章我们学习了以下核心内容：

  1. MongoDB 是什么 —— 一个面向文档的 NoSQL 数据库，使用 BSON 格式存储数据，具有灵活 Schema、水平扩展、高性能等特点。
  2. 核心概念映射 —— Database ↔ 数据库、Collection ↔ 表、Document ↔ 行、Field ↔ 列、_id ↔ 主键。
  3. BSON 与数据类型 —— BSON 是 JSON 的二进制扩展，支持 String、Int32、Int64、Double、Decimal128、Boolean、Date、ObjectId、Array、Object、Null、Binary Data、Timestamp 等丰富的数据类型。
  4. 适用场景 —— MongoDB 适合内容管理、用户画像、IoT、日志存储等灵活 Schema 的场景；不适合强事务和高度关联的场景。实际项目中可以与关系型数据库混合使用。
