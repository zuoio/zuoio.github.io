第11章 Change Streams 与实时数据

在很多业务场景中，我们需要实时感知数据的变化——新订单的通知、库存变动的同步、审计日志的记录等。 传统方案通常需要轮询数据库或引入消息队列，而 MongoDB 提供了原生的 Change Streams 机制， 让应用程序可以直接订阅数据变更事件，实现真正的实时数据推送。

11.1 Change Streams 原理与应用场景

11.1.1 什么是 Change Streams

Change Streams 是 MongoDB 3.6 引入的一种基于 oplog（操作日志）的变更监听机制，允许应用程序订阅数据库中的实时数据变更事件，而无需轮询数据库。

工作原理：

客户端应用 ──watch()──> MongoDB Driver ──> oplog（副本集操作日志）
                                              │
                                              ▼
                                   过滤 + 转换为变更事件
                                              │
                                              ▼
                                   推送给客户端应用
  • oplog 是 MongoDB 副本集用于数据同步的操作日志，记录了所有的写操作
  • Change Streams 本质上是对 oplog 的一个可恢复的、有序的、实时的抽象访问层
  • 与直接读取 oplog 相比，Change Streams 提供了更友好的 API 和更好的可用性

11.1.2 前提条件

条件 说明
副本集或分片集群 Change Streams 依赖 oplog，只有副本集和分片集群才有 oplog
MongoDB 3.6+ 最低版本要求；4.0 支持数据库级别监听；4.0+ 支持集群级别监听
WiredTiger 存储引擎 默认引擎，通常不需要额外配置
readConcern: majority 需要启用 majority 读关注（默认启用）

开发提示：如果你使用的是单节点 MongoDB（非副本集），Change Streams 不可用。 可以参考第9章中的 Docker 单节点副本集配置方法来快速搭建开发环境。

11.1.3 应用场景

场景 说明 示例
实时通知 数据变更时主动推送通知 新订单通知、新评论提醒
数据同步 将 MongoDB 的变更同步到其他系统 同步到 Elasticsearch、Redis 缓存
审计日志 记录所有数据变更操作 谁在什么时间改了什么数据
缓存失效 数据更新时自动清理缓存 商品价格变更时清除 Redis 缓存
ETL 管道 实时数据抽取与转换 实时数据仓库同步
事件驱动架构 基于数据变更触发业务逻辑 订单状态变更触发后续流程

11.1.4 支持的事件类型

事件类型 触发条件 operationType 值
insert 插入新文档 "insert"
update 更新已有文档 "update"
replace 替换整个文档 "replace"
delete 删除文档 "delete"
drop 删除集合 "drop"
rename 重命名集合 "rename"
dropDatabase 删除数据库 "dropDatabase"
invalidate 流失效（如集合被删除） "invalidate"

变更事件文档的结构示例：

{
    _id: { _data: "8263..." },       // Resume Token，用于断点续传
    operationType: "insert",          // 操作类型
    clusterTime: Timestamp(1700000000, 1),  // 集群时间戳
    fullDocument: {                   // 完整文档（insert 时自动包含）
        _id: ObjectId("6571..."),
        name: "新商品",
        price: 99.9
    },
    ns: {                             // 命名空间
        db: "shop",
        coll: "products"
    },
    documentKey: {                    // 文档主键
        _id: ObjectId("6571...")
    }
}

11.2 监听集合变更事件（Shell 篇）

11.2.1 db.collection.watch() 基本用法

watch() 是开启 Change Stream 的核心方法，它返回一个游标，可以持续接收变更事件。

基本用法：

// 监听 products 集合的所有变更
// 注意：这是一个阻塞操作，会持续等待新事件
const cursor = db.products.watch();

// 迭代接收变更事件
while (!cursor.isClosed()) {
    if (cursor.hasNext()) {
        const change = cursor.next();
        printjson(change);
    }
}

快速测试：

打开两个 mongosh 终端窗口：

// ===== 窗口1：启动监听 =====
use shop;
const cursor = db.products.watch();
while (cursor.hasNext()) {
    printjson(cursor.next());
}

// ===== 窗口2：触发变更 =====
use shop;

// 插入文档
db.products.insertOne({ name: "iPhone 16", price: 7999, stock: 100 });

// 更新文档
db.products.updateOne({ name: "iPhone 16" }, { $set: { price: 7499 } });

// 删除文档
db.products.deleteOne({ name: "iPhone 16" });

窗口1 会依次输出 insert、update、delete 三个变更事件。

11.2.2 使用 pipeline 过滤变更事件

watch() 的第一个参数接受一个聚合管道数组，可以用来过滤和转换变更事件。

只监听 insert 操作：

const cursor = db.products.watch([
    { $match: { operationType: "insert" } }
]);

while (cursor.hasNext()) {
    const change = cursor.next();
    print("新商品上架: " + change.fullDocument.name 
          + "，价格: " + change.fullDocument.price);
}

只监听特定条件的变更（如价格大于 1000 的商品）：

const cursor = db.products.watch([
    {
        $match: {
            $or: [
                // insert 事件：新文档价格 > 1000
                {
                    operationType: "insert",
                    "fullDocument.price": { $gt: 1000 }
                },
                // update 事件：只关注价格字段的变更
                {
                    operationType: "update",
                    "updateDescription.updatedFields.price": { $exists: true }
                },
                // delete 事件
                { operationType: "delete" }
            ]
        }
    },
    {
        $project: {
            operationType: 1,
            documentKey: 1,
            "fullDocument.name": 1,
            "fullDocument.price": 1,
            updateDescription: 1
        }
    }
]);

while (cursor.hasNext()) {
    printjson(cursor.next());
}

监听多种操作类型：

const cursor = db.orders.watch([
    {
        $match: {
            operationType: { $in: ["insert", "update"] },
            // 只关注状态为"已支付"或"已发货"的订单变更
            $or: [
                { "fullDocument.status": { $in: ["PAID", "SHIPPED"] } },
                { "updateDescription.updatedFields.status": { $in: ["PAID", "SHIPPED"] } }
            ]
        }
    }
]);

while (cursor.hasNext()) {
    const change = cursor.next();
    if (change.operationType === "insert") {
        print("新订单: " + change.fullDocument._id + " 状态: " + change.fullDocument.status);
    } else {
        print("订单更新: " + change.documentKey._id 
              + " 新状态: " + change.updateDescription.updatedFields.status);
    }
}

11.2.3 fullDocument 选项

默认情况下，update 事件只包含变更的字段描述（updateDescription），不包含完整文档。通过设置 fullDocument: "updateLookup" 可以在 update 事件中获取更新后的完整文档。

// 不使用 fullDocument 选项时，update 事件的内容
{
    operationType: "update",
    documentKey: { _id: ObjectId("...") },
    updateDescription: {
        updatedFields: { price: 7499 },    // 只有变更的字段
        removedFields: [],
        truncatedArrays: []
    }
    // 没有 fullDocument 字段
}

// 使用 fullDocument: "updateLookup" 后
const cursor = db.products.watch(
    [],     // 空的 pipeline
    { fullDocument: "updateLookup" }
);

// 此时 update 事件会包含完整文档
{
    operationType: "update",
    fullDocument: {                         // 完整的更新后文档
        _id: ObjectId("..."),
        name: "iPhone 16",
        price: 7499,
        stock: 100
    },
    updateDescription: {
        updatedFields: { price: 7499 },
        removedFields: [],
        truncatedArrays: []
    }
}

注意：fullDocument: "updateLookup" 会额外执行一次查询来获取当前文档，因此：

  • 获取到的是当前最新的文档，而不是变更发生那一刻的文档
  • 如果文档在变更后、查询前被再次修改或删除，可能获取到不一致的结果
  • 会增加额外的查询开销

MongoDB 6.0+ 新增的 fullDocument 选项：

// "whenAvailable"：尽量返回完整文档，无法获取时返回 null
const cursor = db.products.watch([], { fullDocument: "whenAvailable" });

// "required"：必须返回完整文档，无法获取时事件被丢弃
const cursor2 = db.products.watch([], { fullDocument: "required" });

11.2.4 监听数据库级别和集群级别的变更

监听整个数据库的变更（MongoDB 4.0+）：

// 监听 shop 数据库下所有集合的变更
use shop;
const cursor = db.watch([
    { $match: { operationType: { $in: ["insert", "update", "delete"] } } }
]);

while (cursor.hasNext()) {
    const change = cursor.next();
    print("[" + change.ns.coll + "] " + change.operationType 
          + " -> " + tojson(change.documentKey));
}

监听整个集群的变更（MongoDB 4.0+）：

// 监听所有数据库、所有集合的变更
const client = db.getMongo();
const cursor = client.watch([
    {
        $match: {
            // 排除系统数据库
            "ns.db": { $nin: ["admin", "local", "config"] }
        }
    }
]);

while (cursor.hasNext()) {
    const change = cursor.next();
    print("[" + change.ns.db + "." + change.ns.coll + "] " 
          + change.operationType);
}

11.2.5 完整的 Shell 监听示例

以下是一个完整的订单变更监听脚本：

use shop;

// 准备测试数据
db.orders.drop();

// 定义变更处理函数
function handleOrderChange(change) {
    const timestamp = new Date().toISOString();
    
    switch (change.operationType) {
        case "insert":
            print(`[${timestamp}] 📦 新订单创建`);
            print(`  订单ID: ${change.fullDocument._id}`);
            print(`  用户: ${change.fullDocument.userId}`);
            print(`  金额: ¥${change.fullDocument.totalAmount}`);
            break;
            
        case "update":
            print(`[${timestamp}] 🔄 订单更新`);
            print(`  订单ID: ${change.documentKey._id}`);
            if (change.updateDescription.updatedFields.status) {
                print(`  新状态: ${change.updateDescription.updatedFields.status}`);
            }
            if (change.fullDocument) {
                print(`  完整文档: ${tojson(change.fullDocument)}`);
            }
            break;
            
        case "delete":
            print(`[${timestamp}] ❌ 订单删除`);
            print(`  订单ID: ${change.documentKey._id}`);
            break;
            
        default:
            print(`[${timestamp}] ⚠️ 未知操作: ${change.operationType}`);
    }
    print("---");
}

// 启动监听（带过滤和完整文档查找）
print("开始监听订单变更...");
const cursor = db.orders.watch(
    [
        {
            $match: {
                operationType: { $in: ["insert", "update", "delete"] }
            }
        }
    ],
    {
        fullDocument: "updateLookup"
    }
);

while (cursor.hasNext()) {
    handleOrderChange(cursor.next());
}

使用 MongoTemplate 监听变更：

import com.mongodb.client.model.changestream.FullDocument;
import org.springframework.data.mongodb.core.ChangeStreamOptions;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.aggregation.Aggregation;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.stereotype.Service;
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.atomic.AtomicBoolean;

@Service
public class OrderChangeStreamService {

    private static final Logger log = LoggerFactory.getLogger(OrderChangeStreamService.class);

    private final MongoTemplate mongoTemplate;
    private final ExecutorService executor = Executors.newSingleThreadExecutor();
    private final AtomicBoolean running = new AtomicBoolean(false);

    public OrderChangeStreamService(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    @PostConstruct
    public void startListening() {
        running.set(true);
        executor.submit(this::listenToChanges);
        log.info("订单变更监听已启动");
    }

    @PreDestroy
    public void stopListening() {
        running.set(false);
        executor.shutdown();
        log.info("订单变更监听已停止");
    }

    private void listenToChanges() {
        // 构建 ChangeStreamOptions
        ChangeStreamOptions options = ChangeStreamOptions.builder()
                // 只监听 insert 和 update 操作
                .filter(Aggregation.newAggregation(
                        Aggregation.match(
                                Criteria.where("operationType")
                                        .in("insert", "update", "replace")
                        )
                ))
                // update 事件也返回完整文档
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                .build();

        // 开始监听 orders 集合
        mongoTemplate.changeStream("orders", options, Order.class)
                .forEach(event -> {
                    if (!running.get()) return;

                    String opType = event.getOperationType().getValue();
                    Order order = event.getBody();

                    log.info("收到订单变更事件: type={}, orderId={}, status={}",
                            opType,
                            order != null ? order.getId() : "N/A",
                            order != null ? order.getStatus() : "N/A");

                    // 根据操作类型执行不同的业务逻辑
                    if (order != null) {
                        handleOrderChange(opType, order);
                    }
                });
    }

    private void handleOrderChange(String operationType, Order order) {
        switch (operationType) {
            case "insert" -> log.info("新订单创建: orderId={}, userId={}, amount={}",
                    order.getId(), order.getUserId(), order.getTotalAmount());
            case "update", "replace" -> {
                if ("PAID".equals(order.getStatus())) {
                    log.info("订单已支付，触发发货流程: orderId={}", order.getId());
                    // 这里可以调用发货服务、发送通知等
                }
                if ("SHIPPED".equals(order.getStatus())) {
                    log.info("订单已发货，通知用户: orderId={}", order.getId());
                    // 发送短信/推送通知
                }
            }
        }
    }
}

11.3.2 方式二：使用 MessageListenerContainer

Spring Data MongoDB 提供了 MessageListenerContainer，它以更"Spring 化"的方式管理 Change Stream 监听器的生命周期。

配置 MessageListenerContainer：

import com.mongodb.client.model.changestream.FullDocument;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.messaging.DefaultMessageListenerContainer;
import org.springframework.data.mongodb.core.messaging.MessageListenerContainer;

@Configuration
public class ChangeStreamConfig {

    @Bean
    MessageListenerContainer messageListenerContainer(MongoTemplate mongoTemplate) {
        DefaultMessageListenerContainer container =
                new DefaultMessageListenerContainer(mongoTemplate);
        // 自动启动
        container.setAutoStartup(true);
        return container;
    }
}

定义 MessageListener 并注册监听：

import com.mongodb.client.model.changestream.ChangeStreamDocument;
import com.mongodb.client.model.changestream.FullDocument;
import org.bson.Document;
import org.springframework.boot.context.event.ApplicationReadyEvent;
import org.springframework.context.event.EventListener;
import org.springframework.data.mongodb.core.ChangeStreamOptions;
import org.springframework.data.mongodb.core.aggregation.Aggregation;
import org.springframework.data.mongodb.core.messaging.ChangeStreamRequest;
import org.springframework.data.mongodb.core.messaging.Message;
import org.springframework.data.mongodb.core.messaging.MessageListener;
import org.springframework.data.mongodb.core.messaging.MessageListenerContainer;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.stereotype.Component;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Component
public class OrderChangeListener implements MessageListener<ChangeStreamDocument<Document>, Order> {

    private static final Logger log = LoggerFactory.getLogger(OrderChangeListener.class);

    private final MessageListenerContainer container;

    public OrderChangeListener(MessageListenerContainer container) {
        this.container = container;
    }

    /**
     * 应用启动完成后注册监听
     */
    @EventListener(ApplicationReadyEvent.class)
    public void registerListener() {
        // 构建 ChangeStreamOptions
        ChangeStreamOptions options = ChangeStreamOptions.builder()
                .filter(Aggregation.newAggregation(
                        Aggregation.match(
                                Criteria.where("operationType").in("insert", "update")
                        )
                ))
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                .build();

        // 构建 ChangeStreamRequest
        ChangeStreamRequest<Order> request = ChangeStreamRequest.builder(this)
                .collection("orders")            // 监听的集合名
                .options(options)                 // 监听选项
                .build();

        // 注册到容器
        container.register(request, Order.class);
        log.info("Order ChangeStream 监听器已注册");
    }

    /**
     * 接收到变更消息时的回调
     */
    @Override
    public void onMessage(Message<ChangeStreamDocument<Document>, Order> message) {
        Order order = message.getBody();
        ChangeStreamDocument<Document> raw = message.getRaw();

        if (order == null || raw == null) return;

        String opType = raw.getOperationType().getValue();

        log.info("收到订单变更: type={}, orderId={}, status={}",
                opType, order.getId(), order.getStatus());

        switch (order.getStatus()) {
            case "PAID" -> {
                log.info("处理支付完成事件，准备通知仓库发货...");
                // 调用仓库服务
            }
            case "SHIPPED" -> {
                log.info("处理发货事件，准备通知用户...");
                // 发送通知
            }
            case "COMPLETED" -> {
                log.info("订单完成，更新统计数据...");
                // 更新统计
            }
        }
    }
}

同时注册多个监听器（监听不同集合）：

@Component
public class MultiCollectionChangeListener {

    private static final Logger log = LoggerFactory.getLogger(MultiCollectionChangeListener.class);

    private final MessageListenerContainer container;

    public MultiCollectionChangeListener(MessageListenerContainer container) {
        this.container = container;
    }

    @EventListener(ApplicationReadyEvent.class)
    public void registerAll() {
        // 监听订单集合
        registerOrderListener();
        // 监听商品集合
        registerProductListener();
    }

    private void registerOrderListener() {
        MessageListener<ChangeStreamDocument<Document>, Order> listener =
                message -> {
                    Order order = message.getBody();
                    if (order != null) {
                        log.info("[订单] {} - {}", order.getId(), order.getStatus());
                    }
                };

        ChangeStreamRequest<Order> request = ChangeStreamRequest.builder(listener)
                .collection("orders")
                .options(ChangeStreamOptions.builder()
                        .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                        .build())
                .build();

        container.register(request, Order.class);
    }

    private void registerProductListener() {
        MessageListener<ChangeStreamDocument<Document>, Product> listener =
                message -> {
                    Product product = message.getBody();
                    if (product != null) {
                        log.info("[商品] {} 库存变更: {}", product.getName(), product.getStock());
                    }
                };

        ChangeStreamRequest<Product> request = ChangeStreamRequest.builder(listener)
                .collection("products")
                .options(ChangeStreamOptions.builder()
                        .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                        .build())
                .build();

        container.register(request, Product.class);
    }
}

11.3.3 方式三：@Tailable 注解与 Capped Collection

除了 Change Streams，MongoDB 还支持通过 Tailable Cursor（可追踪游标）实时读取 Capped Collection（固定大小集合）中的新数据。这种方式更轻量，适合日志类场景。

Capped Collection 的特点：

特性 说明
固定大小 创建时指定最大字节数和最大文档数
插入有序 文档按插入顺序存储，类似环形缓冲区
自动淘汰 达到上限时自动删除最旧的文档
高性能插入 不需要索引即可保证插入顺序
限制 不能删除单个文档，不能更新导致文档大小增长

步骤1：创建 Capped Collection

// Shell 中创建
db.createCollection("notifications", {
    capped: true,
    size: 10485760,    // 10MB
    max: 10000         // 最多10000条文档
});

// 验证
db.notifications.isCapped();  // true

或在 Spring Boot 中通过代码创建：

11.3.4 结合 SSE（Server-Sent Events）实现浏览器端实时推送

SSE 是一种 HTTP 协议原生支持的服务器推送技术，非常适合与 Change Streams 或 Tailable Cursor 结合使用。

方式A：Change Streams + SSE（推荐用于生产环境）

import org.springframework.data.mongodb.core.ChangeStreamOptions;
import org.springframework.data.mongodb.core.ReactiveMongoTemplate;
import org.springframework.http.MediaType;
import org.springframework.http.codec.ServerSentEvent;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;
import reactor.core.publisher.Flux;

import com.mongodb.client.model.changestream.FullDocument;
import org.springframework.data.mongodb.core.aggregation.Aggregation;
import org.springframework.data.mongodb.core.query.Criteria;

import java.time.Duration;

@RestController
@RequestMapping("/api/sse")
public class OrderSseController {

    private final ReactiveMongoTemplate reactiveMongoTemplate;

    public OrderSseController(ReactiveMongoTemplate reactiveMongoTemplate) {
        this.reactiveMongoTemplate = reactiveMongoTemplate;
    }

    /**
     * SSE 端点：实时推送订单变更事件
     * 客户端通过 EventSource 连接此端点即可接收实时数据
     * 
     * 使用方式：GET /api/sse/orders
     */
    @GetMapping(value = "/orders", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<Order>> streamOrderChanges() {
        ChangeStreamOptions options = ChangeStreamOptions.builder()
                .filter(Aggregation.newAggregation(
                        Aggregation.match(
                                Criteria.where("operationType").in("insert", "update")
                        )
                ))
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                .build();

        return reactiveMongoTemplate.changeStream("orders", options, Order.class)
                .map(event -> {
                    Order order = event.getBody();
                    String eventType = event.getOperationType() != null
                            ? event.getOperationType().getValue()
                            : "unknown";

                    return ServerSentEvent.<Order>builder()
                            .id(order != null ? order.getId() : null)
                            .event(eventType)
                            .data(order)
                            .build();
                });
    }

    /**
     * SSE 端点：实时推送指定用户的订单变更
     * 
     * 使用方式：GET /api/sse/orders/user/{userId}
     */
    @GetMapping(value = "/orders/user/{userId}", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<Order>> streamUserOrders(@PathVariable String userId) {
        ChangeStreamOptions options = ChangeStreamOptions.builder()
                .filter(Aggregation.newAggregation(
                        Aggregation.match(
                                Criteria.where("operationType").in("insert", "update")
                                        .and("fullDocument.userId").is(userId)
                        )
                ))
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                .build();

        return reactiveMongoTemplate.changeStream("orders", options, Order.class)
                .map(event -> ServerSentEvent.<Order>builder()
                        .event(event.getOperationType().getValue())
                        .data(event.getBody())
                        .build()
                );
    }

    /**
     * SSE 心跳：防止连接超时
     * 每30秒发送一个心跳事件
     */
    @GetMapping(value = "/orders/heartbeat", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<Order>> streamWithHeartbeat() {
        ChangeStreamOptions options = ChangeStreamOptions.builder()
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP)
                .build();

        Flux<ServerSentEvent<Order>> dataStream =
                reactiveMongoTemplate.changeStream("orders", options, Order.class)
                        .map(event -> ServerSentEvent.<Order>builder()
                                .event("order-change")
                                .data(event.getBody())
                                .build());

        // 心跳流：每30秒发送一个空事件
        Flux<ServerSentEvent<Order>> heartbeat = Flux.interval(Duration.ofSeconds(30))
                .map(tick -> ServerSentEvent.<Order>builder()
                        .event("heartbeat")
                        .build());

        // 合并数据流和心跳流
        return Flux.merge(dataStream, heartbeat);
    }
}

11.4 断点续传：Resume Token 机制

11.4.1 什么是 Resume Token

Resume Token 是 Change Stream 中每个变更事件的唯一标识符，包含在事件的 _id 字段中。它的作用是：

  • 标记位置：记录当前消费到了哪个变更事件
  • 断点续传：应用重启后，可以从上次的位置继续消费，不丢失事件
  • 容错恢复：网络闪断或服务故障后恢复监听
// Resume Token 的结构示例
{
    _id: {
        _data: "8263F1A2B3000000012B022C0100296E5A1004..."
    }
}

Resume Token 的有效期取决于 oplog 的大小。当 oplog 被覆盖后，对应的 Resume Token 就会失效。 默认情况下 oplog 大小约为磁盘空间的 5%，通常可以保留数小时到数天的数据。

11.4.2 如何保存和恢复 Resume Token

在 Shell 中使用 Resume Token：

// 1. 开始监听，保存最后一个 Token
var lastToken = null;
var cursor = db.orders.watch([], { fullDocument: "updateLookup" });

while (cursor.hasNext()) {
    var change = cursor.next();
    lastToken = change._id;  // 保存 Resume Token
    printjson(change);
    
    // 将 Token 持久化到一个专门的集合中
    db.resumeTokens.updateOne(
        { _id: "orders_stream" },
        { $set: { token: lastToken, updatedAt: new Date() } },
        { upsert: true }
    );
}

// 2. 恢复监听时，使用保存的 Token
var savedToken = db.resumeTokens.findOne({ _id: "orders_stream" });
if (savedToken) {
    var cursor = db.orders.watch(
        [],
        {
            fullDocument: "updateLookup",
            resumeAfter: savedToken.token  // 从上次的位置继续
        }
    );
    print("从 Resume Token 恢复监听");
} else {
    var cursor = db.orders.watch([], { fullDocument: "updateLookup" });
    print("全新开始监听");
}

startAfter vs resumeAfter：

选项 说明 使用场景
resumeAfter 从指定 Token 之后的下一个事件开始 正常恢复，Token 对应的事件存在于 oplog 中
startAfter 从指定 Token 之后开始，即使流已经 invalidate 集合被删除重建后恢复监听
startAtOperationTime 从指定的集群时间戳开始 不知道 Token 但知道大概时间

11.4.3 防止消息丢失的策略

在生产环境中，需要考虑以下可能导致消息丢失的场景：

风险场景 解决策略
应用崩溃，未保存最新 Token 定期（或每条消息后）持久化 Token
oplog 被覆盖，Token 失效 增大 oplog 大小；配合全量同步使用
处理消息时发生异常 先处理消息，成功后再更新 Token（at-least-once）
网络闪断 Change Streams 自动重试连接
MongoDB 主节点切换 Change Streams 自动切换到新主节点

消息处理的两种语义：

At-most-once（最多一次）：
  1. 保存 Resume Token
  2. 处理消息
  缺点：处理失败时消息丢失

At-least-once（至少一次，推荐）：
  1. 处理消息
  2. 处理成功后保存 Resume Token
  缺点：可能重复处理，需要业务幂等

import org.bson.BsonDocument;
import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.mapping.Document;

@Document(collection = "change_stream_tokens")

Resume Token 存储服务：

import org.bson.BsonDocument;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.time.Instant;

@Service
public class ResumeTokenStore {

    private static final Logger log = LoggerFactory.getLogger(ResumeTokenStore.class);

    private final MongoTemplate mongoTemplate;

    public ResumeTokenStore(MongoTemplate mongoTemplate) {
        this.mongoTemplate = mongoTemplate;
    }

    /**
     * 保存 Resume Token
     */
    public void saveToken(String listenerId, BsonDocument token) {
        Query query = Query.query(Criteria.where("_id").is(listenerId));
        Update update = new Update()
                .set("token", token)
                .set("updatedAt", Instant.now());

        mongoTemplate.upsert(query, update, ChangeStreamToken.class);
        log.debug("Resume Token 已保存: listenerId={}", listenerId);
    }

    /**
     * 获取保存的 Resume Token
     */
    public BsonDocument getToken(String listenerId) {
        ChangeStreamToken tokenDoc = mongoTemplate.findById(listenerId, ChangeStreamToken.class);
        if (tokenDoc != null) {
            log.info("找到已保存的 Resume Token: listenerId={}, updatedAt={}",
                    listenerId, tokenDoc.getUpdatedAt());
            return tokenDoc.getToken();
        }
        log.info("未找到 Resume Token，将从头开始监听: listenerId={}", listenerId);
        return null;
    }

    /**
     * 删除 Resume Token
     */
    public void deleteToken(String listenerId) {
        Query query = Query.query(Criteria.where("_id").is(listenerId));
        mongoTemplate.remove(query, ChangeStreamToken.class);
    }
}

带断点续传的 Change Stream 消费者：

import com.mongodb.client.model.changestream.FullDocument;
import org.bson.BsonDocument;
import org.springframework.data.mongodb.core.ChangeStreamOptions;
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.aggregation.Aggregation;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.stereotype.Service;
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.atomic.AtomicBoolean;

@Service
public class ResilientOrderChangeConsumer {

    private static final Logger log = LoggerFactory.getLogger(ResilientOrderChangeConsumer.class);
    private static final String LISTENER_ID = "orders_change_consumer";
    private static final int MAX_RETRY_DELAY_MS = 30_000;

    private final MongoTemplate mongoTemplate;
    private final ResumeTokenStore tokenStore;
    private final ExecutorService executor = Executors.newSingleThreadExecutor();
    private final AtomicBoolean running = new AtomicBoolean(false);

    public ResilientOrderChangeConsumer(MongoTemplate mongoTemplate, ResumeTokenStore tokenStore) {
        this.mongoTemplate = mongoTemplate;
        this.tokenStore = tokenStore;
    }

    @PostConstruct
    public void start() {
        running.set(true);
        executor.submit(this::consumeWithRetry);
        log.info("订单变更消费者已启动");
    }

    @PreDestroy
    public void stop() {
        running.set(false);
        executor.shutdown();
        log.info("订单变更消费者已停止");
    }

    /**
     * 带重试和断点续传的消费循环
     */
    private void consumeWithRetry() {
        int retryDelay = 1000;

        while (running.get()) {
            try {
                consume();
                retryDelay = 1000; // 成功运行后重置重试延迟
            } catch (Exception e) {
                log.error("Change Stream 消费异常，{}ms 后重试: {}", retryDelay, e.getMessage());
                try {
                    Thread.sleep(retryDelay);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    return;
                }
                // 指数退避，最大30秒
                retryDelay = Math.min(retryDelay * 2, MAX_RETRY_DELAY_MS);
            }
        }
    }

    private void consume() {
        // 尝试从上次的位置恢复
        BsonDocument savedToken = tokenStore.getToken(LISTENER_ID);

        ChangeStreamOptions.ChangeStreamOptionsBuilder builder = ChangeStreamOptions.builder()
                .filter(Aggregation.newAggregation(
                        Aggregation.match(
                                Criteria.where("operationType")
                                        .in("insert", "update", "replace", "delete")
                        )
                ))
                .fullDocumentLookup(FullDocument.UPDATE_LOOKUP);

        // 如果有保存的 Token，从断点恢复
        if (savedToken != null) {
            builder.resumeAfter(savedToken);
            log.info("从 Resume Token 恢复监听");
        }

        ChangeStreamOptions options = builder.build();

        // 开始消费
        mongoTemplate.changeStream("orders", options, Order.class)
                .forEach(event -> {
                    if (!running.get()) return;

                    try {
                        // 1. 先处理消息（at-least-once 语义）
                        processEvent(event.getOperationType().getValue(), event.getBody());

                        // 2. 处理成功后，保存 Resume Token
                        BsonDocument resumeToken = event.getRaw().getResumeToken();
                        if (resumeToken != null) {
                            tokenStore.saveToken(LISTENER_ID, resumeToken);
                        }

                    } catch (Exception e) {
                        log.error("处理变更事件失败: {}", e.getMessage(), e);
                        // 不保存 Token，下次重启时会重新处理这条消息
                        // 业务处理需要保证幂等性
                    }
                });
    }

    private void processEvent(String operationType, Order order) {
        if (order == null) {
            log.warn("收到空文档事件，operationType={}", operationType);
            return;
        }

        switch (operationType) {
            case "insert" -> {
                log.info("处理新订单: id={}, userId={}, amount={}",
                        order.getId(), order.getUserId(), order.getTotalAmount());
                // 发送新订单通知
                // notificationService.sendNewOrderNotification(order);
            }
            case "update", "replace" -> {
                log.info("处理订单更新: id={}, status={}",
                        order.getId(), order.getStatus());

                switch (order.getStatus()) {
                    case "PAID" -> {
                        // 通知仓库备货
                        // warehouseService.prepareShipment(order);
                    }
                    case "SHIPPED" -> {

关键设计要点：

设计点 说明
指数退避重试 连接失败时逐步增加重试间隔（1s → 2s → 4s → ... → 30s），避免对数据库造成压力
At-least-once 语义 先处理消息，成功后才保存 Token；失败时下次会重新处理
幂等性要求 由于可能重复处理，业务逻辑必须保证幂等（如使用唯一约束、状态机检查等）
优雅关闭 通过 @PreDestroy 设置 running=false，让消费循环安全退出
Token 持久化 将 Token 存储在 MongoDB 的专用集合中，应用重启后可以恢复
