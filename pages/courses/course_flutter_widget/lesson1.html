1. Flutter 三棵树：Widget Tree、Element Tree、RenderObject Tree

Flutter 框架的 UI 渲染体系建立在三棵树之上。理解它们之间的关系，是掌握 Flutter 性能优化和组件工作原理的基础。

1.1 三棵树的概念

树 核心职责 特性
Widget Tree 描述 UI 的配置信息（"蓝图"） 不可变（immutable），轻量级，频繁重建
Element Tree 管理 Widget 和 RenderObject 之间的关系（"管家"） 可变，持久存在，负责生命周期管理
RenderObject Tree 负责实际的布局（layout）和绘制（paint） 可变，承担真正的渲染计算工作

类比理解：可以把 Widget 想象成建筑的设计图纸，Element 是施工项目经理，RenderObject 是真正搬砖盖楼的工人。每次设计变更（Widget 重建），项目经理（Element）会判断哪些地方需要改动，只通知相关工人（RenderObject）去修改，而不是把整栋楼拆了重建。

1.2 三棵树的工作流程

当 Flutter 构建 UI 时：

  1. 开发者编写 Widget 代码，形成 Widget Tree
  2. Flutter 框架调用 Widget.createElement() 为每个 Widget 创建对应的 Element，形成 Element Tree
  3. 对于需要渲染的 Widget（RenderObjectWidget 子类），Element 调用 createRenderObject() 创建 RenderObject，形成 RenderObject Tree

当 UI 需要更新时：

  1. 新的 Widget Tree 被创建
  2. Element 通过 Widget.canUpdate() 方法（比较 runtimeType 和 key）判断是否可以复用
  3. 如果可以复用，只更新 RenderObject 的属性；如果不能复用，则销毁旧的 Element 和 RenderObject，创建新的

1.3 代码示例：观察三棵树

import 'package:flutter/material.dart';

void main() {
  runApp(const TreeDemoApp());
}

class TreeDemoApp extends StatelessWidget {
  const TreeDemoApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      home: Scaffold(
        appBar: AppBar(title: const Text('三棵树演示')),
        body: const Center(
          child: TreeDemoWidget(),
        ),
      ),
    );
  }
}

class TreeDemoWidget extends StatefulWidget {
  const TreeDemoWidget({super.key});

  @override
  State<TreeDemoWidget> createState() => _TreeDemoWidgetState();
}

class _TreeDemoWidgetState extends State<TreeDemoWidget> {
  Color _color = Colors.blue;
  int _rebuildCount = 0;

  @override
  Widget build(BuildContext context) {
    _rebuildCount++;
    return Column(
      mainAxisAlignment: MainAxisAlignment.center,
      children: [
        // 这个 Container 对应了三棵树中各自的节点：
        // Widget: Container（配置信息：颜色、尺寸）
        // Element: StatelessElement（管理 Container 的生命周期）
        // RenderObject: RenderDecoratedBox + RenderConstrainedBox（实际渲染）
        Container(
          width: 150,
          height: 150,
          decoration: BoxDecoration(
            color: _color,
            borderRadius: BorderRadius.circular(16),
          ),
          child: Center(
            child: Text(
              '重建次数: $_rebuildCount',
              style: const TextStyle(color: Colors.white, fontSize: 16),
            ),
          ),
        ),
        const SizedBox(height: 24),
        ElevatedButton(
          onPressed: () {
            setState(() {
              // 调用 setState 后，Widget Tree 会重建
              // 但 Element Tree 会复用已有 Element（因为 runtimeType 和 key 没变）
              // RenderObject 只更新变化的属性（颜色），而非重新创建
              _color = _color == Colors.blue ? Colors.orange : Colors.blue;
            });
          },
          child: const Text('切换颜色（触发重建）'),
        ),
        const SizedBox(height: 16),
        // 通过 context（即当前 Element）可以观察树结构
        ElevatedButton(
          onPressed: () {
            // 遍历 Element Tree，打印其中的 Widget 信息
            _printElementTree(context);
          },
          child: const Text('打印 Element Tree 信息'),
        ),
      ],
    );
  }

  void _printElementTree(BuildContext context) {
    debugPrint('===== Element Tree 部分结构 =====');
    // visitChildElements 可以遍历当前 Element 的子节点
    context.visitChildElements((Element element) {
      debugPrint('Element: ${element.runtimeType}');
      debugPrint('  → Widget: ${element.widget.runtimeType}');
      if (element is RenderObjectElement) {
        debugPrint('  → RenderObject: ${element.renderObject.runtimeType}');
      }
      // 继续遍历子节点（只展开一层示意）
      element.visitChildElements((child) {
        debugPrint('  子 Element: ${child.runtimeType}');
        debugPrint('    → Widget: ${child.widget.runtimeType}');
      });
    });
    debugPrint('================================');
  }
}

1.4 Widget 的不可变性与高效重建

一个容易让初学者困惑的问题是：Widget 频繁重建不会影响性能吗？

答案是：通常不会。原因在于：

  • Widget 只是轻量级的配置对象（类似一个 Dart 类的实例），创建和销毁的成本极低
  • 真正昂贵的操作是布局和绘制，这些工作由 RenderObject 完成
  • Element 充当了"差异比较器"的角色，只有属性真正变化时才会更新 RenderObject

最佳实践：尽量使用 const 构造函数声明 Widget。被 const 修饰的 Widget 在整个应用生命周期内只创建一次，并在重建时被直接复用，不会重复执行 build 方法。

// ✅ 推荐：使用 const，避免不必要的重建
const Text('这段文字不会变化')

// ❌ 不推荐：每次 build 都会创建新的 Text 实例
Text('这段文字不会变化')

2. StatelessWidget 与 StatefulWidget 的本质区别

2.1 StatelessWidget：无状态组件

StatelessWidget 是一个不可变的组件。一旦创建，它的属性就不会改变。它只依赖传入的参数（构造函数的参数）和 BuildContext 来构建 UI。

核心特征：

  • 没有可变状态
  • build 方法的输出完全由构造函数参数和 BuildContext 决定
  • 当父组件重建并传入新参数时，StatelessWidget 也会重建
import 'package:flutter/material.dart';

/// 一个自定义的用户信息卡片 —— 典型的 StatelessWidget 使用场景
/// 它只负责展示传入的数据，自身不维护任何状态
class UserProfileCard extends StatelessWidget {
  final String name;
  final String email;
  final String avatarUrl;
  final VoidCallback? onTap;

  const UserProfileCard({
    super.key,
    required this.name,
    required this.email,
    required this.avatarUrl,
    this.onTap,
  });

  @override
  Widget build(BuildContext context) {
    final theme = Theme.of(context);

    return Card(
      elevation: 4,
      shape: RoundedRectangleBorder(borderRadius: BorderRadius.circular(12)),
      child: InkWell(
        onTap: onTap,
        borderRadius: BorderRadius.circular(12),
        child: Padding(
          padding: const EdgeInsets.all(16),
          child: Row(
            children: [
              CircleAvatar(
                radius: 30,
                backgroundImage: NetworkImage(avatarUrl),
              ),
              const SizedBox(width: 16),
              Expanded(
                child: Column(
                  crossAxisAlignment: CrossAxisAlignment.start,
                  children: [
                    Text(
                      name,
                      style: theme.textTheme.titleMedium?.copyWith(
                        fontWeight: FontWeight.bold,
                      ),
                    ),
                    const SizedBox(height: 4),
                    Text(
                      email,
                      style: theme.textTheme.bodyMedium?.copyWith(
                        color: Colors.grey[600],
                      ),
                    ),
                  ],
                ),
              ),
              const Icon(Icons.chevron_right, color: Colors.grey),
            ],
          ),
        ),
      ),
    );
  }
}

2.2 StatefulWidget：有状态组件

StatefulWidget 是一个配合 State 对象使用的组件。Widget 本身仍然是不可变的，但它关联的 State 对象是可变且持久的。

核心机制：

  • StatefulWidget 本身只是配置，真正的状态存储在 State<T> 对象中
  • State 对象的生命周期绑定在 Element 上，而非 Widget 上
  • 当 Widget 重建时，Element 会复用已有的 State 对象，并调用 didUpdateWidget
  • 通过 setState() 通知框架状态发生变化，触发 build 方法重新执行
import 'package:flutter/material.dart';

/// 一个可交互的计时器组件 —— 典型的 StatefulWidget 使用场景
/// 它需要维护内部状态（当前秒数、是否运行中）
class StopwatchWidget extends StatefulWidget {
  final String label;

  const StopwatchWidget({super.key, this.label = '计时器'});

  @override
  State<StopwatchWidget> createState() => _StopwatchWidgetState();
}

class _StopwatchWidgetState extends State<StopwatchWidget> {
  int _seconds = 0;
  bool _isRunning = false;
  late final Stopwatch _stopwatch;

  // _ticker 用 Stream 模拟每秒触发
  Stream<int>? _tickerStream;
  // 用于取消订阅
  Object? _subscription;

  @override
  void initState() {
    super.initState();
    _stopwatch = Stopwatch();
  }

  void _startTimer() {
    if (_isRunning) return;
    setState(() => _isRunning = true);
    _stopwatch.start();
    // 使用 Stream.periodic 每秒更新一次
    _tickerStream = Stream.periodic(
      const Duration(seconds: 1),
      (count) => count,
    );
    _subscription = _tickerStream!.listen((_) {
      if (mounted) {
        setState(() {
          _seconds = _stopwatch.elapsed.inSeconds;
        });
      }
    });
  }

  void _stopTimer() {
    _stopwatch.stop();
    (_subscription as dynamic)?.cancel();
    setState(() => _isRunning = false);
  }

  void _resetTimer() {
    _stopTimer();
    _stopwatch.reset();
    setState(() => _seconds = 0);
  }

  @override
  void dispose() {
    _stopTimer();
    super.dispose();
  }

  String _formatTime(int totalSeconds) {
    final minutes = (totalSeconds ~/ 60).toString().padLeft(2, '0');
    final seconds = (totalSeconds % 60).toString().padLeft(2, '0');
    return '$minutes:$seconds';
  }

  @override
  Widget build(BuildContext context) {
    return Card(
      elevation: 4,
      shape: RoundedRectangleBorder(borderRadius: BorderRadius.circular(16)),
      child: Padding(
        padding: const EdgeInsets.all(24),
        child: Column(
          mainAxisSize: MainAxisSize.min,
          children: [
            // widget 属性可以访问 StatefulWidget 传入的参数
            Text(widget.label, style: Theme.of(context).textTheme.titleMedium),
            const SizedBox(height: 16),
            Text(
              _formatTime(_seconds),
              style: const TextStyle(
                fontSize: 48,
                fontWeight: FontWeight.w300,
                fontFamily: 'monospace',
              ),
            ),
            const SizedBox(height: 16),
            Row(
              mainAxisAlignment: MainAxisAlignment.center,
              children: [
                ElevatedButton.icon(
                  onPressed: _isRunning ? _stopTimer : _startTimer,
                  icon: Icon(_isRunning ? Icons.pause : Icons.play_arrow),
                  label: Text(_isRunning ? '暂停' : '开始'),
                ),
                const SizedBox(width: 12),
                OutlinedButton.icon(
                  onPressed: _resetTimer,
                  icon: const Icon(Icons.replay),
                  label: const Text('重置'),
                ),
              ],
            ),
          ],
        ),
      ),
    );
  }
}

2.3 如何选择 StatelessWidget 还是 StatefulWidget？

场景 选择 原因
纯展示组件，不随用户交互改变 StatelessWidget 无需管理状态
组件内部有需要随时间/交互变化的数据 StatefulWidget 需要 setState 触发重建
组件依赖动画控制器 StatefulWidget AnimationController 需要 TickerProvider
数据完全由外部管理（如状态管理库） StatelessWidget 组件自身不持有状态

注意事项：不要因为"可能以后需要状态"就预先使用 StatefulWidget。遵循最小权限原则——能用 StatelessWidget 就用 StatelessWidget，需要时再重构为 StatefulWidget。IDE 通常提供快速重构工具。

3. Widget 的生命周期

Widget 的生命周期主要体现在 State 对象上（因为 StatelessWidget 只有一个 build 方法，没有复杂的生命周期）。

3.1 完整的生命周期方法

createState() → initState() → didChangeDependencies() → build()
                                                           ↓
                              [Widget 更新] → didUpdateWidget() → build()
                                                           ↓
                          [依赖变化] → didChangeDependencies() → build()
                                                           ↓
                              [停用] → deactivate() → [复用 或 dispose()]

各方法详解：

方法 调用时机 常见用途
createState() StatefulWidget 首次插入树时 创建 State 对象
initState() State 对象创建后、build 之前调用一次 初始化变量、订阅 Stream、创建 Controller
didChangeDependencies() initState 之后、以及依赖的 InheritedWidget 变化时 获取依赖（如 Theme.of、MediaQuery.of）
build() 每次需要重新构建 UI 时 返回 Widget 树描述
didUpdateWidget() 父组件重建导致本 Widget 配置更新时 对比新旧 widget 参数，做出相应调整
deactivate() State 从树中移除时（可能临时） 很少使用，用于清理与树相关的引用
dispose() State 永久从树中移除时 释放资源：取消订阅、关闭 Controller、释放动画

3.2 代码示例：生命周期方法完整演示

import 'package:flutter/material.dart';

void main() {
  runApp(const LifecycleDemoApp());
}

class LifecycleDemoApp extends StatelessWidget {
  const LifecycleDemoApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      theme: ThemeData(
        colorSchemeSeed: Colors.indigo,
        useMaterial3: true,
      ),
      home: const LifecycleHomePage(),
    );
  }
}

class LifecycleHomePage extends StatefulWidget {
  const LifecycleHomePage({super.key});

  @override
  State<LifecycleHomePage> createState() => _LifecycleHomePageState();
}

class _LifecycleHomePageState extends State<LifecycleHomePage> {
  bool _showChild = true;
  int _counter = 0;

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('生命周期演示')),
      body: Padding(
        padding: const EdgeInsets.all(16),
        child: Column(
          children: [
            Row(
              children: [
                ElevatedButton(
                  onPressed: () => setState(() => _showChild = !_showChild),
                  child: Text(_showChild ? '移除子组件' : '添加子组件'),
                ),
                const SizedBox(width: 12),
                ElevatedButton(
                  onPressed: () => setState(() => _counter++),
                  child: const Text('更新参数'),
                ),
              ],
            ),
            const SizedBox(height: 24),
            if (_showChild)
              LifecycleChildWidget(counter: _counter),
          ],
        ),
      ),
    );
  }
}

class LifecycleChildWidget extends StatefulWidget {
  final int counter;

  const LifecycleChildWidget({super.key, required this.counter});

  @override
  State<LifecycleChildWidget> createState() {
    debugPrint('🔵 createState() —— 创建 State 对象');
    return _LifecycleChildWidgetState();
  }
}

class _LifecycleChildWidgetState extends State<LifecycleChildWidget> {
  late TextEditingController _controller;
  final List<String> _lifecycleLog = [];

  void _log(String message) {
    debugPrint(message);
    // 在 initState 阶段不能调用 setState
    if (mounted && _lifecycleLog.isNotEmpty) {
      setState(() => _lifecycleLog.add(message));
    } else {
      _lifecycleLog.add(message);
    }
  }

  @override
  void initState() {
    super.initState();
    // ✅ 在此初始化控制器、订阅、一次性配置
    _controller = TextEditingController(text: '初始文本');
    _log('🟢 initState() —— 初始化资源，只调用一次');
  }

  @override
  void didChangeDependencies() {
    super.didChangeDependencies();
    // 在这里可以安全地使用 Theme.of(context)、MediaQuery.of(context) 等
    final brightness = Theme.of(context).brightness;
    _log('🟡 didChangeDependencies() —— 当前主题亮度: $brightness');
  }

  @override
  void didUpdateWidget(covariant LifecycleChildWidget oldWidget) {
    super.didUpdateWidget(oldWidget);
    // 当父组件传入新参数时，对比新旧值做出响应
    if (oldWidget.counter != widget.counter) {
      _log('🟠 didUpdateWidget() —— counter: ${oldWidget.counter} → ${widget.counter}');
    }
  }

  @override
  void deactivate() {
    debugPrint('🔴 deactivate() —— State 从树中暂时移除');
    super.deactivate();
  }

  @override
  void dispose() {
    // ✅ 在此释放资源，防止内存泄漏
    _controller.dispose();
    debugPrint('⚫ dispose() —— 永久销毁，释放资源');
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    _log('🔵 build() —— 构建 UI');
    return Card(
      child: Padding(
        padding: const EdgeInsets.all(16),
        child: Column(
          crossAxisAlignment: CrossAxisAlignment.start,
          children: [
            Text(
              '当前 counter: ${widget.counter}',
              style: Theme.of(context).textTheme.titleLarge,
            ),
            const Divider(),
            const Text('生命周期日志:', style: TextStyle(fontWeight: FontWeight.bold)),
            const SizedBox(height: 8),
            // 展示生命周期调用记录
            Container(
              height: 200,
              padding: const EdgeInsets.all(8),
              decoration: BoxDecoration(
                color: Colors.grey[100],
                borderRadius: BorderRadius.circular(8),
              ),
              child: ListView.builder(
                itemCount: _lifecycleLog.length,
                itemBuilder: (context, index) {
                  return Text(
                    _lifecycleLog[index],
                    style: const TextStyle(fontSize: 13, fontFamily: 'monospace'),
                  );
                },
              ),
            ),
          ],
        ),
      ),
    );
  }
}

  • 点击"移除子组件"→ 在控制台观察 deactivate 和 dispose 被调用
  • 点击"添加子组件"→ 观察完整的 createState → initState → didChangeDependencies → build 流程

3.3 生命周期注意事项与最佳实践

注意事项 1：initState() 中不能使用 context 来访问 InheritedWidget（如 Theme.of(context)），因为此时组件尚未完全挂载到树中。应在 didChangeDependencies() 中执行此类操作。

注意事项 2：build() 方法可能被频繁调用，绝对不要在 build 中执行耗时操作、发起网络请求或创建 Future/Stream。这些应该在 initState 中完成。

最佳实践：dispose() 中必须释放所有资源。常见需要释放的资源：

  • TextEditingController / ScrollController / AnimationController → 调用 .dispose()
  • StreamSubscription → 调用 .cancel()
  • Timer → 调用 .cancel()
  • FocusNode → 调用 .dispose()

4. BuildContext 的作用与使用场景

4.1 BuildContext 的本质

BuildContext 是一个抽象接口，实际上就是 Element 对象本身。每个 Widget 的 build 方法都会接收一个 BuildContext 参数，它代表了该 Widget 在 Element Tree 中的位置。

// Flutter 源码中的定义（简化）
abstract class Element implements BuildContext {
  // Element 实现了 BuildContext 接口
  // 所以 context 本质上就是 Element
}

4.2 BuildContext 的核心作用

作用一：向上查找祖先 Widget 的数据

Flutter 中的 InheritedWidget 机制（如 Theme、MediaQuery、Navigator）依赖 BuildContext 来实现数据的"向上查找"。

// 通过 context 获取主题数据
final theme = Theme.of(context);

// 通过 context 获取屏幕尺寸信息
final screenSize = MediaQuery.of(context).size;

// 通过 context 获取导航器
Navigator.of(context).push(...);

// 通过 context 获取 Scaffold 的状态（用于打开 Drawer 等）
Scaffold.of(context).openDrawer();

作用二：确定 Widget 在树中的位置

// 获取当前组件在屏幕上的精确位置和尺寸
final RenderBox box = context.findRenderObject() as RenderBox;
final Offset position = box.localToGlobal(Offset.zero);
final Size size = box.size;

作用三：遍历子树

context.visitChildElements((Element element) {
  // 遍历当前节点的子 Element
});

4.4 BuildContext 的常见陷阱

陷阱1：使用了错误的 context

// ❌ 错误：Scaffold.of(context) 在这里找不到 Scaffold
// 因为 context 对应的是 Scaffold 本身，而非 Scaffold 的子组件
@override
Widget build(BuildContext context) {
  return Scaffold(
    body: Center(
      child: ElevatedButton(
        onPressed: () {
          // 这个 context 是 Scaffold 的 context，往上找不到 Scaffold
          Scaffold.of(context).openDrawer(); // 报错！
        },
        child: const Text('打开 Drawer'),
      ),
    ),
  );
}

// ✅ 正确做法：使用 Builder 获取 Scaffold 子树的 context
@override
Widget build(BuildContext context) {
  return Scaffold(
    drawer: const Drawer(child: Center(child: Text('抽屉内容'))),
    body: Center(
      child: Builder(
        builder: (scaffoldContext) {
          // scaffoldContext 是 Scaffold 的子节点，可以向上找到 Scaffold
          return ElevatedButton(
            onPressed: () {
              Scaffold.of(scaffoldContext).openDrawer(); // 正常工作
            },
            child: const Text('打开 Drawer'),
          );
        },
      ),
    ),
  );
}

陷阱2：在异步回调中使用过期的 context

// ❌ 危险：异步操作完成后，Widget 可能已被销毁
onPressed: () async {
  final data = await fetchData();
  Navigator.of(context).push(...); // context 可能已无效
}

// ✅ 安全做法：在使用前检查 mounted
onPressed: () async {
  final data = await fetchData();
  if (!context.mounted) return; // Flutter 3.7+ 支持
  Navigator.of(context).push(...);
}

最佳实践：始终记住 BuildContext 代表的是 Widget 在树中的位置。.of(context) 方法会从该位置向上查找。如果你需要的对象在 context 的下方或同级，你将找不到它。

5. Key 的种类与使用时机

5.1 为什么需要 Key？

Flutter 在更新 Widget Tree 时，会比较新旧 Widget 来决定是否复用 Element。默认的比较策略是：相同的 runtimeType 且 key 相同（或都为 null）时复用。

当列表中的元素发生位置变化（如排序、删除、插入）时，仅靠 runtimeType 无法正确匹配新旧 Widget，可能导致状态错乱。此时就需要 Key 来帮助 Flutter 正确识别每个 Widget。

/// 每个列表项有自己的内部状态（随机颜色）
class ColorfulListItem extends StatefulWidget {

5.3 Key 的种类

Flutter 提供了多种 Key 类型，适用于不同场景：

ValueKey —— 基于值的 Key

当列表项有一个唯一的业务标识（如 ID、名称）时使用。

// 使用 ValueKey 解决上面的排序问题
ListView(
  children: items.map((item) {
    return ColorfulListItem(
      key: ValueKey(item), // 使用水果名称作为 Key
      title: item,
    );
  }).toList(),
)

ObjectKey —— 基于对象引用的 Key

当列表项对应一个数据对象，且对象本身可作为唯一标识时使用。

class User {
  final int id;
  final String name;
  User(this.id, this.name);
}

// ObjectKey 基于对象的 == 运算符进行比较
ListView(
  children: users.map((user) {
    return UserListTile(
      key: ObjectKey(user), // 使用整个 User 对象作为 Key
      user: user,
    );
  }).toList(),
)

UniqueKey —— 始终唯一的 Key

UniqueKey 每次创建都生成一个新的唯一值。使用它意味着告诉 Flutter 永远不要复用这个 Element。

// UniqueKey 适用于需要强制重建的场景
AnimatedSwitcher(
  duration: const Duration(milliseconds: 300),
  child: Text(
    '$_counter',
    // 每次 _counter 变化时，都生成新的 UniqueKey
    // 这样 AnimatedSwitcher 才能识别为"不同的子组件"并播放切换动画
    key: ValueKey(_counter), // 这里用 ValueKey 更合适
    style: const TextStyle(fontSize: 48),
  ),
)

// UniqueKey 更适合"确保每次都重新创建"的场景
Widget build(BuildContext context) {
  return MyExpensiveWidget(
    key: UniqueKey(), // 强制每次 build 都销毁旧的、创建新的
  );
}

注意：UniqueKey 放在 build 方法中会导致每次重建都创建新 Element，慎用！

GlobalKey —— 全局唯一且可跨树访问的 Key

GlobalKey 是功能最强大也最"重量级"的 Key。它可以：

  1. 在整个应用中唯一标识一个 Widget
  2. 访问对应 Element 的 State 对象
  3. 访问对应的 RenderObject（获取尺寸、位置等）
  4. 实现 Widget 在不同父节点间的"移动"（保留状态）
import 'package:flutter/material.dart';

5.4 Key 使用总结

Key 类型 使用场景 示例
ValueKey 列表项有唯一业务值（ID、名称等） ValueKey(user.id)
ObjectKey 列表项绑定唯一数据对象 ObjectKey(user)
UniqueKey 需要强制 Widget 每次都重新创建 动画切换、强制刷新
GlobalKey 需要跨组件访问 State 或 RenderObject 表单验证、获取组件尺寸

最佳实践：

  1. 大部分时候不需要 Key。只有在列表元素可能改变顺序、新增或删除，并且列表项带有自身状态时，才需要显式指定 Key
  2. 优先使用 ValueKey 或 ObjectKey，它们是轻量级的
  3. GlobalKey 成本较高（需要全局注册表维护），不要在 build 方法中频繁创建，应存储为类成员变量
  4. 绝对不要在 build 方法内部 new GlobalKey()，否则每次重建都会创建新的 GlobalKey，导致 State 丢失

本课小结

本课系统讲解了 Flutter Widget 体系的五个核心知识点：

  1. 三棵树架构：Widget Tree 描述配置、Element Tree 管理生命周期与复用、RenderObject Tree 执行布局与绘制。Widget 的频繁重建是安全且廉价的，因为 Element 层做了智能的差异复用。

  2. StatelessWidget vs StatefulWidget：本质区别在于是否需要维护可变状态。StatelessWidget 适合纯展示组件，StatefulWidget 的状态存储在 State 对象中，通过 setState 触发 UI 更新。

  3. 生命周期：initState（初始化）→ didChangeDependencies（依赖变化）→ build（构建 UI）→ didUpdateWidget（参数更新）→ deactivate（停用）→ dispose（销毁）。资源初始化放 initState，资源释放放 dispose。

  4. BuildContext：本质是 Element 对象的引用，代表 Widget 在树中的位置。所有 .of(context) 方法都是从该位置向上查找。注意使用正确的 context，异步回调中要检查 mounted。

  5. Key 系统：帮助 Flutter 在 Widget 更新时正确匹配新旧 Widget。ValueKey 和 ObjectKey 用于列表项标识，UniqueKey 用于强制重建，GlobalKey 用于跨组件访问 State。
